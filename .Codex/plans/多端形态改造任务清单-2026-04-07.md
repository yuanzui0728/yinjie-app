## 1. 当前代码现状摘要

基于当前仓库实际入口，已确认以下事实：

### 1.1 已有基础

- `apps/desktop` 已经是独立桌面壳
- `apps/android-shell`、`apps/ios-shell` 已经建立原生壳目录
- `apps/app/src/runtime/platform.ts` 已经存在 `desktop / android / ios / web` 平台抽象
- `DesktopRuntimeGuard` 已经开始区分桌面本地控制与远程服务可达性
- `packages/contracts` 已支持注入 `coreApiBaseUrlProvider`

### 1.2 当前主要缺口

- `apps/app/src/routes/setup-page.tsx` 仍只是跳转页，尚未承担真正的 desktop/mobile setup 分流
- `packages/contracts/src/client.ts` 仍把 `http://localhost:39091` 作为硬默认值
- `ProfilePage` 仍是通用页面，尚未按 PC / mobile 明确区分能力区域
- `apps/app` 虽有平台能力抽象，但页面编排层还没有真正按“宿主端 / 连接端”分线

结论：

- 现在不是从零开始
- 但还没有把“平台抽象”升级为“产品线架构”

## 2. 改造总原则

所有任务执行前先遵守下面 4 条，不满足就不应开工：

1. 每个任务先归类为 `desktop-host`、`mobile-client`、`shared-business` 三类之一。
2. 不允许把桌面本地能力直接暴露到手机版页面。
3. 不允许新增任何“默认 localhost 就是正确环境”的假设。
4. 页面层只消费 capability，不直接写平台分支。

---

7. 合规与治理闭环

## 4. 实际改造任务清单

## Group A：平台模型与配置模型收敛

目标：

- 把“平台识别”提升为“产品线和部署模式识别”

### A1. 定义统一运行语义

归类：

- `shared-business`

任务：

- 在 `apps/app/src/runtime/` 收敛正式类型：
  - `AppChannel = "desktop" | "mobile" | "web"`
  - `DeploymentMode = "local-hosted" | "remote-connected"`
  - `HostRole = "host" | "client"`
- 保留现有 `AppPlatform = "desktop" | "android" | "ios" | "web"`，但不再让页面直接依赖它
- 增加能力字段：
  - `canConfigureRemoteService`
  - `canExportDiagnostics`
  - `canManageProvider`
  - `canScanBootstrapCode`
  - `canOpenExternalLinks`

涉及位置：

- `apps/app/src/runtime/platform.ts`
- `apps/app/src/runtime/adapters/*`

验收：

- 页面层获取的是统一 runtime context，不再手搓平台判断

### A2. 定义统一 runtime context

归类：

- `shared-business`

任务：

- 新增统一选择器，例如 `resolveAppRuntimeContext()`
- 输出至少包含：
  - `platform`
  - `channel`
  - `deploymentMode`
  - `hostRole`
  - `capabilities`

验收：

- `Setup`、`Profile`、`Runtime Guard` 使用同一份上下文，不重复拼装判断

### A3. 重构 AppRuntimeConfig 模型

归类：

- `shared-business`

任务：

- 明确区分：
  - `apiBaseUrl`
  - `socketBaseUrl`
  - `environment`
  - `bootstrapSource`
  - `channel`
- 区分“本地默认值”和“必填远程值”
- 引入配置状态：
  - `unconfigured`
  - `configured`
  - `validated`

涉及位置：

- `apps/app/src/runtime/runtime-config.ts`
- `apps/app/src/runtime/runtime-config-store.ts`
- `apps/app/src/lib/runtime-config.ts`

验收：

- 移动端不会因为缺配置而偷偷回落到 localhost

## 5. 依赖关系与推荐顺序

严格推荐顺序如下：

1. A 平台模型与配置模型
2. E 网络与会话模型统一
3. B 启动流重构
4. C 桌面能力边界收口
5. D 移动能力边界落地
6. F 壳层接入与发布链闭环
7. G 合规与治理闭环
8. H 文档与回归收口

原因：

- 如果 A/E 不先做，后面的页面改造都会建立在错误默认值上
- 如果 B 不先做，用户入口链仍然是错的
- 如果 C/D 不收口，功能边界还会持续互相污染

## 7. 我的执行策略

后续我实际动手时，默认按以下原则执行：

1. 先做共享模型，再做页面，不倒序。
2. 优先把错误默认值清掉，再做新功能。
3. 桌面和移动的差异放在 feature / runtime adapter 层，不散落到业务页。
4. 每次改造都尽量形成一个可验证的小闭环，不做只改一半的中间状态。
