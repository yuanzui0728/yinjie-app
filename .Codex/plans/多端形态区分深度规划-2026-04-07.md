# 隐界多端形态区分深度规划

日期：2026-04-07
范围：Windows、macOS、iOS、Android 的产品形态、技术架构、代码边界、发布路径与迁移顺序

## 1. 结论

后续不要再按“四个平台各做一套”推进，而应明确收敛为两条产品线：

- `PC 版`：Windows + macOS
- `手机版`：iOS + Android

这两条线共享同一个世界观、同一套业务契约、同一份账号体系、同一份数据模型，但不是同一种运行模式。

必须先承认这一点：

- PC 版本质上是 `自托管本地世界容器`
- 手机版本质上是 `远程连接世界客户端`

也就是：

- Windows / macOS 要像微信 PC 版一样，强调稳定、连续在线、窗口化、多任务、本地运行时与运维能力
- iOS / Android 要像微信手机版一样，强调触达、轻启动、推送、拍照/上传、碎片化使用与系统级集成

如果继续把四端都当成“同一套 WebView 壳 + 少量条件判断”，最终会出现三个问题：

1. 桌面端能力污染移动端
2. 移动端的合规与体验要求被桌面节奏拖慢
3. 代码里会长期存在“本地 Core API”和“远程 Core API”两种假设混杂的问题

所以这次规划的核心不是“怎么支持更多平台”，而是“怎么把平台差异收敛成清晰的产品线边界”。

---

## 2. 产品线定义

## 2.1 PC 版定义

平台：

- Windows
- macOS

角色定位：

- 主世界入口
- 主配置入口
- 本地自部署入口
- 数据与运行时控制中心

使用心智：

- 类似微信 PC 版，但比微信 PC 版更重，因为它不只是聊天壳，还承担本地世界运行容器职责
- 用户会在 PC 上完成首启、服务初始化、provider 配置、数据诊断、备份恢复、长期聊天与管理操作

运行模式：

- 桌面壳内自动拉起本地 `core-api`
- `runtime-data`、SQLite、日志、诊断、备份都在本机
- 支持本地 provider / 远程 provider 配置

结论：

- PC 版不是“移动版的大屏适配”
- PC 版是隐界真正的宿主端

## 2.2 手机版定义

平台：

- iOS
- Android

角色定位：

- 日常陪伴入口
- 移动消息入口
- 即时内容消费与轻互动入口

使用心智：

- 类似微信手机版，是主频使用端
- 但不承担本地世界托管职责

运行模式：

- 连接远程 Core API
- 不在手机本地拉起 `core-api`
- 不暴露本地 provider、runtime data、备份恢复、桌面诊断等系统能力

结论：

- 手机版不是“桌面版 UI 缩小”
- 手机版是针对移动系统能力重做的一条客户端产品线

## 2.3 关系定义

建议产品关系固定为：

- `PC 版 = 世界宿主端`
- `手机版 = 世界连接端`

默认用户路径应为：

1. 在 Windows 或 macOS 首次建立自己的隐界实例
2. 完成基础 setup、provider、账号初始化
3. 再用 iOS / Android 连接这个已存在的世界

这和微信的关系类似，但更进一步：

- 微信 PC 版依赖手机登录
- 隐界 PC 版应是主宿主，手机版依赖 PC 版或远程部署实例提供世界服务

## 3. 平台能力边界

## 3.1 只属于 PC 版的能力

以下能力必须明确标为 `desktop-only`：

- 本地 Core API 启停、重启、探活
- runtime data 路径展示
- provider 配置与连通性测试
- 系统日志与 diagnostics 导出
- snapshot 备份与恢复
- evals、experiments、reports
- 本地数据库迁移与修复
- 本地服务端口、sidecar、进程错误诊断
- 本地世界初始化向导

原因：

- 这些能力天然依赖本地进程、本地文件系统、本地运行目录
- 强行搬到手机没有产品价值，反而会引入审核和系统限制问题

## 3.2 只属于手机版的能力

以下能力必须明确标为 `mobile-only` 或 `mobile-first`：

- 推送通知
- 相册选择、拍照、文件上传
- 分享面板、剪贴板、外链调起
- 移动网络下的断线重连优化
- 前后台切换与省电策略适配
- 安全存储恢复登录态
- 通知权限申请与系统提示
- 深链接、Universal Links、Android App Links

原因：

- 这些能力属于移动操作系统入口价值
- 桌面虽然也能做，但不是主路径

## 3.3 两条线共享的能力

必须保持共享的部分：

- 账号体系
- Bearer session 机制
- `packages/contracts` typed contracts
- 业务语义与接口语义
- 会话、群聊、朋友圈、发现、通讯录、角色详情
- moderation / block / report 规则
- 视觉品牌、内容世界观、文案基调

共享原则：

- 共享的是业务与契约
- 不共享的是运行时假设和交互细节

## 4. 目标架构

## 4.1 推荐总体架构

后续建议固定为以下形态：

### 产品层

- `Yinjie Desktop`
  - Windows
  - macOS
- `Yinjie Mobile`
  - iOS
  - Android

### 前端层

- `apps/app`
  - 继续作为“用户产品 UI 主体”
  - 但内部按 `desktop-shell` 与 `mobile-shell` 分流
- `apps/admin`
  - 保持为系统管理端
  - 不进入移动首发包

### 壳层

- `apps/desktop`
  - Tauri 2
  - 对应 Windows / macOS
- `apps/ios-shell`
  - Capacitor iOS 容器
- `apps/android-shell`
  - Capacitor Android 容器

### 服务层

- `crates/core-api`
  - PC 版：本地 sidecar / 本地宿主服务
  - 手机版：远程部署服务
- `crates/inference-gateway`
  - 仍作为服务能力组成部分

### 共享层

- `packages/contracts`
- `packages/config`
- `packages/ui`

## 4.2 运行模式矩阵

| 维度 | PC 版 | 手机版 |
|------|------|------|
| 用户定位 | 世界宿主端 | 世界连接端 |
| 平台 | Windows / macOS | iOS / Android |
| Core API 来源 | 本地 sidecar | 远程服务 |
| 首启重点 | 建世界、配 provider、检查 runtime | 连接世界、登录、恢复会话 |
| 数据目录 | 本机 app data | 本地仅缓存和安全存储 |
| 高级运维 | 支持 | 不支持 |
| 推送 | 次要 | 核心 |
| 上传拍照 | 可选 | 核心 |
| 商店审核 | 低 | 高 |
| 断网策略 | 可容忍本地服务启动等待 | 必须快速反馈与重试 |

## 5. 代码结构收敛方案

## 5.1 现状问题

当前 `apps/app` 已经在做平台抽象，但仍有一个关键风险：

- 页面层仍残留“桌面运行时是默认形态”的历史假设

这会导致：

- `/setup` 同时承担桌面 setup 与移动接入职责
- `Profile`、root guard、runtime hooks 里混入过多桌面控制逻辑
- Android / iOS 虽然有壳，但产品边界不够硬

## 5.2 推荐目录思路

建议后续将 `apps/app` 内部按产品线分层：

- `src/features/bootstrap/`
  - 平台识别、运行配置、会话恢复、首启路由
- `src/features/desktop/`
  - 本地 Core API、runtime data、diagnostics、provider setup
- `src/features/mobile/`
  - 远程连接、推送注册、媒体能力、移动入口恢复
- `src/features/shared/`
  - chat、moments、contacts、discover、profile 基础业务
- `src/runtime/`
  - adapter 层，只描述能力，不暴露平台细节到业务页

关键要求：

- 页面不直接判断 `isDesktopRuntimeAvailable()`
- 页面只消费 capability flags

例如：

- `canManageLocalCoreApi`
- `canUseRemoteBootstrap`
- `canUseSecureStorage`
- `canReceivePush`
- `canPickMedia`
- `canExportDiagnostics`

## 5.3 Setup 路由重定义

`/setup` 不能再是“桌面配置页的别名”，应改成统一入口，但内容按产品线分流：

### Desktop Setup

内容：

- 检查本地 Core API
- 检查 runtime data
- 配置 provider
- 诊断 sidecar / 端口 / 日志
- 进入世界

### Mobile Setup

内容：

- 输入或扫码导入远程 Core API 地址
- 校验 `GET /health`
- 校验 `GET /system/status`
- 可选导入邀请码/实例码
- 进入登录或 onboarding

结论：

- 路由可以同名
- 但产品任务必须完全不同

## 6. 平台专项规划

## 6.1 Windows

定位：

- PC 版主发平台
- 最先形成完整安装交付

原因：

- 目标用户对本地自部署接受度更高
- Tauri + sidecar + installer 更容易形成闭环
- Windows 用户更容易接受“本地世界容器”这个概念

Windows 的重点不是 UI，而是交付闭环：

- `core-api.exe` sidecar 随包分发
- `.msi` / NSIS 安装器
- 首启诊断清晰
- 升级不丢 `runtime-data`
- 未签名和 Defender 风险有明确处理方案

建议角色：

- 首发宿主平台
- 内测和种子用户主平台

## 6.2 macOS

定位：

- PC 版第二优先平台
- 面向开发者、设计师和苹果生态用户

与 Windows 的差异：

- 更强调 `.app/.dmg` 体验
- 签名、公证、Gatekeeper 是强约束
- Apple Silicon 优先，Intel 次之

macOS 的重点：

- `core-api` 内置 sidecar
- `.icns`、签名、公证
- 首启权限与诊断文案
- sandbox / 路径 / 可执行权限问题收敛

建议角色：

- 与 Windows 同属 PC 版
- 但交付链更严格，节奏可以慢半拍

## 6.3 iOS

定位：

- 手机版高优先平台
- 代表最终移动体验标准

iOS 的关键原则：

- 不本地拉起 `core-api`
- 不暴露桌面系统运维能力
- 优先解决审核、登录态、安全存储、推送、内容治理

iOS 的重点：

- 远程 Core API 接入
- Keychain session 存储
- Push / APNs
- Safe Area、键盘、深链接
- 账号删除、举报、拉黑、法律页、提审材料

建议角色：

- 手机版旗舰平台
- 产品体验基准平台

## 6.4 Android

定位：

- 手机版覆盖平台
- 面向更广用户群和更复杂设备环境

Android 的关键原则与 iOS 相同：

- 只做远程客户端
- 不本地托管世界

Android 的重点：

- 远程连接配置
- Keystore 安全存储
- FCM 推送
- 媒体上传、文件、分享
- 机型兼容、通知权限、后台限制
- AAB、签名、崩溃监控

建议角色：

- 手机版规模平台
- 与 iOS 共用手机版能力骨架，但原生适配项独立推进

## 7. 产品体验差异化原则

## 7.1 PC 版像什么

PC 版应接近：

- 微信 PC 版的稳定会话体验
- 本地工具的可控感
- 桌面应用的持续在线感

应该强化：

- 左侧会话列表 + 多列布局
- 长时间挂起在线
- 更强的信息密度
- 运行状态可感知
- 世界维护与系统控制

不应该照搬：

- 纯移动 tab 信息架构
- 过度依赖底部导航

## 7.2 手机版像什么

手机版应接近：

- 微信手机版的单手操作与碎片化响应
- 社交 App 的高频触达

应该强化：

- 启动快
- 推送进入即达目标页
- 底部 tabs
- 聊天、朋友圈、发现作为高频主路径
- 图片、分享、轻发布

不应该承担：

- provider 管理
- 复杂诊断
- 本地服务维护

## 7.3 一致性要求

两条线即使交互不同，也必须保证以下一致：

- 同一角色与世界状态
- 同一对话和群聊语义
- 同一账号与 session 规则
- 同一法律与安全规则
- 同一品牌气质

用户感受应该是：

- 我在不同设备进入的是同一个隐界
- 但每个设备承担的职责不同

## 8. 运行与部署策略

## 8.1 推荐官方支持模型

建议正式定义为三种部署关系：

### 模式 A：桌面自托管

- Windows / macOS 本地运行 Core API
- 适合作为默认推荐模式

### 模式 B：远程自部署

- Core API 部署在用户自己的服务器或家中主机
- iOS / Android 连接该实例
- PC 版也可作为远程连接客户端存在，但不是首发重点

### 模式 C：混合模式

- 用户先在 PC 本地建立世界
- 再通过内网穿透、反代或迁移导出转到远程部署
- 手机版持续连接远程实例

## 8.2 不建议的模式

不建议投入首期的模式：

- iOS 本地嵌入 `core-api`
- Android 本地常驻 `core-api`
- 四端都允许“本地世界宿主”

原因：

- 系统限制和维护复杂度过高
- 产品叙事不清晰
- 会显著拖慢商店发布

## 9. 发布与商业化视角

## 9.1 PC 版发布逻辑

PC 版发布关注：

- 安装包
- sidecar
- 签名
- 升级
- 备份迁移
- 日志诊断

衡量标准：

- 用户不用懂技术也能装起来
- 安装后世界能自己运行起来

## 9.2 手机版发布逻辑

手机版发布关注：

- 商店审核
- 推送可用
- 登录可靠
- 网络稳定
- 隐私合规
- 崩溃率与机型兼容

衡量标准：

- 用户能像装普通聊天 App 一样装上并登录
- 不要求用户理解本地部署细节

## 10. 推荐实施顺序

## Phase 0：统一口径

先把团队内部口径定死：

- Windows / macOS = PC 版
- iOS / Android = 手机版
- PC 版本地宿主，手机版远程连接

这一步必须先完成，否则所有实现还会反复摇摆。

## Phase 1：桌面线闭环

目标：

- 先把 PC 版做成真正可交付的宿主端

工作：

- Windows sidecar + 安装器闭环
- macOS sidecar + `.dmg` + 签名公证准备
- `apps/app` 的桌面 setup、runtime guard、profile runtime 区域继续收敛
- 备份恢复、provider、diagnostics 保持只在桌面可见

验收：

- 用户能在 Windows 或 macOS 上独立建立自己的世界

## Phase 2：移动线连接闭环

目标：

- 做出真正可用的手机版

工作：

- `apps/app` 启动流按 desktop / mobile 分流
- 接入统一远程 runtime config
- iOS / Android 壳读取并注入服务地址
- 安全存储恢复 session
- 聊天、朋友圈、发现、通讯录跑通远程模式

验收：

- 用户能从手机稳定连接已存在的世界实例

## Phase 3：移动原生能力闭环

工作：

- 推送
- 媒体上传
- 分享与深链接
- 通知权限
- 后台恢复策略

验收：

- 手机版达到高频使用标准

## Phase 4：合规与发布闭环

工作：

- iOS 提审材料
- Android 商店材料
- 账号删除、举报、拉黑、法律页面与后台处理链
- 崩溃监控和日志脱敏

验收：

- 手机版具备正式上架条件

## 11. 仓库级执行建议

## 11.1 现在就应固定的边界

后续新增功能时，先判断它属于哪一类：

- `desktop-host`
- `mobile-client`
- `shared-business`

没有分类就不要开工。

## 11.2 建议新增的显式概念

建议在代码和文档中都引入以下正式术语：

- `host platform`
  - 仅 PC 版
- `client platform`
  - 仅手机版
- `deployment mode`
  - `local-hosted`
  - `remote-connected`
- `channel`
  - `desktop`
  - `mobile`

这样可以避免继续出现：

- “移动端也要不要能启动 core-api？”
- “桌面是不是也该和手机共用同一个 setup？”

这种模糊讨论。

## 11.3 当前仓库与目标的映射关系

现状其实已经接近正确方向：

- `apps/desktop` 已经是 PC 版壳
- `apps/android-shell`、`apps/ios-shell` 已经是手机版壳
- `apps/app` 已经是共用产品 UI 主体

缺的不是“再建一套端”，而是：

- 把 `apps/app` 从“桌面迁移中的主前端”彻底提升为“双产品线共用前端”
- 把桌面能力和移动能力从页面逻辑里彻底切开

---

## 12. 最终建议

最终建议很明确：

1. 产品定义上，立即停止把四端并列讨论，改成 `PC 版` 与 `手机版` 两线讨论。
2. 技术架构上，立即固定 `PC 本地宿主 / 手机远程连接` 的主路线。
3. 代码治理上，立即把平台差异收敛为 capability 和 channel，而不是页面层的散乱条件判断。
4. 发布策略上，PC 版先解决 sidecar 与安装闭环，手机版先解决远程接入、推送、安全存储和合规。
5. 用户心智上，明确传达：
   - Windows / macOS 像微信 PC 版，但更像“你的隐界主机”
   - iOS / Android 像微信手机版，是你随身进入隐界的入口

这条路线的好处是：

- 世界观统一
- 产品职责清晰
- 桌面与移动互不拖累
- 平台能力边界稳定
- 后续新增 iPad、Web Remote、Linux Desktop 时也能自然挂到既有框架内

这应该作为后续多端演进的总纲，而不是某个平台的临时说明。
