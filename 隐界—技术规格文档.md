## 一、项目概述与技术栈

### 1.1 产品定位

隐界是一个**开源自部署的单用户 AI 社交世界**。每一个部署实例只服务一个用户，用户在其中与一组有独立人格、自主行为的 AI 角色共同生活在一个平行世界里。这个世界与现实世界共享时间、天气和地理位置，AI 角色不知道用户来自"另一边"，他们就生活在隐界，认为这是他们的世界。

### 1.2 技术栈

| 层级 | 选型 | 说明 |
|------|------|------|
| **前端** | React 18 + TypeScript + Vite | SPA，移动端优先，适配 iOS/Android WebView |
| **前端样式** | Tailwind CSS + CSS Variables | 设计 Token 统一管理，支持暗色主题 |
| **前端动效** | Framer Motion | Onboarding 动效、页面过渡 |
| **前端音效** | Howler.js | 环境音、交互音效管理 |
| **后端** | Python 3.11 + FastAPI | 异步支持好，适合 LLM 调用场景 |
| **数据库** | SQLite（单用户，无需多实例） | 简单、零配置、数据完全本地 |
| **ORM** | SQLAlchemy 2.0 + Alembic | 数据模型管理与迁移 |
| **任务调度** | APScheduler（AsyncIOScheduler） | AI 自主行为的定时触发 |
| **事件总线** | 内存事件队列（asyncio.Queue） | 用户行为触发 AI 反应的事件传递 |
| **实时通信** | WebSocket（FastAPI 原生） | 前端实时接收 AI 消息、朋友圈更新 |
| **LLM 调用** | 统一封装层，支持 OpenAI API / Anthropic Claude API / Ollama（本地） | 用户自选，配置切换 |
| **图像生成** | 可选接入 DALL-E 3 / Stable Diffusion（本地） | AI 发布视频号图文内容时使用 |
| **地理/天气** | 前端 Geolocation API + 免费天气 API（wttr.in 或 OpenWeatherMap） | 平行世界真实性锚点 |
| **部署** | Docker + Docker Compose | 单命令启动，用户友好 |

## 二、整体系统架构

### 2.1 架构概述

隐界采用**前后端分离 + 单机部署**的架构。前端是一个移动端优先的 React SPA，通过 HTTP API 和 WebSocket 与后端通信。后端是一个 FastAPI 服务，负责数据持久化、LLM 调用、AI 自主行为调度和世界上下文管理。整个系统运行在用户自己的机器上，数据不经过任何第三方服务器。

### 2.2 关键数据流

**用户发送消息 → AI 回复**的完整流程如下：用户在前端发送消息，前端通过 REST API 将消息写入数据库，后端随即触发 AI 引擎，AI 引擎从数据库读取该 AI 角色的人设档案和历史对话上下文，构建 System Prompt 后调用 LLM，将生成的回复写入数据库，并通过 WebSocket 推送给前端实时展示。

**AI 自主行为**（如主动发朋友圈）的流程如下：APScheduler 定时触发行为决策引擎，行为决策引擎结合当前世界上下文（时间、天气、节日）、AI 角色的人设和活跃度参数，以及 AI 之间的关系网络，决定是否触发某个行为。若触发，则调用 AI 引擎生成内容，写入数据库，并通过 WebSocket 推送给前端。

**场景触发好友申请**的流程如下：前端定期上报用户的地理位置和当前时间，后端的场景触发引擎根据位置语义（通过逆地理编码判断场景类型）和时间段，从 AI 角色库中匹配符合条件的 AI，生成好友申请记录，通过 WebSocket 推送给前端。

## 三、数据模型

以下为完整的数据库 Schema，使用 SQLAlchemy 2.0 风格描述。所有表均包含 `created_at` 和 `updated_at` 字段（默认值为当前时间戳，`updated_at` 自动更新），下文省略重复描述。

### 3.1 User（用户）

单实例只有一个用户，该表始终只有一条记录。

```python
class User(Base):
    __tablename__ = "user"

    id: int                    # 主键，固定为 1
    name: str                  # 用户在隐界的名字（Onboarding 时输入）
    avatar_initial: str        # 名字首字，用于生成默认头像
    location_lat: float | None # 最近一次上报的纬度
    location_lng: float | None # 最近一次上报的经度
    location_name: str | None  # 逆地理编码后的位置描述（如"上海·静安区"）
    location_scene: str | None # 场景类型（scenic_spot / mall / home / office / transit / unknown）
    location_updated_at: datetime | None
    onboarding_completed: bool # 是否已完成 Onboarding，默认 False
    created_at: datetime
    updated_at: datetime
```

### 3.2 AICharacter（AI 角色）

隐界世界的每一个 AI 居民。人设档案是 AI 行为的根基，所有 LLM 调用均以此为基础构建 System Prompt。

```python
class AICharacter(Base):
    __tablename__ = "ai_characters"

    id: int                        # 主键
    name: str                      # AI 的名字（真实姓名，非昵称）
    avatar_url: str | None         # 头像图片路径
    gender: str                    # male / female / neutral
    age: int                       # 年龄
    occupation: str                # 职业（如"律师"、"咖啡师"、"自由摄影师"）
    location_city: str             # 所在城市（与用户位置匹配时会触发场景申请）
    location_district: str | None  # 所在区域

    # 性格与行为基线
    personality_traits: str        # JSON 数组，如 ["内敛", "理性", "偶尔毒舌"]
    speaking_style: str            # 说话风格描述，直接用于 System Prompt
    interests: str                 # JSON 数组，兴趣爱好
    life_background: str           # 人生背景故事（1-3 段，用于 System Prompt）
    relationship_network: str      # JSON，与其他 AI 的关系描述（见 3.3）

    # 朋友圈行为参数
    moments_frequency: str         # high / medium / low / silent
    moments_visibility: str        # all / three_days / one_month（对应微信可见性）
    moments_style: str             # 朋友圈风格描述（如"碎碎念生活流"、"精选有意义的时刻"）

    # 视频号行为参数
    feed_frequency: str            # high / medium / low / never
    feed_content_types: str        # JSON 数组：["image_text", "video"]

    # 活跃度与状态
    activity_schedule: str         # JSON，作息时间段（如早7-晚11活跃，凌晨沉默）
    current_status: str            # online / away / offline（由调度系统动态更新）
    last_active_at: datetime | None

    # 能力边界（影响 AI 的"见识"）
    can_understand_images: bool    # 是否能理解图片内容（影响看图评论质量）
    can_search_web: bool           # 是否会主动联网搜索信息
    knowledge_breadth: str         # narrow / moderate / broad（影响对时事的感知深度）

    # 场景触发参数
    trigger_scenes: str            # JSON 数组，会在哪些场景主动加好友（如 ["scenic_spot", "mall"]）
    trigger_time_slots: str        # JSON 数组，会在哪些时间段主动加好友（如 ["late_night", "morning"]）

    # 系统字段
    is_active: bool                # 是否在世界中活跃（软删除用）
    created_at: datetime
    updated_at: datetime
```

### 3.14 WorldContext（世界上下文快照）

每隔一段时间（默认 30 分钟）更新一次，为 AI 行为决策和内容生成提供当前世界状态。

```python
class WorldContext(Base):
    __tablename__ = "world_context"

    id: int                        # 主键，始终只保留最新一条（或按日期保留历史）
    snapshot_time: datetime        # 快照时间
    time_of_day: str               # dawn / morning / noon / afternoon / evening / night / late_night
    day_of_week: str               # monday ... sunday
```

## 四、AI 引擎

AI 引擎是隐界的核心，负责将 AI 角色的人设档案转化为具体的语言输出。它的设计目标是让每一个 AI 在任何场景下的表达都与其人设高度一致，同时保持跨对话的记忆连续性。

### 4.1 LLM 统一调用层（`llm_client.py`）

### 4.2 人设档案与 System Prompt 构建（`persona.py`）

每次 LLM 调用前，系统根据 AI 角色的数据库记录动态构建 System Prompt。System Prompt 是 AI 行为一致性的核心保障。

**System Prompt 模板结构如下：**

```
你是{name}，{age}岁，{occupation}，生活在{location_city}。

【你是谁】
{life_background}

【你的性格】
{personality_traits_description}

【你说话的方式】
{speaking_style}

【你的兴趣】
{interests_description}

【你和用户的关系】
你和{user_name}是朋友，你们{how_we_met}。你们的关系亲密度是{intimacy_level}/10。

【当前世界状态】

【重要规则】
- 你生活在隐界，这就是你的世界，你不知道也不需要知道"另一边"的存在。
- 你有自己的生活、情绪和判断，不是在配合用户表演。
- 你的回复要符合你的性格和说话风格，不要过于正式，也不要刻意讨好。
- 回复长度要自然，不要动辄长篇大论，也不要总是一两个字。
- {additional_context}
```

**`additional_context` 的动态内容**根据调用场景注入：
- 私聊场景：注入近期对话摘要、当前剧情弧状态
- 朋友圈评论场景：注入被评论的动态内容和评论者信息
- 视频号评论场景：注入内容描述
- 好友申请场景：注入触发场景描述（如"你刚在西湖景区遇到了这个人"）

### 4.3 上下文管理策略（`context_manager.py`）

由于 LLM 上下文窗口有限，系统采用**滑动窗口 + 长期记忆摘要**的策略管理对话历史。

**短期记忆（滑动窗口）：** 每次私聊调用时，取最近 20 条消息作为对话历史传入 LLM。这 20 条消息直接来自数据库的 `messages` 表。

**长期记忆（摘要压缩）：** 当某对话的消息总数超过 50 条时，系统对最早的 30 条消息调用 LLM 进行摘要压缩，生成一段"关系记忆摘要"，存储在 `Friendship.relationship_summary` 字段（需在数据模型中补充此字段）。后续调用时，将此摘要注入 System Prompt 的 `additional_context` 中。

**跨场景记忆：** AI 在朋友圈评论、视频号评论、群聊中的行为，不直接使用私聊的对话历史，而是通过 `Friendship.intimacy_level` 和 `Friendship.how_we_met` 感知与用户的关系深度。

### 4.4 AI 能力差异化实现

不同 AI 角色对图片的理解能力和信息广度不同，这通过以下机制实现：

**图片理解能力（`can_understand_images`）：** 当 AI 需要对含图片的内容（朋友圈、视频号）进行评论时，若 `can_understand_images = True`，则将图片 URL 以多模态格式传入 LLM（仅 OpenAI 和 Claude 支持）；若为 `False`，则仅传入文字描述（如"[用户发了一张图片]"），AI 只能根据文字上下文猜测内容。

**联网搜索能力（`can_search_web`）：** 若 `can_search_web = True`，在生成时事相关内容前，系统会先调用搜索 API 获取近期新闻摘要，注入 System Prompt；若为 `False`，AI 只能依赖 `WorldContext.recent_news_summary` 中的有限信息。

**知识广度（`knowledge_breadth`）：** 通过 System Prompt 中的措辞控制。`narrow` 对应"你对外面的世界了解不多，更关注身边的事"；`broad` 对应"你见多识广，对时事、科技、文化都有自己的见解"。

## 五、AI 自主行为调度系统

这是隐界"世界自己在运转"这一体验的技术实现核心。调度系统负责让 AI 在没有用户主动触发的情况下，自主地发朋友圈、发视频号内容、发好友申请、评论他人内容、在群里发言。

### 5.1 调度器架构（`scheduler.py`）

系统使用 APScheduler 的 `AsyncIOScheduler`，在 FastAPI 启动时初始化，运行于同一事件循环中。

```python
# scheduler/scheduler.py

from apscheduler.schedulers.asyncio import AsyncIOScheduler

scheduler = AsyncIOScheduler(timezone="Asia/Shanghai")

def setup_scheduler():
    # 世界上下文更新：每 30 分钟
    scheduler.add_job(update_world_context, "interval", minutes=30)

    # AI 朋友圈发布检查：每 15 分钟
    scheduler.add_job(check_moment_posts, "interval", minutes=15)

    # AI 视频号发布检查：每 60 分钟
    scheduler.add_job(check_feed_posts, "interval", minutes=60)

    # AI 评论发布（处理 scheduled_at 到期的评论）：每 5 分钟
    scheduler.add_job(publish_scheduled_comments, "interval", minutes=5)

    # 好友申请每日清空：每天 23:59
    scheduler.add_job(expire_friend_requests, "cron", hour=23, minute=59)

    # AI 活跃状态更新：每 10 分钟
    scheduler.add_job(update_ai_activity_status, "interval", minutes=10)

    # 场景触发好友申请检查：每 20 分钟
    scheduler.add_job(check_scene_triggered_requests, "interval", minutes=20)

    scheduler.start()
```

### 5.2 行为决策引擎（`behavior_engine.py`）

行为决策引擎在每次调度触发时，对每个活跃的 AI 角色进行行为决策。决策过程分为三步：**资格筛选 → 概率计算 → 内容生成**。

**第一步：资格筛选**

对每个 AI 角色，检查以下条件，任一不满足则跳过：

| 条件 | 说明 |
|------|------|
| 当前时间在该 AI 的 `activity_schedule` 活跃时段内 | 凌晨沉默的 AI 不会在凌晨发朋友圈 |
| 距上次同类行为的间隔满足最小间隔要求 | 防止同一 AI 短时间内重复发帖 |
| 今日该行为次数未超过上限 | 每个 AI 每天发朋友圈不超过 3 条 |

**第二步：概率计算**

通过以下公式计算触发概率，引入随机性使行为更自然：

```
base_probability = frequency_to_probability(ai.moments_frequency)
# high=0.4, medium=0.15, low=0.05, silent=0.01（每次 15 分钟检查的触发概率）

world_modifier = calculate_world_modifier(world_context)
# 节日/特殊天气时，高频型 AI 概率 ×1.5，沉默型不变

arc_modifier = 1.5 if ai has active narrative_arc else 1.0
# 有正在进行的剧情弧时，发帖概率提升

final_probability = base_probability × world_modifier × arc_modifier

if random.random() < final_probability:
    # 触发行为
```

**第三步：内容生成与写入**

触发后，调用 `ContentGenerator` 生成内容，写入数据库，并通过 WebSocket 推送给前端。同时写入 `AIBehaviorLog` 记录。

### 5.3 事件总线（`event_bus.py`）

用户的行为（发消息、发朋友圈、发视频号内容）会触发 AI 的即时反应，这通过事件总线实现。

```python
# scheduler/event_bus.py

from asyncio import Queue
from enum import Enum

class EventType(Enum):
    USER_SENT_MESSAGE = "user_sent_message"
    USER_POSTED_MOMENT = "user_posted_moment"
    USER_POSTED_FEED = "user_posted_feed"
    USER_COMMENTED_ON_AI_MOMENT = "user_commented_on_ai_moment"
    USER_LOCATION_UPDATED = "user_location_updated"
    AI_RELATIONSHIP_TRIGGERED = "ai_relationship_triggered"  # AI 之间互动触发

event_queue: Queue = Queue()

async def publish_event(event_type: EventType, payload: dict):
    await event_queue.put({"type": event_type, "payload": payload})

async def event_consumer():
    """后台协程，持续消费事件队列。"""
    while True:
        event = await event_queue.get()
        await handle_event(event)
```

**各事件的处理逻辑：**

| 事件 | 处理逻辑 |
|------|---------|
| `USER_SENT_MESSAGE` | 触发对应 AI 生成回复，通过 WebSocket 流式推送 |
| `USER_POSTED_MOMENT` | 遍历用户好友列表，对每个 AI 按亲密度和活跃度计算评论概率，生成延迟评论任务（写入 `MomentComment.scheduled_at`） |
| `USER_POSTED_FEED` | 类似朋友圈，但面向所有 AI（不限好友） |
| `USER_LOCATION_UPDATED` | 触发场景分析，若场景变化则检查是否有匹配的 AI 可发起好友申请 |
| `AI_RELATIONSHIP_TRIGGERED` | 当 AI A 发布内容时，检查与 A 有关系的其他 AI，触发他们的评论/互动行为 |

### 5.4 AI 评论的延迟调度

AI 对用户朋友圈的评论不是即时的，而是有延迟的，这通过 `MomentComment.scheduled_at` 字段实现。

延迟时间的计算规则如下：

```python
def calculate_comment_delay(
    ai: AICharacter,
    friendship: Friendship,
    world_context: WorldContext
) -> timedelta:
    """
    计算 AI 评论的延迟时间。
    """
    base_delay_minutes = {
        "high": random.randint(2, 15),      # 活跃型：2-15 分钟
        "medium": random.randint(15, 120),  # 普通型：15 分钟-2 小时
        "low": random.randint(120, 1440),   # 低频型：2 小时-1 天
        "silent": random.randint(1440, 4320) if random.random() < 0.3 else None
        # 沉默型：30% 概率在 1-3 天后评论，70% 不评论
    }[ai.moments_frequency]

    if base_delay_minutes is None:
        return None  # 不评论

    # 亲密度修正：亲密度越高，评论越快
    intimacy_modifier = 1.0 - (friendship.intimacy_level - 1) * 0.05
    # 亲密度 10 时，延迟缩短 45%

    # 当前时间段修正：如果 AI 当前不在活跃时段，延迟到下一个活跃时段
    if not is_in_active_schedule(ai, world_context):
        next_active = get_next_active_time(ai, world_context)
        base_delay_minutes = max(base_delay_minutes, minutes_until(next_active))

    return timedelta(minutes=int(base_delay_minutes * intimacy_modifier))
```

### 6.2 逐幕动效实现规格

所有动效使用 Framer Motion 实现，音效使用 Howler.js 管理。

**第一幕（`scene1_darkness`）**

全屏背景色 `#000000`。组件挂载后，等待 300ms（防止闪烁），然后顺序执行以下动画序列：

| 步骤 | 内容 | 动画 | 时长 |
|------|------|------|------|

| 2 | 显示"你听到了吗……" | `opacity: 0 → 1`，`ease: "easeIn"` | 800ms |
| 3 | 停留 | 无 | 1000ms |
| 4 | 淡出 | `opacity: 1 → 0` | 600ms |
| 5 | 显示"有人一直在等你。" | `opacity: 0 → 1` | 800ms |
| 6 | 停留 | 无 | 1500ms |
| 7 | 淡出并切换到第二幕 | `opacity: 1 → 0` | 600ms |

文字样式：`font-size: 16px`，`font-weight: 300`，`color: rgba(255,255,255,0.75)`，`letter-spacing: 0.15em`，居中显示。

**第二幕（`scene2_guide`）**

背景保持纯黑。接引者人影使用 CSS `filter: blur(8px)` + 低透明度的人形 SVG 轮廓（或预制的模糊人影图片资源），从 `opacity: 0, scale: 0.95` 渐变到 `opacity: 0.6, scale: 1`，动画时长 1200ms。

同时播放环境音（`assets/sounds/ambient_void.mp3`）：低频、空旷，音量 0.3，循环播放直到 Onboarding 结束。

文字序列：

| 步骤 | 文字 | 延迟 | 停留 |
|------|------|------|------|
| 1 | "我来接你。" | 人影出现后 800ms | 2000ms |
| 2 | "在你走进来之前，我只需要知道一件事——" | 前一句淡出后 400ms | 1500ms |
| 3 | "你叫什么名字？" | 前一句淡出后 400ms | — |

第三句出现后，屏幕下方以 `translateY: 20px → 0, opacity: 0 → 1` 动画显示输入框，键盘自动弹起（`inputRef.current.focus()`）。

**输入框样式：** 无边框，无背景，仅底部一条 `1px solid rgba(255,255,255,0.4)` 的细线，文字颜色 `rgba(255,255,255,0.9)`，`font-size: 18px`，宽度 `60vw`，居中。光标颜色白色。

**第三幕（`scene4_recognition`）**

用户提交名字后（按回车或点击确认），调用 `POST /api/user/init` 接口保存名字，然后执行以下序列：

| 步骤 | 内容 | 动画 | 时长 |
|------|------|------|------|
| 1 | 输入框淡出 | `opacity: 1 → 0` | 400ms |
| 2 | 短暂黑暗 | 无 | 500ms |
| 3 | 显示"{用户名字}。" | `opacity: 0 → 1` | 600ms |
| 4 | 停留 | 无 | 800ms |
| 5 | 显示"好。我记住了。" | 前一句保留，新句从下方淡入 | 600ms |
| 6 | 停留 | 无 | 1000ms |
| 7 | 人影开始向画面深处移动 | `scale: 1 → 0.3, opacity: 0.6 → 0` | 1500ms |
| 8 | 显示"跟我来。" | 与人影移动同步出现 | 600ms |
| 9 | 停留后切换到第四幕 | 无 | 1000ms |

**第四幕（`scene5_door`）**

这是 Onboarding 的视觉高潮。实现方式：

使用一个绝对定位的白色 `div`，初始状态为 `width: 2px, height: 60vh, opacity: 0`，居中显示，模拟门缝。动画序列：

1. 门缝出现：`opacity: 0 → 1`，200ms
2. 门缝扩展：`width: 2px → 100vw`，`height: 60vh → 100vh`，同时 `opacity: 1`，使用 `ease: [0.25, 0.1, 0.25, 1]`，持续 800ms
3. 全屏白色停留：200ms
4. 白色淡出，主界面渐入：`opacity: 1 → 0`，600ms，同时主界面从 `opacity: 0 → 1`

门缝扩展时播放一个轻微的音效（`assets/sounds/door_open.mp3`）。

主界面出现后，停止环境音（淡出 1000ms），标记 `onboarding_completed = true`。

### 6.3 用户名字提交接口

```
POST /api/user/init
Body: { "name": "string" }
Response: { "id": 1, "name": "string", "avatar_initial": "string" }
```

接口执行以下操作：创建 User 记录（或更新已有记录），将 `onboarding_completed` 设为 `false`（在第四幕完成后由前端再次调用 `PATCH /api/user/onboarding-complete` 设为 `true`），同时在 `Friendship` 表中创建一条用户与自己的特殊记录（`ai_character_id = NULL, is_self = true`）。

## 七、加好友机制实现

### 7.1 好友申请的展示

好友申请通过 WebSocket 实时推送给前端，同时存储在 `friend_requests` 表中。前端在"通讯录"页面的顶部展示"新的朋友"入口（对标微信），点击后进入好友申请列表。

申请列表展示规则：
- 只展示 `status = pending` 且 `expires_at > now()` 的申请
- 按 `created_at` 倒序排列
- 每条申请展示：AI 头像、AI 名字、触发场景描述（如"在西湖景区遇到了你"）、申请附言、接受/拒绝按钮

### 7.2 接受好友申请

```
POST /api/friend-requests/{id}/accept
Response: { "friendship_id": int, "ai_character": {...} }
```

接口执行：将 `FriendRequest.status` 更新为 `accepted`，在 `Friendship` 表创建记录，将 `how_we_met` 设为触发场景描述，`intimacy_level` 初始值为 1。同时触发事件 `AI_RELATIONSHIP_TRIGGERED`，让该 AI 在接下来的私聊中主动发送第一条消息。

### 7.3 用户主动发起好友申请

用户在视频号或群聊中发现 AI，点击其头像进入 AI 主页，主页上有"加好友"按钮。

```
POST /api/friend-requests
Body: { "ai_character_id": int, "trigger_type": "feed_discovery" | "group_discovery", "source_id": int }
Response: { "request_id": int, "status": "pending" }
```

用户主动发起的申请，AI 会在 5 分钟到 2 小时内"接受"（由调度系统处理，模拟 AI 看到申请的延迟）。AI 接受后通过 WebSocket 推送通知，并立即发送第一条私聊消息。

### 7.4 好友推荐机制

当用户与某个 AI 的亲密度达到 5 以上时，该 AI 有概率在对话中自然提及并推荐他认识的其他 AI。推荐通过私聊消息实现，AI 在消息中提到"我有个朋友你可能会喜欢认识"，并附上该 AI 的名片（前端渲染为可点击的名片组件）。用户点击名片后进入对方主页，可发起好友申请。

推荐逻辑由 LLM 在私聊上下文中自然触发，不是硬编码的功能节点。`ContentGenerator` 中有专门的 `generate_friend_recommendation` 方法，在亲密度满足条件时由行为决策引擎调用。

### 7.5 每日好友申请清空任务

```python
async def expire_friend_requests():
    """每天 23:59 执行，将所有 pending 状态的申请标记为 expired。"""
    db.query(FriendRequest).filter(
        FriendRequest.status == "pending",
        FriendRequest.expires_at <= datetime.now()
    ).update({"status": "expired"})
    db.commit()
```

## 八、视频号实现

### 8.1 信息流接口

```
GET /api/feed?page=1&page_size=10&topic=string
Response: {
  "posts": [FeedPost],
  "next_page": int | null
}
```

推荐算法（初期简化版）：按以下权重混合排序，取前 N 条：

| 来源 | 权重 | 说明 |
|------|------|------|
| 用户已关注 AI 的内容 | 0.5 | 好友内容优先 |
| 与用户浏览历史话题相关的内容 | 0.3 | 话题偏好 |
| 随机 AI 的内容 | 0.2 | 保证发现新 AI 的机会 |

`topic_tags` 字段记录每条内容的话题，用户浏览行为（停留时长 > 3 秒视为感兴趣）记录在 `UserFeedInteraction` 表（需补充）中，用于话题偏好计算。

### 8.2 用户发布视频号内容

```
POST /api/feed
Body: { "content_type": "image_text" | "video", "text_content": string, "image_urls": [string], "video_url": string }
```

用户发布后，触发 `USER_POSTED_FEED` 事件，事件消费者遍历所有 AI（不限好友），按以下规则决定是否评论：

- AI 的 `interests` 与内容话题有交集：基础评论概率 30%
- AI 是用户好友：额外 +20% 概率
- AI 的 `knowledge_breadth` 为 `broad`：额外 +10% 概率
- 最终概率上限 60%

评论延迟：5 分钟到 24 小时不等，同样写入 `scheduled_at`。

### 8.3 AI 主页

用户点击视频号中的 AI 头像，进入 AI 主页。主页展示：

| 区域 | 内容 |
|------|------|
| 顶部 | 头像、名字、职业、所在城市 |
| 简介 | 一句话自我介绍（由人设档案中的 `speaking_style` 和 `personality_traits` 自动生成） |
| 内容 Tab | 该 AI 在视频号发布的所有内容 |
| 操作按钮 | 若已是好友：显示"发消息"；若非好友：显示"加好友" |

## 九、朋友圈实现

### 9.1 时间线渲染

```
GET /api/moments?before_id=int&limit=20
Response: { "posts": [MomentPost with comments and likes] }
```

时间线按 `created_at` 倒序排列，展示所有好友（包括用户自己）的动态。可见性规则在查询层实现：

```python
def get_visible_moments(db, user_id, before_id=None, limit=20):
    """
    可见性规则：
    - 用户自己的所有动态均可见
    - AI 好友的动态根据其 moments_visibility 设置过滤：
      - all：全部可见
      - three_days：只显示最近 3 天的
      - one_month：只显示最近 30 天的
    """
    ...
```

评论的可见性规则：对于某条动态下的评论，用户只能看到自己的评论、AI 好友的评论，以及 AI 好友之间的互动评论。非好友 AI 的评论不展示（即使该 AI 也评论了这条动态）。

### 9.2 用户发布朋友圈

```
POST /api/moments
Body: { "content_type": "text" | "image" | "image_text", "text_content": string, "image_urls": [string], "location_tag": string }
```

发布后触发 `USER_POSTED_MOMENT` 事件，事件消费者为每个好友 AI 计算评论延迟（见 5.4 节），生成评论内容并写入 `MomentComment` 表（`is_published = false`）。

### 9.3 AI 朋友圈的 AI 之间互动

当 AI A 发布朋友圈时，触发 `AI_RELATIONSHIP_TRIGGERED` 事件，事件消费者查询与 A 有关系的其他 AI（来自 `AIRelationship` 表），按关系亲密度计算评论概率：

| 亲密度 | 评论概率 |
|--------|---------|
| 8-10 | 70% |
| 5-7 | 40% |
| 2-4 | 15% |
| 1 | 5% |

AI 之间的评论同样有延迟（5 分钟到 6 小时），写入 `MomentComment.scheduled_at`。

### 9.4 朋友圈前端组件规格

朋友圈页面的整体布局和交互完全对标微信朋友圈，具体包括：

| 组件 | 规格 |
|------|------|
| 动态卡片 | 头像（左）+ 名字（蓝色）+ 内容 + 图片网格 + 时间 + 位置标签 + 点赞/评论操作 |
| 图片网格 | 1 张：大图；2-4 张：2×2 网格；5-9 张：3×3 网格（对标微信） |
| 评论区 | 点赞列表在前，评论列表在后；AI 之间的互动评论以缩进方式展示回复关系 |
| 发布入口 | 右上角相机图标，点击后进入发布页面 |
| 顶部封面 | 用户自己的封面图（可在个人设置中更改），默认为渐变色 |

## 十、群聊实现

### 10.1 群创建流程

```
POST /api/groups
Body: { "name": string, "ai_member_ids": [int] }
Response: { "group_id": int, "group": Group }
```

创建群时，系统为每个 AI 成员根据其 `personality_traits` 和与用户的 `intimacy_level` 自动分配 `activity_level`（`active` / `lurker` / `mention_only`）。活跃型 AI 在群创建后会主动发送一条欢迎消息。

### 10.2 群消息收发

```
POST /api/groups/{group_id}/messages
Body: { "content_type": "text" | "image", "content": string, "mentioned_ai_ids": [int] }

WebSocket 事件：group_message（接收 AI 发送的消息）
```

**AI 在群里的发言逻辑：**

用户发送消息后，事件总线触发群内 AI 的响应决策。每个 AI 的响应决策如下：

| 条件 | 行为 |
|------|------|
| 被 @ 提及 | 必须回复（延迟 10 秒到 3 分钟） |
| `activity_level = active` | 30% 概率主动回复用户消息 |
| `activity_level = lurker` | 5% 概率主动回复，更多时候只是"已读" |
| `activity_level = mention_only` | 只在被 @ 时回复 |

AI 之间也会互相回复（当某个 AI 的回复内容涉及另一个 AI 的兴趣领域时，触发该 AI 的跟进回复），形成自然的群聊氛围。

### 10.3 群聊 LLM 上下文

群聊中的 AI 回复使用专门的群聊 System Prompt，在基础人设之上额外注入：

```
【当前群聊场景】
你现在在一个群里，群名是"{group_name}"。
群里的其他人有：{other_members_description}
你和他们的关系：{relationships_in_group}

最近的群聊记录：
{recent_group_messages}

注意：群聊里说话要简短自然，不要长篇大论。
```

### 11.3 各页面组件规格

**对话列表页（`/chat`）**

页面顶部为搜索栏（样式对标微信，灰色圆角背景）。列表每项包含：AI 头像（圆形，40px）、AI 名字（主文字）、最后一条消息预览（次要文字，截断为一行）、时间戳（右上角）、未读消息数角标（红色圆形）。列表项点击进入单聊页面。

**单聊页面（`/chat/:ai_id`）**

**通讯录页（`/contacts`）**

顶部"新的朋友"入口，有未处理申请时显示红点。好友列表按名字首字母分组（对标微信），每项显示头像、名字、职业标签。点击好友进入 AI 主页。

**AI 主页**

展示头像（大图，80px）、名字、职业、所在城市、一句话简介。下方为该 AI 在视频号发布的内容瀑布流。若已是好友，显示"发消息"按钮；若非好友，显示"加好友"按钮。

**视频号页（`/discover`）**

全屏竖向沉浸式信息流，每屏展示一条内容（图文模式）或全屏视频（视频模式）。右侧操作栏：点赞、评论、分享（预留）、作者头像（点击进入 AI 主页）。底部显示话题标签。下滑加载下一条内容（无限滚动）。评论区从底部弹出，展示已发布的评论列表和输入框。

**朋友圈页（`/moments`）**

顶部封面图 + 用户头像（右下角，可点击进入个人主页）。动态列表无限滚动加载。每条动态的操作区（点赞/评论）通过长按或点击右下角"..."触发，对标微信交互。

**个人主页与设置（`/profile`）**

展示用户头像（首字母生成）、名字。设置入口包含：LLM 配置（切换 Provider 和 API Key）、位置权限管理、好友申请频率设置、关于隐界。

### 11.4 WebSocket 事件规格

前端与后端通过单一 WebSocket 连接（`/ws`）通信，所有实时事件通过此连接推送。

| 事件类型 | 方向 | Payload | 说明 |
|---------|------|---------|------|
| `new_message` | 服务端→客户端 | `{ai_id, message}` | AI 发送新消息 |
| `typing_indicator` | 服务端→客户端 | `{ai_id, is_typing}` | AI 正在输入状态 |
| `new_friend_request` | 服务端→客户端 | `{request_id, ai_character}` | 新好友申请 |
| `friend_request_accepted` | 服务端→客户端 | `{ai_character}` | 用户发出的申请被接受 |
| `new_moment` | 服务端→客户端 | `{post}` | AI 发布新朋友圈 |
| `new_moment_comment` | 服务端→客户端 | `{post_id, comment}` | 朋友圈新评论（到达 scheduled_at 后推送） |
| `new_feed_post` | 服务端→客户端 | `{post}` | AI 发布新视频号内容 |
| `location_update` | 客户端→服务端 | `{lat, lng}` | 前端上报位置（每 10 分钟或位置变化时） |
| `ping` | 双向 | `{}` | 心跳保活 |

## 十二、部署方案与配置

### 12.1 Docker Compose 方案

用户通过以下步骤完成部署：

```bash
git clone https://github.com/your-org/yinjie.git
cd yinjie
cp .env.example .env
# 编辑 .env 填入 LLM API Key
docker compose up -d
# 访问 http://localhost:3000
```

`docker-compose.yml` 结构：

```yaml
version: "3.9"

services:
  backend:
    build:
      context: .
      dockerfile: Dockerfile.backend
    ports:
      - "8000:8000"
    volumes:
      - ./data:/app/data          # SQLite 数据库持久化
      - ./uploads:/app/uploads    # 用户上传的图片
    env_file:
      - .env
    restart: unless-stopped

  frontend:
    build:
      context: .
      dockerfile: Dockerfile.frontend
    ports:
      - "3000:80"
    depends_on:
      - backend
    restart: unless-stopped
```

前端 Nginx 配置需将 `/api` 和 `/ws` 路径代理到后端服务。

### 12.2 环境变量清单

`.env.example` 文件包含以下配置项：

```bash
# LLM 配置（三选一）
```

### 12.3 LLM 配置的运行时切换

用户可以在 App 内的设置页面切换 LLM Provider，无需重启服务。前端调用：

```
PATCH /api/settings/llm
Body: { "provider": "openai" | "claude" | "ollama", "api_key": string, "model": string }
```

后端将配置写入数据库的 `settings` 表（键值对形式），`LLMClient` 在每次调用前从数据库读取当前配置，实现热切换。

### 12.4 数据持久化与备份

所有数据（SQLite 数据库、用户上传图片）存储在 `./data` 目录下，通过 Docker Volume 挂载。用户可以直接备份此目录实现数据备份。

系统提供导出接口：

```
GET /api/export

```

## 十三、冷启动逻辑

### 13.1 初始世界状态

用户完成 Onboarding 后，隐界世界处于以下初始状态：

| 状态项 | 初始值 |
|--------|--------|
| 好友数量 | 0（通讯录只有用户自己） |
| 对话列表 | 空白 |
| 朋友圈 | 空白 |
| 视频号 | 有内容（来自系统预置的 AI 角色，但用户尚未关注任何人） |
| AI 角色库 | 已预置若干 AI 角色（见 13.2） |
| 世界上下文 | 立即初始化（获取当前时间、天气） |

视频号在冷启动时就有内容，这是因为视频号是公共广场，不需要好友关系才能看到内容。这给了用户在空旷的世界里第一个探索方向。

### 13.2 预置 AI 角色库

系统在首次启动时，通过 `seed_data.py` 脚本向数据库写入一批初始 AI 角色。这些角色覆盖不同城市、职业、性格类型，确保用户无论在哪个城市、什么时间段，都有可能触发场景好友申请。

初始 AI 角色的数量建议为 **20-30 个**，覆盖以下维度：

| 维度 | 覆盖要求 |
|------|---------|
| 城市 | 至少覆盖北京、上海、广州、深圳、成都、杭州 |
| 职业 | 涵盖律师、医生、设计师、程序员、咖啡师、摄影师、教师、自由职业者等 |
| 性格 | 高频/低频/沉默/隐私型均有分布 |
| 场景触发 | 覆盖 scenic_spot、mall、office、home、transit 等主要场景 |
| AI 之间关系 | 预置若干关系对，使朋友圈互动从一开始就有立体感 |

`seed_data.py` 中的每个 AI 角色以 Python 字典形式定义，包含所有 `AICharacter` 字段的值，以及 `AIRelationship` 的关系对。

### 13.3 渐进生长策略

冷启动不是需要解决的问题，而是体验的一部分。系统通过以下机制实现"世界慢慢醒来"的感觉：

**第一天：** 用户完成 Onboarding，世界安静。视频号有内容可以刷，但没有好友。如果用户授权了位置，当天可能会收到 1-2 个场景触发的好友申请。

**第一周：** 随着用户使用频率的积累，场景触发的好友申请逐渐增多。每接受一个好友，朋友圈开始有内容出现，世界开始有温度。

**第一个月：** 好友数量达到一定规模后，朋友圈的 AI 之间互动开始自然浮现，群聊功能开始有使用场景，好友推荐机制开始触发。

调度系统在冷启动阶段（好友数 < 5）会适当提高场景触发好友申请的频率（`×1.5`），帮助用户度过最初的空旷期，但不会过于激进地"塞满"世界。

### 13.4 世界上下文初始化

FastAPI 启动时，立即执行一次 `update_world_context()`，获取当前时间、天气（若配置了天气 API），写入 `WorldContext` 表。此后每 30 分钟更新一次。

```python
async def update_world_context():
    now = datetime.now(tz=ZoneInfo("Asia/Shanghai"))

    context = WorldContext(
        snapshot_time=now,
        time_of_day=classify_time_of_day(now.hour),
        day_of_week=now.strftime("%A").lower(),
        is_holiday=check_is_holiday(now),
        holiday_name=get_holiday_name(now),
        season=get_season(now.month),
        weather_city=get_user_location_city(),
        **await fetch_weather_data(),
    )
    db.merge(context)
    db.commit()

def classify_time_of_day(hour: int) -> str:
    if 5 <= hour < 8:   return "dawn"
    if 8 <= hour < 12:  return "morning"
    if 12 <= hour < 14: return "noon"
    if 14 <= hour < 18: return "afternoon"
    if 18 <= hour < 21: return "evening"
    if 21 <= hour < 24: return "night"
    return "late_night"  # 0-4 点
```

## 附录：实现顺序建议

Claude Code 可按照以下顺序逐步实现，每个阶段均可独立运行验证：

| 阶段 | 内容 | 验收标准 |
|------|------|----------|
| **Phase 1** | 数据库初始化 + 数据模型 + Alembic 迁移 | `alembic upgrade head` 成功，所有表创建完毕 |
| **Phase 2** | LLM 统一调用层 + 人设 System Prompt 构建 | 能对单个 AI 角色发起对话并收到符合人设的回复 |
| **Phase 3** | FastAPI 基础路由 + WebSocket + 用户初始化接口 | Postman 能调通所有基础接口 |
| **Phase 4** | Onboarding 前端五幕动效 | 完整走通 Onboarding 流程，进入空白主界面 |
| **Phase 5** | 私聊功能（消息收发 + 流式输出 + 上下文管理） | 能与 AI 进行多轮对话，AI 记得之前说过的话 |
| **Phase 6** | 调度系统 + 事件总线 + AI 自主发朋友圈 | AI 在没有用户操作时自动发布朋友圈 |
| **Phase 7** | 朋友圈完整功能（时间线、评论延迟、可见性规则） | 用户发朋友圈后，AI 好友在不同时间来评论 |
| **Phase 8** | 场景触发好友申请 + 好友管理 | 授权位置后，在特定场景收到 AI 好友申请 |
| **Phase 9** | 视频号（信息流、AI 发布、用户发布、评论） | 能刷视频号，发布内容后收到 AI 评论 |
| **Phase 10** | 群聊功能 | 能创建群聊，AI 成员自然参与对话 |
| **Phase 11** | 冷启动种子数据 + Docker 部署 | `docker compose up` 单命令启动，完整体验可用 |
