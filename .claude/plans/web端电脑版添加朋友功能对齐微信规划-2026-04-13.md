# web端电脑版添加朋友功能对齐微信规划

## 1. 目标

- 为 `apps/app` 的 Web 桌面布局补齐独立的“添加朋友”能力，不再把桌面端入口错误地落到移动式 `新的朋友` 页面。
- 交互结构、视觉节奏、状态设计尽量贴近微信电脑版当前主流体验：
  - 从顶部 `+` 菜单进入
  - 独立工作区
  - 搜索输入驱动的单结果/空结果页
  - 查看资料
  - 填写验证信息
  - 发送好友申请
- 不是机械复制微信的品牌资源；要抄的是信息架构、交互节奏、布局层级、状态切换和按钮关系。业务语义仍映射为隐界世界：
  - 微信号 / 手机号 -> `隐界号 / 角色名`
  - 用户 -> `世界角色`
  - 添加到通讯录 -> 现有好友申请 / 好友关系链路

## 2. 当前现状

### 2.1 前端已有入口，但桌面主流程缺失

- `apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx`
  - 桌面消息页 `+` 菜单已有 `添加朋友`，但现在直接跳到 `/friend-requests`。
- `apps/app/src/routes/contacts-page.tsx`
  - 移动端快捷菜单也有 `添加朋友`，同样跳 `/friend-requests`。
- `apps/app/src/routes/friend-requests-page.tsx`
  - 当前页面是明显偏移动端的“新的朋友”，不适合作为桌面端“添加朋友”入口页。

### 2.2 数据能力并不空白，但没有桌面编排

- 后端已具备：
  - `GET /api/social/friends`
  - `GET /api/social/friend-requests`
  - `POST /api/social/friend-requests/send`
- 前端已具备：
  - `listCharacters`
  - `getCharacter`
  - `sendFriendRequest`
  - `getFriends`
  - `getFriendRequests`
- 现有缺口不是“不能加好友”，而是没有微信电脑版那套完整的桌面搜索添加链路。

### 2.3 现有行为存在不一致

- `character-detail-page.tsx` 与 `chat-details-page.tsx` 里有“加到通讯录”，其中一部分使用 `autoAccept: true`，这更像“直接加入通讯录”，不是微信式“发送验证申请”。
- `desktop-chat-details-panel.tsx` 则发送 pending 请求，不自动通过。
- 结论：
  - 后续必须统一桌面端“添加朋友”语义。
  - 微信电脑版主流程应以“发送申请”作为默认，不应继续混用自动通过。

## 3. 目标产品定义

## 3.1 路由与入口

- 新增桌面独立路由：
  - `apps/app/src/routes/desktop-add-friend-page.tsx`
  - 路径建议：`/desktop/add-friend`
- 桌面入口全部改为跳转这个新路由：
  - 消息页 `+` 菜单
  - 通讯录页桌面快捷入口里的“添加朋友”
  - 搜一搜无结果区的“去添加朋友” CTA
- `新的朋友` 继续保留在 `/friend-requests`，但桌面版后续应同步桌面化，避免成功后跳回移动页面。

## 3.2 页面结构

- 采用微信电脑版常见的独立工具页结构，而不是移动端整页列表。
- 页面建议使用 `DesktopUtilityShell` 承载，分成三块：
  - 左侧窄栏：入口说明、最近搜索、常见操作入口
  - 中间主工作区：搜索框、搜索状态、结果卡、验证弹层
  - 右侧信息栏：我的隐界号、我的二维码占位、添加说明

## 3.3 交互目标

- 初始态
  - 自动聚焦搜索框
  - 中央展示“搜索隐界号/角色名添加朋友”的引导态
  - 右侧展示“我的隐界号”和二维码占位说明
- 搜索态
  - 回车触发搜索
  - 搜索中展示桌面骨架屏
  - 结果为空展示微信式空态文案和建议操作
- 命中结果
  - 展示单个主结果卡，强调头像、昵称、隐界号、签名、来源
  - 若已是好友，主按钮为 `发消息`
  - 若已发送待处理申请，主按钮为 `已发送`
  - 若未建立关系，主按钮为 `添加到通讯录`
- 添加动作
  - 点击后打开桌面验证弹层
  - 允许编辑验证语
  - 确认后调用 `sendFriendRequest`
  - 成功后停留当前页并切换状态为 `已发送`

## 4. 微信式映射原则

- 抄微信电脑版的交互，不硬塞微信不存在的隐界概念到主链路。
- 但也不伪造我们没有的数据：
  - 不展示手机号、QQ号这类系统并不存在的字段
  - 统一改成 `隐界号 / 角色名 / 备注关键词`
- 不做微信移动端的“雷达加朋友 / 手机通讯录 / 面对面建群”扩展，只覆盖微信 PC 端核心可见主链路：
  - 搜索
  - 查看资料
  - 发送验证
  - 查看新的朋友

## 5. 前端实现方案

## 5.1 新增桌面工作区组件

- 新增：
  - `apps/app/src/features/desktop/contacts/desktop-add-friend-workspace.tsx`
  - `apps/app/src/features/desktop/contacts/desktop-add-friend-result-card.tsx`
  - `apps/app/src/features/desktop/contacts/desktop-add-friend-send-dialog.tsx`
  - `apps/app/src/features/desktop/contacts/desktop-add-friend-route-state.ts`
- 职责拆分：
  - `workspace`
    - 状态编排
    - 查询与 mutation
    - 路由 hash 同步
  - `result-card`
    - 命中结果展示
    - 已好友 / 已申请 / 可添加 三态按钮
  - `send-dialog`
    - 微信式验证信息弹层
    - 表单校验与提交按钮

## 5.2 搜索策略

- 首期直接复用 `listCharacters(baseUrl)` 全量角色列表，在前端做桌面搜索。
- 这是当前最务实的方案，因为项目仍是单世界实例，角色规模可控。
- 搜索匹配建议按优先级排序：
  1. 精确命中隐界号
  2. 精确命中角色名
  3. 角色名前缀命中
  4. 签名 / 简介 / 关系字段模糊命中
- 只展示最优主结果，右侧或下方可选展示“更多可能结果”。
- 搜索工具函数建议抽到：
  - `apps/app/src/features/contacts/contact-utils.ts`
  - 或新建 `desktop-add-friend-search.ts`

## 5.3 状态派生

- 页面需要把三份现有数据合并：
  - `listCharacters`
  - `getFriends`
  - `getFriendRequests`
- 派生出统一关系状态：
  - `friend`
  - `pending`
  - `available`
  - `blocked`
- 结果卡只关心这个派生状态，不直接散落判断。

## 5.4 验证弹层

- 验证弹层建议字段：
  - 只读显示：目标头像、昵称、隐界号
  - 可编辑字段：`验证信息`
  - 默认文案：`${ownerName} 请求添加你为朋友`
- 交互要求：
  - 打开时聚焦文本域
  - `Esc` 关闭
  - `Cmd/Ctrl + Enter` 提交
  - 成功后关闭弹层并回填主页面状态

## 5.5 入口改造

- `apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx`
  - `add-friend` 从 `/friend-requests` 改为 `/desktop/add-friend`
- `apps/app/src/routes/contacts-page.tsx`
  - 桌面布局下新增真正的“添加朋友”快捷入口
  - `新的朋友` 保持现有语义，不与“添加朋友”混用
- `apps/app/src/features/search/desktop-search-launcher.tsx`
  - 无命中提示里增加桌面“去添加朋友”动作

## 5.6 桌面样式方向

- 风格目标不是现在仓库里常见的圆角卡片堆叠，而是更接近微信 PC：
  - 标题栏简洁
  - 主搜索区平直、留白更多
  - 结果区域弱卡片感
  - 选中与主按钮使用微信式绿色强调
- 需要控制几个关键视觉点：
  - 搜索框高度与按钮尺寸
  - 空状态插图/说明的留白
  - 结果卡的头像比例、行距、主次信息层级
  - 验证弹层的宽度、输入框和底部按钮区

## 6. 后端与契约补口

## 6.1 首期不需要新表

- 当前 `FriendRequestEntity` 已够用。
- 不需要新增好友搜索表，不需要新增关系实体。

## 6.2 需要补的契约字段

- 当前 `sendFriendRequest` 只有：
  - `characterId`
  - `greeting`
  - `autoAccept`
- 建议扩展一个可选来源字段：
  - `source?: "manual_search" | "chat_detail" | "character_profile" | "shake"`
- 后端可以先继续复用 `FriendRequestEntity.triggerScene` 做兼容存储，不必新增物理列。
- 原因：
  - “新的朋友”列表需要区分“来自搜索添加”与“来自场景触发”
  - 桌面端后续做“新的朋友”工作区时要还原微信式来源文案

## 6.3 默认语义统一

- 桌面添加朋友主链路统一走 pending 请求：
  - `autoAccept` 默认不用
- `autoAccept: true` 只保留给特定直达场景，后续应收敛：
  - 例如内部运营开关或某些系统角色快速接入
- 否则同一个产品里会同时存在“发申请”和“直接成为好友”两套互相打架的认知。

## 9. 验收标准

- 桌面端点击 `+ -> 添加朋友` 后，进入独立桌面工作区，而不是移动版“新的朋友”页。
- 搜索框支持键盘优先操作：
  - 自动聚焦
  - 回车搜索
  - `Esc` 关闭弹层或清理当前临时态
- 命中角色后，页面能准确区分：
  - 已是好友
  - 已发送申请
  - 可添加
- 点击 `添加到通讯录` 后，有微信式验证信息步骤，不是直接静默成功。
- 发送后状态立即刷新，避免重复发请求。
- 整体桌面视觉第一眼接近微信电脑版工具页，而不是当前项目里偏卡片化的工作台。

## 10. 风险与注意事项

- 当前没有服务端搜索接口，首期前端本地搜索在角色数量很大时会退化；但以现在项目规模看，优先级低于把桌面主链路先做出来。
- `friend-requests-page.tsx` 目前明显偏移动，如果不一并桌面化，添加成功后的后续体验会断裂。
- 当前仓库里存在 `autoAccept` 的历史分支，落地时必须先统一产品语义，否则用户会觉得“有时直接加上，有时要等验证”。
- 如果后续新增路由或契约字段，实施时要同步更新根目录 `AGENTS.md` 中的结构说明。
- 按仓库规则，本任务实施阶段不生成测试文件。

## 11. 建议执行顺序

1. 先做桌面路由和工作区壳，替换错误入口。
2. 再做搜索、结果卡和关系状态派生。
3. 然后补验证弹层与发送申请。
4. 最后把 `新的朋友` 的桌面体验收口，完成桌面闭环。
