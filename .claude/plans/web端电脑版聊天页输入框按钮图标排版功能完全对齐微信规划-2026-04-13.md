# Web 端电脑版聊天页输入框按钮图标排版功能完全对齐微信规划

## 任务目标

- 把 `apps/app` Web 端电脑版聊天页输入框区域，按当前微信电脑端 `4.1.8` 的稳定心智重做。
- 这次规划只覆盖聊天页输入框附近的：
  - 按钮图标
  - 排版结构
  - 展开/收起
  - 发送与换行
  - 附件、表情、截图、语音输入等入口的交互承载
- 不覆盖移动端，不新增测试文件，不改后端接口。

  - `apps/app/src/features/chat/conversation-thread-panel.tsx`

- 桌面聊天工作区：

### 当前桌面输入区的问题

- 当前桌面输入区仍然明显不像微信电脑端：
  - 工具条是“图标 + 中文标签”的按钮，不是纯图标工具列
  - 输入控件是单行 `input`，不是可折叠/可展开的文本编辑区
  - 整体结构是：
    - 上方横排文字工具条
    - 下方灰底单行输入框
    - 右侧条件性出现的发送按钮
  - 这和微信 `4.1.8` 的“白色输入卡片 + 左下图标列 + 右上展开 + 更克制的视觉”差距很大
- 当前桌面分支把过多信息直白暴露出来：
  - `表情 / 图片 / 文件 / 截图 / 收藏 / 聊天记录 / 语音输入`
  - 全部使用有文字标签的工具按钮
  - 第一眼更像工具面板，不像聊天输入框
- 当前发送区也不对：
  - 文本为空时直接留一个空白占位，而不是微信式固定发送区
  - 输入框高度固定，无法承载真正的多行输入体验

### 当前已经具备、可以复用的能力

- 表情面板：
  - 已有 `StickerPanel`
- 图片/文件发送：
  - 已支持选择、草稿预览、拖拽、粘贴、发送
- 截图：
  - 已支持 `Ctrl/Cmd + Shift + S`
  - 已有截图编辑器与发送/复制能力
- 语音输入：
  - 已有桌面 `dictation` 模式
- 回复预览：
  - 已支持回复条
- `@` 提及：
  - 已有桌面 mention picker
- 发送快捷键偏好：
  - 已有 `apps/app/src/store/chat-preferences-store.ts`

### 当前实现的结构性风险

- `ChatComposer` 同时承载：
  - 移动端
  - 桌面端
  - 截图编辑
  - 语音输入
  - 桌面收藏发送
  - 提及选择
- 如果继续直接在这个文件里堆微信桌面输入框改造，后续会越来越难维护。
- 桌面输入框微信化，应该优先拆成桌面专用子组件，而不是继续在一个 JSX 大分支里硬改。

## 目标产品定义

## 1. 总体原则

- 这次不是“换一套图标”。
- 要按微信电脑端当前心智统一重做这几个层面：
  - 工具列位置
  - 图标表达
  - 输入区层级
  - 发送区稳定性
  - 展开后的多行编辑体验
  - 语音输入在桌面端的入口与反馈
- 不抄微信品牌素材，不复制官方图像资源。
- 抄的是：
  - 布局
  - 层级
  - 按钮顺序
  - 状态切换
  - 肌肉记忆

## 2. 目标排版

### 2.1 外层结构

- 桌面聊天输入区改成一整块微信式白色输入卡片。
- 结构分成三层：
  - 顶部：回复预览 / 附件草稿 / 桌面语音状态提示
  - 中部：文本编辑区
  - 底部：左侧工具图标列 + 右侧发送区
- 不再保留“上方工具条 + 下方小灰输入框”的旧结构。

### 2.2 文本编辑区

- 当前桌面 `input` 改成 `textarea`。
- 默认是“收起态”：
  - 高度控制在单行到双行的桌面微信感受区间
  - 视觉上仍像标准聊天输入区，不是大编辑器
- 点击右上角“展开”后进入“展开态”：
  - 高度显著增大
  - 允许更自然地输入长文本
  - 进入更接近微信 `4.1.8` 的多行编辑体验
- 输入区右上角固定放一个“展开/收起”图标按钮。

### 2.3 工具列

- 工具列移到输入区左下角。
- 改成纯图标按钮，不显示中文标签。
- 图标 hover / active 只给轻量底色，不要当前这种接近“按钮条”的存在感。
- 图标按钮建议统一使用：
  - `28px ~ 30px` 点击热区
  - `16px ~ 18px` 图标尺寸
  - 统一灰阶线性图标

### 2.4 发送区

- 发送按钮固定在右下角，不再因为输入框是否有内容而消失。
- 文本为空时：
  - 按钮保留位置
  - 置灰不可发，或保持弱可用态
- 文本有内容时：
  - 变成微信式绿色主按钮
- 这样可以避免当前界面在“有字/没字”之间来回抖动。

## 3. 目标按钮集合

### 3.1 输入框左下角保留的主按钮

- `表情`
- `收藏`
- `图片`
- `文件`
- `截图`
- `语音输入`

### 3.2 不应继续放在输入区的按钮

- `语音通话`
- `视频通话`
- `聊天信息`
- `查找聊天记录`

这些入口应继续留在聊天头部右上角，不和输入区混放。

### 3.3 按钮顺序

- 首选顺序：
  - `表情`
  - `收藏`
  - `图片`
  - `文件`
  - `截图`
  - `语音输入`
- 说明：
  - 这个顺序综合了公开 `4.1.8` 体验信息和当前项目已实现能力
  - 其中“聊天记录”不纳入输入区主工具列，避免和头部搜索重复

## 4. 目标功能定义

### 4.1 表情

- 点击后在输入区上方弹出表情面板。
- 桌面端继续使用已有 `StickerPanel`。
- 但入口视觉改成微信式单图标，不再显示“表情”文字。

### 4.2 收藏

- 点击后弹出收藏选择面板。
- 继续复用当前桌面收藏分享能力：
  - `DesktopFavoritePicker`
- 但面板锚点改成微信式输入区左下角悬浮展开。

### 4.3 图片

- 单独保留“图片”按钮，不与“文件”共用一个入口。
- 点击直接打开图片选择。
- 保留：
  - 粘贴图片
  - 拖拽图片
  - 图片草稿条

### 4.4 文件

- 单独保留“文件”按钮。
- 点击打开文件选择。
- 保留现有文件草稿和发送能力。

### 4.5 截图

- 保留当前桌面截图能力和截图编辑器。
- 图标按钮进入方式保持微信心智：
  - 点击图标可进入截图
  - 快捷键继续支持 `Ctrl/Cmd + Shift + S`
- 截图编辑器属于功能已够用、UI 还可后续再抛光的区域，本轮先不重做整套编辑器。

### 4.6 语音输入

- 语音输入入口固定在输入区左下角，使用小话筒图标。
- 点击后：
  - 若当前未在识别中，则开始桌面语音输入
  - 若当前正在识别，则停止识别
- 状态反馈尽量贴近微信：
  - 不在工具列上铺太多文案
  - 主要状态提示收敛到输入区上缘的弱提示区
- 继续保留当前 `dictation` 模式和识别结果“插入输入框”能力。

### 4.7 展开 / 收起

- 输入区右上角增加一个独立图标按钮：
  - 收起态点击后进入展开态
  - 展开态点击后回到收起态
- 展开态行为：
  - `Enter` 优先换行
  - 发送通过发送按钮或约定快捷键触发
- 收起态行为：
  - 保持微信式快速聊天心智
  - `Enter` 仍按当前发送偏好决定是否发送

## 5. 交互细则

### 5.1 快捷键

- 收起态：
  - 沿用当前 `sendMessageShortcut`
  - 默认 `Enter` 发送，`Ctrl/Cmd + Enter` 换行
- 展开态：
  - `Enter` 换行
  - `Ctrl/Cmd + Enter` 发送
- 原因：
  - 这更符合微信 `4.1.8` “展开后允许自然回车换行”的心智
  - 同时避免用户在长文本模式里误发

### 5.2 IME 与提及

- `textarea` 替换 `input` 后，要重点处理：
  - 中文输入法合成态
  - `@` mention 的光标位置计算
  - `Enter` 在候选词确认时不能误发
- 提及面板视觉不需要大改，但定位逻辑要适配 `textarea`。

### 5.3 焦点管理

- 打开表情 / 收藏后：
  - 不要让输入区闪烁失焦
- 关闭弹层后：
  - 焦点回到 `textarea`
- 切换到截图或文件选择后：
  - 返回时尽量恢复输入焦点和光标

### 5.4 发送按钮行为

- 点击发送时：
  - 单聊、群聊继续走原有发送逻辑
- 发送中：
  - 按钮进入 loading / disabled
- 回复态发送成功后：
  - 清空回复条
- 不因为视觉改造改动消息协议

## 6. 当前实现与目标差距

### 差距 1：工具按钮形态不对

- 当前是“带文字的小按钮”
- 目标应是“纯图标工具列”

### 差距 2：输入控件不对

- 当前是单行 `input`
- 目标应是 `textarea + 收起/展开态`

### 差距 3：发送区不稳定

- 当前空文本时发送按钮消失
- 目标应是固定右下角发送区

### 差距 4：桌面与移动实现耦合过深

- 当前一个组件里同时维护两套复杂界面
- 目标应优先拆桌面专用子组件，降低继续改造风险

### 差距 5：工具入口层级不对

- 当前“聊天记录”混在输入区工具列里
- 微信电脑端更稳定的心智是：
  - 输入区只保留编辑相关工具
  - 搜索和通话回到头部

## 7. 实施方案

### Phase 1：拆桌面专用输入组件

目标：把桌面微信化输入区从 `ChatComposer` 的巨大分支里剥出来。

涉及文件：

- `apps/app/src/components/chat-composer.tsx`
- 新增建议：
  - `apps/app/src/components/desktop-chat-composer.tsx`
  - 或
  - `apps/app/src/components/chat-composer/desktop-chat-composer.tsx`

实施点：

- 保留 `ChatComposer` 作为统一入口。
- 桌面分支改为转发到独立桌面组件。
- 移动端逻辑不动。

### Phase 2：重做桌面输入区结构

目标：先把布局改对。

实施点：

- 把桌面输入控件从 `input` 改为 `textarea`
- 新增桌面本地 UI 状态：
  - `expanded`
  - 可能需要 `isComposing`
- 布局改成：
  - 上层内容条
  - 中间编辑区
  - 下层工具列与发送区

### Phase 3：图标与弹层对齐微信

目标：把第一眼观感改成微信电脑端。

实施点：

- 统一改成纯图标按钮
- 删除工具按钮上的中文标签
- 重排顺序：
  - `表情`
  - `收藏`
  - `图片`
  - `文件`
  - `截图`
  - `语音输入`
- 收藏、表情弹层锚点改到左下工具列

### Phase 4：发送 / 换行 / 展开逻辑收口

目标：让行为也像微信，而不是只像皮肤。

实施点：

- 收起态与展开态分开处理键盘逻辑
- 修复 `textarea` 下的提及逻辑
- 处理输入法合成态下的 `Enter`
- 发送按钮常驻

### Phase 5：边缘能力回归

目标：确保现有桌面能力不丢。

实施点：

- 回归验证：
  - 表情发送
  - 收藏发送
  - 图片发送
  - 文件发送
  - 拖拽文件
  - 粘贴图片
  - 截图快捷键
  - 语音输入
  - 回复态发送
  - 单聊 / 群聊一致性

用途：

## 9. 风险与注意事项

### 风险 1：`textarea` 会影响现有光标与 mention 逻辑

- 这是这轮最容易出问题的点。
- 必须把：
  - 光标同步
  - 选区恢复
  - `@` 命中范围
  - 中文输入法合成态
  一并处理。

### 风险 2：桌面截图编辑器层级较复杂

- 这部分状态已经够重。
- 本轮不建议把截图编辑器也一起推倒重做。

### 风险 3：复用面过广

- `ChatComposer` 的桌面分支不只在主聊天页使用。
- 执行时要先确认：
  - 独立聊天窗
  - 群成员选择页等
  是否真的需要复用“微信式聊天输入框”
- 如果某些场景只是表单输入，不应被强行换成聊天输入样式。

## 10. 最终落地口径

- 这轮改造完成后，桌面聊天输入区应达到以下结果：
  - 第一眼就是微信电脑端输入框，而不是项目自定义工具条
  - 左下是纯图标编辑工具列
  - 右上有展开 / 收起
  - 中间是可伸缩文本区
  - 右下发送区稳定不跳变
  - 通话 / 聊天信息 / 聊天记录不再混在输入区

## 11. 验收标准

- 打开桌面单聊 / 群聊时，输入区结构和当前微信电脑端心智一致。
- 输入区工具列不再出现文字标签。
- 能正常发送：
  - 文本
  - 表情
  - 图片
  - 文件
  - 截图
  - 收藏转发文本
- 语音输入可从输入区左下角图标启用。
- 展开态支持自然换行，收起态保持快速发送。
- 不新增测试文件，不改移动端外观，不引入后端变更。
