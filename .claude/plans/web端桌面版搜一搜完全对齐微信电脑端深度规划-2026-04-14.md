# Web端桌面版搜一搜完全对齐微信电脑端深度规划

## 目标定义

这次不是继续给现有 `搜一搜` 页面补几个分类，而是把桌面 Web 版全局搜索重做成接近微信电脑端当前心智的产品。

目标有六个：
- 搜索入口、下拉联想、结果页、跳转回原上下文形成一条完整链路。
- 桌面版搜索体验优先对齐微信电脑端，而不是沿用当前项目的“工具页 + 指标侧栏”思路。
- 全局聊天记录搜索不再靠前端拉全量消息硬建索引，正式方案转向按需聚合检索。
- 搜索结果覆盖桌面端真实高频内容：聊天记录、联系人、公众号、公众号文章、收藏、小程序、朋友圈、广场动态/视频号。
- 与聊天页内 `Ctrl/Cmd + F` 的“当前会话查找聊天记录”能力清晰分层，不互相串味。
- 保持单世界模型，不把这次需求扩成一套独立搜索基础设施，但为后续扩展预留稳定接口。

范围只覆盖 `apps/app` 桌面 Web 版，以及为此必须补齐的 `packages/contracts` / `api` 能力。移动端不在本规划内。

## 外部对标依据

截至 2026-04-14，我补查了几条公开资料，可以确认几个对桌面实现有约束力的事实：

- 2021-09-14，微信 Windows `3.4.0` 内测版已在左侧边栏新增 `搜一搜` 入口，说明桌面端搜索不是附属功能，而是一级工作区。
  - 参考：IT之家《微信 Windows 3.4.0 内测版发布：侧边栏新增“搜一搜”》
  - 链接：https://m.ithome.com/html/575474.htm
- 2023-06-20，微信 Windows `3.9.6` 内测版已支持“按群成员和日期搜索聊天记录”，说明桌面端聊天搜索不是简单关键字过滤，而是结构化检索。
  - 参考：IT之家《微信 3.9.6 Windows 内测版更新：可按群成员和日期搜索聊天记录》
  - 链接：https://www.ithome.com/0/701/093.htm
- 2025-10-28，微信 Linux `4.1.0` 更新日志明确提到：
  - 本地搜索联系人可匹配标签、描述；
  - 全局搜索体验优化，增加最近使用的小程序、收藏等内容；
  - 全局搜索聊天记录支持多选及操作；
  - 搜一搜搜索结果支持跳朋友圈。
  - 参考：cnBeta 转快科技《微信Linux 4.1.0更新 终于能清理存储空间了》
  - 链接：https://www.cnbeta.com.tw/articles/tech/1533190.htm

基于这些信息，可以得到一个清晰结论：

- 微信电脑端搜索不是单一结果页，而是“主搜索框 + 下拉联想 + 全局结果页 + 上下文回跳”的组合。
- 搜索结果不只查联系人和会话名，还查聊天记录、收藏、小程序、朋友圈等更宽的桌面资产。
- 桌面端全局搜索和当前会话内搜索是两套能力，但交互语义一致，都是轻输入、快收敛、可直接落到原对象。

## 当前仓库现状

### 已有能力

- 全局搜索路由已经存在：
  - `apps/app/src/router.tsx`
  - `/tabs/search`
- 桌面端导航已经有 `搜一搜`：
  - `apps/app/src/features/shell/desktop-nav-config.ts`
- 全局快捷键已经存在：
  - `apps/app/src/features/shell/desktop-shell.tsx`
  - `Ctrl/Cmd + K -> /tabs/search`
- 聊天页和通讯录页顶部搜索框已经有桌面下拉面板：
  - `apps/app/src/features/search/desktop-search-launcher.tsx`
  - 当前支持：
    - 历史搜索
    - 联系人 / 世界角色轻联想
    - 打开搜一搜
- 全局搜索页已经有桌面工作区：
  - `apps/app/src/routes/search-page.tsx`
  - `apps/app/src/features/search/desktop-search-workspace.tsx`
- 当前会话内搜索能力已经存在：
  - `packages/contracts/src/client.ts`
    - `searchConversationMessages`
    - `searchGroupMessages`
  - `api/src/modules/chat/message-search.utils.ts`
  - `apps/app/src/features/chat/chat-message-search-panel.tsx`

### 当前实现为什么还不像微信电脑端

问题不是“没有搜一搜”，而是“现有搜一搜长成了错误的产品形态”。

#### 1. 搜索入口是分裂的

- 聊天页顶部搜索框、本地联想下拉、左侧导航 `搜一搜`、`Ctrl/Cmd + K` 都是独立拼的。
- 用户心智上看不到一条连续链路：
  - 输入
  - 联想
  - 进入全局搜索
  - 过滤某类结果
  - 回到原对象

#### 2. 全局结果页布局明显不像微信电脑端

当前 `DesktopSearchWorkspace` 是：
- 左栏分类 + 最近搜索
- 中间大卡片搜索框 + 卡片结果
- 右栏统计卡 + 提示文案

这更像内部工具台，不像微信电脑端的搜索工作区。最大问题有两个：
- 视觉层级太“后台化”；
- 右侧统计和搜索提示不产生任何真实操作价值。

#### 3. 聊天记录搜索策略错了

当前 `useSearchIndex` 在输入关键字后，会：
- 先拉 `getConversations`
- 再对每个会话调用：
  - `getConversationMessages`
  - 或 `getGroupMessages`
- 然后在前端本地全文匹配

这条链路的问题很大：
- 流量重
- 首次结果慢
- 会话越多越差
- 结果排序不可控
- 与已有 `message-search` 接口重复建设

更关键的是：仓库其实已经有会话级搜索接口，但全局搜一搜完全没复用。

#### 4. 搜索覆盖面不对

当前全局搜索主要覆盖：
- 会话标题 / 最后一条消息
- 联系人 / 世界角色
- 公众号账号
- 朋友圈
- 广场动态

缺失或严重不足的桌面核心内容：
- 聊天记录真正的服务端聚合搜索
- 公众号文章级结果
- 收藏 / 笔记
- 最近使用 / 我的小程序
- 视频号内容流
- 更完整的朋友圈回跳

#### 5. 数据覆盖也不完整

- `getFeed(1, 20, baseUrl)` 只查了第一页 `20` 条内容。
- 公众号只查账号摘要，不查文章。
- 收藏、笔记、小程序状态明明都存在，但没接进全局搜索。

所以当前“搜一搜”给用户的感受是：有入口，有页面，但不像微信，也不够全。

## 与微信电脑端的关键差异

### 差异 1：没有形成统一搜索会话

微信电脑端的桌面搜索，本质上是一个统一搜索会话：
- 从主搜索框开始
- 下拉联想先承接
- 明确命中则直接打开对象
- 模糊命中则进入全局结果页
- 在结果页继续切类型、继续定位

当前仓库则是：
- 输入框是一套状态
- 下拉是一套数据
- 搜索页是一套索引
- 聊天页查找记录又是一套逻辑

这会导致：
- 交互碎
- 代码重复
- 搜索规则不一致

### 差异 2：结果页的形态不对

微信电脑端全局搜索强调：
- 轻量头部
- 平铺结果
- 分组清晰
- 结果项本身就是操作入口

当前实现强调：
- 指标卡
- 说明卡
- 介绍型空页面

这会直接稀释搜索效率。

### 差异 3：微信的“快结果”和“深结果”是分层的

桌面搜索至少有两层：

- 快结果：
  - 最近搜索
  - 联系人/会话直达
  - 最近使用小程序
  - 收藏直达
- 深结果：
  - 聊天记录片段
  - 公众号文章
  - 朋友圈命中
  - 更完整的内容流

当前项目把这两层几乎混成了一层，既没有做足够快的直达，也没有做足够完整的深检索。

### 差异 4：微信桌面搜索更重“回到原上下文”

用户真正需要的不是看见搜索结果，而是：
- 打开那个人
- 回到那条消息
- 打开那篇文章
- 跳到那条朋友圈
- 打开那个收藏或小程序

当前项目部分结果能跳，但回跳策略不一致，很多只是“打开列表页”，不是“落到对象本身”。

## 核心产品决策

### 决策 1：桌面搜一搜保留独立工作区，但入口统一收口

不取消 `/tabs/search`，因为桌面端已经有左侧导航 `搜一搜`。

但必须统一入口心智：
- 聊天页顶部搜索框
- 通讯录顶部搜索框
- 左侧导航 `搜一搜`
- `Ctrl/Cmd + K`

这四个入口全部进入同一套搜索会话模型。

### 决策 2：把搜索拆成“联想层”和“结果层”

联想层解决“快直达”：
- 联系人
- 最近聊天
- 世界角色
- 公众号
- 最近使用小程序
- 最近搜索
- 收藏快捷命中

结果层解决“全局深搜”：
- 聊天记录
- 联系人
- 公众号
- 公众号文章
- 收藏 / 笔记
- 小程序
- 朋友圈
- 广场动态 / 视频号

### 决策 3：正式方案不再用前端全量消息索引

当前 `useSearchIndex` 的全量消息拉取策略只能算过渡做法，不能作为桌面微信化的正式形态。

正式方向分两步：

- 第一阶段可先复用现有会话级 `message-search` 接口做前端聚合扇出，快速验证 UI。
- 第二阶段必须补统一的桌面全局搜索聚合接口，避免前端在几十上百个会话上并发打洞。

### 决策 4：全局搜一搜和当前会话搜索保持分层，不合并

不建议把桌面全局搜一搜和聊天页内 `查找聊天记录` 做成同一块 UI。

两者关系应该是：
- 全局搜一搜：跨域找对象
- 当前会话搜索：在当前线程里精确找消息

但它们应该共享：
- 关键词高亮规则
- 消息摘要规则
- 日期格式
- 定位回跳机制

### 决策 5：桌面版先对齐微信“能搜什么”和“怎么搜”，再补高级操作

按照微信近几版更新，聊天记录多选和批量操作已经是桌面能力。

但在本项目里，这部分不应抢首批主链路。建议节奏：
- P0/P1：先把搜索链路和结果质量做对
- P2：再补聊天记录多选、批量收藏、批量转发、复制等桌面操作

## 目标产品形态

## 1. 入口层

### 1.1 聊天页 / 通讯录顶部搜索框

目标形态：
- 输入框始终在原位，不跳视觉
- 聚焦后立即展开联想面板
- 不输入时展示：
  - 最近搜索
  - 最近使用小程序
  - 最近收藏 / 最近访问对象
- 输入后按分组展示联想：
  - 联系人
  - 会话
  - 世界角色
  - 公众号
  - 收藏
  - 小程序
- 面板底部固定一行：
  - `搜一搜 “关键词”`

行为规则：
- `Enter`：
  - 若高亮某联想项，直接打开
  - 否则进入全局结果页
- `↑/↓`：
  - 在联想项间移动
- `Esc`：
  - 关闭面板

### 1.2 左侧导航 `搜一搜`

点击后进入：
- `/tabs/search`
- 自动聚焦主搜索框
- 若存在上一次搜索会话，则恢复：
  - 关键词
  - 激活分类
  - 上次滚动位置

### 1.3 `Ctrl/Cmd + K`

行为改成：
- 若当前不在输入控件内：
  - 进入 `/tabs/search`
  - 焦点落到主搜索框
- 若已经在 `/tabs/search`：
  - 只重新聚焦

## 2. 全局结果页

### 2.1 页面结构

桌面搜一搜结果页不再沿用现有 `DesktopUtilityShell` 三栏结构。

目标结构应该更接近微信电脑端：
- 顶部：扁平搜索框 + 轻量操作
- 次级：分类切换条
- 主体：结果分组列表

明确移除：
- 左侧“搜索范围”工具栏
- 右侧“搜索概览 / 提示”侧栏
- 大面积介绍型空态卡片

### 2.2 空状态

空状态不是说明文案页，而是可操作首页。

建议展示四块：
- 最近搜索
- 最近使用小程序
- 最近收藏
- 快捷分类入口
  - 聊天记录
  - 联系人
  - 公众号
  - 朋友圈
  - 小程序

### 2.3 有关键词后的结果页

顶部固定：
- 搜索框
- 分类栏

分类建议：
- 全部
- 聊天记录
- 联系人
- 公众号
- 文章
- 收藏
- 小程序
- 朋友圈
- 广场动态
- 视频号

结果区规则：
- `全部` 下按分组展示
- 其他分类下展示单一列表
- 每组默认展示前 `5~8` 条
- 组头有 `查看更多`
- 结果项 hover 可见快捷操作

## 3. 结果项设计

### 3.1 聊天记录

展示字段：
- 主标题：命中发送者 / 会话名
- 副信息：会话名 + 时间
- 摘要：命中的消息片段
- 辅助标签：
  - 单聊 / 群聊
  - 图片 / 文件 / 链接 / 语音 / 表情

点击行为：
- 打开对应线程
- 定位到消息锚点
- 高亮目标消息

二期 hover 操作：
- 收藏
- 转发
- 复制
- 多选

### 3.2 联系人 / 世界角色

展示字段：
- 头像
- 备注名 / 角色名
- 关系、地区、来源、标签

点击行为：
- 打开 `/character/$characterId`

排序权重：
- 已是联系人 > 世界角色
- 备注名命中 > 原名命中 > 标签命中

### 3.3 公众号

账号结果和文章结果分开。

账号结果展示：
- 头像
- 账号名
- `订阅号 / 服务号`
- handle / 简介

点击行为：
- 打开账号主页

### 3.4 公众号文章

展示字段：
- 文章标题
- 账号名
- 摘要
- 发布时间

点击行为：
- 打开 `/official-accounts/articles/$articleId`
- 桌面端落在公众号工作区右侧阅读器

### 3.5 收藏 / 笔记

展示字段：
- 标题
- 分类
- 摘要
- 标签 / 更新时间

点击行为：
- 收藏消息 -> 打开原对象
- 笔记 -> 打开笔记工作区 / 独立笔记窗口

### 3.6 小程序

展示字段：
- 小程序图标
- 名称
- slogan / 场景描述
- 状态：
  - 最近使用
  - 已加入我的小程序

点击行为：
- 打开 `/tabs/mini-programs?miniProgram=:id`

### 3.7 朋友圈

展示字段：
- 作者
- 文本摘要
- 发布时间
- 命中类型：
  - 正文
  - 评论
  - 位置

点击行为：
- 优先跳到对应动态
- 若是好友独立时间线更合适，则进入：
  - `/desktop/friend-moments/$characterId`
- 否则回落：
  - `/tabs/moments#moment=:id`

### 3.8 广场动态 / 视频号

展示字段：
- 作者
- 内容摘要
- surface 标签：
  - 广场动态
  - 视频号

点击行为：
- 跳到对应内容流并定位目标项

## 搜索域规划

### P0 必须覆盖

- 聊天记录
- 联系人 / 世界角色
- 公众号账号
- 公众号文章
- 收藏 / 笔记
- 最近使用 / 我的小程序
- 朋友圈

### P1 补齐

- 广场动态
- 视频号
- 更完整的服务号消息对象搜索

### 不建议首批做

- 管理后台数据
- 设置项
- 系统配置项
- 云世界平台管理对象

桌面搜一搜应该先专注用户主前台内容，而不是把内部系统入口全塞进去。

## 搜索排序与召回策略

## 1. 联想层排序

联想层强调秒开和直达，排序建议：
- 完全前缀命中
- 标题/备注命中
- 最近使用权重
- 当前场景权重
  - 从聊天页打开时，最近聊天更高
  - 从通讯录打开时，联系人更高

## 2. 结果层排序

### 全部页

分组顺序建议：
1. 聊天记录
2. 联系人
3. 公众号
4. 文章
5. 收藏
6. 小程序
7. 朋友圈
8. 广场动态
9. 视频号

原因很简单：
- 桌面端高频第一位永远是找消息和找人。
- 内容消费类结果不该压过消息和联系人。

### 组内排序

统一策略：
- 精确标题命中 > 摘要命中 > 附属字段命中
- 最近活跃 > 更早活跃
- 已建立关系 > 弱关系

聊天记录额外规则：
- 会话标题命中略高于消息正文命中
- 最近 7 天消息命中可轻微加权
- 附件消息按类型加标签，不提高基础排名

## 技术方案

## 1. 前端架构重组

### 1.1 统一搜索会话模型

新增统一桌面搜索状态，不再让：
- launcher
- search page
- chat page local state

各自维护不同关键词和分类。

建议新增：

## 2. 数据源接入策略

### 2.1 聊天记录

正式方向：
- 不再用 `getConversationMessages` / `getGroupMessages` 全量拉消息
- 第一阶段复用：
  - `searchConversationMessages`
  - `searchGroupMessages`
- 第二阶段补统一接口：
  - `POST /api/search/desktop`
  - 或 `GET /api/search/desktop`

建议最终采用统一接口，而不是长期前端扇出。

原因：
- 前端并发量不可控
- 结果分页和总数不稳定
- 很难做统一排序
- 很难做多选操作的后续能力

### 2.2 联系人

联想层：
- `getFriends`
- `listCharacters`

结果层：
- 仍可先走前端匹配
- 若后续联系人量扩大，再补服务端搜索接口

### 2.3 公众号与文章

账号层已有：
- `listOfficialAccounts`

文章层已有：
- `getOfficialAccount`
- `getOfficialAccountArticles`
- `getOfficialAccountArticle`

但缺全局文章检索入口。

建议正式补：
- `GET /api/official-accounts/articles/search`
  - `keyword`
  - `accountId?`
  - `cursor?`
  - `limit?`

否则前端必须把所有号文章都扫一遍，桌面体验会很差。

### 2.4 收藏 / 笔记

已有数据：
- `getFavorites`
- `getFavoriteNotes`
- `desktop-favorites-storage`

这一块非常适合优先接入，因为：
- 已有桌面本地态
- 用户价值高
- 也符合微信近版本“搜索收藏”的方向

### 2.5 小程序

已有数据主要在本地：
- `useMiniProgramsState`
- `mini-programs-storage`
- `mini-programs-data`

桌面联想和结果页都应直接接入：
- 最近使用
- 我的小程序
- 全部目录

### 2.6 朋友圈 / 广场动态 / 视频号

当前问题：
- 朋友圈可以搜
- 但广场动态只查第一页
- 视频号尚未单独接入搜一搜

建议正式补后端查询参数：
- `GET /api/feed?surface=feed|channels&keyword=&page=&limit=`
- `GET /api/moments?keyword=&cursor=&limit=`

否则搜一搜结果会天然缺内容。

## 3. 统一后端接口建议

如果要把桌面搜一搜真正做成稳定能力，建议在 `api/` 新增独立搜索聚合模块。

建议新增：
- `api/src/modules/search/`
- `packages/contracts/src/search.ts`

建议接口：

```ts
POST /api/search/desktop
{
  keyword: string;
  scope: "all" | "messages" | "contacts" | "officialAccounts" | "articles" | "favorites" | "miniPrograms" | "moments" | "feed" | "channels";
  cursorByScope?: Record<string, string | undefined>;
  limitByScope?: Record<string, number | undefined>;
}
```

返回：

```ts
{
  keyword: string;
  scopes: Array<{
    scope: string;
    total: number;
    nextCursor?: string;
    items: SearchResultItem[];
  }>;
}
```

这样做的价值：
- 统一排序口径
- 统一分页
- 统一总数
- 统一扩展新搜索域
- 前端可以更像微信那样稳定展示分组结果

## 4. 路由状态建议

当前 `/tabs/search` 只用 hash 保存：
- `q`
- `category`
- `source`

对桌面正式版本不够。

建议升级成 URL search 参数或统一 session state，至少补：
- `q`
- `scope`
- `source`
- `focus`
- `entry`

其中：
- `focus`
  - 恢复主输入框焦点态
- `entry`
  - 标记来自聊天、通讯录、左侧导航还是快捷键

对外可分享的状态放 URL；
纯交互态放 store。

## 风险与约束

### 风险 1：前端扇出搜索会把桌面端拖慢

如果继续维持“前端对每个会话单独搜”的方案，数据量一大就会卡。

结论：
- 只能作为验证阶段方案
- 不能作为正式长期方案

### 风险 2：内容流接口当前不足以支撑全局搜索

尤其是：
- `feed` 只有分页第一页接入
- `official articles` 没有全局检索接口

结论：
- 搜一搜深搜一旦做实，后端必须补接口

### 风险 3：现有桌面空态和视觉体系会拖慢重构

当前搜索页已经写了很多侧栏和提示卡。

结论：
- 不建议继续修修补补
- 应直接按新骨架重排

## 验收标准

满足以下条件，才算桌面搜一搜进入“像微信电脑端”的可交付状态：

- 聊天页、通讯录页、左侧导航、`Ctrl/Cmd + K` 四个入口使用同一套搜索会话。
- 聚焦搜索框时能看到最近搜索、收藏、小程序等快入口。
- 输入关键词后，下拉联想支持键盘上下选择和回车打开。
- 进入 `/tabs/search` 后，不再出现当前这种左栏分类 + 右栏统计的工具页形态。
- 全局结果至少覆盖：
  - 聊天记录
  - 联系人
  - 公众号
  - 公众号文章
  - 收藏
  - 小程序
  - 朋友圈
- 聊天记录搜索不再使用全量消息加载建索引作为正式方案。
- 搜索结果点击后，优先落到原对象，而不是泛泛跳去列表首页。
- 朋友圈结果能直接跳到目标动态或对应好友时间线。

## 最终结论

桌面版 `搜一搜` 当前最大的问题不是“结果还不够多”，而是整套产品结构就不是微信电脑端那一型。

正确路线不是继续给 `desktop-search-workspace.tsx` 加分类卡和说明文案，而是：
- 统一入口
- 拆开联想层与结果层
- 重做结果页骨架
- 把聊天记录搜索从前端全量索引迁回按需聚合
- 把收藏、小程序、文章、朋友圈这些桌面真实资产接进来

只有这样，桌面搜一搜才能从“有个页面”升级成“像微信电脑端那样真正好用的搜索工作区”。
