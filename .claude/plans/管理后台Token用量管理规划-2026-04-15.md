# 管理后台 Token 用量管理规划

## 1. 目标

- 在实例级管理后台新增一套可运营的 AI 用量中心。
- 能看到单个对象花了多少 `token`、折算花了多少钱、用了多少次请求。
- 能看到整座世界在指定时间范围内的总消耗、趋势和分布。
- 能按时间统计：今天、昨天、近 7 天、近 30 天、本月、自定义区间，并支持按天 / 周 / 月聚合。
- 后续可继续扩展到预算预警、限额和云平台跨实例汇总。

## 2. 先明确口径

### 2.1 当前实例后台不是“多真实用户后台”

- 现在 Core API 已经是单世界模型：`1 个实例 = 1 个世界主人`。
- 所以在当前 `apps/admin` 里，不能按“多个真实用户账号”做统计，因为实例里本来就只有一个世界主人。
- 这次如果你说“每一个人的 token”，在实例级后台里最合理的定义应该是：
  - 每个角色
  - 每个会话对象
  - 每个功能场景

### 2.2 推荐的一期口径

- “各自花费了多少 token”：
  - 默认按 `Character` 统计
  - 同时支持按 `Conversation / Group / 场景类型` 切换
- “整体花了多少 token”：
  - 统计整个实例在时间范围内的总输入 token、总输出 token、总 token、总估算费用
- “按时间统计”：
  - 以 `occurredAt` 为准
  - 数据库存 UTC 存储
  - 后台展示按当前管理端时区显示

### 2.3 如果你真正想看“每个世界主人”的账

- 那不应该放在当前 `apps/admin`。
- 那应该是 `apps/cloud-console` + `apps/cloud-api` 的平台级账本功能。
- 建议这次先把实例级账本做稳，二期再做跨实例汇总。

## 4. 数据模型设计

## 4.1 核心原则

- 不做“覆盖式统计表”，先做“不可变账本表”。
- 每次 AI 请求结束后写入一条账本记录，再通过聚合查询生成统计。
- 这样后面改统计口径、改页面、改报表都不用补底层数据。

## 4.3 为什么价格要落库快照

- 模型价格后面可能会变。
- 如果只存 token，不存“当时按什么价格算”，历史金额会漂移。
- 所以建议在账本行里直接存：
  - 当次输入单价
  - 当次输出单价
  - 当次估算费用

## 5. 价格与费用计算规则

## 5.1 价格来源

- 一期不依赖外部实时价格。
- 价格由后台手工配置，避免接口抖动和价格源不一致。
- 价格配置建议先存在 `SystemConfig`：
  - `token_pricing_catalog`

### 建议结构

```ts
type TokenPricingCatalog = {
  currency: "CNY" | "USD";
  items: Array<{
    model: string;
    inputPer1kTokens: number;
    outputPer1kTokens: number;
    enabled: boolean;
    note?: string;
  }>;
};
```

## 5.2 费用计算

- `estimatedInputCost = promptTokens / 1000 * inputPer1kTokens`
- `estimatedOutputCost = completionTokens / 1000 * outputPer1kTokens`
- `estimatedCost = estimatedInputCost + estimatedOutputCost`

## 5.3 一期的推荐边界

- 先只对“返回 token usage 的文本模型请求”做准确统计。
- 对以下场景先单独记账，但默认不并入 token 总览，避免假精确：
  - 语音转写
  - 语音合成
  - 某些不返回 token 的第三方调用
- 这些场景可以先记录为：
  - 有请求记录
  - 无 token 明细
  - 后续二期再扩成“综合 AI 成本中心”

## 6. 后端采集方案

## 6.1 采集入口

- 核心落点应放在：
  - `api/src/modules/ai/ai-orchestrator.service.ts`
- 原因：
  - 绝大多数模型调用都集中在这里
  - 可以避免每个业务模块各自重复埋点

## 6.2 需要补齐的能力

- 当前代码里虽然能拿到部分 `usage`，但没有持久化。
- 当前 `GenerateReplyResult` 只有 `tokensUsed`，对后台统计不够。
- 建议把返回结构升级成更完整的 usage 对象，例如：

```ts
type AiUsageMetrics = {
  promptTokens?: number;
  completionTokens?: number;
  totalTokens?: number;
  raw?: Record<string, unknown> | null;
};
```

- 同时给所有 AI 调用增加一个统一上下文：

```ts
type AiUsageContext = {
  surface: "app" | "admin" | "scheduler" | "system";
  scene: string;
  scopeType: "character" | "conversation" | "group" | "world" | "admin_task";
  scopeId?: string;
  characterId?: string;
  conversationId?: string;
  groupId?: string;
  ownerId?: string;
};
```

## 8. 后台前端规划

- 第三屏：排行榜与分布

- “按角色”

- “按场景”

## 9. 分阶段实施建议

## Phase 1：先把账记准

- 新增账本实体
- 在 `AiOrchestratorService` 内落 usage 采集
- 把文本模型主路径都接上
- 提供 overview / trend / records 三个接口
- 管理后台先做一个可用页面

### Phase 1 完成后你就能看到

- 总 token
- 总费用
- 按角色花费排行
- 按时间走势
- 单次请求明细

## Phase 2：把运营能力补齐

- 增加价格配置界面
- 增加按模型 / 场景 / 计费来源统计
- 增加 CSV 导出
- 增加异常请求筛选

## Phase 3：从“统计”升级到“管理”

- 增加预算
  - 日预算
  - 月预算

  - 仅告警

## 10. 风险与关键决策

## 10.1 最大风险

- 现在很多调用虽然拿到了结果，但没有完整 usage 上下文。
- 如果不先统一埋点口径，后面统计一定会有缺口。

## 10.2 最关键决策

- 决策一：
  - 一期先做“实例内按角色 / 会话 / 场景统计”
  - 不做“跨实例按真实用户汇总”
- 决策二：
  - 一期先做文本 token 账本
  - 语音与非 token 型成本二期再并表
- 决策三：
  - 历史账金额按“入账时价格快照”固定
  - 不做历史金额实时回算

## 12. 我给你的推荐结论

- 这件事最值得做，而且应该尽快做，因为它会直接影响后续模型选择、角色运营和成本可控性。
- 最稳的落地方式不是一上来做“限额”，而是先做“可追溯账本 + 时间趋势 + 角色分布 + 金额估算”。
- 一期只要把文本模型主链路打通，你马上就能回答三个最关键的问题：
  - 最近一段时间总共烧了多少 token？
  - 哪些角色 / 会话最贵？
  - 花费是在什么时间段突然上升的？

## 13. 二期平台化扩展

- 如果你后面要看“所有云世界主人”各自花了多少，就复用同一套账本结构，上移到：
  - `apps/cloud-api`
  - `apps/cloud-console`
- 到那时新增的主统计维度会是：
  - 手机号
  - 云世界 ID
  - 世界实例地址
  - 世界主人

---

当前建议的正式实施顺序：

1. 先做实例级 `Token 用量` 页面与账本表。
2. 先统计文本模型主链路，不把语音成本硬塞进 token 报表。
3. 页面主口径统一写成“角色 / 会话 / 场景”，不要写“用户”。
4. 等实例级跑稳后，再决定要不要把这套能力上移到云平台。
