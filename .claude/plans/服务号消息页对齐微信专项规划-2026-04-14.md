# 服务号消息页对齐微信专项规划

日期：2026-04-14
范围：`/tabs/chat#officialView=service-account...` 对应的桌面消息工作区、移动端服务号线程、会话列表里的服务号/订阅号入口

## 一、先给结论

当前这套“左侧会话列表 + 中间服务号线程 + 右侧固定文章阅读栏”的方案，方向上就不是微信。

要抄微信，这一块要改成：

1. 服务号先是“一个消息会话”，不是一个独立工具工作区。
2. 文章阅读是“进入阅读态”，不是常驻在右边的第三栏。
3. 标题、标签、说明、操作按钮必须降级，消息内容和文章内容才是主角。
4. 公众号/服务号/订阅号入口在消息列表里要和普通会话同一视觉系统，不能像一组单独拼进去的业务卡片。

这意味着后面不该继续在现有三栏结构上修补，而要做一次结构级收口。

## 二、当前实现为什么不像微信

### 1. 服务号被做成了“业务工作台”，而不是“聊天会话”

当前入口链路：

- 消息列表：[apps/app/src/routes/chat-list-page.tsx](/home/ps/claude/yinjie-app/apps/app/src/routes/chat-list-page.tsx:865)
- 桌面消息工作区：[desktop-chat-workspace.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx:1543)
- 服务号线程：[official-account-service-thread.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/official-accounts/service/official-account-service-thread.tsx:254)

问题：

- 服务号打开后不是标准消息区，而是定制化的业务面板。
- 线程头部长期展示说明、标签、按钮组，这些在微信里都属于次级信息。
- 文章和消息同时驻留在桌面三栏里，主次关系反了。

### 2. 右侧文章栏是“常驻读者面板”，不是微信式阅读态

当前 `selectedServiceAccountId + selectedOfficialArticleId` 会把线程和文章同时保留在画面上。

结果：

- 宽度一紧，右侧阅读区先坏。
- 文章正文会出现一列列竖向掉字的观感。
- 用户会觉得这是一个后台工作台，而不是微信消息页。

微信的核心不是“三栏更强”，而是“当前只让你聚焦一个主任务”。

### 3. 服务号/订阅号入口和普通会话不是一套规则

相关组件：

- [official-service-conversation-card.tsx](/home/ps/claude/yinjie-app/apps/app/src/components/official-service-conversation-card.tsx:8)
- [subscription-inbox-card.tsx](/home/ps/claude/yinjie-app/apps/app/src/components/subscription-inbox-card.tsx:7)
- [official-accounts-entry-card.tsx](/home/ps/claude/yinjie-app/apps/app/src/components/official-accounts-entry-card.tsx:5)

虽然最近已经做过一轮对齐，但本质上它们仍然是“定制卡片”，不是“标准会话行的特例”。

微信里这类入口的感觉是：

- 本质仍然属于消息列表的一项
- 只在头像、名称、预览文案、未读语义上有差异
- 不是一块颜色和层级都明显不同的业务卡

### 4. 当前路由语义也在推着页面跑偏

当前 hash 语义：

- [desktop-chat-route-state.ts](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/chat/desktop-chat-route-state.ts:1)

现在的 `officialView=service-account&accountId=...&articleId=...` 语义，天然偏向“线程与文章并存”。

这对微信式页面不友好，因为微信更像：

- 先打开服务号消息
- 再从消息里进入文章阅读态
- 需要时再返回消息流

也就是“一个主视图 + 一个进入态”，而不是“两个主视图并列”。

## 三、目标页面应该长什么样

## 目标 A：桌面端消息列表

目标不是做“官方消息专区”，而是把公众号相关入口彻底混入微信式聊天列表。

要求：

1. `订阅号消息`、`世界服务台`、`公众号` 三类入口都使用和普通会话同一行高、同一时间列、同一未读位规则。
2. 服务号头像可以保留品牌化图标，但不能抬高整行密度。
3. 分隔线、hover、active、已读/免打扰态必须和普通聊天完全一致。
4. 不再出现“这几行像另外一个模块”的感觉。

## 目标 B：桌面端服务号线程

打开服务号后，右侧主区域默认显示“消息线程”，不是“线程 + 阅读器并排”。

目标结构：

1. 顶部是微信式轻量标题栏
2. 中间是消息流
3. 文章卡片仍然存在于消息流中
4. 点击文章卡片后，主区域切换到文章阅读态
5. 从文章阅读态返回后，回到原消息流滚动位置

要点：

- 服务号说明文案不能长期占据头部视觉中心
- `公众号主页 / 到手机继续 / 免打扰` 这类操作不能并排占据主标题区
- 这些动作要收进口袋式工具区或 `...` 菜单

## 目标 C：桌面端文章阅读态

桌面端文章应该是“消息工作区里的阅读态”，不是第三栏。

建议形态：

1. 文章阅读占据原主内容区
2. 左侧保留消息列表
3. 服务号线程被阅读态替换，不与阅读器并排
4. 阅读区使用固定阅读宽度，正文居中
5. 顶部只保留返回、标题来源、更多操作

这更接近微信的任务节奏：

- 看消息时专心看消息
- 进文章时专心读文章
- 回来时回到消息

## 目标 D：移动端服务号页

移动端其实已经比桌面端更接近微信，但还要继续收紧：

1. 顶栏只保留返回、标题、更多
2. 说明文案继续弱化
3. 服务消息气泡与文章卡片保持微信式轻量层级
4. 打开文章必须进入独立文章页，不做内嵌双态并排

## 四、建议的结构改造

## Phase 1：先改路由语义和视图关系

目标：把“并排双主视图”改成“单主视图 + 阅读态”。

改动：

1. 保留 `officialView=service-account`
2. `articleId` 的语义从“右栏当前文章”改成“当前处于阅读态的文章”
3. `DesktopChatWorkspace` 不再把服务号线程和文章阅读器同时渲染
4. 服务号消息页改成：
   - 无 `articleId`：显示线程
   - 有 `articleId`：显示文章阅读态

目标：去掉“服务号工具工作台”感。

3. `公众号主页 / 到手机继续 / 免打扰` 收进：
   - 标题栏图标
   - 或 `...` 下拉菜单
4. 线程消息区使用标准消息流节奏，不额外挂一个大块介绍卡

4. 卡片摘要、元信息、按钮层级全面收轻
5. 需要“打开独立窗口”时，复用已有文章窗口路由，不在主态常驻第三栏

## 五、具体交互口径

### 桌面端

1. 点击 `世界服务台`
2. 进入服务号消息线程
3. 点击文章卡片
4. 主区域切到文章阅读态
5. 点击返回
6. 回到服务号线程

禁止继续保留的交互：

1. 线程和文章长时间并排常驻
2. 头部长期展示一排业务按钮
3. 为了“信息更全”牺牲阅读宽度

### 移动端

1. 点击 `世界服务台`
2. 进入服务号消息页
3. 点击文章卡片
4. 跳独立文章页
5. 返回后回到服务号消息页

## 六、验收标准

满足以下条件才算“抄微信抄对了”：

1. `960 / 1024 / 1280 / 1440` 桌面宽度下，不再出现文章标题或正文竖排掉字。
2. 服务号页默认只存在一个主任务面：消息线程或文章阅读，不再双主视图并排。
3. 服务号头部从“业务工作台”变回“微信式轻标题栏”。
4. 消息列表里的服务号/订阅号/公众号入口，不再看起来像额外拼接的卡片模块。
5. 移动端打开服务号和文章的节奏，和微信一样是“线程页 -> 独立文章页”。

## 七、推荐执行顺序

不要再从样式边角开始。

正确顺序应该是：

1. 先做 Phase 1，改视图关系
2. 再做 Phase 2，收线程壳
3. 再做 Phase 3，重做阅读态
4. 最后做 Phase 4，统一列表入口

原因很简单：

只要三栏并排还在，后面所有“抄微信”的样式努力都会继续被错误结构抵消。

## 八、下一步建议

下一刀不要再调 spacing。

直接开做：

1. 把桌面服务号页从“线程 + 右栏文章”改成“线程 / 文章二选一”
2. 把服务号头部按钮组收进口袋菜单
3. 把右侧文章阅读抽成独立阅读态组件

这三步做完，页面方向才会真的回到微信上。
