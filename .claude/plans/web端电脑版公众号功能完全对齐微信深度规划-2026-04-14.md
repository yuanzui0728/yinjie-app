# Web端电脑版公众号功能完全对齐微信深度规划

## 目标定义

这次不是继续给现有公众号页面“加功能”，而是把桌面 Web 版公众号能力重组到接近微信电脑版的产品形态。

目标有四个：
- 公众号在桌面端同时具备“通讯录目录”和“消息入口”两套语义。
- 订阅号消息与服务号消息彻底分流，表现层对齐微信电脑版。
- 文章阅读、账号主页、服务消息线程共用一套桌面工作区，不再各自长成独立页面。
- 交互、布局、信息密度、状态语义尽量 1:1 对齐微信电脑版，但不直接照搬微信品牌素材。

范围只覆盖 `apps/app` 的桌面 Web 版表现，以及为此必须补齐的 `packages/contracts` / `api` 能力。移动端不在本规划内。

## 外部对标依据

截至 2026-04-14，我补查了几条公开资料，能确认几个稳定事实：

- 2019 年微信 PC 测试版把公众号阅读从“账号列表页 -> 下一页文章页”改成了同屏三栏工作区，强调“点击账号后直接在右侧展示”，不再跳下一级页面。
  - 参考：快科技转载太平洋电脑网《公众号和小程序重磅升级 微信PC版新版体验》
  - 链接：https://news.mydrivers.com/1/664/664889.htm
- 2020 年微信公众号主页的移动端改版已经明确把账号主页拆成“消息 / 视频 / 服务”三个模块，说明公众号主页不是单纯的文章列表，而是“内容 + 服务”的复合入口。
  - 参考：36 氪《微信再改版：公众号视频新增观看入口，你的微信年账单也来了》
  - 链接：https://36kr.com/p/1013981130054281
- 2022 年微信 Windows 3.7.5.5 测试版明确提到“服务号消息默认免打扰”“订阅号消息样式调整”，说明桌面端对服务号和订阅号的消息策略本来就不同。
  - 参考：IT之家《微信 Windows 测试版 3.7.5.5 发布：群视频通话时可共享屏幕，服务号消息默认免打扰》
  - 链接：https://www.ithome.com/0/626/780.htm
- 2025 年公开报道再次提到订阅号内容分发已经不再是纯时间序，而是会围绕阅读效率和偏好做排序优化。
  - 参考：太平洋科技《微信想让公众号再次伟大》
  - 链接：https://g.pconline.com.cn/x/1915/19152412.html

基于这些资料，可以得到一个对桌面版实现有约束力的结论：

- 微信电脑版公众号不是“通讯录里的静态资料页”。
- 它本质上是“目录入口 + 消息入口 + 阅读器 + 服务面板”的组合体。
- 订阅号是聚合消息。
- 服务号是低打扰会话。
- 桌面形态强调同屏切换，不强调频繁跳页。

### 现状为什么还不像微信电脑版

现有实现已经有骨架，但产品组织方式还是错的。

- 公众号目录工作区过重，像“后台式资料页”。
  - 左栏有统计卡、筛选胶囊、卡片化账号列表。
  - 中栏是大号账号简介区 + 统计卡 + 文章卡片列表。
  - 右栏是文章阅读器。
  - 代码位置：`apps/app/src/features/desktop/official-accounts/desktop-official-accounts-workspace.tsx:336`
- 消息列表虽然插进了订阅号和服务号，但只是“插队显示”，没有和普通会话合并成统一桌面消息条目模型。
  - 当前是 `subscriptionInbox` 和 `serviceConversations` 先渲染，再渲染普通会话。
  - 这意味着排序、置顶、搜索、免打扰表现都不是统一语义。
  - 代码位置：`apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx:244`、`1232`
- 订阅号消息工作区现在是“按账号分组的卡片列表 + 右侧阅读器”。
  - 这更像内部演示页，不像微信电脑版的信息流。
  - 当前还会在打开订阅号聚合页后直接把整批未读推送全部置已读。
  - 代码位置：`apps/app/src/features/desktop/official-accounts/desktop-subscription-workspace.tsx:29`、`81`、`165`
- 服务号线程目前只是最简单的气泡流。
  - 桌面态没有详情抽屉，没有免打扰操作，没有“打开公众号主页/服务菜单”的顶部动作。
  - 打开线程就把整条线程全部标已读，策略过粗。
  - 代码位置：`apps/app/src/features/official-accounts/service/official-account-service-thread.tsx:40`、`64`、`82`
- 后端虽然已经有 `menuPayload` 和 `follow.isMuted` 字段，但契约没暴露，前端没消费。
  - `api/src/modules/official-accounts/official-account.entity.ts:30`
  - `api/src/modules/official-accounts/official-account-follow.entity.ts:16`

## 与微信电脑版的关键差异

### 差异 1：消息模型没有统一

微信电脑版用户感知到的是一个统一消息列表：
- 普通单聊
- 群聊
- 订阅号消息
- 服务号消息

当前仓库实际是两套并排数据：
- `getConversations()`
- `getOfficialAccountMessageEntries()`

这本身没问题，问题在于前端没有建立统一的桌面消息条目视图模型，所以表现层只能“插队渲染”，无法自然实现：
- 统一排序
- 统一搜索
- 统一置顶
- 统一免打扰视觉
- 统一右键菜单

### 差异 2：订阅号仍是“按账号分组”，不是“聚合信息流”

微信系订阅号的核心不是“进入某个号再看文章”，而是“先在订阅流里消费，再进入单号深看”。

当前桌面端 `DesktopSubscriptionWorkspace` 还是：
- 先按号分组
- 每个号下面再列文章

这会带来三个问题：
- 信息流感太弱。
- 同一时间多号更新时缺少全局排序。
- 未读语义只能粗暴按组或整页处理。

### 差异 3：服务号没有桌面“低打扰会话”语义

微信桌面端对服务号的核心不是“像普通人聊天”，而是：
- 默认低打扰
- 仍然出现在消息列表
- 以通知 / 卡片 / 服务入口为主

当前实现虽然 `OfficialAccountFollowEntity` 已有 `isMuted` 字段，但没有暴露给前端，也没有在左侧消息行或线程头部表现出来。

### 差异 4：桌面工作区还是“多个独立页面”，不是“一个壳里的多个状态”

目前桌面端的路由虽然复用了工作区组件，但心智还是：
- 公众号列表页
- 公众号详情页
- 文章页
- 服务号线程页
- 订阅号页

微信电脑版更接近：
- 通讯录里的公众号目录工作区
- 消息里的公众号消息工作区
- 右侧文章阅读器是被多个入口复用的同一块区域

也就是说，桌面态应该以“工作区状态切换”为主，不应该继续强化“页面跳转”。

## 核心产品决策

### 决策 1：继续保持公众号与聊天域分离，但在前端做统一消息条目层

不建议把 `ConversationType` 硬扩成 `official_account`。

原因：
- 现有 `direct / group` 会话链路已经和 AI 回复、群逻辑、消息详情、背景设置深度耦合。
- 公众号的订阅号聚合和服务号线程都不是这套语义。
- 为了复制微信桌面体验，不需要把后端领域模型强行揉成一张表。

结论：
- 后端继续分域。
- 前端新增统一的 `DesktopMessageEntry` 视图模型。
- 在消息列表层解决排序、搜索、激活态、右键菜单问题。

### 决策 2：订阅号消息视为一个固定“特殊会话”

订阅号消息在消息列表里表现为一个固定条目，但它不是普通聊天会话。

它需要有自己的状态：
- 最后活动时间
- 未读数
- 置顶状态
- 隐藏状态
- 最后阅读位置

因此不能继续只靠 `OfficialAccountSubscriptionInboxSummary` 顶一下 UI。

建议新增独立状态表：
- `OfficialAccountEntryState`
- 主键语义：`ownerId + entryKind + accountId?`
- `entryKind` 取值：`subscription_inbox | service_account`

这个表只服务桌面消息体验，不污染普通 `Conversation`。

### 决策 3：服务号默认免打扰，但允许显式取消

现有 follow 表已经有 `isMuted` / `mutedAt`，直接复用，不另起一套 mute 模型。

规则建议：
- 新关注服务号时，默认 `isMuted=true`
- 订阅号不参与该规则
- 消息列表上服务号未读只显示小红点，不抢普通聊天的数字提醒层级
- 线程头部和右键菜单都可以切换免打扰

### 决策 4：桌面端文章阅读器做成跨入口共享面板

不再让文章详情页在桌面端承担“页面”角色，而是只作为深链接入口。

桌面真实行为应该是：
- 从公众号目录点文章，右栏打开阅读器
- 从订阅号聚合流点文章，右栏打开阅读器
- 从服务号卡片点文章，右栏打开阅读器
- 深链接进入 `/official-accounts/articles/:id` 时，自动还原到正确工作区状态

## 目标产品形态

### 1. 通讯录里的“公众号”

它承担的是目录和资料语义，不承担主消息消费。

桌面目标结构：
- 左栏：公众号列表，密度更接近微信 PC，而不是现在的大卡片。
- 中栏：账号主页，顶部是头像、名称、认证、关注、发消息、更多。
- 中栏主体：按账号类型切标签。
  - 订阅号：默认 `消息`
  - 服务号：默认 `消息`
  - 如果有 `menuPayload`，显示 `服务`
  - `视频` 暂不实现，但布局预留
- 右栏：文章阅读器或服务内容面板

这里的关键不是“把当前三栏保留”，而是把中栏从“资料看板”改成“微信式账号主页”。

### 2. 消息里的“订阅号消息”

它是一个固定消息条目，不是目录页入口。

点击后，内容区改成两段式：
- 中栏：订阅号聚合流
- 右栏：文章阅读器

聚合流目标形态：
- 默认按“最后推送时间”逆序
- 列表项主信息是文章，而不是账号卡
- 同一账号可连续出现多篇，不强制按账号折叠
- 行内露出账号名、标题、摘要、时间、未读态
- 顶部提供搜索和轻筛选

这和现在的“按账号分组卡片块”是两套产品。

### 3. 消息里的“服务号会话”

它在左侧消息列表里和普通会话一起出现，但默认免打扰。

点击后，内容区改成：
- 中栏：服务号消息线程
- 右栏：文章阅读器或账号详情抽屉

线程目标形态：
- 顶部显示账号名、认证状态、免打扰状态、更多菜单
- 中部是服务消息流，支持 `text`、`article_card`
- 底部不提供用户输入框
- 点击文章卡片只更新右栏，不跳出当前线程

## 数据与契约规划

### 现有可复用能力

- `OfficialAccountDeliveryEntity` 已能承载订阅号投递。
- `OfficialAccountServiceMessageEntity` 已能承载服务号消息。
- `OfficialAccountFollowEntity.isMuted` 可直接承载服务号免打扰。
- `menuPayload` 已预留，可继续扩展成服务菜单。

### 必补的契约字段

建议扩展 `packages/contracts/src/official-accounts.ts`：

- `OfficialAccountSummary`
  - 新增 `isMuted?: boolean`
  - 新增 `mutedAt?: string`
  - 新增 `menu?: OfficialAccountMenu | null`
- `OfficialAccountServiceConversationSummary`
  - 新增 `isMuted: boolean`
  - 新增 `mutedAt?: string`
  - 新增 `lastActivityAt?: string`
- `OfficialAccountSubscriptionInbox`
  - 保留现有 `groups` 兼容移动端
  - 新增 `feedItems: OfficialAccountDeliveryItem[]`
  - 新增 `lastActivityAt?: string`
- 新增 `OfficialAccountEntryState`
  - `entryKind`
  - `accountId?`
  - `isPinned`
  - `pinnedAt?`
  - `isHidden`
  - `hiddenAt?`
  - `lastReadAt?`

### 建议新增后端路由

- `GET /api/official-accounts/entry-states`
- `PATCH /api/official-accounts/entry-states`
- `PATCH /api/official-accounts/:id/preferences`
  - 首轮只接 `isMuted`
- `GET /api/official-accounts/:id/menu`
  - 也可以直接并入 `GET /api/official-accounts/:id`

### 已读策略必须调整

当前问题：
- 进入订阅号页就批量 `markSubscriptionInboxRead()`
- 进入服务号线程就批量 `markServiceMessagesRead()`

桌面版目标策略：
- 订阅号按 `deliveryId` 粒度已读
- 服务号按“打开线程后读取至最新可见消息”已读
- 聚合入口未读数来自未读 item 汇总，不是“只要打开过页面就清零”

## 前端结构规划

### 1. 引入统一桌面消息条目适配层

新增一个纯前端适配层，把这些源数据映射成同一列表模型：

- 普通会话 `Conversation`
- 订阅号聚合入口 `subscription_inbox`
- 服务号入口 `service_account`

目标结果：
- 搜索逻辑统一
- 排序逻辑统一
- 置顶逻辑统一
- 右键菜单统一
- 渲染组件只关心 `DesktopMessageEntry`

建议新增位置：
- `apps/app/src/features/desktop/chat/desktop-message-entry-types.ts`
- `apps/app/src/features/desktop/chat/use-desktop-message-entries.ts`

### 2. 重写桌面订阅号工作区

保留现有路由，不保留现有分组 UI。

建议重构 `DesktopSubscriptionWorkspace`：
- 左侧不再是“账号块”
- 改成订阅消息流列表
- 顶部提供搜索框、未读筛选、刷新按钮
- 右侧继续复用 `OfficialArticleViewer`

阅读逻辑：
- 点击消息流项时调用 `POST /api/official-accounts/deliveries/:deliveryId/read`
- 列表项进入激活态后同步右栏
- 右栏文章滚动位置按 `articleId` 本地记忆

### 3. 重写桌面服务号线程

`OfficialAccountServiceThread` 需要从“单栏气泡页”变成“线程 + 右栏”的中心区域组件。

建议拆成：
- `DesktopOfficialServiceWorkspace`
- `DesktopOfficialServiceHeader`
- `DesktopOfficialServiceThread`
- `DesktopOfficialServiceSidePanel`

行为目标：
- 顶部能直接打开账号主页
- 顶部能切换免打扰
- 顶部能进入更多菜单
- 文章卡片只更新右栏
- 右栏可在“文章阅读”和“账号详情/服务菜单”之间切换

### 4. 公众号目录页回归“目录角色”

现有 `DesktopOfficialAccountsWorkspace` 不应该再承担桌面公众号的全部职责。

它的目标应该收口为：
- 查找公众号
- 查看账号主页
- 预览最近文章
- 一键进入服务号消息

建议修改：
- 去掉统计卡式大面板
- 列表密度改为微信 PC 风格
- 中栏首屏按钮改成 `发消息` / `关注` / `更多`
- 服务号账号默认入口优先打开消息线程，而不是只看资料

## 详细交互规则

### 消息列表排序

统一规则建议：

1. 置顶项优先
2. 未隐藏项按 `lastActivityAt` 倒序
3. 订阅号聚合入口参与同一排序，不固定钉死在会话顶部
4. 服务号沿用同一排序，但默认免打扰

这点很关键，因为当前“先渲染公众号消息，再渲染普通会话”的做法会让桌面感非常假。

### 消息列表视觉

建议：
- 订阅号消息使用固定图标和更像系统会话的单行样式
- 服务号消息行和普通会话行尽量共形，只增加认证/免打扰提示
- 免打扰状态下未读用小红点，不抢大红角标

### 账号主页标签

桌面账号主页建议按类型表现：

- 订阅号
  - `消息`
  - `服务` 仅在 `menuPayload` 存在时显示
- 服务号
  - `消息`
  - `服务`

`视频` 暂时只留布局槽位，不在本轮实现，否则会把计划扩散到视频消息建模。

### 深链接恢复

桌面端所有公众号相关路由都应变成“恢复工作区状态”而不是“真正切页”：

- `/official-accounts/:accountId`
  - 恢复目录工作区，聚焦该号
- `/official-accounts/articles/:articleId`
  - 优先恢复最近相关工作区
  - 恢复失败时回到目录工作区并打开文章
- `/official-accounts/service/:accountId`
  - 恢复消息工作区并激活该服务号线程

## 分阶段实施

### Phase 1：统一消息条目层

目标：
- 不改大 UI 先改消息模型

产出：
- 新增 `OfficialAccountEntryState`
- 新增统一桌面消息条目适配器
- 左侧消息列表实现公众号消息与普通会话的统一排序、统一搜索

### Phase 2：订阅号聚合流重做

目标：
- 把桌面订阅号从“账号分组卡片”改成真正信息流

产出：
- `feedItems`
- `deliveryId` 粒度已读
- 搜索 / 未读筛选
- 右栏文章阅读联动

### Phase 3：服务号线程桌面化

目标：
- 把服务号变成低打扰桌面会话，而不是简单静态页

产出：
- 顶部动作区
- 免打扰切换
- 右栏阅读器 / 详情面板
- 线程未读策略收敛

### Phase 4：目录页微信化

目标：
- 把通讯录里的公众号目录收口为“查找与资料入口”

产出：
- 密集列表
- 账号主页标签
- 服务号快捷发消息
- 服务菜单接入

### Phase 5：桌面深链接与状态恢复

目标：
- 所有公众号路由在桌面端都像微信 PC 一样只是切换工作区状态

产出：
- 路由恢复逻辑
- 右栏阅读器状态同步
- 跨入口一致性收敛

## 推荐提交顺序

为了符合仓库“小步提交”的规则，建议后续按下面顺序实现：

1. `contracts + api`：补 entry state、mute 契约、订阅流 feedItems
2. `desktop chat`：统一消息条目层与左侧排序
3. `desktop subscription`：重写订阅号工作区
4. `desktop service`：重写服务号线程与右栏联动
5. `desktop directory`：公众号目录与主页微信化
6. `router`：深链接恢复与桌面状态同步

## 验收标准

- 桌面消息列表里的订阅号和服务号不再是“插入块”，而是统一消息条目。
- 服务号默认免打扰，并且能在桌面端显式切换。
- 订阅号消息不再按账号分组展示，而是按推送信息流展示。
- 进入订阅号/服务号不会粗暴清空全部未读，而是按条目或可见范围已读。
- 从目录、订阅号、服务号三处打开文章，都进入同一套右栏阅读器。
- 桌面深链接进入公众号详情、文章、服务号线程时，恢复的是工作区状态，不是孤立页面。

## 一句话结论

当前仓库已经有公众号功能的 60% 骨架，但产品结构还停留在“资料页 + 演示页”阶段。要真正抄到微信电脑版，重点不是继续加卡片和按钮，而是把公众号重新组织成“统一消息条目层 + 订阅号聚合流 + 服务号低打扰会话 + 共享右栏阅读器”这四件事。
