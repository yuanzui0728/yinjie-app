# Web 端电脑版 AI 语音通话规划

## 1. 目标
- 在 `apps/app` 的 Web 桌面端落地“语音通话”体验，用 AI 角色与用户进行语音往返对话。
- 本次实现口径不是 WebRTC 真双向实时音频，而是“按回合录音”：
  - 用户按住或点击说话，录一段音频
  - 后端调用 STT 转文字
  - 复用现有聊天回复链路生成角色文本回复
  - 后端调用 TTS 合成角色语音
  - 前端自动播放角色语音，并在聊天流里落消息
- 先只做 Web 桌面端单聊 AI 角色，不做群语音，不做视频，不做真人对真人通话。
- 不新增表，不生成测试文件，尽量复用现有聊天、语音和桌面工作台结构。

## 2. 现状结论

### 2.1 后端主链路其实已经基本具备
- `api/src/modules/ai/ai.controller.ts`
  - 已有 `POST /api/ai/transcriptions`
  - 已有 `POST /api/ai/speech`
- `api/src/modules/ai/ai-orchestrator.service.ts`
  - 已有 `transcribeAudio()`
  - 已有 `synthesizeSpeech()`
  - 已接 OpenAI 兼容语音转写 / 语音合成能力
- `api/src/modules/chat/voice-calls.service.ts`
  - 已有 `createTurn()`
  - 流程已经是：
    - 转写用户音频
    - 调 `chatService.sendMessage()` 走现有单聊 AI 回复
    - 对 AI 回复做 TTS
    - 返回 `assistantAudioUrl`
- `api/src/modules/chat/chat.controller.ts`
  - 已有 `POST /api/chat/voice-calls/turns`
- 结论：
  - 后端不需要从零设计接口，重点是补稳定性和前端接线。

### 2.2 前端桌面“通话”还是占位工作台
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 桌面端点“语音通话”后，只会切到 `DesktopDirectCallPanel`
- `apps/app/src/features/desktop/chat/desktop-direct-call-panel.tsx`
  - 目前是状态模拟 UI，只有静音、扬声器、通知对方、转手机等占位交互
  - 没有麦克风录音、没有 `createVoiceCallTurn()` 请求、没有 AI 音频播放
- `apps/app/src/features/chat/group-chat-thread-panel.tsx`
  - 群聊通话同样是占位，不适合作为本次首版目标

### 2.3 前端已经有可复用的录音基础，但它服务的是“语音输入”而不是“语音通话”
- `apps/app/src/features/chat/use-speech-input.ts`
  - 已封装浏览器录音和转写
  - 当前主要给输入框语音输入 / 语音消息使用
- `apps/app/src/components/chat-composer.tsx`
  - Web 桌面端已支持“语音转文字输入”
- `packages/contracts/src/client.ts`
  - 已有 `createVoiceCallTurn(payload)`
- 结论：
  - 录音、FormData 上传、浏览器权限处理可以复用
  - 但需要为“通话态”单独做 session 状态机，不能把输入框语音能力直接硬塞到通话面板里

### 2.4 当前产品定义必须收口
- 本次应该定义为：
  - `桌面 AI 语音通话工作台`
  - `单聊`
  - `一问一答回合制`
  - `聊天消息与语音播报同步`
- 本次不做：
  - 真正持续占线的 RTC 音频流
  - 后台持续监听
  - 双人并发抢说、打断、回声消除、网络抖动补偿
  - 群聊语音
  - 视频通话

## 3. 推荐产品口径

### 3.1 用户体验
- 用户在桌面单聊顶部点击“语音通话”
- 聊天区域切入“AI 语音通话面板”
- 用户点击“按住说话”或“开始说话”
- 录音结束后显示：
  - “正在转写”
  - “角色正在思考”
  - “正在生成语音”
- AI 回复回来后：
  - 聊天流写入用户文本消息与角色文本消息
  - 通话面板自动播放角色语音
  - 面板显示本轮文字摘要、时长、播放状态
- 用户可以继续下一轮，形成持续的“通话感”

### 3.2 首版必须牺牲的点
- 不追求全双工，只做半双工
- AI 播放时默认禁止继续录音，避免状态冲突
- 先不做“打断 AI 说话”
- 先不做“切换音色 / 语速 / 降噪参数”
- 先不做“通话记录单独落库”，仍以普通聊天消息为主记录源

## 4. 推荐改造方案

### 4.1 把桌面通话面板从“占位卡片”升级成真正的 AI 语音工作台
- 目标文件：
  - `apps/app/src/features/desktop/chat/desktop-direct-call-panel.tsx`
  - 可选拆分新文件：
    - `apps/app/src/features/desktop/chat/desktop-ai-voice-call-panel.tsx`
    - `apps/app/src/features/desktop/chat/use-desktop-ai-voice-call.ts`
- 建议不要继续在现有占位状态上硬补逻辑。
- 更稳的做法：
  - 保留当前视觉框架
  - 把内部逻辑拆成“AI 语音通话 session”
- session 最少要有这些状态：
  - `idle`
  - `requesting_permission`
  - `recording`
  - `transcribing`
  - `thinking`
  - `synthesizing`
  - `playing`
  - `error`

### 4.2 前端直接调用现有 `/chat/voice-calls/turns`
- 目标文件：
  - `packages/contracts/src/client.ts`
  - `apps/app/src/features/desktop/chat/use-desktop-ai-voice-call.ts`
- 请求体建议继续使用现有 `FormData`：
  - `file`
  - `conversationId`
  - `characterId` 可选
- 响应直接消费 `VoiceCallTurnResult`
- 前端不要自己再拆 STT / 发消息 / TTS 三次请求
- 原因：
  - 后端已经把这一轮语音对话编排好了
  - 保持一次请求一个回合，状态更简单

### 4.3 录音能力复用 `MediaRecorder`，但不要复用输入框的交互层
- 可复用：
  - `use-speech-input.ts` 里的 MIME 选择逻辑
  - 麦克风权限错误映射
  - 录音时长统计
- 不建议直接复用：
  - 输入框 dictation 的“插入文本”流程
  - 移动端“按住说话”弹层
- 原因：
  - 语音通话需要的是“录音一轮 -> 发整轮 -> 自动播 AI”
  - 输入框语音需要的是“录音 -> 转文字 -> 编辑再发送”

### 4.4 通话面板要新增一个明确的“单轮结果区”
- 目标文件：
  - `apps/app/src/features/desktop/chat/desktop-direct-call-panel.tsx`
- 建议新增三块信息：
  - 当前状态区
    - 麦克风权限
    - 录音中 / 处理中 / 播放中
  - 本轮摘要区
    - 用户说了什么
    - 角色回复了什么
  - 语音播报区
    - 播放 / 暂停
    - 重播
    - 本轮总耗时
- 这样用户即使没听清，也能直接看文字，不会把功能做成黑盒

### 4.5 聊天线程页与通话工作台要共享同一消息事实源
- 目标文件：
  - `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - `apps/app/src/features/chat/use-conversation-thread.ts`
  - `apps/app/src/features/desktop/chat/use-desktop-ai-voice-call.ts`
- 本轮语音通话成功后，不需要前端手工插消息到列表
- 因为 `/chat/voice-calls/turns` 内部已经调用 `chatService.sendMessage()`
- 前端只要：
  - 请求成功后记住 `userMessageId` / `assistantMessageId`
  - 触发消息列表刷新，或依赖 socket 已有更新
  - 将视图滚到最新消息
- 这样可以保证：
  - 打字聊天与语音通话共用一条消息历史
  - 退出通话后仍能看到完整上下文

### 4.6 自动播放要做成“可控播放”，不能只是一把 `audio.play()`
- 目标文件：
  - `apps/app/src/features/desktop/chat/use-desktop-ai-voice-call.ts`
  - 可复用 `apps/app/src/components/chat-message-list.tsx` 里的音频播放处理方式
- 要处理的状态：
  - 自动播放失败
  - 用户手动暂停
  - 本轮播放结束
  - 下一轮开始时停止上一轮音频
- 建议维护一个 `HTMLAudioElement` ref
- 必须暴露：
  - `playLatestReply()`
  - `pauseReply()`
  - `replayReply()`
  - `stopReply()`

### 4.7 错误模型要按阶段拆开，不要只给一个“通话失败”
- 建议分为：
  - `permission_denied`
  - `recording_failed`
  - `upload_failed`
  - `transcription_failed`
  - `reply_failed`
  - `synthesis_failed`
  - `playback_failed`
- 对用户展示时写成明确中文：
  - “麦克风权限被拒绝”
  - “语音上传失败，请检查网络”
  - “本轮语音未识别出有效文字”
  - “角色回复生成失败”
  - “语音已生成，但浏览器未能自动播放”

## 5. 建议实施顺序

### Phase 1：打通单聊桌面 AI 语音通话主链路
- 新增桌面通话 session hook
- 在桌面通话面板接入录音、上传、自动播放
- 调用现有 `/chat/voice-calls/turns`
- 成功后自动刷新聊天消息
- 验收结果：
  - 单聊里能完成一整轮“说话 -> AI 回复并播报”

### Phase 2：把当前占位操作收口为真实可用操作
- 保留：
  - 静音麦克风
  - 播放器开关 / 暂停
  - 返回聊天
  - 结束通话
- 移除或降级：
  - “通知对方”
  - “手机接力”
  - “等待对方加入”
- 原因：
  - 这些语义属于真人通话，不属于 AI 语音通话
- 验收结果：
  - UI 文案不再误导用户以为是 RTC 真人呼叫

### Phase 3：补细节体验
- 新增通话中的轮次记录
- 播放中展示波形 / 动效
- 支持快捷键：
  - 空格按住说话，松开发送
  - `Esc` 取消本轮
- 支持“重播上一条 AI 语音”
- 验收结果：
  - 桌面端具备连续对话效率

### Phase 4：再评估扩展面
- 角色音色配置透传
- 通话中断 AI 播报
- 群聊 AI 语音轮询式发言
- 视频通话入口重命名或并行规划

## 6. UI 文案和交互口径建议
- 顶部标题：
  - `AI 语音通话`
- 副标题：
  - `当前为回合制语音对话，AI 会在每轮结束后语音回复`
- 主按钮：
  - `按住说话`
  - 或桌面语义更强的 `开始说话`
- 处理中提示：
  - `正在转写`
  - `正在生成回复`
  - `正在合成语音`

  - `结束语音通话`
- 不建议继续使用：
  - `等待接听`
  - `通知对方`
  - `双方已接通`

## 8. 验收标准
- Web 桌面端单聊点击“语音通话”后，进入 AI 语音工作台，而不是跳手机接力语义。
- 用户可完成一轮录音并成功发送到 `/api/chat/voice-calls/turns`。
- 成功后聊天列表里能看到：
  - 用户文本消息
  - AI 文本回复消息
- AI 回复语音会自动播放，且可以手动重播 / 暂停。
- 切回聊天页或重新进入会话，历史消息仍完整可见。
- 录音、转写、合成、播放任一失败时，界面能给出明确错误提示。
- 群聊和视频通话入口首版不误导，不冒充已实现能力。

## 9. 风险与注意事项
- 当前工作区已有与语音能力相关的未提交改动：
  - `api/src/modules/ai/*`
  - `api/src/modules/chat/chat.controller.ts`
  - `api/src/modules/chat/chat.module.ts`
  - `api/src/modules/chat/voice-calls.service.ts`
  - `apps/app/src/features/chat/group-chat-thread-panel.tsx`
  - `apps/app/src/features/desktop/chat/desktop-direct-call-panel.tsx`
  - `apps/app/src/features/desktop/chat/desktop-group-call-panel.tsx`
- 开发时必须先读懂这些改动，不能回退。
- 浏览器自动播放可能被策略拦截，因此“自动播失败后可手动播”必须作为兜底。
- 如果 provider 没配好 TTS 或 STT key，本功能表面上会像“通话失败”，所以桌面面板最好把失败阶段说清楚。
- 当前 `voice-calls.service.ts` 只支持 `direct` 单聊，本次首版不要把群聊 UI 一起放开。
