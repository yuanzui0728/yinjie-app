# Web 端手机版 AI 数字人视频通话功能规划

## 1. 目标
- 只覆盖 `apps/app` 的 Web 手机版单聊场景。
- 视频通话对象是 `AI 数字人`，不是真人 RTC。
- 要把现有“语音回合 + 数字人 UI 包装”升级成“真正接数字人 provider 的手机版视频通话”。
- 首期仍保持半双工，不做真人对真人双向实时音视频。

## 2. 现状结论

### 2.1 前端页面已经存在，但目前不是真数字人链路
- `apps/app/src/routes/chat-video-call-page.tsx`
  - 已有 `/chat/$conversationId/video-call` 路由。
- `apps/app/src/features/chat/mobile-ai-call-screen.tsx`
  - `mode="video"` 已经有完整移动端视频页壳：
    - 远端数字人主舞台
    - 本地摄像头预览小窗
    - 按住说话
    - 最近一轮文案
    - 播报中 / 思考中 / 聆听中状态
  - 但远端舞台当前只是头像/图片 + 动效，不是真实数字人视频输出。
- `apps/app/src/features/chat/use-self-camera-preview.ts`
  - 已有本地前置摄像头预览能力。
  - 当前只服务本地小窗，不参与 AI 回复链路。

### 2.2 当前后端只有语音回合接口，没有数字人视频接口
- `api/src/modules/chat/chat.controller.ts`
  - 当前只有 `POST /api/chat/voice-calls/turns`。
- `api/src/modules/chat/voice-calls.service.ts`
  - 当前流程是：
    - 上传录音
    - STT 转写
    - 写入聊天消息
    - 生成 AI 回复
    - TTS 合成
    - 返回 `assistantAudioUrl`
- `packages/contracts/src/speech.ts`
  - 当前 `VoiceCallTurnResult` 只有语音回合结果，没有数字人 session、视频流、播放器状态等字段。

### 2.3 结论
- 现在的手机版 `video-call` 本质仍是：
  - `语音通话接口`
  - `数字人风格 UI`
- 如果需求是“接的是 AI 数字人”，核心工作不在前端排版，而在：
  - 新增数字人 provider 抽象
  - 新增数字人 session / turn API
  - 新增前端播放器与状态同步协议

## 3. 产品口径

### 3.1 对用户的真实表述
- 页面标题继续叫 `视频通话`。
- 副文案改成：
  - `当前为半双工 AI 数字人视频通话，你说一轮，TA 回一轮。`
- 页面内必须明确：
  - 远端是 `AI 数字人`
  - 本地小窗是 `我的摄像头预览`
  - 本地摄像头默认只用于用户自我预览，不作为数字人回复输入

### 3.2 首版不要宣称的能力
- 不宣称真人实时接听。
- 不宣称 WebRTC 双向视频。
- 不宣称 AI 能实时看见你的摄像头内容。
- 不做“对方正在输入”“对方已接通”这类真人 RTC 语义。

## 4. 推荐技术方案

### 4.1 分两层推进
- `MVP`：
  - 保留现有移动端视频页。
  - 页面和交互继续可用。
  - 新增数字人协议骨架，但允许远端先以静态舞台/短视频片段承载。
- `正式版`：
  - 真正接通 AI 数字人 provider。
  - 远端舞台消费 provider 返回的真实数字人视频流、播放器地址或渲染任务结果。

### 4.2 不建议做 WebRTC
- 当前远端不是人类终端，而是 AI 数字人服务。
- 更合理的模型是：
  - 前端负责本地录音和本地摄像头预览
  - 后端负责一轮编排
  - 数字人 provider 负责渲染和播放资源
- 这类场景优先使用：
  - `playerUrl`
  - `streamUrl`
  - `HLS`
  - `mp4/segment`
  - `iframe/embed`
- 不优先引入双向 RTC 基础设施。

## 5. 后端设计

### 5.1 新增模块建议
- 新增 `api/src/modules/chat/digital-human-calls/`
- 建议文件：
  - `digital-human-calls.controller.ts`
  - `digital-human-calls.service.ts`
  - `digital-human-provider.ts`
  - `digital-human.types.ts`

### 5.2 provider 抽象职责
- 创建数字人会话
- 提交一轮用户语音
- 驱动数字人播报 AI 回复
- 查询当前渲染/播放状态
- 关闭会话并回收资源

### 5.3 推荐 API
- `POST /api/chat/digital-human-calls/sessions`
- `GET /api/chat/digital-human-calls/sessions/:sessionId`
- `DELETE /api/chat/digital-human-calls/sessions/:sessionId`
- `POST /api/chat/digital-human-calls/sessions/:sessionId/turns`

### 5.4 推荐 session 入参
- `conversationId`
- `characterId`
- `surface: "mobile_video_call"`
- `previewMode: "camera_preview_only"`
- `providerConfig?`

### 5.5 推荐 session 返回
- `sessionId`
- `conversationId`
- `characterId`
- `characterName`
- `transport`
  - 取值建议：`player_url | hls | mp4 | iframe`
- `playerUrl?`
- `streamUrl?`
- `posterUrl?`
- `status`
  - 取值建议：`connecting | ready | busy | closed | error`
- `capabilities`
  - `supportsRealtimeStream`
  - `supportsInterrupt`
  - `supportsSubtitle`

### 5.6 推荐 turn 接口职责
- 一次请求承接一整轮：
  - 接收用户录音
  - STT 转写
  - 生成 AI 回复
  - TTS 或 provider 内部语音合成
  - 提交给数字人 provider 渲染
  - 返回本轮结果和播放信息

### 5.7 推荐 turn 返回
- `sessionId`
- `conversationId`
- `characterId`
- `userTranscript`
- `assistantText`
- `assistantAudioUrl?`
- `renderStatus`
  - 取值建议：`queued | rendering | ready | failed`
- `playerUrl?`
- `streamUrl?`
- `subtitle?`
- `provider`
- `transcriptionDurationMs`
- `generationDurationMs`
- `renderDurationMs`
- `totalDurationMs`
- `userMessageId`
- `assistantMessageId`

### 5.8 编排建议
- 不建议让前端自己串：
  - `/ai/transcriptions`
  - `/chat/send`
  - `/ai/speech`
  - `/digital-human/render`
- 应由后端统一编排成一个 `turn` 接口。
- 理由：
  - 前端状态更简单
  - 失败恢复集中
  - 更容易做 provider 切换和降级

## 6. 契约层设计

### 6.1 `packages/contracts` 新增内容
- 在 `packages/contracts/src/speech.ts` 增加：
  - `DigitalHumanCallSession`
  - `DigitalHumanCallTurnResult`
  - `DigitalHumanRenderStatus`
- 在 `packages/contracts/src/client.ts` 增加：
  - `createDigitalHumanCallSession()`
  - `getDigitalHumanCallSession()`
  - `closeDigitalHumanCallSession()`
  - `createDigitalHumanCallTurn()`

### 6.2 与现有语音通话的关系
- `createVoiceCallTurn()` 继续保留，给纯语音通话使用。
- `video-call` 页面不再直接依赖 `VoiceCallTurnResult`。
- 避免把数字人视频能力硬塞进现有语音契约，造成后续字段混乱。

## 7. 前端设计

### 7.1 页面保留现有壳，替换底层数据源
- 继续复用：
  - `apps/app/src/routes/chat-video-call-page.tsx`
  - `apps/app/src/features/chat/mobile-ai-call-screen.tsx`
  - `apps/app/src/features/chat/use-self-camera-preview.ts`
- 新增：
  - `apps/app/src/features/chat/use-digital-human-call-session.ts`
  - `apps/app/src/features/chat/digital-human-player.tsx`

### 7.2 页面状态机建议
- `idle`
- `creating_session`
- `session_ready`
- `requesting_mic_permission`
- `recording`
- `uploading_turn`
- `thinking`
- `rendering_digital_human`
- `playing`
- `error`
- `closed`

### 7.3 页面行为
- 进入页面时：
  - 拉取单聊信息
  - 创建数字人 session
  - 创建失败则给出明确错误，不要静默退回语音页
- 按住说话时：
  - 继续使用现有录音体验
  - 松开后调用数字人 turn 接口
- 播放阶段：
  - 若 provider 返回 `playerUrl/streamUrl`，由 `DigitalHumanPlayer` 承接播放
  - 若 provider 只返回音频，页面降级到当前“数字人舞台 + TTS 播报”模式

### 7.4 本地摄像头策略
- 首版只做本地预览。
- 不上传摄像头视频给后端。
- 权限拒绝时：
  - 不阻断数字人视频通话主流程
  - 只关闭本地小窗并显示提示

### 7.5 播放器组件职责
- 统一承接不同 provider 传输形式：
  - `video src`
  - `hls`
  - `iframe`
  - `poster + audio` 降级
- 统一对外暴露：
  - `playing`
  - `buffering`
  - `ended`
  - `failed`

## 8. 分阶段实施

### Phase 1：收口现有假视频页面
- 目标：
  - 明确把现有页面定义成“数字人视频通话壳”。
  - 从实现上拆出 `use-digital-human-call-session` 和 `DigitalHumanPlayer` 的接口位。
- 产出：
  - 前端结构不再把 `video-call` 和 `voice-call` 完全绑死在一起。

### Phase 2：新增后端数字人 session/turn 协议
- 目标：
  - 打通数字人会话、单轮请求、状态查询、关闭会话。
- 产出：
  - `chat/digital-human-calls/*` API
  - contracts client 封装

### Phase 3：接入真实数字人 provider
- 目标：
  - 让后端 turn 编排结果真正驱动数字人渲染。
- 产出：
  - provider adapter
  - session 生命周期管理
  - 渲染/播放状态同步

### Phase 4：前端切换到真实数字人播放
- 目标：
  - `mobile-ai-call-screen` 使用数字人 session/turn 接口
  - `DigitalHumanPlayer` 承接远端视频播放
- 产出：
  - 手机版视频通话真正接入 AI 数字人

### Phase 5：细节收口
- 目标：
  - 处理中断、挂断、刷新页面恢复、自动播放限制、provider 超时
- 产出：
  - 更稳定的移动端体验

## 9. 风险与取舍

### 9.1 最大风险
- 当前仓库完全没有数字人 provider 接入代码。
- 所以“接的是 AI 数字人”不是一个页面层任务，关键阻塞在后端新模块和第三方能力。

### 9.2 自动播放限制
- Web 手机端对音视频自动播放限制更严格。
- 必须准备：
  - 首次手势触发播放
  - 播放失败后的重试按钮
  - 清晰错误文案

### 9.3 网络和渲染时延
- 数字人渲染耗时通常高于纯 TTS。
- 产品上要接受：
  - `AI 思考中`
  - `数字人渲染中`
  - `视频准备完成`
- 不能只保留一个“正在回复”状态。

### 9.4 降级策略
- provider 不可用时：
  - 页面允许降级到当前的 `数字人舞台 + 语音播报`
- 但要明确标注：
  - `当前已降级为语音驱动舞台，不是实时数字人视频流`

## 10. 验收标准
- 手机版单聊进入 `/chat/$conversationId/video-call` 后，会先创建数字人 session。
- 用户完成一轮说话后，前端消费的是数字人 turn 接口，而不是旧的语音接口。
- 远端主舞台能播放真实数字人视频结果，或清晰降级到语音驱动舞台。
- 本地摄像头权限拒绝不阻断主链路。
- 挂断后会关闭 session，刷新进入不会留下悬挂会话。
- 页面内所有文案都符合“AI 数字人半双工视频通话”口径，不混入真人 RTC 语义。

## 11. 推荐实施顺序
1. 先拆前端 `video-call` 页面数据层，给数字人 session/turn 留接口位。
2. 再补 `packages/contracts` 与 `api` 的数字人会话协议。
3. 接着实现后端 provider adapter 和 turn 编排。
4. 最后把手机版页面切到真实数字人接口，并补齐降级与恢复逻辑。
