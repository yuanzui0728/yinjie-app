# 世界角色名人预设与后台运营规划

日期：2026-04-12

## 目标

在现有“世界角色”体系里增加一批公众人物预设，首批可先覆盖：

- 马斯克
- 特朗普
- 乔布斯
- 巴菲特
- 任正非
- 稻盛和夫

本次规划要同时满足 4 个结果：

1. 这些人物以“世界角色”的方式进入当前世界，而不是单独做一套平行系统。
2. 后台可以直接看到这些角色，并继续使用现有角色编辑页、角色工厂、角色运行逻辑台、回复逻辑页去修改行为逻辑。
3. 用户端把这些角色当成普通世界角色处理，可以发起好友申请，也可以删除好友。
4. 后台可以删除这些世界角色，但不能误删系统默认保底角色。

## 现状判断

### 已经具备的能力

- 角色 CRUD 已有：
  - `api/src/modules/characters/characters.controller.ts`
  - `api/src/modules/admin/admin.controller.ts`
- 后台已具备角色行为逻辑查看与修改入口：
  - 角色基础资料：`apps/admin/src/routes/character-editor-page.tsx`
  - 角色运行逻辑台：`apps/admin/src/routes/character-runtime-page.tsx`
  - 世界级回复逻辑页：`apps/admin/src/routes/reply-logic-page.tsx`
- 用户端已具备好友申请与删除好友能力：
  - 发起好友申请：`POST /api/social/friend-requests/send`
  - 删除好友：`DELETE /api/social/friends/:characterId`
  - 前端入口已经存在于 `character-detail-page.tsx`、`contacts-page.tsx`
- 通讯录页已经能展示“非好友的世界角色”，用户天然可以从这里把角色加进通讯录。

### 当前缺口

1. 当前真正稳定接入的内置角色只有 `default-characters.ts` 里的 2 个默认角色。
2. `characters.seed.ts` 里虽然有模板角色，但它不是当前单世界内置角色的统一入口，和 `api/src/database/seed.ts` 是两套路子。
3. 现在没有“名人预设库 -> 安装到当前世界 -> 作为普通世界角色运营”的链路。
4. 现在后台虽然有删除按钮，但删除 `Character` 时没有一套安全的角色下线语义，容易留下：
   - friendship / friend_request 脏数据
   - narrative arc 残留
   - ai relationship 残留
   - conversation / message / group member / moments / feed 的引用悬空
5. 当前只靠 `isTemplate` 不够，区分不了：
   - 系统默认保底角色
   - 公众人物预设
   - 后台人工新建角色

## 核心方案

### 方案原则

- 不把名人预设直接塞进 `DEFAULT_CHARACTER_IDS`。
- 不单独做一套“名人聊天系统”，而是复用现有世界角色体系。
- 后台行为逻辑不新开页面，直接复用现有角色编辑、工厂、运行逻辑、回复逻辑。
- “后台删除角色”建议做成“归档下线”语义，而不是裸物理删除。

### 角色分层

建议给 `CharacterEntity` 增加一组来源与生命周期字段，至少包含：

- `sourceType`
  - `default_seed`
  - `preset_catalog`
  - `manual_admin`
- `sourceKey`
  - 例如 `elon_musk`、`donald_trump`
- `lifecycleStatus`
  - `active`
  - `archived`
- `deletionPolicy`
  - `protected`
  - `archive_allowed`

这样可以明确区分：

- `我自己`、`创业者朋友` 这类保底角色：`default_seed + protected`
- 马斯克、特朗普这类预设：`preset_catalog + archive_allowed`
- 后台手工创建角色：`manual_admin + archive_allowed`

## 名人预设接入方式

### 推荐做法

新增一个代码级预设目录，例如：

- `api/src/modules/characters/celebrity-character-presets.ts`

这个文件只负责保存“预设定义”，不直接把它们永久写死到世界实例里。每个预设包含：

- `presetKey`
- `name`
- `avatar`
- `relationship`
- `relationshipType`
- `bio`
- `expertDomains`
- `profile.basePrompt`
- `traits`
- `identity`
- `behavioralPatterns`
- `cognitiveBoundaries`
- 默认活跃时间、动态频率、触发场景

### 为什么不用“启动时自动种进去”

如果把这些名人角色和默认角色一样做成强制 reconcile：

- 后台删掉后，下次 seed 又会回来
- 无法满足“后台删除这些世界角色”的真实诉求

所以更合理的是：

- 默认保底角色继续走 `default-characters.ts`
- 名人预设存在于“预设库”
- 管理员从后台把某个预设安装到当前世界
- 安装后才生成对应 `CharacterEntity`

## 后台侧方案

### 1. 角色中心增加“从预设添加”

建议直接在 `apps/admin/src/routes/characters-page.tsx` 增加预设安装入口，而不是另开新页面。

推荐增加：

- “从名人预设添加”按钮
- 预设选择抽屉/弹层
- 已安装/未安装状态
- 预设来源标签：`默认角色 / 名人预设 / 自定义`
- 删除策略标签：`受保护 / 可下线`

对应后台接口建议增加：

- `GET /api/admin/characters/presets`
- `POST /api/admin/characters/presets/:presetKey/install`

安装动作做的事情：

1. 从预设定义生成 `CharacterEntity`
2. 写入 `sourceType=preset_catalog`
3. 写入 `sourceKey`
4. 写入 `lifecycleStatus=active`
5. 为该角色初始化 blueprint，保证后台工厂与运行逻辑页立即可用

### 2. 行为逻辑直接复用现有后台

安装后的名人角色不需要新逻辑页，直接进入现有工作流：

- 基础资料与 prompt：`/characters/$characterId`
- 配方与发布映射：`/characters/$characterId/factory`
- 运行态与生活逻辑：`/characters/$characterId/runtime`
- 世界级回复链路查看：`/reply-logic`

也就是说，这部分需求现在架子已经有了，本次主要是把“名人预设角色”接进这套架子里。

### 3. 后台删除语义

建议把当前“删除角色”改成：

- 默认保底角色：禁止删除，只允许编辑
- 名人预设角色 / 自定义角色：允许归档下线

后台按钮文案也建议改掉：

- 受保护角色：`不可删除`
- 可下线角色：`下线角色`

这样比“物理删除”稳定，因为历史消息、内容流和关系记录不至于直接悬空。

## 用户端方案

### 1. 好友申请

用户侧不需要单独做一套名人预设交互，只要这些角色成为 active 的世界角色，就会自动接入现有链路：

- 通讯录“世界角色”区展示
- 角色详情页展示
- `添加到通讯录`
- `好友申请`

这部分主要依赖现有：

- `apps/app/src/routes/contacts-page.tsx`
- `apps/app/src/routes/character-detail-page.tsx`
- `POST /api/social/friend-requests/send`

### 2. 删除好友

安装后的名人角色一旦成为好友，删除好友仍然走现有链路：

- `DELETE /api/social/friends/:characterId`

这部分已经有，不需要为名人预设单开。

### 3. 归档角色的前台表现

若后台把某个角色“下线/归档”，前台需要统一处理：

- 不再出现在通讯录“世界角色”可添加列表
- 不再参与摇一摇、场景触发、朋友圈生成、视频号生成、主动消息等调度
- 已经删除好友的自然无影响
- 如果历史上有聊天记录，页面可显示“该角色已下线”而不是直接炸 404

## 删除与下线的技术语义

### 推荐采用“归档下线”而不是物理删除

原因很直接：

- `Conversation.participants`
- `Message.senderId`
- `GroupMember.memberId`
- `GroupMessage.senderId`
- `MomentPost.authorId`
- `FeedPost.authorId`
- `Friendship.characterId`
- `FriendRequest.characterId`
- `NarrativeArc.characterId`
- `AIRelationship.characterIdA / characterIdB`
- `CharacterBlueprint.characterId`

这些地方都可能引用角色。如果直接物理删角色，清理成本高，而且很容易留下历史数据断链。

### 归档时建议同步处理

1. 角色本身标记为 `archived`
2. 好友申请：
   - 关闭 pending request
3. 好友关系：
   - 非默认角色统一标记 `removed`
4. AI 关系网：
   - 删除或忽略该角色相关关系
5. 叙事弧：
   - 标记为 `completed` 或 `paused`
6. 调度器：
   - 所有角色选择逻辑过滤 `archived`
7. 前台列表：
   - `GET /api/characters`
   - 通讯录世界角色列表
   - 摇一摇/场景相遇候选
   都过滤 `archived`

## 公众人物预设的内容口径

建议这些人物预设不要做成“实时新闻播报器”，而要做成“角色化公众人物映射”：

- 重点刻画说话风格
- 决策方式
- 价值偏好
- 行为边界
- 兴趣领域

同时在 `cognitiveBoundaries` 里明确：

- 不承诺实时事实一定最新
- 对超出设定边界的问题要克制回答
- 不伪装成真实在线真人

这样可以降低：

- 最新事实漂移
- 角色失真
- 政治人物口径不稳

## 验收口径

完成后应满足：

1. 后台可以从预设库一键把“马斯克”“特朗普”等角色安装到当前世界。
2. 安装后这些角色会出现在后台角色中心，且可进入：
   - 角色编辑页
   - 角色工厂
   - 角色运行逻辑台
   - 回复逻辑页
3. 用户可以在通讯录的世界角色区看到这些角色，并能发起好友申请。
4. 成为好友后，用户可以像普通世界角色一样删除好友。
5. 后台可以下线这些角色，而不会误删 `我自己` / `创业者朋友` 这类默认保底角色。
6. 角色下线后不会继续参与调度，也不会在前台继续以可添加角色身份出现。

## 我建议的落地顺序

按性价比排序，建议这样做：

1. 先做“代码级预设库 + 后台安装到世界”
2. 再做后台来源标签与快速入口
3. 再补前台世界角色与好友链路的状态收口
4. 最后把删除改成安全归档

这样可以最快得到一个可运营版本，同时避免一开始就把角色删除问题做成数据事故。
