范围：`apps/app` 的 Web 桌面版运行路径

## 任务定义

- 目标不是再做一轮泛泛“看看有没有问题”，而是把 Web 桌面版按真实使用路径拆成可执行批次，逐批找出并修掉：
  - 运行时崩溃
  - 空白页 / 死路由
  - 错误高亮 / 错误跳转 / 错误回退
  - 状态不同步 / 选中态漂移 / 刷新后回退
  - 桌面专属弹窗、独立窗口、侧栏、更多菜单失效
- 本轮只覆盖 Web 桌面版，不扩散到：
  - 移动端主路径
  - Tauri 原生壳特有问题
  - 后端 API 语义重构
- 本轮输出必须是“可直接执行”的工程计划，而不是产品脑暴。

- 后续执行重点必须从编译错误转到：
  - 路由状态
  - 本地状态与查询状态同步
  - 桌面专属 UI 交互闭环

## 完成定义

满足以下条件才算本轮完成：

1. 桌面主路径下的高频入口都能进入、切换、回退，不出现空白页和明显死路。
2. 左侧导航、工作区选中态、更多菜单、右侧面板不出现错高亮和错位状态。
3. 独立窗口、图片查看器、聊天文件、聊天记录、视频号直播伴侣等桌面扩展页都有闭环回退。
4. 不再存在“点按钮即抛异常”的确定性 bug。
5. 每个修复批次都做小步提交。
6. `typecheck`、`build`、`lint` 在批次结束时保持通过。

## bug 分级口径

### P0

- 点操作直接报错
- 进入页面空白
- 页面进入后无法回退
- 工作区主路径断掉
- 删除 / 隐藏 / 退出后仍指向无效对象

### P1

- 导航高亮错
- 左右栏选中态错误
- 同一路由切换参数不生效
- 查询刷新后用户操作态被覆盖
- 桌面弹层或独立窗口行为不一致

### P2

- 文案误导
- 状态提示滞后
- 轻微布局错位
- 图标、标题、副标题与实际状态不一致

## 每批次固定验证

### 静态验证

- `pnpm --filter @yinjie/app typecheck`
- `pnpm --filter @yinjie/app build`
- `pnpm --filter @yinjie/app lint`

### 代码复核口径

- 查所有 `useEffect` 是否把“初始化逻辑”误写成“持续同步逻辑”
- 查所有 `useMemo` 是否在依赖变化后仍保留旧选中态
- 查所有 `navigate({ to })` 是否和当前桌面语义一致
- 查所有桌面路由 fallback 是否能在参数缺失时自救
- 查所有弹层 / 菜单关闭逻辑是否在路由切换后自动复位

## 提交策略
