# Web端桌面版和手机版公众号功能规划

## 任务目标
- 为 `apps/app` 的 Web 桌面版和 Web 手机版补齐“公众号”能力，产品语义、入口位置、页面排版、消息承接方式尽量向微信靠齐。
- 这次规划不是单做一个资讯列表页，而是把“公众号”作为一套完整能力来规划：
  - 公众号列表
  - 公众号主页
  - 文章详情
  - 订阅关系
  - 订阅号消息聚合
  - 服务号消息会话
- 规划必须兼容当前项目的单世界模型，不能引入“实例内多用户”语义。

## 先定几个关键判断

### 1. 公众号不应该塞进发现页主入口
参考微信的使用习惯，公众号更接近“通讯录 + 消息”体系，而不是“发现”体系。

对当前仓库的落地建议：
- 移动端：`公众号` 入口放进 `通讯录` 固定服务项，不放进 `发现`。
- 桌面端：不新增 Shell 一级导航，不把 `公众号` 塞成和 `消息 / 通讯录 / 朋友圈 / 发现` 平级的新主栏目。
- 双端都通过：
  - `通讯录 -> 公众号列表`
  - `消息 -> 订阅号消息 / 服务号会话`
 这两条路径进入。

### 2. 必须一开始就区分“订阅号”和“服务号”
如果不区分，后面很容易把能力做歪。

建议按微信语义拆成两类：
- `subscription`
  - 重点是文章推送
  - 在消息页里收口到 `订阅号消息` 聚合容器
  - 单个号默认不长期占据普通会话列表主位
- `service`
  - 重点是服务通知、菜单、模板消息、和用户的一对一交互
  - 以独立会话形式进入消息列表
  - 详情页里保留“进入公众号主页 / 查看历史消息 / 服务菜单”

### 3. 不能只做前端假壳
当前仓库虽然已有：
- `discover/feed`
- `moments`
- `chat`

但这些都不能直接替代公众号：
- `feed` 更像公共动态流，不像公众号文章流
- `moments` 是熟人内容，不是机构账号内容
- `chat` 当前只有 `direct / group` 两类会话，承接不了“服务号会话”和“订阅号聚合消息”

所以公众号首版就需要补：
- 新实体
- 新契约
- 新接口
- 新页面
- 新会话语义

## 当前现状与可复用基础

### 1. 当前前端结构
可复用的入口和布局基础已经有了：
- `apps/app/src/routes/contacts-page.tsx`
  - 已有“固定服务项 + 联系人列表”的通讯录结构，适合加 `公众号`
- `apps/app/src/routes/chat-list-page.tsx`
  - 移动端已是微信式消息列表骨架，适合增加 `订阅号消息`
- `apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx`
  - 已经有桌面工作区模式，适合承接“公众号会话 / 订阅号工作区”
- `apps/app/src/router.tsx`
  - 当前已有独立详情页和二级页模式，适合新增公众号列表、主页、文章详情路由

### 2. 当前后端结构
后端目前最接近公众号的数据模型是 `feed`，但只能部分复用思路：
- `api/src/modules/feed/feed-post.entity.ts`
  - 说明仓库已经接受“内容流 + 评论预览”这套数据返回方式
- `api/src/modules/chat/conversation.entity.ts`
  - 当前 `type` 只有 `direct / group`
  - 这意味着公众号落地时要补新的会话类型或新的会话子语义

### 3. 当前差距
当前产品没有以下能力：
- 公众号账号实体
- 文章实体
- 订阅关系
- 公众号主页
- 公众号文章阅读页
- 订阅号消息聚合
- 服务号消息会话
- 公众号菜单

## 目标产品形态

### 1. 移动端形态
移动端对齐微信的核心原则：
- 底部 Tab 不新增 `公众号`
- `公众号` 放在 `通讯录` 内部固定入口
- 文章详情是独立阅读页
- 订阅号消息从 `消息` 列表进入

建议的移动端信息架构：
- `通讯录`
  - `新的朋友`
  - `群聊`
  - `标签`
  - `公众号`
- 点击 `公众号`
  - 进入公众号列表页
- 点击某个公众号
  - 进入公众号主页
- 点击文章
  - 进入文章详情页
- `消息` 列表
  - 新增 `订阅号消息`
  - 服务号作为独立会话出现

### 2. 桌面端形态
桌面端对齐微信的核心原则：
- 不新增 Shell 一级导航
- 公众号仍然挂在 `通讯录` 和 `消息` 体系下
- 阅读文章时优先使用“工作区阅读”，而不是手机式整页跳转感

建议的桌面信息架构：
- `通讯录`
  - 左栏固定服务项里增加 `公众号`
  - 点击后进入桌面公众号工作区
- `消息`
  - 左栏消息列表增加 `订阅号消息`
  - 服务号作为独立会话卡片
- 桌面公众号工作区采用三栏：
  - 左栏：公众号列表 / 搜索 / 已关注筛选
  - 中栏：当前公众号资料、最近文章、历史推送
  - 右栏：文章阅读面板

## 双端具体页面规划

### 1. 公众号列表页
目标：像微信一样先让用户“找到号”，而不是先看到一堆文章。

移动端：
- 入口来自 `通讯录`
- 页面结构：
  - 顶部返回栏
  - 搜索框
  - 最近关注
  - 全部公众号列表
- 列表项字段：
  - 头像
  - 名称
  - 简介一句
  - 类型标签：`订阅号 / 服务号`

桌面端：
- 作为 `通讯录` 的一个工作区态
- 左栏为公众号列表，支持搜索
- 中栏展示账号摘要和最近文章
- 不建议首版做复杂分类树

### 2. 公众号主页
目标：对齐微信“公众号资料页 + 历史消息入口”。

页面内容建议：
- 封面 / 头像 / 名称 / 微信号样式的唯一标识
- 简介
- 账号类型
- 关注 / 取消关注
- 最近文章列表
- 历史消息入口
- 服务号菜单入口

移动端：
- 单页上下结构
- 顶部为资料区，底部为最近文章

桌面端：
- 中栏顶部资料区
- 下方文章列表
- 右栏常驻文章阅读区

### 3. 文章详情页
目标：尽量贴近微信文章阅读页，而不是普通博客页。

首版建议包含：
- 标题
- 作者 / 公众号名
- 发布时间
- 头图
- 正文富文本
- 相关推荐或“该号更多内容”
- 底部操作条：
  - 返回公众号主页
  - 复制链接
  - 收藏（可后置）
  - 在聊天中转发（后置）

移动端排版建议：
- 顶部使用简洁返回栏
- 正文最大宽度不要过窄
- 图片、引用、分隔符、二级标题需要有公众号文章阅读感
- 底部固定轻操作条

桌面端排版建议：
- 右栏作为阅读器
- 中栏保留文章列表，不强制整页切走
- 支持从列表切换上一篇 / 下一篇

### 4. 订阅号消息
这是最容易被忽略，但最像微信的一层。

移动端：
- 在 `消息` 列表中增加 `订阅号消息`
- 它不是一个真实公众号，而是一个聚合容器
- 进入后展示：
  - 各订阅号最近推送
  - 按时间倒序
  - 每个号展示最近一篇或最近多篇

桌面端：
- 在桌面消息左栏中增加一条特殊列表项：`订阅号消息`
- 点击后进入桌面订阅号工作区：
  - 左栏：订阅号分组或账号列表
  - 中栏：推送流
  - 右栏：文章阅读

### 5. 服务号会话
服务号不能混成订阅号流。

建议行为：
- 服务号在消息页里表现为独立会话
- 点击进入 `/chat/$conversationId`
- 聊天线程里支持：
  - 文本通知
  - 文章卡片消息
  - 菜单点击后触发系统回复

首版不建议做：
- 支付
- 表单
- 外部网页跳转编排
- 复杂 H5 小程序能力

## 数据结构规划

### 1. 新增实体
建议新增以下实体：

- `OfficialAccount`
  - 公众号基础资料
  - 核心字段建议：
    - `id`
    - `name`
    - `avatar`
    - `handle`
    - `description`
    - `accountType`：`subscription | service`
    - `coverImage`
    - `isVerified`
    - `menuPayload`
    - `isEnabled`
    - `lastPublishedAt`

- `OfficialAccountArticle`
  - 文章实体
  - 核心字段建议：
    - `id`
    - `accountId`
    - `title`
    - `summary`
    - `coverImage`
    - `authorName`
    - `contentPayload`
    - `publishedAt`
    - `isPinned`
    - `readCount`

- `OfficialAccountFollow`
  - 世界主人对公众号的关注关系
  - 核心字段建议：
    - `id`
    - `ownerId`
    - `accountId`
    - `followedAt`
    - `isMuted`
    - `mutedAt`

- `OfficialAccountDelivery`
  - 推送投递记录
  - 核心字段建议：
    - `id`
    - `ownerId`
    - `accountId`
    - `articleId`
    - `deliveredAt`
    - `readAt`
    - `deliveryKind`
      - `subscription_digest`
      - `service_notice`

### 2. 会话结构调整
当前 `packages/contracts/src/chat.ts` 里的 `ConversationType = "direct" | "group"` 不够用。

建议扩成：
- `direct`
- `group`
- `official_account`

同时给公众号会话再补子语义：
- `officialAccountKind`
  - `subscription_inbox`
  - `service`

理由：
- `订阅号消息` 是特殊聚合会话，不适合伪装成普通 direct
- 服务号是一对一服务型会话，也不能混进角色私聊语义

### 3. 消息类型调整
当前 `MessageType` 也不足以表达公众号推送。

建议新增：
- `article_card`
- `official_menu`

用途：
- `article_card`
  - 在订阅号消息和服务号会话中推送文章摘要
- `official_menu`
  - 用于服务号底部菜单点击后的系统回显或菜单型消息

### 3. API 路由建议
建议提供：

- `GET /api/official-accounts`
  - 获取公众号列表
- `GET /api/official-accounts/:id`
  - 获取公众号主页信息
- `POST /api/official-accounts/:id/follow`
  - 关注公众号
- `POST /api/official-accounts/:id/unfollow`
  - 取消关注

  - 获取某个公众号的历史文章
- `GET /api/official-accounts/articles/:articleId`
  - 获取文章详情
- `GET /api/official-accounts/subscription-inbox`
  - 获取订阅号消息聚合流
- `POST /api/official-accounts/articles/:articleId/read`
  - 标记文章已读
- `GET /api/official-accounts/service-conversations`
  - 获取服务号会话摘要

如果服务号首版要支持菜单，还需要：
- `POST /api/official-accounts/:id/menu/:menuKey`

## 交互细节建议

### 1. 关注链路
- 在公众号主页显式提供 `关注 / 已关注`
- 关注后：
  - 订阅号立即进入 `订阅号消息`
  - 服务号可立即创建会话或仅在首次推送时出现

### 2. 已读与红点
- `订阅号消息` 要有总红点
- 聚合流里的每个公众号要有未读状态
- 服务号会话复用现有 `unreadCount`

### 3. 搜索
首版只做：
- 搜公众号名
- 搜文章标题

不建议首版做：
- 全文搜索
- 按作者 / 标签 / 时间复杂过滤

### 4. 内容生产
首版不规划公众号后台发布端。

建议先采用两种来源：
- 预置种子数据
- 管理后台后续追加“公众号管理”模块

## 分阶段实施建议

### Phase 1：只做阅读链路骨架
目标：先让双端都能完整走通“找号 -> 看主页 -> 看文章”。

范围：
- 新增公众号实体与文章实体
- 新增列表、主页、文章详情接口
- 移动端通讯录入口
- 桌面端公众号工作区
- 文章详情阅读页 / 阅读器

这一阶段不做：
- 订阅号消息聚合
- 服务号会话
- 菜单
- 红点系统

为什么这么拆：
- 能先把最重的阅读体验和页面排版做出来
- 不会一上来就把聊天层改复杂

### Phase 2：补消息体系，对齐微信核心使用路径
目标：补齐“消息里看到公众号”的关键体验。

范围：
- `ConversationType` 扩容
- 订阅号消息聚合流
- 服务号会话
- 文章卡片消息
- 已读 / 未读

这一阶段完成后，产品才真正接近微信公众号。

### Phase 3：补服务号能力
目标：让服务号不只是一个会推文章的壳。

范围：
- 底部菜单
- 系统回复
- 服务通知模板
- 历史消息页

首版仍然不要做：
- 支付
- 表单工作流
- 外部业务平台编排

## 验收标准
- 移动端不新增底部 Tab，但能从 `通讯录` 顺畅进入公众号体系。
- 桌面端不新增 Shell 一级导航，但能从 `通讯录` 和 `消息` 使用公众号。
- 公众号文章在移动端和桌面端都具备明显的“微信文章阅读感”，不是普通资讯卡片页。
- 订阅号和服务号在产品表现上有明确区别，不混成一套消息流。
- 服务号消息能进入会话体系，订阅号消息能进入聚合体系。
- 现有聊天、朋友圈、广场动态能力不被公众号改造污染。

## 主要风险与决策
- 最大改动点不在 UI，而在会话语义。`direct / group` 两类会话不够承接公众号。
- 如果一开始不区分 `subscription / service`，后续一定返工。
- 如果把公众号硬塞进 `发现`，虽然开发快，但会和“微信一致”的目标直接冲突。
- 文章详情页如果只用现有通用卡片拼，会做成博客感，不会像微信文章页；这部分需要单独组件族。

## 推荐执行顺序
1. 先补实体、契约、列表页、主页、文章详情，完成 Phase 1。
2. 再改消息契约和会话体系，接入订阅号消息与服务号会话。
3. 最后补菜单、历史消息、已读策略等服务号增强能力。

- 真正进入实现时，这属于结构变更：
  - 新模块
  - 新实体
  - 新路由
  - 新契约
- 届时要立即同步更新 `AGENTS.md` 中的模块、路由、实体说明。
