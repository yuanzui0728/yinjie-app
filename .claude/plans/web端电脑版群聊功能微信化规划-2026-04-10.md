# Web端电脑版群聊功能微信化规划

## 任务目标
- 目标范围只覆盖 `apps/app` 的 Web 电脑版群聊工作区，不动移动端主路径。
- 目标体验对齐微信电脑端群聊的布局、信息架构和操作路径，但不做逐像素、逐文案、逐素材照抄。
- 这份规划是对现有聊天右侧栏规划的群聊专项深化，重点回答四件事：
  - 桌面群聊主工作区应该长什么样
  - 群聊信息和聊天记录侧栏应该怎么落
  - 当前前后端能力缺什么
  - 应该按什么顺序改，才能先像微信，再补齐深层功能

### 当前已经有的真实能力
- 群聊创建、改名、群公告更新
- 群成员读取
- 单个成员添加
- 群消息收发
- 图片 / 文件 / 名片 / 位置卡片附件
- 群聊置顶、已读、清空、隐藏、退出
- 我在本群昵称修改
- 群聊消息搜索的前端本地过滤版

### 当前桌面群聊的主要问题
- 中间聊天区结构不对。
  - 头部下面还挂着一条成员胶囊横条，这不像微信电脑端主聊天区。
- 右侧信息区不对。
  - 当前桌面右栏仍是占位壳，不是可交付的群聊信息侧栏。
- 聊天记录入口不对。
  - 现在群记录搜索是独立全页，更像移动端二级页，不像微信电脑端右侧内联侧栏。
- 建群流程不对。
  - 当前 `create-group-page.tsx` 是整页移动端表单，不像微信电脑端从消息页里直接发起的建群流程。
- 会话列表表达不对。
  - 群会话卡片还没有群头像、发言者前缀、免打扰状态、`@我` 提醒这类微信式信号。
- 深层消息交互不够。
  - 还没有消息右键菜单、引用回复、转发、撤回、多选、聊天文件聚合这些桌面端群聊常见能力。

## 产品基线

### 先说边界
- 不建议做“完全抄袭微信”的逐像素复制。
- 正确口径应该是：
  - 布局结构对齐微信电脑端
  - 交互路径对齐微信电脑端
  - 信息分层对齐微信电脑端
  - 视觉 token、组件实现、命名继续保留隐界自己的代码体系

### 单世界约束下的群聊口径
- 当前实例是单世界模型，只有一个世界主人，群成员本质上是：
  - 世界主人本人
  - 世界角色 / AI 角色
- 因此不应该假装存在完整的“多人真实 IM 群管理体系”。
- 这会影响规划取舍：
  - 首版不做群主转让
  - 不做管理员层级
  - 不做入群审批
  - 不做二维码进群
  - 不做真实多端同步的人类在线态
- 但以下能力仍然有价值，而且符合微信式桌面群聊体验：
  - 建群
  - 添加角色
  - 移除角色
  - 改群名 / 改公告
  - 查群记录
  - 置顶 / 免打扰 / 隐藏 / 清空 / 退出
  - 我在本群昵称
  - 群文件 / 附件聚合
  - 消息右键操作

## 目标体验

### 1. 总体布局
- 桌面群聊工作区统一收敛为微信电脑端熟悉的三段式：
  - 左侧会话列表
  - 中间群聊线程
  - 右侧侧栏
- 右侧侧栏不是常驻三栏硬占位，而是“关闭时收起，打开时从右侧占 320px 左右”的微信式工作区。
- 群聊详情和聊天记录都在右侧栏里完成，不跳全页。

### 2. 中间群聊线程
- 顶部头部只保留：
  - 群名
  - 成员数
  - 右上角三按钮：`聊天记录 / 通话 / 更多`
- 去掉头部下方那条成员胶囊横条。
- 消息区保持微信电脑端阅读节奏：
  - 时间居中
  - 群聊消息显示发言人昵称
  - 自己消息右对齐
  - 他人消息左对齐
  - 系统消息居中
- 输入区收敛为桌面 IM 结构：
  - 左下工具排
  - 文本输入区
  - 发送入口
  - 支持附件、表情、语音转文字

### 3. 右侧栏双模式
- 右侧栏统一只有两种模式：
  - `history`：群聊记录检索
  - `details`：群聊信息
- 两者复用同一容器，不再把群聊信息做成独立桌面详情页。

### 4. 左侧群会话列表
- 群聊卡片要更像微信：
  - 群头像优先显示组合头像或群头像
  - 预览文案优先显示 `发送者: 内容`
  - 未读数保持右侧红点 / 红 badge
  - 置顶仍置于顶部
  - 免打扰要显示静音态
  - 后续补 `@我` / `有人@所有人` 的特殊提醒样式

## 桌面群聊目标信息架构

### A. 中间线程区

#### 头部
- 左侧：
  - 群名
  - 成员数
- 右侧：
  - `聊天记录`
  - `通话`
  - `更多`

#### 消息区
- 消息时间分隔
- 群成员昵称
- 文本、图片、文件、名片、位置卡片
- 高亮定位消息
- 后续补右键菜单与引用回复

#### 输入区
- 第一行工具条建议收敛为：
  - 表情
  - 图片 / 文件
  - 截图入口占位
  - 语音转文字
  - 更多能力占位
- 第二行是文本输入
- 发送规则：
  - 默认 `Enter` 发送
  - `Shift + Enter` 换行
- 桌面端建议补：
  - 拖拽文件到消息区发送
  - 粘贴图片直接发送

### B. 群聊记录侧栏

#### 顶部
- 标题：`聊天记录`
- 副标题：当前群名
- 关闭按钮

#### 主体
- 搜索输入框
- 搜索结果列表
- 每项展示：
  - 发送者
  - 时间
  - 命中片段
- 点击结果后：
  - 关闭搜索结果高亮态
  - 中间消息区滚动到对应消息
  - 对目标消息闪烁高亮

#### 首版实现方式
- 首版先复用 `getGroupMessages`
- 继续用前端本地过滤
- 但 UI 不再走独立页，而是嵌入右侧栏

#### 二期升级
- 服务端搜索接口
- 分页 / 游标
- 大群长历史搜索
- 附件类型过滤

### C. 群聊信息侧栏

#### 1. 顶部资料区
- 群头像
- 群名
- 成员数
- 群公告摘要

#### 2. 成员区
- 微信式成员宫格
- 成员头像 + 名称
- 末尾保留：
  - `+` 添加成员
  - `-` 移除成员

#### 3. 常用入口区
- `查找聊天记录`
- `聊天文件`
- `群公告`

#### 4. 设置区
- `消息免打扰`
- `置顶聊天`
- `保存到通讯录`
- `我在本群的昵称`
- `显示群成员昵称`
- `聊天背景`

#### 5. 风险操作区
- `隐藏聊天`
- `清空聊天记录`
- `删除并退出`

### D. 建群入口
- 不再把桌面建群体验做成整页移动端表单。
- 建议桌面端改成微信式轻流程：
  - 从左侧 `+` 菜单点 `发起群聊`
  - 弹出桌面联系人选择面板
  - 支持搜索联系人
  - 右侧展示已选成员
  - 底部确认创建
- 如果短期不想新做弹层，也至少要把 `create-group-page.tsx` 的桌面态单独改成双栏选择器，而不是沿用移动端页面。

## 推荐的数据与接口改造

### 1. 群设置模型
- 建议扩展 `Group` / `GroupEntity`：
  - `isMuted`
  - `mutedAt`
  - `isSavedToContacts`
  - `savedToContactsAt`
  - 可选：`showMemberNicknames`
- 注意：
  - 由于当前是单世界单主人实例，这些字段可以直接挂在 `Group` 上，不必再抽 owner-group settings 子表。

### 2. 群成员管理
- 保留现有 `POST /api/groups/:id/members`
- 新增：
  - `POST /api/groups/:id/members/batch`
  - `DELETE /api/groups/:id/members/:memberId`
- 理由：
  - 桌面建群和后续加人都是批量选择心智，单个接口太弱。

### 3. 群记录检索
- 新增：
  - `GET /api/groups/:id/messages/search?q=&cursor=&limit=`
- 返回内容建议：
  - `messageId`
  - `senderName`
  - `text`
  - `type`
  - `createdAt`
  - `snippet`
- 首版仍可先用本地过滤，但接口必须列入 P1。

### 4. 消息交互元数据
- 建议扩展 `GroupMessage` / `GroupMessageEntity`：
  - `replyToMessageId`
  - `replyPreviewPayload`
  - `recalledAt`
  - `recalledBy`
  - `mentionsPayload`
- 对应新增接口：
  - `POST /api/groups/:id/messages/:messageId/recall`
  - 后续可选：`POST /api/groups/:id/messages/forward`

### 5. 会话列表摘要
- 当前群聊在会话列表里仍按普通会话摘要展示，不够微信化。
- 建议扩展 `ConversationListItem` 的群聊摘要能力：
  - `avatar`
  - `avatarMembers`
  - `previewPrefix`
  - `mentionMe`
  - `mentionAll`
- 这样左侧群会话卡片就不用再额外拉一次成员列表。

## 推荐实施顺序

### Phase 1：把外形和主路径改对
目标：先让桌面群聊看起来就是微信电脑端那一类产品。

改造点：
- 统一群聊头部三按钮
- 去掉成员胶囊横条
- 右侧 `history / details` 侧栏接入真实内容
- 群详情和群记录逻辑从全页路由抽出来，复用到桌面侧栏
- 左侧群会话卡片补预览前缀
- 建群页补桌面态

交付结果：
- 用户从桌面消息页进入群聊后，主工作区结构和操作路径基本成立。

### Phase 2：把群设置和成员管理做成真能力
目标：不再只是微信式壳子。

改造点：
- 群免打扰后端化
- 保存到通讯录
- 批量加人
- 移除成员
- 群搜索服务端化

交付结果：
- 群右侧信息栏大部分开关和入口都是真功能，不再主要依赖本地偏好。

### Phase 3：把桌面端深交互补齐
目标：拉开与“只有基础聊天”的差距。

改造点：
- 消息右键菜单
- 引用回复
- 拖拽 / 粘贴发送
- 聊天文件聚合
- `@成员` 能力

交付结果：
- 桌面群聊进入真正可高频使用的状态。

### Phase 4：补高级 IM 细节
目标：解决“像微信，但还没到成熟桌面 IM”的尾差。

改造点：
- 撤回
- 多选
- 转发
- `@我` 特殊提醒
- 群头像组合图

## 视觉与交互要求

### 1. 收平
- 群聊桌面态不要继续保留当前大面积米色渐变和圆角悬浮卡片感。
- 目标应该更接近：
  - 浅灰白背景
  - 细分隔线
  - 更少阴影
  - 更稳定的平直层级

### 2. 群信息侧栏
- 不要像“设置页”。
- 要像微信电脑端那种：
  - 头像 / 标题
  - 成员宫格
  - 行列表
  - 风险区

### 3. 桌面建群弹层
- 不要做移动端长页面在桌面上的简单放大。
- 要做成更适合鼠标的：
  - 搜索
  - 列表
  - 已选区
  - 底部确认区

## 验收标准
- 桌面群聊主线程不再显示头部下方成员胶囊横条。
- 点击 `聊天记录` 后，在当前页面右侧打开群记录搜索侧栏，不跳独立全页。
- 点击 `更多` 后，在当前页面右侧打开群信息侧栏，不跳独立全页。
- 群信息侧栏至少能完成：
  - 改群名
  - 改群公告
  - 置顶
  - 我在本群昵称
  - 清空
  - 隐藏
  - 退出
- 左侧群会话卡片能区分群聊，不再只显示无前缀的通用摘要。
- 桌面发起群聊流程不再沿用移动端整页表单。

## 风险与取舍

### 1. 最大风险不是样式，而是信息架构
- 如果只把颜色和圆角改得像微信，但仍然保留：
  - 全页群详情
  - 全页群记录搜索
  - 移动端式建群页
- 最终结果还是不像微信电脑端。

### 2. 单世界模型决定了不要过度照搬多用户群管理
- 不要为了“像微信”而硬做一整套无意义的群主管理权限体系。
- 当前产品真正需要的是：
  - 世界主人管理角色群聊
  - 不是模拟企业微信群治理

### 3. 服务端搜索和消息交互最好尽早列入里程碑
- 小群用前端本地过滤还勉强可用。
- 一旦历史消息变长，桌面群记录搜索和引用回复迟早会成为刚需。

## 结论
- 当前仓库已经具备桌面群聊的基础骨架，但仍停在“能聊”的层级，没有进入“像微信电脑端”的层级。
- 正确推进方式不是继续堆单点按钮，而是先重排桌面群聊的信息架构：
  - 左侧会话卡片表达
  - 中间线程头部和输入区
