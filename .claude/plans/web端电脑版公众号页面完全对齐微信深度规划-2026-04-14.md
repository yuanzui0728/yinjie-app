# Web 端电脑版公众号页面完全对齐微信深度规划

## 任务目标
- 将桌面 Web 端 `公众号` 收敛为“微信电脑版公众号窗口”而不是“通讯录里的资料页 + 右侧内嵌正文”。
- 对齐的重点不是单个按钮，而是四件事：
  - 公众号主窗口的信息架构
  - 文章阅读与主窗口的关系
  - 展示方式切换
  - 消息入口、通讯录入口、文章窗口三者之间的连续性
- 用户明确要求的目标行为必须成立：
  - 文章详情点击后跳出新的页面查看
  - 页面顶部有展示方式切换按钮
  - 切换后的主模式为“上面常看的号，下面文章”
- 不复制腾讯素材、图标和插画，只复刻结构、节奏、状态切换和桌面交互。

## 当前实现现状

### 已有能力，不要误判为缺失
- 桌面消息页已经有 `公众号` 入口，能在消息工作区右侧打开公众号页。
- 订阅号聚合流和服务号线程都已经做了桌面右侧工作区。
- 公众号工作区已有：
  - 搜索
  - 关注筛选
  - 服务号 / 订阅号动作入口
  - 文章阅读
  - 收藏
  - 到手机继续
- 桌面端已有成熟的独立窗口基础设施，可直接复用到公众号文章阅读。

### 现状问题，按代码落点拆解

#### 1. `desktop-official-accounts-workspace.tsx` 仍然是“目录页 + 详情页 + 内嵌正文”
- 左栏是账号清单，中栏是账号资料，右栏是内嵌文章。
- 这更像“运营后台里的内容目录页”，不像当前微信电脑版的公众号工作窗口。
- 尤其不符合你要求的“文章点击后跳出新的页面查看”。

#### 2. 桌面文章页实际上不存在独立阅读页
- `official-account-article-page.tsx` 在桌面态会直接重定向回 `/tabs/contacts#pane=official-accounts`。
- 这意味着桌面端现在没有真正的“公众号文章新页面”，只有内嵌阅读面板。

#### 3. 没有展示方式切换
- 当前公众号页只有：
  - 搜索
  - 全部 / 已关注筛选
  - 消息 / 资料两个 Tab
- 没有“按账号看”与“按文章看”的显式切换，更没有“上面常看的号，下面文章”的精选模式。

#### 4. 当前左栏模型是“全部账号目录”，不是“微信式阅读分发页”
- 现在左栏主心智是查找公众号账号。
- 微信当前公开可见的内容分发心智更偏：
  - 常读账号优先
  - 文章卡片为主
  - 账号目录作为一种可切换查看方式

#### 5. 订阅号、服务号、公众号主页三套桌面逻辑还是分裂的
- 订阅号聚合流是一套布局。
- 服务号线程是一套布局。
- 公众号目录又是一套布局。
- 用户感知仍是“几个公众号功能页的组合”，不是“一个微信电脑版公众号系统”。

#### 6. 文章阅读器现在是详情侧栏，不是桌面浏览窗口
- `OfficialArticleViewer` 当前是一个嵌入组件，默认承载在右侧主工作区。
- 没有浏览器式头部，也没有独立窗口尺寸、返回源上下文、最近浏览链路。

## 目标产品定义

### 一句话目标
- 进入桌面 `公众号` 后，用户感知应该是“微信电脑版公众号窗口”，而不是“通讯录中的一个扩展详情页”。

### 目标结构
- 桌面公众号统一收口为两个层次：
  - `公众号主窗口`
  - `公众号文章窗口`
- 主窗口负责：
  - 展示方式切换
  - 常看账号
  - 文章流
  - 账号目录
  - 服务号 / 订阅号入口
- 文章窗口负责：
  - 正文阅读
  - 收藏 / 分享 / 默认浏览器打开
  - 回到来源工作区

### 核心原则
- 账号页和文章页彻底解耦。
- 主窗口以“阅读分发”优先，不以“账号资料”优先。
- 文章点击默认跳新页面，不再占主窗口右栏。
- 主窗口必须有显式展示方式切换。
- “常看的号 + 文章流”必须成为默认主视图，而不是附属功能。

## 目标信息架构

### 主窗口顶部
- 左侧标题：`公众号`
- 中间保留搜索框
- 右侧新增展示方式切换按钮

### 展示方式定义

#### 1. 精选流模式
- 默认模式
- 顶部区域：`常看的号`
  - 以横向账号条或宫格展示高频阅读账号
  - 展示 6 到 10 个，支持“更多”
- 下方区域：`文章流`
  - 按文章卡片直接浏览
  - 不是先点账号再看文章
  - 卡片包括封面、标题、摘要、账号名、发布时间、未读状态
- 这是你要求的“上面有常看的号，下面是文章”
- 也是最接近微信当前公开内容分发心智的模式

#### 2. 账号模式
- 用于按公众号查找和管理
- 左侧改成账号目录列表
- 中部是当前账号主页摘要
- 右侧不再内嵌正文，改为账号最近文章列表 / 资料卡
- 点击文章仍然跳到新页面

### 主窗口中部 / 右部角色
- 精选流模式下：
  - 主区就是文章流，不再需要现在这种右侧正文面板
- 账号模式下：
  - 保留账号资料和最近文章列表
  - 文章点击也跳独立窗口
- 结论：
  - 现有 `DesktopOfficialAccountsWorkspace` 的“三栏固定布局”需要解体

## 文章阅读模型

### 目标行为
- 点击任意文章卡片后，打开新的桌面文章页面，而不是在当前主窗口右侧替换内容。
- 桌面原窗口保留当前滚动位置、选中账号、展示模式和文章流位置。
- 文章新页面关闭后，用户仍回到原来的公众号主窗口上下文。

### 为什么必须这么改
- 用户明确要求如此。
- 当前稳定微信 PC 公开资料已经不适合继续假设“聊天旁常驻浏览侧栏”是正式口径。
- 微信桌面端近年的内容阅读更像“浏览任务”，独立页面 / 独立窗口更符合桌面心智。

### 目标页面形态
- 推荐做成独立桌面窗口路由，而不是简单浏览器标签页：
  - 路由建议：`/desktop/official-article-window`
- 窗口顶部应包含：
  - 文章来源公众号
  - 返回来源
  - 收藏
  - 分享 / 复制链接
  - 用默认浏览器打开
- 正文区沿用 `OfficialArticleViewer` 的内容块，但改造成更浏览器式容器

### 与现有能力的关系
- 直接复用聊天独立窗口和图片独立窗口的基础设施
- 新增：
  - `desktop-official-article-window-route-state.ts`
  - `openDesktopOfficialArticleWindow(...)`
  - `desktop-official-article-window-page.tsx`

## 路由与状态模型

### 当前问题
- 现在桌面公众号状态分散在：
  - `/tabs/chat#officialView=official-accounts`
  - `/tabs/contacts#pane=official-accounts`
  - `/official-accounts/articles/$articleId`（桌面又会被重定向回去）
- 结果是“能打开”，但不是“一个系统”。

### 目标模型
- 桌面公众号需要两套状态：

```ts
type DesktopOfficialHubRouteState = {
  mode?: "feed" | "accounts";
  accountId?: string;
  articleId?: string;
  section?: "frequent" | "articles";
};

type DesktopOfficialArticleWindowRouteState = {
  articleId: string;
  accountId?: string;
  sourcePath?: string;
  sourceHash?: string;
  title?: string;
};
```

### 路由策略
- 主窗口仍落在既有工作区里：
  - 消息页：`/tabs/chat#officialView=official-accounts&mode=feed...`
  - 通讯录：`/tabs/contacts#pane=official-accounts&mode=feed...`
- 文章阅读统一走新窗口路由：
  - `/desktop/official-article-window#articleId=...`
- 移动端继续沿用现有 `/official-accounts/articles/$articleId`

### 核心原则
- 主窗口负责“选什么”
- 新页面负责“读什么”
- 不再让桌面文章页重定向回内嵌面板

## 分阶段实施计划

### Phase 1：主窗口状态与模式切换收口
- 目标：
  - 给桌面公众号引入 `mode=feed|accounts`
  - 所有桌面入口统一把公众号打开到同一套状态模型
- 动作：
  - 扩展消息页的 `desktop-chat-route-state`
  - 扩展通讯录的 `desktop-contacts-route-state`
  - `DesktopOfficialAccountsWorkspace` 退化为模式路由器

### Phase 2：精选流模式落地
- 目标：
  - 做出“上面常看的号，下面文章”的主页面
- 动作：
  - 顶部常看账号条
  - 下方文章流卡片
  - 顶部展示模式切换按钮
  - 搜索在 `feed` 模式下优先搜账号名与文章标题

### Phase 3：账号模式重构
- 目标：
  - 保留按账号查找和资料查看能力
- 动作：
  - 账号列表改成轻量目录
  - 账号详情改成摘要页，不再内嵌正文
  - 服务号 / 订阅号动作入口继续保留

### Phase 4：文章独立窗口
- 目标：
  - 任意文章都能跳新页面
- 动作：
  - 新建桌面文章窗口路由
  - 复用独立窗口基础设施
  - 文章窗口头部补“返回来源 / 浏览器打开 / 收藏 / 分享”

### Phase 5：消息页与通讯录入口统一
- 目标：
  - 消息列表入口和通讯录入口打开的是同一个公众号系统
- 动作：
  - `/tabs/chat` 的公众号入口默认进 `feed`
  - `/tabs/contacts` 的公众号入口也默认进 `feed`
  - 如果用户从账号目录进入，则保留 `mode=accounts`

### Phase 6：桌面细节收尾
- 目标：
  - 补齐微信式使用感
- 动作：
  - 缩窄窗口时隐藏部分侧栏
  - 键盘切换账号 / 文章
  - 右键菜单增加“在新窗口打开文章”
  - 最近浏览与回跳来源恢复

## 视觉与交互口径

### 主窗口
- 白底主画布，灰辅色，选中态绿色强调，跟微信 PC `4.1.8` 公开风格一致
- 不再使用当前偏“运营看板”的统计卡片堆叠
- 展示方式切换按钮放在标题区右侧，做成微信式轻按钮而不是大标签卡

### 精选流模式
- 常看账号区比文章区更短、更轻
- 文章卡片区承担主视觉
- 卡片密度要比现在订阅号聚合流更高，减少“巨卡片感”

### 账号模式
- 账号目录更轻、更扁平
- 账号主页更像微信资料页，不像内容运营页

### 文章窗口
- 参考微信内置浏览器 / 桌面阅读页气质
- 不做沉重三栏
- 重点是正文可读性、顶部动作、回源链路

## 风险与决策点

### 1. 展示方式切换按钮的微信原始形态公开资料不完整
- 结论：
  - 按“微信风格推断”做，不伪装成已公开确认的官方细节
  - 但产品行为必须做，因为这是你的明确要求，也是整个页面信息架构成立的关键

### 2. “常看的号”排序没有现成后端字段
- 结论：
  - 第一阶段以前端行为数据和已读 / 未读 / 最近文章做启发式排序
  - 后续如有必要再补后端聚合接口

### 3. 文章新页面是“浏览器新标签”还是“独立桌面窗口”
- 结论：
  - 桌面壳环境优先独立窗口
  - Web 环境 fallback 到 `window.open`
  - 不建议继续内嵌右栏

## 验证清单
- 从消息页点击 `公众号` 入口，应默认进入 `feed` 模式
- 顶部存在展示方式切换，能在 `feed / accounts` 间切换
- `feed` 模式下，上方能看到常看的号，下方是文章流
- 点击文章后，必须打开新的桌面文章页面
- 关闭文章页后，主窗口仍保留原来的账号、滚动和模式
- 从账号模式点击文章，也必须打开新页面
- 从服务号 / 订阅号跳回公众号主页时，仍回到同一套桌面公众号系统

## 最终结论
- 这次公众号微信化不能继续在现有 `DesktopOfficialAccountsWorkspace` 上小修小补。
- 真正的改造方向是：
  - 主窗口从“账号资料页”改成“可切换的公众号分发窗口”
  - 文章从“右侧嵌入阅读”改成“新页面 / 新窗口阅读”
  - 默认主模式改成“常看的号 + 文章流”
- 只有这样，桌面端公众号才会从“通讯录功能块”真正变成“微信电脑版公众号窗口”。
