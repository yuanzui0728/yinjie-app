# 单世界架构收口计划

## 目标
把项目彻底收口为“单世界实例 + 远程客户端壳 + 官方云平台控制面”这一套唯一架构，删除所有旧多用户、本地托管、历史登录、兼容契约、废弃页面和误导性命名，只保留当前产品逻辑下真正需要的产物。

目标架构固定为：
- `api/`：每个实例只服务 1 个世界主人
- `apps/app/`：唯一业务前端
- `apps/desktop/`、`apps/android-shell/`、`apps/ios-shell/`：只作为壳承载 `apps/app`
- `apps/admin/`：单世界实例运维后台
- `apps/cloud-api/` + `apps/cloud-console/`：官方云世界平台与控制面

## 收口原则
- 不做“兼容保留”，旧逻辑一旦确认废弃就直接删除
- 不再保留“也许以后还会用”的旧接口、旧字段、旧文案、旧路由
- 先删调用链末端的死代码，再删共享契约，再删实体与文档，避免半路断链
- 每完成一个阶段就同步更新 `AGENTS.md`，确保项目规则和实际结构一致
- 不新增测试文件，验证依赖现有构建、类型检查和人工回归

## 现阶段必须删除的旧东西
### 1. 旧多用户语义
- `userId` 作为实例隔离条件的前端参数、契约字段、WebSocket 载荷
- `authorId` / `creatorId` / `senderId` 中仅用于“用户上下文传参”的兼容写法
- `session-store` 这类面向多用户登录历史的命名和包装
- 旧“当前用户 / 用户列表 / 删除用户 / 用户管理”文案与路由

### 2. 旧认证与历史接口
- `/auth/login`、`/auth/init`、`/auth/me`、`/auth/users/:id` 这一整套历史接口契约与调用包装
- `SessionPayload` 驱动的旧前端登录态设计
- 一切“先登录，再进入实例”的历史流程

### 3. 旧本地托管 / 本地 Core API 语义
- `local-hosted`、`hostRole=host`、`canManageLocalCoreApi=true` 相关设计预留
- 桌面端“启动 / 停止 / 重启 Core API”的共享运行时接口层
- 文案中“桌面端可托管本地 Core API”或“本地 Core API 控制”的残留表达

### 4. 旧产品结构残留
- `login-page` 及其相关跳转、说明、兼容逻辑
- 管理后台中已退役但仍挂在仓库里的历史页面和入口
- 与单世界产品无关的废弃 README、脚本说明、部署说明

### 5. 误导性部署产物
- 默认部署里缺失但文档仍暗示存在的服务组合
- 对“云平台会自动创建世界实例”的错误暗示

## 分阶段执行
### Phase 1：冻结唯一架构口径
目标：先把“项目现在只支持什么”写死，避免边删边摇摆。

执行项：
- 更新 `AGENTS.md`
- 明确只有两类后端：
  - 世界实例后端 `api/`
  - 官方云平台 `apps/cloud-api/`
- 明确客户端全部为 remote-connected 壳
- 明确不再支持历史登录模型和多用户实例
- 明确云平台当前职责只是“手机号绑定 + 世界记录 + 地址分发 + 平台审核”，不是自动编排实例

完成标准：
- 仓库规则、README、部署说明、产品口径完全一致

### Phase 2：删除旧认证与会话体系
目标：把“用户登录产品”彻底收口成“世界主人产品”。

执行项：
- 删除 `packages/contracts` 中历史 auth 接口与类型
- 删除前端对 `loginUser`、`initUser`、`getCurrentUser`、`updateUser`、`completeOnboarding` 等旧封装的支持
- 将 `world-owner-store` 里的 `token`、`userId` 兼容暴露改为只保留当前真实需要的数据
- 删除 `session-store` 兼容别名，统一改为 `world-owner-store`
- 删除 `login-page` 路由、页面和入口引用

完成标准：
- 前端不再出现“登录”“当前用户”“用户会话”主语
- 世界主人初始化只走 `GET/PATCH /api/world/owner`

### Phase 3：删除前端与契约层的多用户兼容参数
目标：把所有“虽然没用但还留着”的 `userId` 风格参数清干净。

执行项：
- 清理 `packages/contracts/src/chat.ts`
- 清理 `packages/contracts/src/social.ts`
- 清理 `packages/contracts/src/moments.ts`
- 清理 `packages/contracts/src/feed.ts`
- 清理 `packages/contracts/src/ws.ts`
- 清理 `packages/contracts/src/client.ts` 中接受 `userIdOrBaseUrl`、`payloadOrBaseUrl` 的兼容签名
- 前端调用点统一改为只传 `baseUrl` 或纯业务参数
- 聊天会话 ID 推断逻辑不再依赖 `userId_` 前缀假设

完成标准：
- `apps/app/` 不再需要为了兼容旧模型而读取 `userId`
- `packages/contracts/src` 中不再保留无意义的 `userId?` 兼容字段

### Phase 4：删除本地托管与桌面旧运行时残留
目标：彻底把客户端定义成“壳”，不再保留本地 Core API 管理幻觉。

执行项：
- 简化 `apps/app/src/runtime/platform.ts`，删除 `local-hosted` / `host` 相关分支
- 删除 `packages/ui` 中桌面 Core API 启停相关共享接口、hook 和类型
- 清理桌面端守卫、诊断、文案中所有“启动本地 Core API”的历史痕迹
- 保留桌面端只做：
  - 远程地址配置
  - 健康检查
  - 壳级诊断

完成标准：
- 项目中不存在“桌面端托管后端”语义
- 桌面端运行时接口只表达“远程连接状态”

### Phase 5：删除后端实体和控制器里的旧多用户痕迹
目标：让代码语义和业务语义一致，而不只是“运行时约束单世界”。

执行项：
- 审核所有仍保留 `userId` 的实体字段
- 对必须保留物理表兼容的字段：
  - 保留列
  - 重命名代码注释和使用方式，明确其为“世界主人兼容列”
- 对不再需要外部传入的查询参数：
  - 删除控制器层 `authorId` / `userId` 风格查询入口
- 审核 `Conversation`、`Friendship`、`FriendRequest`、`NarrativeArc` 等模块
- 删除仅用于迁移过渡的服务方法或注释

完成标准：
- 后端公开接口完全不暴露多用户实例语义
- 实体若因物理表兼容保留旧字段，代码注释必须明确其兼容性质

### Phase 6：收口管理后台、云平台和部署
目标：只留下和现有产品真正匹配的后台与部署产物。

执行项：
- 删除 Admin 内仍残留但不再可达的历史页面文件
- Admin 路由只保留实例运维相关页面
- 校正文案，避免把 Admin 说成用户管理后台
- 补齐根部署说明：
  - 世界实例部署
  - Admin 部署
  - 云平台部署
  - 云平台控制台部署
- 明确云平台当前是“控制面 + 记录中心”，不是自动实例编排器

完成标准：
- 新人只看仓库结构和 README 就能正确理解系统组成
- 默认部署说明不再误导

## 推荐落地顺序
1. 先做 Phase 1，统一口径
2. 再做 Phase 2 和 Phase 3，先删前端和契约层的旧接口
3. 然后做 Phase 4，收掉本地托管幻想
4. 再做 Phase 5，清理后端公开语义和实体残留
5. 最后做 Phase 6，把后台、云平台、部署和文档全部收口

## 每阶段验证方式
- `pnpm --filter @yinjie/app build`
- `pnpm --filter @yinjie/admin build`
- `pnpm --filter @yinjie/cloud-console build`
- `pnpm --filter @yinjie/cloud-api build`
- `pnpm --filter @yinjie/desktop typecheck`
- 人工检查 Setup、Onboarding、Profile、聊天、云世界入口、Admin 首页

## 风险点
- `packages/contracts` 被多端共享，删兼容签名时会连带影响 App、Admin、Cloud Console
- 某些数据库字段虽然语义过时，但物理表可能还要保留兼容，不能直接删列
- 桌面端共享 UI 里可能还有“已改为远程模式但接口名还很旧”的情况，需要先替换调用再删底层
- 云平台容易被误改成“自动开世界”，但当前代码并没有编排能力，规划时不能把未来能力当现状

## 最终完成定义
- 项目里不存在历史登录产品语义
- 项目里不存在多用户实例语义
- 项目里不存在客户端本地托管 Core API 语义
- 所有对外文档和后台文案都只描述当前真实架构
- 仓库只保留单世界架构下仍然会被运行、构建、发布、运维的产物
