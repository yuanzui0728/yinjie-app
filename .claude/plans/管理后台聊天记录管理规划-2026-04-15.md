# 管理后台聊天记录管理规划

日期：2026-04-15

## 1. 目标

- 在 `apps/admin` 新增一个“聊天记录管理”工作台，集中查看世界主人与各个角色的真实对话记录。
- 这套能力的定位不是“聊天页搬运”，而是给后续产品升级、Prompt 调整、角色优化、内容策略迭代提供真实依据。
- 首版聚焦 `世界主人 <-> 角色` 的单聊记录，先把最核心的样本池、检索、阅读和分析能力做扎实。
- 全程保持单世界约束，不重新引入“实例内多用户管理”语义。

## 2. 现状结论

### 2.1 已有能力

- `api/src/modules/admin/reply-logic-admin.service.ts`
  - 已经能拿到会话列表与会话快照。
  - 已经能读取 direct conversation / group conversation 的可见消息。
- `api/src/modules/chat/chat.service.ts`
  - 已经支持单聊消息读取、按消息窗口定位、按关键词/类型/发送者/日期搜索。
- `api/src/modules/chat/group.service.ts`
  - 已经有群聊版本的消息读取与搜索逻辑，可作为二期复用基础。
- `api/src/modules/analytics/ai-usage-ledger.service.ts`
  - 已经能按 `conversationId` / `groupId` 统计 Token、请求数、成本、模型、场景。

### 2.2 当前缺口

- 现有 `reply-logic` 更偏“回复调试台”，不是“聊天记录管理台”。
- 没有后台专用的会话列表筛选、排序、分页和沉淀视图。
- 没有把聊天记录和 Token 成本、角色画像、回复逻辑联动起来。
- 没有“产品分析视角”的摘要指标，例如：
  - 某角色近 7/30 天对话活跃度
  - 用户/角色消息占比
  - 主动消息占比
  - 附件消息占比
  - 大致回复时延
- 当前消息搜索主要是“单会话内搜索”，还没有“跨全部角色的全局检索”方案。

### 2.3 重要约束

- 该功能面向“世界主人与角色”的样本管理，不做旧式 `users-page.tsx` 那种多用户后台。
- `clear` 目前只是通过 `lastClearedAt` 隐藏历史，并没有物理删除消息。
- `delete` 会物理删除消息，后台无法恢复。
- `recall` 是把原消息替换成系统消息，因此后台看到的应是“撤回结果”，不是原始文本。

## 3. 产品定位

建议新增后台一级路由：

- 路由：`/chat-records`
- 中文名称：`聊天记录`
- 页面定位：`会话档案 + 搜索定位 + 迭代分析`

它与现有页面的关系：

- `回复逻辑`：回答“为什么这次会这么回”
- `聊天记录`：回答“真实世界里都聊了什么”
- `Token 用量`：回答“这些对话花了多少成本”

三者应该互相跳转，而不是合并成一个超大页面。

## 4. 首版范围（V1）

### 4.1 必做能力

1. 会话列表
- 只看世界主人与角色的 direct conversation。
- 支持按角色、最后活跃时间、消息量排序。
- 支持显示 `隐藏会话`。
- 支持显示“仅最近 7 天 / 30 天有消息的会话”。

2. 会话详情
- 中央区域展示完整时间线。
- 支持按时间倒序加载更早消息。
- 支持按消息类型过滤：`text / proactive / image / file / voice / sticker / system`。
- 支持跳转到指定消息上下文。

3. 会话内搜索
- 支持关键词、发送者、消息类型、日期范围搜索。
- 搜索结果点击后，时间线跳到对应消息附近。

4. 右侧分析面板
- 会话基础信息：角色、会话创建时间、最近活跃时间、总消息数。
- 行为指标：用户消息数、角色消息数、主动消息数、附件消息数、系统消息数。
- 粗粒度效率指标：近 7/30 天消息量、大致首响时延、中位回复时延。
- 成本指标：总请求数、总 Token、估算成本、主要模型、主要 scene。

5. 联动入口
- 从聊天记录跳到：
  - `回复逻辑` 对应会话
  - 角色运行逻辑台
  - Token 用量页，并自动带上 `conversationId`

### 4.2 首版不做

- 不做跨所有消息的全文全局搜索。
- 不做群聊、公众号、订阅号消息的统一接入。
- 不做 AI 自动生成“需求总结 / 问题聚类 / 情绪分析”。
- 不做后台改写消息、批量删除消息等高风险操作。

## 5. 推荐交互结构

延续现有管理后台工作台风格，采用三栏布局：

### 左栏：会话与筛选

- 顶部总览卡：
  - 总会话数
  - 近 7 天活跃会话数
  - 近 30 天总消息数
  - 近 30 天估算成本
- 筛选区：
  - 角色
  - 时间范围
  - 是否显示隐藏会话
  - 是否包含清空前历史
- 会话列表项：
  - 角色名
  - 最后一条消息预览
  - 最后活跃时间
  - 近 30 天消息数
  - 未清空历史标记 / 已隐藏标记

### 中栏：时间线

- 气泡流不必完全微信化，重点是“清晰、连续、可检索、可定位”。
- 每条消息展示：
  - 发送方
  - 时间
  - 类型
  - 正文/附件摘要
- 搜索命中后高亮关键词，并支持“定位前后各 N 条消息”。
- 对 `includeClearedHistory=true` 的历史区间加一层“已被用户清空但仍保留在数据库”的提示条。

### 右栏：洞察与联动

- `概览` Tab：消息结构、活跃趋势、回复效率。
- `成本` Tab：Token、模型、scene、失败请求。
- `角色` Tab：角色摘要、最近在线状态、跳转到运行逻辑台。

## 6. 后端方案

### 6.1 新增 Admin 读取服务

不建议继续把逻辑塞进 `ReplyLogicAdminService`，建议新增：

- `api/src/modules/admin/chat-records-admin.service.ts`
- `api/src/modules/admin/chat-records-admin.types.ts`

原因：

- `reply-logic` 已经偏大，继续堆会让“调试逻辑”和“历史档案”耦合过深。
- 聊天记录管理本质是一个新的后台读模型。

### 6.2 推荐接口

建议新增以下管理后台接口：

- `GET /api/admin/chat-records/overview`
  - 返回总会话数、活跃会话数、近 7/30 天消息数、近 30 天成本等。
- `GET /api/admin/chat-records/conversations`
  - 分页返回 direct conversation 列表。
  - 支持 `characterId`、`dateFrom`、`dateTo`、`includeHidden`、`page`、`pageSize`、`sortBy`。
- `GET /api/admin/chat-records/conversations/:id`
  - 返回会话元信息与聚合指标。
- `GET /api/admin/chat-records/conversations/:id/messages`
  - 分页读取消息时间线。
  - 支持 `cursor`、`direction=older|newer`、`limit`、`includeClearedHistory`。
- `GET /api/admin/chat-records/conversations/:id/search`
  - 复用现有消息搜索条件。
  - 返回命中消息及其跳转锚点。
- `GET /api/admin/chat-records/conversations/:id/token-usage`
  - 返回该会话的 Token 用量聚合与最近请求记录。

### 6.3 数据来源

首版不建议改表，全部基于现有实体派生：

- `ConversationEntity`
- `MessageEntity`
- `CharacterEntity`
- `AiUsageLedgerEntity`

需要注意两类历史：

- 用户当前“可见历史”：`createdAt > lastClearedAt`
- 数据库内“完整留存历史”：不过滤 `lastClearedAt`

后台应同时支持这两种口径，并在 UI 上明确区分。

### 6.4 性能策略

- 会话列表走分页，不要一次性把所有消息拉进后台页面。
- 会话详情走 cursor 分页，不延续目前“全量加载后截窗”的模式。
- 单会话搜索首版可以先复用现有搜索规则，但要封装在 admin service 内，避免页面直接依赖 `ChatService` 的用户端语义。
- 跨全部会话的全局全文搜索建议放二期，再考虑 SQLite FTS5 或额外索引表。

## 7. 前端方案

### 7.1 需要新增/调整的文件

- `apps/admin/src/routes/chat-records-page.tsx`
- `apps/admin/src/router.tsx`
- `apps/admin/src/components/root-layout.tsx`
- `apps/admin/src/lib/admin-api.ts`

如果要共享面板组件，可再补：

- `apps/admin/src/components/chat-records/`

### 7.2 页面节奏

建议分两步做：

1. 第一轮先把“查得到、看得清、跳得准”做出来
- 会话列表
- 会话详情
- 会话搜索
- Token 联动

2. 第二轮再把“看完能沉淀洞察”补上
- 趋势卡
- 回复效率指标
- 高活跃/低活跃角色对比
- 和 `reply-logic`、`character-runtime` 的联动跳转

## 8. 分阶段实施建议

### Phase 1：打通基础档案页

- 后端建 `chat-records` admin service 和基础接口。
- 前端建 `聊天记录` 页面、列表、详情、搜索。
- 完成从 Token 用量和回复逻辑页的跳转。

完成标准：

- 能稳定查看任意角色的 direct conversation。
- 能在后台搜索某个关键词并跳转到对应聊天位置。

### Phase 2：补产品分析视角

- 增加近 7/30 天消息趋势。
- 增加主动消息占比、附件占比、回复时延。
- 增加按角色排序和高频活跃会话筛选。

完成标准：

- 产品和运营能不翻全量聊天，也先看出“谁最活跃、谁最沉默、谁最费 Token、谁回复最慢”。

### Phase 3：扩展样本池

- 接入群聊记录。
- 接入公众号/服务号消息。
- 增加导出能力：Markdown / JSON。

完成标准：

- 管理后台不只看单聊，还能回看更复杂的世界互动样本。

### Phase 4：做高阶洞察

- AI 自动摘要单会话阶段变化。
- AI 聚类“高频需求 / 高频投诉 / 高频无聊话题 / 触发失败样本”。
- 支持人工标记“值得复盘”的样本。

这阶段再考虑是否新增表，例如：

- `AdminConversationInsightNote`
- `AdminConversationTag`

首版不建议提前建。

## 9. 我建议的优先级

如果只做一轮最值得的版本，我建议优先级是：

1. `聊天记录列表 + 会话详情 + 会话搜索`
2. `会话维度 Token 用量联动`
3. `基础行为指标`
4. `群聊 / 导出 / AI 摘要`

原因很直接：

- 真正能支撑产品迭代的第一步，不是花哨分析，而是先拿到可回看的真实样本。
- 有了稳定样本池，后面的 Prompt 调整、角色调优、场景优化才不会凭感觉。

## 10. 验收标准

- 后台能按角色快速定位到对应单聊。
- 能检索到某段历史并跳回上下文。
- 能区分“当前可见历史”和“清空前历史”。
- 能看到每个会话的基础消息结构和 Token 成本。
- 能从聊天记录页跳到回复逻辑页和角色运行逻辑台。
- 不引入多用户语义，不破坏单世界约束。
