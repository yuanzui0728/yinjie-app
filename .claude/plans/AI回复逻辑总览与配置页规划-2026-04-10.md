# AI回复逻辑总览与配置页规划

## 1. 目标
- 在管理后台提供一个单独页面，把“当前项目里 AI 人实际如何回复”完整、真实、可追踪地展示出来。
- 页面展示的不是产品文案版解释，而是运行时真实链路：状态门控、上下文窗口、prompt 拼装、记忆注入、路由判定、模型调用参数、写回动作。
- 同一页面内完成核心配置修改，避免角色画像、运行时参数、记忆、状态和调试入口散落在多个页面。
- 首版先解决“看清楚真实逻辑”和“修改现有可编辑项”，第二阶段再把现在散落在代码里的硬编码常量收口成可配置项。

## 2. 现状结论

### 2.1 当前真实回复链路已经存在，但没有统一视图
- 单聊主链路：
  - `chat.gateway -> chat.service.sendMessage() -> ai.classifyIntent() -> ai.generateReply() -> message save`
- 群聊主链路：
  - `group.service.sendOwnerMessage() -> triggerAiReplies() -> ai.generateReply()`
- 朋友圈 / 广场动态 / 摇一摇 / 场景相遇 / 主动提醒：
  - 都是复用 `ai.generateReply()` 或 `ai.generateMoment()`，但各自的触发条件、延迟、概率、上下文构造方式不同。
- 问题不在“没有逻辑”，而在“逻辑分散在 gateway / service / scheduler / prompt-builder / narrative 里，后台没有总览页”。

### 2.2 影响回复的关键输入分散在三类地方
- 角色实体顶层字段：
  - `isOnline`
  - `currentActivity`
  - `activeHoursStart` / `activeHoursEnd`
  - `activityFrequency`
  - `momentsFrequency`
  - `feedFrequency`
- `profile` JSON：
  - `basePrompt`
  - `systemPrompt`
  - `traits`
  - `expertDomains`
  - `identity`
  - `behavioralPatterns`
  - `cognitiveBoundaries`
  - `reasoningConfig`
  - `memory`
- 运行时派生数据：
  - 会话可见消息历史
  - `lastClearedAt`
  - 世界快照
  - narrative arc
  - 世界主人自定义 API Key / base
  - 分类路由结果

### 2.3 当前“可编辑”和“真实生效”之间存在断层
- 角色页能改大部分 `profile`，但看不到这些字段最终如何组装成 prompt。
- 主 App 只能看到少量角色信息，看不到真实上下文窗口、记忆、路由判定。
- 许多实际影响回复的常量仍是硬编码：
  - 状态延迟文案和等待时长
  - 上下文窗口公式
  - `temperature` / `max_tokens`
  - 群聊回复概率
  - 记忆压缩触发阈值
  - 主动提醒调度时间
  - narrative 进度阈值
- 结果是：
  - 能改角色，但不知改动是否真的生效。
  - 能看到部分状态，但不知道状态如何影响回复。
  - 真正最关键的“effective prompt / effective history / effective branch”完全不可见。

### 2.4 当前页面最缺的是“实际链路快照”和“草稿预演”
- 用户想要看的不是静态 schema，而是：
  - 这个角色此刻为什么会慢回 / 不慢回
  - 这条消息当前会走单聊还是临时拉群
  - 最终送给模型的 system prompt 是什么
  - 实际截取了哪几条历史
  - 最近记忆摘要是什么
  - narrative 现在处在哪个阶段
- 现有后台没有任何页面能回答这些问题。

## 3. 页面定位
- 页面应放在 `apps/admin`，而不是主 App。
- 推荐新路由：`/reply-logic`
- 推荐页面名称：
  - 中文：`AI回复逻辑`
  - 英文副标题：`Runtime Reply Control Plane`
- 这是一个“单页控制台”，不是一个简单表单页。
- 单页内允许使用顶部锚点、分段卡片、右侧抽屉和局部折叠，但不再拆成多个二级页面。

## 4. 页面目标拆解

### 4.1 首要目标：把“真实逻辑”看清楚
- 展示实际链路图，而不是产品示意图。
- 所有展示数据都来自后端运行时快照接口，不允许前端自己拼一套解释版逻辑。
- 每一段数据都标明来源：
  - `seed`
  - `admin_saved`
  - `runtime_derived`
  - `system_config`
  - `env_fallback`
  - `hardcoded`

### 4.2 第二目标：在同页修改
- 同页修改角色画像、记忆、状态、活跃时间、专家领域、prompt 字段。
- 同页修改全局 provider / model / 若干运行时参数。
- 保存前能先看“改动后的运行快照”，避免盲改。

### 4.3 第三目标：支持调试和预演
- 支持输入一条候选用户消息，模拟：
  - 当前状态门控
  - intent 分类
  - 是否转临时群聊
  - 最终 prompt
  - 最终窗口消息
  - 实际模型参数
- 可选执行一次真 preview，但应和“只做编排模拟”分开。

## 5. 页面信息架构

### 5.1 顶部总览栏
- 展示世界级运行信息：
  - 当前 provider endpoint
  - 当前 model
  - API key 来源
  - 世界快照时间 / 季节 / 天气 / 节日
  - scheduler 最近一次状态更新时间
  - 当前选中对象：角色 / 会话 / 模拟输入
- 快捷操作：
  - 刷新快照
  - 仅预演
  - 执行真实 preview
  - 保存改动

### 5.2 左侧对象选择区
- 选择维度：
  - 按角色查看
  - 按会话查看
  - 按群聊查看
- 支持筛选：
  - 默认角色
  - 模板角色
  - 已成为好友角色
  - 最近活跃会话
- 选择对象后，主内容区切到对应快照。

### 5.3 主内容区六个核心板块

#### A. 实际链路
- 用时间线或流程卡展示当前对象会走哪条链：
  - 单聊直回
  - 忙碌/睡眠提示后延迟回复
  - 临时拉群
  - 正式群聊随机回复
  - 主动提醒
- 每一节点展示：
  - 判定条件
  - 当前命中结果
  - 数据来源

#### B. 当前有效输入
- 展示本次回复真正依赖的输入值：
  - `currentActivity`
  - `isOnline`
  - `activeHours`
  - `expertDomains`
  - `basePrompt`
  - `systemPrompt`
  - `traits`
  - `coreMemory`
  - `recentSummary`
  - `forgettingCurve`
  - `worldContext`
  - `lastChatAt`
  - `conversation type`
  - `participants`
  - `lastClearedAt`

#### C. Effective Prompt
- 展示 `PromptBuilderService` 组装后的每个段落：
  - identity
  - personality_and_tone
  - behavioral_patterns
  - cognitive_boundaries
  - internal_reasoning
  - collaboration_routing
  - memory
  - current_context
  - group_chat
  - rules
- 支持两种视图：
  - 分段结构视图
  - 最终拼接文本视图
- 支持 diff：
  - 当前已保存版本
  - 草稿版本

#### D. 上下文窗口
- 展示会话历史如何被截取：
  - 可见历史总数
  - `lastClearedAt`
  - 当前 `forgettingCurve`
  - 计算出的窗口大小
  - 实际送入模型的最后 N 条消息
- 每条消息标明：
  - 来源用户 / 角色 / 系统
  - 是否因为清空历史被排除
  - 附件是否退化成占位文本
- 必须明确告诉操作者：
  - “这几条会进模型”
  - “这几条不会进模型”

#### E. 记忆与关系推进
- 展示：
  - `memorySummary`
  - `coreMemory`
  - `recentSummary`
  - 最近一次压缩时机
  - 下次压缩触发条件
  - narrative progress
  - milestones
- 提供操作：
  - 手工编辑 `coreMemory`
  - 手工编辑 `recentSummary`
  - 一键重新压缩当前会话生成 `recentSummary`
  - 一键清空 `recentSummary`

#### F. 分支决策与调度常量
- 展示现有硬编码规则：
  - 睡眠 / 工作 / 通勤提示
  - 睡眠延迟 12-22 秒
  - 忙碌延迟 8-15 秒
  - 群聊回复概率：
    - `high = 0.7`
    - `normal = 0.4`
    - `low = 0.2`
  - 群聊回复延迟 5-30 秒
  - 记忆压缩条件：`history.length % 10 === 0`
  - narrative 阈值：4 / 8 / 12 / 18 / 24
  - 主动提醒：每天 20:00
- 首版先可见。
- 第二阶段再把其中一部分升级为可编辑。

### 5.4 右侧配置抽屉
- 分三组：
  - `角色配置`
  - `回复运行配置`
  - `记忆与调试`
- 保持同页，不跳转到 Character Editor。
- 但数据保存尽量复用现有角色接口和 provider 接口，减少重复表单后台。

## 6. 页面里的“可看”和“可改”边界

### 6.1 首版必须能看清楚的内容
- 当前状态如何决定延迟回复。
- 当前消息会走哪条分支。
- 当前角色画像最终如何变成 prompt。
- 当前会话真实进模型的历史窗口。
- 当前记忆和 narrative 处于什么状态。
- 当前 provider / model / key 来源。
- 当前项目里仍是硬编码、尚未配置化的常量。

### 6.2 首版必须能直接改的内容
- 角色顶层字段：
  - `name`
  - `relationship`
  - `expertDomains`
  - `triggerScenes`
  - `isOnline`
  - `activeHoursStart`
  - `activeHoursEnd`
  - `activityFrequency`
  - `currentActivity`
- `profile` 字段：
  - `basePrompt`
  - `systemPrompt`
  - `traits`
  - `identity`
  - `behavioralPatterns`
  - `cognitiveBoundaries`
  - `reasoningConfig`
  - `memorySummary`
  - `memory.coreMemory`
  - `memory.recentSummary`
  - `memory.forgettingCurve`
- 全局运行配置：
  - provider endpoint
  - provider model
  - provider api style
  - provider key 占位状态

### 6.3 第二阶段再开放编辑的内容
- `generateReply()` 里的全局 `temperature`
- `generateReply()` 里的 `max_tokens`
- 上下文窗口公式参数
- 群聊回复概率映射
- 状态提示文案与延迟范围
- 记忆压缩阈值
- 主动提醒 cron 时间
- narrative milestone 阈值

### 6.4 不建议首版就放开的内容
- 直接显示真实 API Key 明文
- 任意修改历史消息正文
- 任意修改 narrative 原始记录
- 在前端手写 prompt 模板 DSL

## 7. 关键产品原则

### 7.1 “实际逻辑页”必须来自后端快照，而不是前端复制逻辑
- 页面要看清“实际逻辑”，就不能让前端自己重写一份 `if/else`。
- 必须在后端新增一个 `ReplyLogicInspectorService`，复用真实运行代码生成快照。
- 页面只消费快照 DTO。

### 7.2 只读真相和可编辑草稿必须并存
- 页面要同时展示：
  - 当前已生效版本
  - 正在编辑但未保存的草稿版本
- 草稿修改后应立即支持“重新计算快照”，但不必立刻落库。

### 7.3 先做“观测 + 预演”，再做“全面配置化”
- 若一上来就把全部硬编码改成系统配置，改动面会过大。
- 正确顺序：
  - 先建立统一快照和总览页
  - 再逐步把硬编码抽成配置项

## 8. 后端设计

### 8.1 新增统一的回复逻辑快照服务
- 建议文件：
  - `api/src/modules/ai/reply-logic-inspector.service.ts`
  - 或 `api/src/modules/admin/reply-logic.admin.service.ts`
- 责任：
  - 基于真实角色、会话、世界状态和候选消息，计算当前实际回复快照。
  - 不直接生成业务消息，不修改数据库，默认只做 inspect。

### 8.2 推荐新增 DTO
- `packages/contracts/src/reply-logic.ts`
- 建议结构：
  - `ReplyLogicOverview`
  - `ReplyLogicCharacterSnapshot`
  - `ReplyLogicConversationSnapshot`
  - `ReplyLogicSimulationRequest`
  - `ReplyLogicSimulationResult`
  - `ReplyLogicRuntimeConfig`

### 8.3 推荐新增 admin-only 接口
- `GET /api/admin/reply-logic/overview`
  - 获取全局运行摘要、provider 摘要、scheduler 摘要、全局常量摘要。
- `GET /api/admin/reply-logic/characters/:characterId`
  - 获取某个角色的有效画像、状态、记忆、默认 prompt 快照。
- `GET /api/admin/reply-logic/conversations/:conversationId`
  - 获取某个会话的有效上下文窗口、prompt、narrative、分支信息。
- `POST /api/admin/reply-logic/simulate`
  - 输入角色 / 会话 / 候选用户消息 / 草稿配置覆盖，返回模拟结果。
- `PATCH /api/admin/reply-logic/runtime-config`
  - 修改首版允许编辑的全局运行参数。
- `POST /api/admin/reply-logic/characters/:characterId/rebuild-memory`
  - 基于当前可见历史重新压缩记忆。

### 8.4 尽量复用现有接口，不重复造轮子
- 角色保存优先复用：
  - `PATCH /api/characters/:id`
- provider 保存优先复用：
  - `PUT /api/system/provider`
- 新接口只解决“总览快照、模拟、重建记忆、运行配置化”这类现有接口没有语义的能力。

### 8.5 需要抽离的共用逻辑
- 为避免“inspect 和真实运行不一致”，建议把以下逻辑从现有 service 中抽 helper：
  - 上下文窗口计算
  - 历史消息可见性过滤
  - 会话历史重建
  - system prompt 组装
  - 当前状态门控判定
  - intent 分类前后的分支摘要
- 推荐新增：
  - `reply-logic.constants.ts`
  - `reply-logic.helpers.ts`

## 9. 前端页面设计

### 9.1 新增路由
- `apps/admin/src/routes/reply-logic-page.tsx`
- `apps/admin/src/router.tsx`
- `apps/admin/src/components/root-layout.tsx`

### 9.2 页面骨架
- 顶部：
  - 页面标题
  - 运行摘要 chip
  - 保存 / 预演 / 刷新按钮
- 左栏：
  - 角色 / 会话选择器
- 中栏：
  - 实际链路
  - prompt
  - 上下文窗口
  - 记忆与 narrative
  - 硬编码常量
- 右栏：
  - 配置表单
  - 草稿 diff
  - 调试输入框

### 9.3 页面交互建议
- 默认选中一个角色，先让页面可打开可看。
- 切换到会话模式后，新增“候选消息”输入框。
- 所有编辑都先进入草稿态。
- 点击“重新计算”后刷新中心面板的 effective snapshot。
- 点击“保存”后再走真实接口。

### 9.4 页面上的三种视图模式
- `当前真实状态`
- `草稿预演状态`
- `真实模型预览`
- 三者必须明确区分，避免用户把“模拟结果”误以为“已上线生效”。

## 10. 推荐实施阶段

### Phase 1：只读总览页
- 新增 admin 路由和页面壳。
- 后端新增 reply-logic overview / character snapshot / conversation snapshot。
- 页面先把真实链路、prompt、上下文窗口、记忆、常量完整展示出来。
- 这一阶段不做保存，只做观察和模拟。

### Phase 2：把现有可编辑项并入同页
- 角色字段直接在右栏编辑。
- provider 配置在右栏编辑。
- 记忆手工编辑、重建、清空并入同页。
- 这一阶段仍不碰大规模配置化。

### Phase 3：加入“候选消息模拟”
- 输入一条候选消息。
- 展示：
  - 状态门控是否命中
  - intent 分类结果
  - 是否会转临时群聊
  - 会邀请哪些角色
  - 最终 effective prompt
  - 最终历史窗口
- 可选再执行一次真实 preview。

### Phase 4：把关键硬编码升级为配置项
- 引入运行时配置实体或 `system_config` 键值。
- 先收口：
  - 回复 temperature
  - max_tokens
  - 群聊概率
  - 延迟区间
  - 记忆压缩阈值
- 页面同页编辑并立即反映到快照。

## 12. 风险与注意事项
- 现在有不少逻辑散落在 `private` 方法和硬编码常量里，首版若不先抽 helper，inspect 结果容易和真实链路漂移。
- 当前聊天主链路里 API key 来源优先级与系统 provider 配置有历史差异，这个页面必须明确展示“当前实际使用来源”，不能只展示系统配置。
- 正式群聊、临时拉群、朋友圈评论、广场动态评论虽然都复用 `generateReply()`，但触发条件不同；首版页面应优先聚焦“单聊 + 正式群聊 + 临时拉群”三条主线，其他链路放成扩展卡片。
- `currentActivity` 与 `isOnline` 目前并不是一套严格一致的状态机，页面应如实展示，而不是替项目虚构统一语义。

## 13. 验收标准
- 打开一个页面后，能直接回答“这个角色为什么现在这样回”。
- 能看到最终 effective prompt，而不是只看到 profile 原始 JSON。
- 能看到真正进入模型的上下文窗口，而不是整段会话列表。
- 能看到当前记忆、压缩规则、narrative 进度和调度规则。
- 能在同页修改角色画像、状态、记忆和 provider 配置。
- 修改后无需离开页面，就能重新计算并看到新的 effective snapshot。
- 页面展示的关键运行数据来自后端真实快照，不是前端手写推断。
