日期：2026-04-11

范围：`apps/app` 的 Web 手机版运行路径

## 任务定义

- 目标不是做一轮“顺手修几个点”，而是把 Web 手机版按真实使用路径拆成可以连续执行的批次，逐批清掉：
  - 空白页 / 死路由 / 不可达入口
  - 返回链路错位 / 错误兜底 Tab / 刷新后回错页
  - 隐藏桌面 Tab 路由误用
  - 会话、搜索、二级页之间的状态不同步
  - 输入区、附件、语音、键盘顶起、滚动定位等移动端专属交互 bug
- 本轮只覆盖 Web 手机版，不扩散到：
  - 桌面 Web 专属工作区问题
  - Android / iOS 原生壳差异
  - 后端接口语义重构
- 本轮输出必须是“可直接执行的工程计划”，不是产品层脑暴。

### 已经修过的手机版高影响问题

- `发现` 首页缺失 `场景相遇` 入口
- `通讯录` 缺失 `星标朋友` 入口
- `公众号主页 / 文章 / 服务号消息 / 订阅号消息` 返回链路错位
- `新的朋友` 从 `通讯录` 进入却回到 `消息`
- `设置 -> 协议文档` 返回会丢失设置上下文
- `发现` / `通讯录` 多个二级页硬编码回父 Tab，破坏真实浏览历史
- 若干入口把手机版带到隐藏桌面 Tab：`/tabs/moments`、`/tabs/feed`、`/tabs/mini-programs`
- `搜一搜` 不记录来源页，返回默认掉回 `消息`

### 这意味着什么

- 剩余问题大概率不是“编不过”，而是“页面能打开，但在手机 Web 实际使用时逻辑不闭环”。
- 后续执行重点必须从编译错误转到：
  - 路由来源与回退链路
  - 页面复用下的本地状态稳定性
  - 手机端专属输入、键盘、滚动和附件能力

## 完成定义

满足以下条件才算本轮完成：

1. 手机版四个主入口 `消息 / 通讯录 / 发现 / 我` 的二级页都能进入、返回、刷新恢复，不出现明显错页。
2. 不再存在“从手机版入口进入，却被带去隐藏桌面 Tab”的错误路径。

4. 输入区、语音、附件、键盘顶起、消息定位等移动端专属能力不出现确定性失效。
5. 每个修复批次都做小步提交。
6. 每个批次结束时 `typecheck`、`lint`、`build` 保持通过。

## bug 分级口径

### P0

- 页面打不开
- 点操作直接报错
- 返回后进入错误页面且无恢复路径
- 真实主路径断掉
- 输入、附件、语音等核心能力完全不可用

### P1

- 来源页回退错误
- 状态刷新后回滚
- 搜索、提醒、深链跳到错误页面
- 二级页虽然能用，但会把用户带离当前上下文

### P2

- 文案与实际状态不一致
- 轻微布局错位
- 提示延迟或局部交互卡顿

- 收口手机版最核心的聊天闭环

执行内容：

提交建议：

### 批次 3：输入区、附件、语音、键盘

目标：

- 收口手机 Web 最容易出现的“能看不能发”问题

- 检查 `use-keyboard-inset` 与各类面板叠加时的布局
- 检查浏览器不支持语音/通知/剪贴板时的降级逻辑

- 手机 Web 下至少图片、文件、文本输入稳定可用
- 键盘弹起不遮挡发送入口，不把消息面板推飞
- 不支持能力时给出正确降级，而不是静默失效

- `fix(app): stabilize mobile composer and keyboard flow`

### 批次 4：发现、内容流、小游戏、小程序

- 收口 `发现` 体系的实际使用路径

### 批次 5：状态同步与边界清理

目标：

- 把前四批修完后剩下的状态漂移和脏状态继续收口
