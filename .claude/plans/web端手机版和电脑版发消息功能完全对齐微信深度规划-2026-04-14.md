# Web 端手机版和电脑版发消息功能完全对齐微信深度规划

## 任务目标

- 以当前 `apps/app` 的聊天发送链路为基础，把“发消息”能力重构成接近微信稳定心智的输入体验。
- 这次规划覆盖：
  - 单聊
  - 群聊
  - 移动端消息输入区
  - 桌面端消息输入区
  - 发送态、失败态、引用态、附件态、语音态、表情态的 UI / UX
- 这次规划不直接要求后端大改，优先用现有接口和现有消息契约收口前端体验。
- 不生成测试文件，实施阶段以手工验证矩阵为主。

## 范围边界

### 本次纳入

- 文本发送
- 换行与发送快捷键
- 回复 / 引用
- 表情发送
- 图片 / 文件 / 名片 / 位置发送
- 语音消息发送
- 移动端加号面板
- 桌面端工具栏
- 发送中的状态反馈
- 发送失败后的反馈与补救

### 本次不纳入

- 真正的微信支付链路：红包、转账
- 真正的实时语音 / 视频通话重做
- 后端消息协议大规模改版
- 输入法层面的平台级兼容 Hack
- 和公众号 / 订阅号专属消息输入区完全统一

## 微信式体验基线

## 1. 先定义什么叫“像微信”

不是简单抄颜色或圆角，而是抄下面这套稳定心智：

- 默认路径永远是“打开会话 -> 直接输入 -> 直接发出”。
- 输入区第一优先级永远是文本，不是工具。
- 工具入口只在需要时出现，不抢正文输入的注意力。
- 文本、语音、表情、附件在视觉上都属于同一个消息输入系统，而不是多个并列功能。
- 输入区的状态切换必须可预期：
  - 文本
  - 语音
  - 表情
  - 更多
  这四类模式之间切换时，始终只有一个主模式。
- 用户不需要理解技术细节，只需要理解：
  - 现在是在输入文字
  - 现在是在按住说话
  - 现在是在选表情
  - 现在是在选更多功能
- 发送后的反馈要轻，不要把输入区炸成工作台。

## 2. 微信式发消息的核心准则

### 2.1 主路径最短

- 进入聊天页后，光标、输入区、发送区的视觉关系必须一眼成立。
- “发一条文字消息”必须是全产品最短操作路径。
- 输入框、发送键、最近消息底部之间不能有多余 UI 噪音。

### 2.2 模式互斥

- 文本输入模式
- 语音消息模式
- 表情模式
- 加号模式

这四个模式必须是互斥的，不能叠很多层。

### 2.3 反馈轻量

- 正在发送：轻提示，不打断输入。
- 发送失败：明确可重试，但不要把整个输入区染成错误界面。
- 引用态：只在输入区上方保留一条很薄的 reply bar。
- 语音态：用强状态，但结束后尽快回到普通输入区。

### 2.4 移动端与桌面端一致的是心智，不是一比一排版

- 移动端像微信手机版：
  - 底部细条输入栏
  - 左侧语音切换
  - 右侧表情 / 加号 / 发送
- 桌面端像微信电脑版：
  - 白色输入卡片
  - 下方工具图标列
  - 多行输入
  - 固定发送按钮
- 两端不要求布局一样，但必须共享同一套发送逻辑和状态规则。

## 当前差距判断

## 1. 移动端差距

### 1.1 输入主路径已经接近，但还不够“微信味”

- 当前移动端主条是：
  - 语音切换
  - 输入框 / 按住说话
  - 表情
  - 发送或加号
- 这条主路径方向是对的。
- 但细节上仍偏“Web 组件拼装感”：
  - 输入条圆角、边框、按钮尺寸和微信不够一致
  - 表情 / 加号 / 语音切换三个按钮的视觉权重太平均
  - 输入区与消息列表之间的贴合感不足

### 1.2 语音消息过于“面板化”

- 当前 `mobile-speech-input-sheet.tsx` 是一个较重的底部大浮层。
- 微信的语音消息心智更接近：
  - 按住说话按钮是主交互
  - 录音反馈是操作态，不是一个完整新页面
- 现在这个大浮层在视觉上更像“AI 语音工具”，不够像普通聊天语音发送。

### 1.3 加号面板功能排序不够微信

- 当前加号面板保留了：
  - 相册
  - 拍摄
  - 视频通话
  - 位置
  - 红包
  - 转账
  - 收藏
  - 名片
  - 文件
  - 语音通话
- 这在“功能数量”上很全，但不符合微信主路径优先级。
- 问题不在功能有没有，而在：
  - 第一屏排序
  - 是否保留未接入能力占位
  - 占位是否压缩了真实高频能力

### 1.4 发送反馈偏散

- 当前发送相关反馈分散在：
  - InlineNotice
  - 底部“正在发送...”
  - 语音识别完成提示
  - 加号面板不可用 fallback 提示
- 这些反馈本身合理，但缺少统一分层：
  - 哪些属于输入条内反馈
  - 哪些属于全局 toast
  - 哪些属于一次性面板说明

## 2. 桌面端差距

### 2.1 当前桌面输入区功能是够的，但视觉和节奏不像微信

- 当前桌面工具按钮是“图标 + 中文标签”的按钮条。
- 当前输入区更像“富工具编辑器”，不是微信式聊天输入卡片。
- 当前视觉重点落在工具，而不是文本编辑区。

### 2.2 桌面发送区还不够稳定

- 发送按钮、展开态、语音输入提示之间已经有很多状态。
- 但桌面主路径还没有收敛成：
  - 看一眼就知道在哪打字
  - 看一眼就知道怎么发送
  - 看一眼就知道附件入口在哪

### 2.3 收藏 / 截图 / 语音输入都做了，但层级没分好

- 这些能力应该是“主输入卡片下面的工具能力”。
- 现在更像一组并列功能键。
- 微信电脑版的感觉是：
  - 主体是输入
  - 工具只是辅助

## 3. 技术差距

### 3.1 输入状态机缺少显式定义

- 当前状态散落在 `chat-composer.tsx` 内多个 `useState`。
- 例如：
  - `stickerPanelOpen`
  - `plusPanelOpen`
  - `mobileSpeechSheetOpen`
  - `attachmentDraft`
  - `desktopPlusMenuOpen`
  - `desktopEditorExpanded`
- 这导致行为上虽然可用，但很难保证多个模式切换时的细节永远正确。

### 3.2 发送能力契约不够统一

- `ChatComposerAttachmentPayload` 当前支持：
  - image
  - file
  - voice
  - contact_card
  - location_card
- 但消息列表已经支持更多展示型 attachment。
- 例如 `note_card` 发送目前是独立链路，不是输入系统的一部分。
- 这会导致“发消息系统”和“聊天内容系统”之间存在一段边界断裂。

### 3.3 输入组件拆分不够

- 后续如果直接继续在 `ChatComposer` 上加规则，很容易把每次微信化都做成风险改动。
- 必须先把移动端与桌面端发送 UI 从同一个巨型组件里拆出稳定边界。

## 目标产品定义

## 1. 移动端目标

### 1.1 底部输入条结构

- 左：
  - 语音 / 键盘模式切换
- 中：
  - 文本输入区或按住说话区
- 右：
  - 表情
  - 文本为空时显示加号
  - 文本非空时显示发送

这条结构保持不变，但要进一步微信化：

- 高度更稳定
- 图标更统一
- 输入框与按钮的视觉密度更贴近微信
- 底部安全区与键盘弹起时的贴边效果更稳

### 1.2 模式切换规则

- 文本模式：
  - 焦点进入输入框
  - 自动收起表情和加号面板
- 语音模式：
  - 自动收起表情和加号面板
  - 不再出现“文本输入框 + 语音 UI 并存”
- 表情模式：
  - 自动切回文本模式
  - 自动收起加号面板
- 加号模式：
  - 自动切回文本模式
  - 自动收起表情面板

### 1.3 语音消息交互

- 目标是更像微信，而不是更像 AI 工具。
- 交互改成两层：
  - 主层：输入条中的“按住说话”按钮
  - 状态层：轻量浮层 / 悬浮反馈
- 不建议继续保留当前这种信息量很重的完整底部大卡片作为默认语音发送界面。

### 1.4 加号面板重排

- 第一屏只保留高频、真实可用的能力：
  - 相册
  - 拍摄
  - 文件
  - 位置
  - 名片
  - 收藏
- 占位能力如：
  - 红包
  - 转账
  - 语音通话
  - 视频通话
 需要降权。

推荐策略：

- 默认第一屏不放“待接入”入口。
- 若必须保留微信式入口位：
  - 放到第二屏
  - 或放在“更多能力”次级页
  - 不影响真实发送功能的首屏效率

### 1.5 发送反馈

- 文本发送：
  - 保持无打断
- 附件发送：
  - 在输入区上方给轻量进度提示
- 失败：
  - 用轻提示 + 消息级失败标记
- 语音转文字：
  - 若只是语音消息，不出现“识别完成”提示
  - 若是语音转写输入，才保留“插入输入框”确认

## 2. 桌面端目标

### 2.1 桌面输入卡片

- 输入区改成微信电脑版风格的白色卡片式承载。
- 结构拆成三层：
  - 顶部：回复条 / 附件草稿 / 错误提示
  - 中部：多行输入区
  - 底部：左侧工具图标 + 右侧发送区域

### 2.2 工具列

- 改成纯图标，不显示常驻中文标签。
- 建议顺序：
  - 表情
  - 收藏
  - 图片
  - 文件
  - 截图
  - 语音输入
- 不把通话、聊天记录、聊天详情放进输入区。

### 2.3 展开 / 收起

- 保留当前“展开输入区”的能力。
- 但展开按钮必须弱化成“输入区右上角的一个次级动作”，而不是抢主要视线。
- 默认收起态就应该足够好用，不能逼用户先展开才能舒服发消息。

### 2.4 发送按钮

- 固定在右下角，不因空文本而消失。
- 空文本：
  - 禁用态
- 有文本：
  - 绿色主按钮
- 附件仅发送时：
  - 附件草稿面板内独立发送，不借用文本发送按钮

### 2.5 快捷键

- 保留当前设置项：
  - `Enter 发送`
  - `Ctrl/Cmd + Enter 发送`
- 但 UI 提示不要太吵：
  - 默认弱提示
  - 只有 hover 或聚焦时再明确展示完整提示

## 3. 统一消息类型模型

### 3.1 文本消息

- 默认输入、默认发送。
- 是所有会话类型的第一路径。

### 3.2 表情消息

- 表情面板属于输入系统，而不是附件系统。
- 发送后应立即关闭面板或维持最近使用态，行为与微信一致：
  - 高频连续发时保持面板
  - 单次发送后保持输入上下文不丢

### 3.3 图片 / 文件

- 统一视为附件发送，但要区分：
  - 快速发送
  - 发送前预览 / 裁剪
- 图片优先更接近微信：
  - 选择后先进入轻量预览草稿
  - 可以多图连续发送

### 3.4 语音消息

- 和“语音转写输入”分开。
- 语音消息是即时发送消息类型。
- 语音转写输入只是输入辅助，不应和语音消息混成一个概念。

### 3.5 名片 / 位置 / 收藏 / 笔记

- 这些属于“结构化消息”。
- 视觉上要像微信中的卡片消息，而不是普通文件消息。
- 发送入口可以分散，但最终都要通过统一的 composer-send pipeline 进入消息列表。

## 目标状态机

## 1. 移动端状态机

### 一级状态

- `text`
- `voice`
- `emoji`
- `plus`

### 二级状态

- `replying`
- `uploading`
- `error`
- `recording`

### 规则

- 一级状态同时只能有一个。
- 二级状态可以叠在一级状态上，但不能反过来控制一级状态。
- 例如：
  - `text + replying`
  - `voice + recording`
  - `plus + uploading`

不允许出现：

- `emoji + plus`
- `voice + emoji`
- `plus + voice`

## 2. 桌面端状态机

### 一级状态

- `typing_collapsed`
- `typing_expanded`
- `emoji`
- `favorites`
- `screenshot`

### 二级状态

- `replying`
- `uploading`
- `speech_dictation`
- `error`

### 规则

- 桌面端可以存在“输入态 + 某个工具浮层”。
- 但输入态外只允许一个主工具浮层。
- 截图编辑必须是更高优先级的独占态。

## 交互与排版细则

## 1. 移动端排版细则

### 1.1 底部输入条

- 高度目标：
  - 单行时稳定在微信式紧凑高度
  - 多行时自然拉高，但不超过约 4 行
- 输入框圆角不宜过大，避免像通用 Web search bar。
- 发送按钮宽度固定，不要因为字体或状态抖动。

### 1.2 回复条

- 位置：
  - 输入条上方
- 形态：
  - 一条薄灰条
  - 左侧彩色竖线
  - 中部显示“回复对象 + 摘要”
  - 右侧关闭
- 不要做成卡片。

### 1.3 表情面板

- 默认高度和微信接近，贴底出现。
- 不与加号面板同时出现。
- 最近使用区要成为第一触点。

### 1.4 加号面板

- 网格规则固定。
- 第一屏只放真实可发能力。
- 卡片、名片、位置等次级选择页要统一标题和返回模式。

## 2. 桌面端排版细则

### 2.1 输入卡片

- 外层背景白色
- 轻边框
- 无重阴影
- 与消息区之间保留微信式呼吸感

### 2.2 工具图标

- 只保留图标
- hover 给浅灰底
- active 给更明确底色
- selected 态只用于：
  - 表情面板打开
  - 收藏面板打开
  - 语音输入进行中

### 2.3 编辑区

- 默认显示 1 到 2 行可见区
- 输入增多后自适应
- 展开态改成更高的编辑区
- 右上角展开按钮使用弱图标，不加文字

### 2.4 附件草稿区

- 图片 / 文件 / 截图草稿统一放在输入卡片顶部
- 每种草稿卡片样式统一
- 草稿区承担：
  - 预览
  - 删除
  - 单独发送
- 不让正文输入区承担附件控制逻辑

## 3. 消息列表配套调整

“发消息像微信”不能只改 composer，消息列表也要配合。

### 3.1 自己发送的消息

- 发送中、失败、已发送状态需要有稳定标记。
- 当前已有 optimistic message，但缺少更像微信的消息级反馈。

建议：

- 发送中：
  - 气泡旁弱 loading
- 失败：
  - 红色感叹号 / 重试入口
- 成功：
  - 不额外显示“已发送”

### 3.2 回复消息

- 当前已有 reply quote card。
- 后续需要统一：
  - 回复卡片高度
  - 引用摘要行数
  - 点击跳转原消息的反馈

### 3.3 语音消息

- 当前已支持 voice attachment 渲染。
- 后续要更像微信：
  - 自己发送和他人发送的气泡样式区分更明确
  - 时长、播放态、未播放提示收敛到同一视觉语言

## 技术实施策略

## 1. 先拆，再改样式

不要继续直接在 `chat-composer.tsx` 巨型分支上堆条件。

推荐拆分为：

- `mobile-chat-composer-shell.tsx`
- `desktop-chat-composer-shell.tsx`
- `chat-composer-core-state.ts`
- `chat-composer-send-pipeline.ts`
- `chat-composer-reply-bar.tsx`
- `chat-composer-attachment-draft-strip.tsx`

`chat-composer.tsx` 最终只保留薄封装，不再承载全部 JSX。

## 2. 发送 pipeline 统一

把以下能力统一成一条“发送入口”：

- text
- sticker
- image
- file
- voice
- contact_card
- location_card
- 后续可扩展 note_card

统一 pipeline 的目标不是改协议，而是统一前端行为：

- optimistic 插入
- pending 状态
- 失败处理
- 成功回写
- 清空输入 / 保留面板策略

## 3. 显式建立 composer 状态机

新增明确的 state model，而不是多个布尔值互撞：

- 移动端主模式
- 桌面端主模式
- 当前是否 replying
- 当前是否 uploading
- 当前是否 speech processing

这样后续做微信式行为时，规则会更清楚。

## 4. 消息级发送状态补齐

当前 `useConversationThread.ts` 已经有 optimistic message，但还不够。

需要补：

- 本地消息 status
  - `sending`
  - `sent`
  - `failed`
- 重试机制
- 失败后不自动丢消息

这一步不一定要改后端，只要前端本地模型先补齐即可。

## 现有契约是否够用

结论：现阶段大体够用，先不建议为“像微信”而先改后端。

### 现有后端 / 契约已满足

- 文本发送
- 图片 / 文件 / 语音附件上传
- 表情消息
- 名片 / 位置卡片
- optimistic 依赖的消息回写

### 现有契约的轻微缺口

- 本地发送状态没有明确契约，只能前端自己补
- `note_card` 没进 `ChatComposerAttachmentPayload`
- 群聊和单聊的发送状态模型需要进一步收口

建议：

- 第一轮只做前端状态模型补齐
- 第二轮再决定是否需要调整共享 contracts

## 验收标准

## 1. 移动端

- 单手操作下，发文字、发语音、发表情、发图片四条路径都明显顺手。
- 输入条在空文本 / 有文本 / 多行文本 / 键盘弹起时不抖动。
- 表情、加号、语音三种模式切换时不会互相叠层。
- 回复一条消息时，reply bar 明确但不抢视线。
- 发送失败时，用户知道这条消息没发出去，也知道怎么补救。

## 2. 桌面端

- 第一眼看上去像微信电脑版聊天输入区，而不是工具工作台。
- 工具列是辅助，不压过输入区。
- 发送按钮固定存在，快捷键规则清晰。
- 截图、图片、文件、收藏这些次级能力不破坏主输入路径。

## 3. 一致性

- 单聊和群聊发送规则一致。
- 自己发送的消息在移动端和桌面端的状态表达一致。
- 语音消息和语音转写输入不会混淆。

## 风险与注意点

## 1. 最大风险不是样式，而是状态耦合

- 如果不先拆组件和状态机，后面每次改一处都会冒出新的边界 bug。

## 2. 不能把“功能很多”误当成“像微信”

- 微信感来自主路径短、模式清晰、反馈轻，而不是入口多。

## 3. 不要一轮内同时大改消息列表和输入协议

- 第一优先级是输入体验与发送状态。
- 不要把这次重构扩成消息系统全面重写。

## 最终建议

- 这项改造应该按“先收口发送模型，再做 UI 对齐”的顺序推进。
- 如果直接先改样式，会很快遇到：
  - 模式互撞
  - 发送失败状态无处安放
  - 移动 / 桌面行为不一致
- 最稳的执行顺序是：
  - 先拆 `ChatComposer`
  - 再统一 send pipeline
  - 先做移动端主输入条
  - 再做桌面输入卡片
  - 最后补消息级发送状态与细节抛光

按这个顺序做，最终出来的不是“长得有点像微信”，而是“用起来真的像微信”。
