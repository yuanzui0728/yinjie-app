# Web端手机版和桌面版视频号功能深度规划

## 结论先行

这块能力不该新开一套“短视频系统”。

当前仓库已经有一套可复用底座：
- 后端已有 `feed` 模块与 `FeedPost / FeedComment / UserFeedInteraction`
- `FeedPost` 已带 `mediaType / mediaUrl`
- 桌面端已经给出一级 `视频号` 入口与 `直播伴侣` 工具入口
- 移动端“发现”页已经承载朋友圈、广场动态这类微信式二级能力

所以首版视频号的正确做法是：
- 继续复用 `feed` 域作为视频号内容域
- 给 `FeedPost` 增加“内容面”语义，明确区分：
  - `feed`：广场动态
  - `channels`：视频号
- 移动端把视频号收口到“发现”
- 桌面端把视频号做成独立一级工作区
- 首轮只做消费链路，不先做完整创作后台

当前真实边界：
- `feed` 现在更像“广场动态”
- `channels` 页面仍是占位
- `mediaType / mediaUrl` 在前端几乎没有被真正消费
- 后端没有“按内容面筛选”的接口
- 前端没有沉浸式视频卡片与桌面频道工作区

### 已有前端能力

已有入口：
- `apps/app/src/routes/feed-page.tsx`
- `apps/app/src/routes/discover-feed-page.tsx`
- `apps/app/src/routes/channels-page.tsx`
- `apps/app/src/routes/live-companion-page.tsx`

已有壳层：
- 桌面端左侧一级导航已有 `视频号`
- 桌面端更多菜单已有 `视频号直播伴侣`
- 移动端 `discover-page.tsx` 已经是微信式入口聚合页

当前问题：
- `channels-page.tsx` 还是说明文案占位
- 移动端发现页没有 `视频号` 入口
- `mobile-shell.tsx` 仍保留了“朋友圈”底部 Tab，和当前产品口径冲突

## 产品目标

### 移动端

目标形态对齐微信视频号：
- 不新增底部 Tab
- `发现 -> 视频号`
- 进入后是沉浸式纵向内容流
- 顶部保留频道切换感，首版先做 `推荐`
- 单条内容包含：
  - 视频封面或直接视频播放
  - 作者头像/昵称
  - 标题或文案
  - 点赞 / 评论 / 转发占位
  - 进入作者主页的后续承接位

### 桌面端

目标形态对齐微信电脑版“视频号”频道：
- 左侧继续保持独立一级 `视频号`
- 工作区采用“推荐列表 + 主播放器 + 互动侧栏”的桌面形态
- 不照搬手机整页跳转
- `直播伴侣` 保持工具语义，继续从“更多”进入

## 核心技术决策

### 决策 1：不新建视频号模块，继续复用 `feed`

理由：
- 仓库已经把 `FeedPost` 列为视频号相关实体
- 已有点赞、评论、互动记录可以直接复用
- 首版目标是尽快把 UI/交互跑通，而不是重新抽象社交内容域

### 决策 2：给 `FeedPost` 增加 `surface`

建议新增字段：
- `surface: "feed" | "channels"`，默认 `feed`

这样可以明确拆开两条产品语义：
- 广场动态不再和视频号混同一条流
- 搜索 / 收藏 / AI 互动 / 内容筛选都能按面区分

### 决策 3：首轮不新建“视频号作者”实体

首轮作者仍复用：
- 世界主人
- 世界居民（Character）

理由：
- 当前产品里内容本来就是世界角色驱动
- “AI 生成的内容 / 视频”天然适合挂在角色名下
- 先把内容消费链路跑通，比先建创作者中心更重要

### 决策 4：首轮先做消费，不做完整创作链路

首轮实现：
- 拉流
- 播放
- 点赞
- 评论
- 收藏
- 桌面工作区
- 移动端发现入口
- 示例 AI 内容初始化

延后实现：
- 上传视频
- 编辑封面
- 视频号主页
- 关注流
- 直播中状态
- 真正的创作后台

## 数据与接口规划

### Feed 契约扩展

建议扩展 `packages/contracts/src/feed.ts`：
- `FeedSurface = "feed" | "channels"`
- `FeedPost.surface`
- `CreateFeedPostRequest.surface?`
- `CreateFeedPostRequest.mediaType?`
- `CreateFeedPostRequest.mediaUrl?`

### Feed 查询扩展

建议扩展 `GET /api/feed`：
- `GET /api/feed?page=1&limit=20&surface=feed`
- `GET /api/feed?page=1&limit=20&surface=channels`

默认值建议：
- 默认 `surface=feed`

原因：
- 现有调用大多数语义都指向“广场动态”
- 新视频号页面显式传 `channels`，边界更清晰

### 示例内容初始化

首轮需要自动灌入一批“AI 视频号示例内容”，否则页面会空。

建议策略：
- 当请求 `surface=channels` 且库内没有视频号内容时
- 后端自动写入 3 到 5 条示例内容
- 内容作者优先用默认角色
- 内容文案明确标记为 AI 生成世界内容
- 媒体先使用公开可访问的演示视频地址占位

这样做的价值：
- 首次启动就能看到真实视频流
- 不依赖手工录入后台
- 未来接入真实 AI 生成视频后，只需要替换内容来源

## 前端落地规划

### 1. 移动端

新增入口：
- `discover-page.tsx` 增加 `视频号`
- 新增路由 `discover/channels`

页面结构：
- 顶部栏：返回 + `视频号`
- 内容区：纵向沉浸式卡片流
- 单卡布局：
  - 上层为视频/封面
  - 下层为作者、文案、标签
  - 右侧动作带：赞、评、收藏、转发占位

交互要点：
- 视频默认静音播放，支持手动开声
- 页面滚动时尽量保持“一屏一主内容”的阅读节奏
- 评论先以内嵌预览 + 快捷输入为主，不做弹层评论区

### 2. 桌面端

替换当前占位 `channels-page.tsx`。

建议桌面工作区结构：
- 左栏：
  - 频道介绍
  - 推荐作者/热门标签
  - 视频号列表缩略导航
- 中栏：
  - 当前选中视频的大播放器
  - 标题、作者、发布时间
  - 点赞、评论、收藏
- 右栏：
  - 评论预览
  - 下一个推荐
  - `直播伴侣` 快捷入口

这样做的原因：
- 更符合桌面端“浏览 + 停留 + 切换”的效率模式
- 不会强行把手机竖屏体验搬到宽屏上

### 3. 收藏与搜索接入

桌面视频号不是孤立页面，应该接入已有桌面能力。

建议接入：
- `desktop-favorites-storage.ts` 新增 `channels` 分类
- `favorites-page.tsx` 增加视频号统计与筛选
- `search-page.tsx` 增加视频号搜索域

## AI 内容策略

短期策略：
- 首批内容由固定示例数据提供
- 文案口径全部按“AI 角色发布的世界内容”撰写

中期策略：
- 接入 AI 视频生产流水线后
- 只需要把 `channels` 内容源替换成真实生成结果
- UI 与交互层不需要推倒重来

## 分阶段实施

### Phase 1：打通双端入口与首屏内容

范围：
- 新增移动端 `discover/channels`
- 桌面端 `channels-page` 从占位改为真实工作区
- `FeedPost` 增加 `surface`
- `GET /api/feed` 支持 `surface`
- 自动初始化视频号示例内容

交付标准：
- 手机和桌面都能看到视频号真实内容
- 点赞、评论、收藏可用

### Phase 2：桌面生态整合

范围：
- 搜索接入视频号
- 收藏接入视频号
- 视频号内容卡片的桌面筛选与推荐增强

### Phase 3：创作与直播

范围：
- 世界主人发布视频号内容
- AI 自动生成视频号内容
- 直播中状态
- `直播伴侣` 真正接入创作能力

## 风险与控制

### 风险 1：广场动态和视频号内容混淆

控制方案：
- 必须引入 `surface`
- 不再用 `mediaType` 硬猜频道归属

### 风险 2：移动端视频体验过重

控制方案：
- 首轮不做复杂手势
- 使用原生 `video` 能力
- 保证浏览器可稳定播放即可

### 风险 3：没有真实 AI 视频素材

控制方案：
- 先注入示例 AI 内容
- 把“内容来源”与“UI/交互”彻底解耦

## 最终建议

这次实现应该按下面的顺序推进：

1. 先补 `feed surface`，把广场动态和视频号正式拆开
2. 再补移动端“发现 -> 视频号”与桌面视频号工作区
3. 最后把收藏 / 搜索 / 直播伴侣入口接进来

这样能保证：
- 改动边界清晰
- 不会把现有广场动态搞乱
- 首版就能看到接近微信的信息架构和浏览节奏
