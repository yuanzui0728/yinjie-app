# 管理后台角色制造与回复生活逻辑规划

## 1. 目标

- 在隐界管理后台把两类能力做成明确产品能力，而不是散落在角色编辑、回复调试、调度器代码里的局部能力。
- 第一类能力：`角色工厂`
  - 能看到 AI 角色“被制造出来”的完整逻辑。
  - 能配置制造过程里的各阶段输入、默认值、覆盖关系、发布结果。
- 第二类能力：`角色运行逻辑台`
  - 能看到每个角色“当前如何回复、当前如何生活”的真实运行逻辑。
  - 能直接配置会影响回复与生活行为的参数，并看到保存后的真实生效快照。

## 2. 现状结论

### 2.1 已有基础

- 管理后台已经有 `角色名册` 和 `角色编辑页`：
  - `apps/admin/src/routes/characters-page.tsx`
  - `apps/admin/src/routes/character-editor-page.tsx`
- 管理后台已经有 `AI 回复逻辑` 页面：
  - `apps/admin/src/routes/reply-logic-page.tsx`
- 后端已经有回复逻辑快照接口：
  - `GET /api/admin/reply-logic/overview`
  - `GET /api/admin/reply-logic/characters/:id`
  - `GET /api/admin/reply-logic/conversations/:id`
- 这一页已经能看到：
  - provider 来源
  - prompt 分段
  - effective prompt
  - 上下文窗口
  - 状态门控
  - narrative / memory 摘要
  - 角色大部分 `profile` 配置，并支持保存

### 2.2 关键缺口

- 现在并没有一个真正的“角色制造流程”产品概念。
  - 当前角色来源只有两种：
    - 启动时默认 seed
    - 管理后台手工 CRUD
  - 但后台看不到：
    - 角色从哪个来源来
    - 哪些字段来自 seed / 手工编辑 / 默认推导
    - 发布前后的版本差异
    - 某个角色是如何从“设定草稿”变成“运行时角色”的
- 现在也没有一个完整的“生活逻辑台”。
  - 生活逻辑分散在：
    - `Character` 顶层字段
    - `profile`
    - `SchedulerService`
    - `reply-logic.constants.ts`
  - 只能看到部分结果，无法看到完整调度链路，也无法集中配置。

### 2.3 当前最重要的技术约束

- `api/src/database/seed.ts` 会在启动时清理非默认角色。
  - 这意味着“后台新建角色/制造角色”目前不是稳定产品能力。
  - 如果不先改这里，角色工厂做出来也会在重启后失真。
- 回复与生活逻辑中一批关键规则仍然是硬编码：
  - 睡眠/忙碌提示文案
  - 延迟时间
  - 群聊回复概率
  - 历史窗口公式
  - 记忆压缩阈值
  - 叙事进度阈值
  - 主动提醒调度时间
- `currentActivity` 当前会被调度器周期性覆盖。
  - 如果后台要支持“可配置”和“可解释”，就必须区分：
    - 当前值
    - 值的来源
    - 是否人工覆盖

## 3. 产品结构建议

- 不建议把这两个诉求继续塞进现有一个超大 `/reply-logic` 页面。
- 建议保留三个层次：

### 3.1 世界级控制台

- 保留现有 `/reply-logic`
- 角色定位：
  - 世界级回复调试台
  - 看某个会话、某个角色在此刻真实走哪条回复链路
  - 看全局 provider、全局规则、会话分支

### 3.2 角色工厂

- 新增路由：
  - `/characters/$characterId/factory`
- 角色定位：
  - 看这个角色是如何被定义、组装、发布出来的
  - 管理“制造逻辑”

### 3.3 角色运行逻辑台

- 新增路由：
  - `/characters/$characterId/runtime`
- 角色定位：
  - 看这个角色当前的回复逻辑、生活逻辑、叙事逻辑
  - 管理“当前行为逻辑”

### 3.4 角色名册作为入口页

- 在 `角色名册` 中，每个角色卡片提供三个入口：
  - `编辑基础资料`
  - `制造逻辑`
  - `运行逻辑`

## 4. 角色工厂规划

## 4.1 页面要回答的问题

- 这个角色从哪里来？
- 这个角色最初是谁定义的？
- 它的身份、关系、语气、边界、记忆、生活策略分别由谁决定？
- 哪些值是默认模板，哪些是人工覆盖？
- 当前已发布版本和草稿版本差异是什么？
- 发布后会落到 `Character` 的哪些字段，如何影响运行时？

## 4.2 页面信息架构

### A. 来源与发布状态

- 角色来源：
  - `default_seed`
  - `manual_admin`
  - `template_clone`
  - `ai_generated`
- 当前状态：
  - `draft`
  - `published`
  - `archived`
- 最近发布人、最近发布时间、当前版本号

### B. 制造流程阶段图

- 建议把角色制造拆成 6 个阶段：
  - `身份与关系`
  - `能力域与边界`
  - `语气与行为`
  - `记忆底稿`
  - `生活策略`
  - `发布映射`
- 每个阶段展示：
  - 输入值
  - 有效值
  - 来源
  - 是否有覆盖
  - 可编辑表单

### C. 字段映射视图

- 明确显示“工厂草稿”如何映射到运行时实体：
  - `Character.name`
  - `Character.relationship`
  - `Character.expertDomains`
  - `Character.profile.basePrompt`
  - `Character.profile.systemPrompt`
  - `Character.profile.traits`
  - `Character.profile.identity`
  - `Character.profile.behavioralPatterns`
  - `Character.profile.cognitiveBoundaries`
  - `Character.profile.reasoningConfig`
  - `Character.profile.memory`
  - `activityFrequency`
  - `momentsFrequency`
  - `feedFrequency`
  - `activeHoursStart` / `activeHoursEnd`
  - `triggerScenes`

### D. 版本与发布记录

- 展示：
  - 草稿版本
  - 最近一次发布版本
  - 发布说明
  - 差异摘要
- 支持：
  - 恢复到某个版本
  - 对比两个版本

## 4.3 页面可配置项

- 身份：
  - 名称
  - 关系
  - 关系类型
  - 职业
  - 背景
  - 核心动机
  - 世界观
- 能力与边界：
  - expertDomains
  - expertiseDescription
  - knowledgeLimits
  - refusalStyle
- 语气与行为：
  - speechPatterns
  - catchphrases
  - emotionalTone
  - responseLength
  - emojiUsage
  - workStyle
  - socialStyle
  - taboos
  - quirks
- 记忆底稿：
  - memorySummary
  - coreMemory
  - recentSummary 初始值
  - forgettingCurve 默认值
- 生活策略：
  - activityFrequency
  - activeHours
  - momentsFrequency
  - feedFrequency
  - triggerScenes

## 4.4 数据模型建议

- 建议新增两张表，而不是继续把“制造过程”混进 `Character.profile`。

### `CharacterBlueprint`

- 角色工厂主表，保存当前草稿与发布状态。
- 建议字段：
  - `id`
  - `characterId`
  - `sourceType`
  - `status`
  - `recipe`
  - `publishedRevisionId`
  - `createdAt`
  - `updatedAt`

### `CharacterBlueprintRevision`

- 保存每次修改或发布的快照。
- 建议字段：
  - `id`
  - `blueprintId`
  - `version`
  - `recipe`
  - `summary`
  - `createdAt`

## 4.5 首版接口建议

- `GET /api/admin/characters/:id/factory`
- `PATCH /api/admin/characters/:id/factory`
- `POST /api/admin/characters/:id/factory/publish`
- `GET /api/admin/characters/:id/factory/revisions`
- `POST /api/admin/characters/:id/factory/revisions/:revisionId/restore`

## 5. 角色运行逻辑台规划

## 5.1 页面要回答的问题

- 这个角色现在为什么这么回？
- 这个角色现在为什么在线 / 离线？
- 这个角色现在为什么处于 `working` / `sleeping` / `free`？
- 它下一次可能发朋友圈、发视频号、触发场景相遇、主动提醒的依据是什么？
- 这套逻辑里哪些是角色级配置，哪些是全局规则，哪些是运行时推导？

## 5.2 页面结构建议

- 建议按 Tab 拆成四块：

### A. 回复逻辑

- 直接复用并下沉现有 `/reply-logic` 的角色能力：
  - 状态门控
  - prompt 分段
  - effective prompt
  - 上下文窗口
  - provider 来源
  - 记忆注入
  - narrative 快照

### B. 生活逻辑

- 这是当前后台缺失最多的部分，要新增完整视图：
  - 在线状态来源
  - 当前活动来源
  - 活跃时间窗口
  - 日常活动策略
  - 朋友圈生成策略
  - 视频号生成策略
  - 场景触发策略
  - 主动提醒策略
- 每块都要展示：
  - 当前配置
  - 真实生效值
  - 规则来源
  - 最近一次执行结果
  - 下一次可能执行时机

### C. 记忆与叙事

- 展示：
  - `memorySummary`
  - `coreMemory`
  - `recentSummary`
  - forgettingCurve
  - narrative progress
  - milestones
  - 最近一次压缩/推进时机
- 支持：
  - 手工编辑
  - 手工触发压缩
  - 手工清空近期摘要

### D. 规则与调试

- 展示角色级与全局级规则：
  - 睡眠 / 忙碌文案
  - 延迟范围
  - 群聊回复概率
  - 历史窗口公式
  - narrative 阈值
  - 主动提醒时间
- 支持预演：
  - 输入一条候选消息
  - 看本次会走哪条分支
  - 看有效 prompt / 历史窗口 / 结果来源

## 5.3 生活逻辑必须新增的解释能力

- 每个运行时值都要显示 `source`：
  - `character_saved`
  - `blueprint_published`
  - `scheduler_derived`
  - `system_config`
  - `env_fallback`
  - `hardcoded`
  - `manual_override`
- 尤其是这几个字段必须解释来源：
  - `isOnline`
  - `currentActivity`
  - `activityFrequency`
  - `momentsFrequency`
  - `feedFrequency`
  - `activeHoursStart` / `activeHoursEnd`
  - `triggerScenes`

## 5.4 角色级可配置项

- 回复逻辑：
  - `basePrompt`
  - `systemPrompt`
  - `traits`
  - `identity`
  - `behavioralPatterns`
  - `cognitiveBoundaries`
  - `reasoningConfig`
  - `memory`
- 生活逻辑：
  - `isOnline` 手工覆盖
  - `currentActivity` 手工覆盖
  - `activityFrequency`
  - `momentsFrequency`
  - `feedFrequency`
  - `activeHoursStart`
  - `activeHoursEnd`
  - `triggerScenes`

## 5.5 全局级可配置项

- 先把现有硬编码规则迁到可配置存储，再开放后台编辑：
  - 睡眠提示文案
  - 忙碌提示文案
  - 睡眠延迟范围
  - 忙碌延迟范围
  - 群聊回复概率
  - 群聊回复延迟
  - 记忆压缩阈值
  - 历史窗口基数与范围
  - narrative milestones
  - 主动提醒执行时间

## 5.6 首版接口建议

- `GET /api/admin/characters/:id/runtime`

## 6. 必须先补的底座

## 6.1 停止 seed 清理自定义角色

- 当前 `api/src/database/seed.ts` 的行为会删除非默认角色。
- 必须改为：
  - 只校准默认角色
  - 不再删除手工创建或工厂发布的角色
- 否则角色工厂没有真实意义。

## 6.2 把硬编码常量移出代码路径

- 当前 `reply-logic.constants.ts` 是只读硬编码。
- 应改为：
  - 代码内保留 fallback 默认值
  - 运行时优先读 `SystemConfig`
  - 后台编辑 `SystemConfig`

## 6.3 给活动状态增加“来源”和“覆盖模式”

- 建议新增运行时语义：
  - `activitySource`
  - `activityOverrideUntil`
  - `isOnlineOverride`
- 否则后台刚修改，调度器下一个周期就把值改回去，用户会误解成“保存没生效”。

## 6.4 给调度器补可观测性

- 需要能看到：
  - 最近一次运行时间
  - 最近一次命中结果
  - 最近一次生成的 moments / channels / proactive message
  - 下次计划运行时间

## 7. 分阶段落地建议

## Phase 0：先把底座修正

- 目标：
  - 让“角色制造”和“角色运行配置”成为稳定能力
- 交付：
  - seed 不再清理自定义角色
  - 回复/生活硬编码常量迁到可配置读取链路
  - 活动状态支持来源与覆盖模式
  - 调度器补最近运行快照

## Phase 1：把“角色运行逻辑台”做出来

- 目标：
  - 先满足“能看、能配每个角色目前的回复和生活逻辑”
- 交付：
  - 新增 `/characters/$characterId/runtime`
  - 现有 `/reply-logic` 的角色快照能力复用到角色维度
  - 新增生活逻辑卡片与配置区
  - 新增全局运行规则配置

## Phase 2：把“角色工厂”做出来

- 目标：
  - 满足“能看、能配角色被制造出来的全过程逻辑”
- 交付：
  - 新增 `CharacterBlueprint` / `CharacterBlueprintRevision`
  - 新增 `/characters/$characterId/factory`
  - 支持草稿、发布、回滚、版本对比

## Phase 3：补调试和预演

- 目标：
  - 让运营可以在改配置前先预演
- 交付：
  - 候选消息预演
  - 生活逻辑预演
  - 工厂发布前 diff 预览
  - 保存后回显真实生效快照

## 8. 我对产品拆分的建议

- 你的两个诉求本质上不是一个页面，而是两个层次：
  - `角色工厂` 解决“这个角色是怎么被做出来的”
  - `角色运行逻辑台` 解决“这个角色现在怎么活、怎么回”
- 现有 `/reply-logic` 不应该废掉。
  - 它适合作为“世界级回复调试台”继续保留。
  - 但不适合继续独自承载“角色制造逻辑”。

## 9. 推荐实施顺序

1. 先修 seed 和硬编码配置读取链路。
2. 先做 `角色运行逻辑台`，因为这部分已有 `/reply-logic` 基础，可最快见效。
3. 再做 `角色工厂`，把当前角色编辑页升级为“工厂草稿 + 发布”模型。
4. 最后补预演、版本 diff 和回滚。

## 10. 预期结果

- 管理后台最终会有三种明确视图：
  - `角色名册`：看有哪些角色
  - `角色工厂`：看角色如何被制造出来
  - `角色运行逻辑台`：看角色当前如何回复、如何生活
- 这样你的两个要求都能满足，而且不会继续把“设计期逻辑”和“运行时逻辑”混在一个表单页里。
