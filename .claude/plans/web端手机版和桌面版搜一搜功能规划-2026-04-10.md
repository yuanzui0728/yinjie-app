# Web端手机版和桌面版搜一搜功能深度规划

## 任务目标

- 在 `apps/app` 内把现有零散的“局部搜索框”升级成统一的微信式 `搜一搜` 能力。
- 同时覆盖：
  - Web 手机版
  - Web 桌面版
- 保持现有单世界模型与现有接口边界，不先引入新的独立搜索服务。
- 首期直接复用现有业务数据：
  - 会话
  - 全量聊天记录
  - 联系人 / 世界角色
  - 公众号
  - 朋友圈
  - 广场动态

## 结论先行

当前仓库并不是“没有搜索”，而是“只有局部过滤，没有统一搜一搜”。

现状问题有三个：

- `apps/app/src/routes/search-page.tsx` 只在桌面布局下渲染，移动端进来是空页面。
- `消息`、`通讯录` 等页面的顶部搜索框，本质只是当前列表过滤，不是微信那种统一搜索入口。
- 现有桌面搜索页只索引：
  - 会话标题 / 最后一条消息
  - 联系人与角色
  - 公众号
  - 朋友圈
  - 广场动态

它没有真正覆盖“聊天记录”这一微信搜一搜里最核心的内容。

所以本次实现的正确方向不是继续补一个搜索输入框，而是做三件事：

1. 抽出双端共享的搜索索引层。
2. 把 `/tabs/search` 改造成真正可用的微信式搜一搜页面。
3. 把移动端 `消息 / 通讯录` 顶部搜索入口接到统一搜一搜，而不是继续停留在局部过滤。

## 产品判断

### 1. 首期不做后端统一搜索接口

不建议一上来改后端做：

- `GET /api/search`
- 或单独全文检索服务

原因：

- 当前数据规模可控
- 已有接口已经足够拼出首版搜一搜
- 先做前端聚合能快速对齐微信交互路径
- 风险更小，不会把本次需求扩成搜索基础设施项目

### 2. 首期必须补“全局聊天记录检索”

如果继续只搜索会话标题和最后一条消息，这个页面虽然叫 `搜一搜`，但用户感知会很弱。

所以首期应该在用户输入关键词后，额外拉取所有会话消息：

- `direct` 会话 -> `getConversationMessages`
- `group` 会话 -> `getGroupMessages`

然后建立全局聊天记录索引。

这比直接上服务端搜索更重，但在当前产品阶段是可接受的，因为：

- 只有进入 `搜一搜` 且输入关键词时才触发
- 当前单世界模型下数据总量有限
- 能显著提升“像微信”的程度

### 3. 手机版和桌面版不应该做成同一套皮肤

数据层应共享，交互壳层必须分开。

建议：

- 共享：
  - 搜索数据聚合
  - 搜索结果分组
  - 搜索历史
  - 关键词匹配 / 高亮 / 预览文本生成
- 分端：
  - 页面布局
  - 默认入口提示
  - 分类栏形态
  - 结果承载方式

## 双端目标形态

### 手机版搜一搜

目标是贴近微信手机端搜一搜：

- 独立页面进入，不放在底部 Tab
- 顶部为返回按钮 + 搜索框
- 空状态展示：
  - 最近搜索
  - 快捷搜索分类
  - 推荐可搜内容说明
- 输入关键词后展示：
  - 最先展示 `聊天记录`
  - 再展示 `联系人`
  - 再展示 `公众号`
  - 再展示 `朋友圈`
  - 再展示 `广场动态`
- 每个分组只先展示前几条，并支持“查看更多”

手机版入口策略：

- `消息` 页顶部搜索框点击后进入 `/tabs/search`
- `通讯录` 页顶部搜索框点击后进入 `/tabs/search`
- 进入页面后自动聚焦搜索输入框
- 如果是带关键词跳转，直接写入初始关键词

### 桌面版搜一搜

目标是贴近微信电脑端的“左分类 + 中结果 + 右辅助面板”的工作区节奏。

建议布局：

- 左侧：
  - 搜索范围概览
  - 分类切换
  - 最近搜索
- 中间：
  - 搜索框
  - 结果列表
  - 按分类聚合展示
- 右侧：
  - 当前检索统计
  - 快捷入口
  - 热门搜索提示文案

桌面端保留：

- 左侧导航里的 `搜一搜`
- `Cmd/Ctrl + K`

同时把当前页面从偏展示型 `DesktopEntryShell` 改成更像微信工作区的结构，不再只是“介绍面 + 结果面”。

## 搜索范围设计

### A. 聊天记录

这是首期最重要的一层。

数据来源：

- `getConversations(baseUrl)`
- 对每个会话再调用：
  - `getConversationMessages(conversationId, baseUrl)`
  - `getGroupMessages(groupId, baseUrl)`

结果形态：

- 标题：发送者名称
- 副标题：会话标题 + 时间
- 摘要：命中的聊天片段
- 点击行为：
  - 单聊 -> `/chat/$conversationId#chat-message-$messageId`
  - 群聊 -> `/group/$groupId#chat-message-$messageId`

匹配字段：

- `message.text`
- `message.senderName`
- 会话标题

### B. 联系人 / 世界角色

数据来源：

- `getFriends`
- `listCharacters`

结果形态：

- 联系人优先于普通世界角色
- 展示：
  - 备注名 / 角色名
  - 关系、地区、来源、标签等辅助信息
- 点击进入 `/character/$characterId`

### C. 公众号

数据来源：

- `listOfficialAccounts`

匹配字段：

- 公众号名称
- handle
- 简介
- 最近文章标题 / 摘要

点击进入：

- `/official-accounts/$accountId`

### D. 朋友圈

数据来源：

- `getMoments`

匹配字段：

- 作者名
- 正文
- 评论文本
- 位置

点击进入：

- 首期仍跳 `/discover/moments` 或 `/tabs/moments`
- 不做“精确定位到某条朋友圈”的复杂路由

### E. 广场动态

数据来源：

- `getFeed`

匹配字段：

- 作者名
- 正文
- 评论预览

点击进入：

- `/discover/feed` 或 `/tabs/feed`

## 分类与排序策略

### 分类

统一分类建议收口为：

- `all`
- `messages`
- `contacts`
- `officialAccounts`
- `moments`
- `feed`

### 排序

#### 默认全局排序

优先级建议：

1. 聊天记录命中
2. 联系人命中
3. 公众号命中
4. 朋友圈命中
5. 广场动态命中

同分类内再按：

- 时间倒序
- 或更强命中优先

#### 分类页排序

- 聊天记录：按消息时间倒序
- 联系人：好友优先，再按拼音/名称
- 公众号：服务号与订阅号混排，命中优先
- 朋友圈 / 广场动态：按发布时间倒序

## 交互细节设计

### 搜索历史

首期使用 `localStorage` 即可。

建议新增独立存储：

- `yinjie.app.search-history`

记录字段：

- `keyword`
- `usedAt`

策略：

- 去重
- 最近 8 条
- 支持单条删除
- 支持全部清空

### 默认状态

无关键词时：

- 不展示大面积空白
- 展示最近搜索
- 展示“可搜索的内容类型”
- 桌面端同时展示数据规模统计

### 关键词高亮

首期只做单关键词高亮，不做复杂分词。

高亮场景：

- 聊天记录片段
- 联系人标题
- 公众号标题
- 内容流摘要

### 防抖与性能

- 输入值用 `useDeferredValue`
- 基础索引查询可常驻
- 全局聊天记录索引仅在有关键词时拉取
- 每次渲染避免重复做全量字符串拼接

- `use-search-index.ts`
  - 统一拉取数据并构建索引

### 页面职责

- `apps/app/src/routes/search-page.tsx`
  - 只负责：
    - 识别桌面/移动布局
    - 读取 / 写入搜索词
    - 渲染对应端页面
- `apps/app/src/features/search/mobile-search-page.tsx`
  - 移动端微信式搜一搜
- `apps/app/src/features/search/desktop-search-page.tsx`
  - 桌面端微信式搜一搜

### 入口接线

要把以下页面的“搜索框”从本地过滤改成统一搜一搜入口：

- `apps/app/src/routes/chat-list-page.tsx`
- `apps/app/src/routes/contacts-page.tsx`

替换策略：

- 移动端：
  - 保持视觉上还是搜索框
  - 但点击后跳转 `/tabs/search`
- 桌面端：
  - 本来已有独立 `搜一搜` 导航，不强制改当前工作区内的局部过滤逻辑

## 推荐分阶段实施

### Phase 1：共享索引层

- 新增 `features/search` 目录
- 抽离结果类型、历史存储、匹配工具
- 抽离基础数据聚合 hook
- 抽离“全局聊天记录索引”查询

### Phase 2：重做 `/tabs/search`

- 移动端页面真正可用
- 桌面端页面重构为工作区布局
- 支持空状态、历史记录、分类筛选、结果高亮

### Phase 3：接通移动端入口

- 消息页顶部搜索框跳统一搜一搜
- 通讯录顶部搜索框跳统一搜一搜
- 保持原页面内其它逻辑不被打断

## 风险与控制

### 风险 1：全局聊天记录搜索请求数上升

风险来源：

- 每个会话都要额外拉消息

控制方式：

- 只有输入关键词时才启用
- 使用 React Query 缓存
- 后续如果数据量增大，再补后端统一搜索接口

### 风险 2：搜索结果跳转路径不完全精准

例如：

- 朋友圈
- 广场动态

当前没有“跳到某一条内容”的精确路由。

控制方式：

- 首期先跳对应列表页
- 后续如有需要再加锚点或详情页

### 风险 3：消息页和通讯录页失去“即时本地过滤”

这是本次有意做的产品选择，因为微信式搜一搜优先级更高。

控制方式：

- 把统一搜一搜做得足够快
- 搜索页回跳路径明确
- 聊天内、群聊内的“查找聊天记录”继续保留

## 本次落地边界

本次会直接实现：

- 双端统一搜一搜
- 全局聊天记录检索
- 搜索历史
- 双端微信式页面布局
- 移动端消息页 / 通讯录页搜索入口接线

本次不做：

- 后端统一全文检索接口
- 搜索推荐服务
- 公众号文章全文检索
- 朋友圈 / 广场动态精确定位到单条内容
- 语音搜索

## 最终实施判断

这次最合理的实现方式是：

- 不改后端结构
- 不新增实体
- 不新增接口
- 直接在 `apps/app` 做统一聚合搜索层

这样能最快把 Web 手机版和桌面版都拉到“像微信搜一搜”的可用水平，而且改动集中、风险可控、回退成本低。
