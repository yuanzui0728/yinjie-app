# 云世界实例编排与唤起平台深度规划

## 1. 先给结论

这次要做的，不是“云世界申请单优化”，而是把 `apps/cloud-api` 从“手机号查表 + 人工录地址”升级成真正的世界控制面：

- 新用户输入手机号并验证成功后，平台自动创建一个新的单世界实例。
- 老用户再次登录时，平台找到他原来的世界；如果实例已休眠，就自动唤起。
- `apps/app` 永远只做一件事：请求“进入我的世界”。
- 世界内部数据仍然只存在各自的 `api/` 实例里，云平台只管生命周期，不托管世界业务数据。

这份规划基于当前仓库真实代码来定稿，目标不是继续写概念，而是把“下一步要改哪些文件、先做哪一段、哪些地方必须先拍板”讲清楚。

## 2. 当前代码真实现状

### 2.1 现有实现已经做到的事

- `apps/cloud-api/src/auth/cloud-auth.controller.ts`
  - 已有 `/cloud/auth/send-code` 与 `/cloud/auth/verify-code`
- `apps/cloud-api/src/cloud/cloud.controller.ts`
  - 客户端现在只有 `/cloud/me/world`
  - 没有“创建或唤起世界”的统一入口
- `apps/cloud-api/src/cloud/cloud.service.ts`
  - 现在本质是“按手机号查 `CloudWorld`/`CloudWorldRequest`”
  - `createWorldRequest()` 仍然是新用户主路径
  - `syncWorldForRequest()` 还是人工录入 `apiBaseUrl` 后把状态切到 `active`
- `apps/cloud-api/src/entities/cloud-world.entity.ts`
  - 只有 `phone/name/status/apiBaseUrl/adminUrl/note`
  - 没有实例维度、心跳、任务、失败原因、目标状态
- `apps/cloud-api/src/entities/cloud-world-request.entity.ts`
  - 现在还是“用户申请单”的主数据之一
- `apps/app/src/routes/welcome-page.tsx`
  - 验证码通过后，先 `getMyCloudWorld`
  - 没有世界时允许用户填 `worldName` 提交 `createMyCloudWorldRequest`
  - 有世界且 `status=active` 且 `apiBaseUrl` 存在才允许进入
- `apps/cloud-console`
  - 当前页面与 API 都围绕“申请单 + 世界地址人工维护”
  - 还是运营录地址台，不是世界管理台

### 2.2 当前设计里的核心问题

- `CloudWorldStatus` 在 `packages/contracts/src/cloud.ts` 里同时服务于“申请单状态”和“世界状态”，语义已经混了。
- 新用户主流程依然是“提交申请”，这和“自动开世界”目标正冲突。
- App 现在必须理解“申请”和“进入世界”两套路径，这会让未来自动编排非常别扭。
- `apiBaseUrl` 现在像人工填空字段，不是由平台稳定产出的世界入口。
- 自动休眠如果只看 `cloud-api` 的访问日志会误判，因为用户进入世界后，后续交互都直接打到世界实例，不再经过 `cloud-api`。

## 3. 这次必须先拍板的 7 个设计决定

### 3.1 主入口必须改成统一的 `resolve world access`

普通用户不再走：

- “查有没有世界”
- “没有就提交申请”

而是统一走：

- `POST /cloud/me/world-access/resolve`

语义永远是：

- “我要进入我的世界”

平台自己决定：

- 首次创建
- 唤起旧世界
- 返回排队中
- 返回失败中
- 返回已可进入

### 3.2 `CloudWorldRequest` 不再做普通用户主流程

推荐调整：

- `CloudWorldEntity` 成为手机号到世界的唯一权威映射。
- `CloudWorldRequestEntity` 降级为“人工工单 / 异常工单 / 审计记录”。
- 普通手机号首次登录时，默认直接进入自动编排，而不是先创建申请单。

也就是说：

- “用户申请开世界”从主流程里移除
- “人工工单”保留给失败、风控、人工审核、异常补录

### 3.3 世界地址必须稳定，不要跟着 VM 动态 IP 变化

推荐直接拍板：

- 每个世界都有一个唯一 `slug`
- `apiBaseUrl` 使用稳定子域名，例如 `https://{slug}.worlds.xxx`
- VM 重建、重启、唤起时，只更新 provider/反向代理目标，不改对外入口

否则会出现两个问题：

- 每次唤起都要重新回填地址
- App 侧缓存、控制台展示、运维排障都会变得非常痛苦

### 3.4 新用户首次不应该再要求先填 `worldName`

当前 `welcome-page.tsx` 让用户在创建申请单前输入 `worldName`，这在自动编排场景里可以拿掉。

更合理的做法：

- 平台先生成一个临时显示名，例如 `隐界世界-手机号后四位`
- 进入世界后，继续沿用已有世界主人命名流程
- 世界真正的人设命名、资料初始化仍在各自世界实例内完成

### 3.5 自动休眠不能只看“有人点过进入”

因为用户进入世界后，和世界持续交互走的是世界实例自己的 `api/`。

所以自动休眠至少要等以下能力具备后再开：

- VM 内 agent 或世界 `api` 能上报 `lastInteractiveAt`
- 或能上报 `activeClientCount`
- 或至少能上报“最近是否仍有前端连接/活动”

因此建议：

- Phase 1 先不做自动休眠，只做手动休眠/唤起
- Phase 2 再补 runtime activity 回调后开启自动休眠

### 3.6 `cloud-api` 首期继续单实例进程内跑 worker

当前 `apps/cloud-api` 是 SQLite + 单实例 NestJS。

在这个前提下，首期最稳的做法不是马上上 Redis/独立队列，而是：

- 任务表落 DB
- 用 NestJS 内部 worker 轮询 `WorldLifecycleJob`
- 做 lease / 超时 / 重试

原因：

- 和当前技术栈最贴
- 上手快，能最快把主链路跑通
- 等平台日请求量上来，再拆出独立 worker 或外部队列

### 3.7 真正要管理的是“世界生命周期”，不是“世界内部数据”

云平台只存：

- 手机号与世界归属
- 实例资源标识
- 生命周期状态
- 任务历史
- 心跳与失败原因

云平台不存：

- 世界主人资料
- API Key
- 角色数据
- 聊天记录
- 朋友圈/视频号数据

这些仍然只存在各自世界实例里的 `api/` 数据库。

## 4. 目标控制面结构

### 4.1 逻辑分层

1. `apps/app`
   - 手机号验证
   - 调 `resolve`
   - 轮询 access session
   - 拿到 `apiBaseUrl` 后进入对应世界

2. `apps/cloud-api`
   - 身份入口
   - 世界归属管理
   - 生命周期任务管理
   - provider 抽象与执行
   - VM/运行时回调
   - 管理后台接口

3. `世界 VM`
   - 运行 `api/`
   - 挂独立数据库/数据盘
   - 启动 world agent 或 bootstrap agent
   - 向 `cloud-api` 上报启动、心跳、健康、活动

4. `provider`
   - VM 创建/开机/关机/删除/查状态
   - 卷与快照
   - 未来可对接 Proxmox / 自建 KVM / 公有云

### 4.2 推荐模块拆分

建议把 `apps/cloud-api/src/` 从现在的扁平结构拆成：

- `auth/`
- `worlds/`
- `world-access/`
- `orchestration/`
- `providers/`
- `runtime-callbacks/`
- `admin/`
- `scheduler/`
- `entities/`

注意：

- 这是结构变化
- 真正落代码时要同步更新 `AGENTS.md`

## 8. 服务端具体逻辑定稿

### 8.1 新用户首次进入

1. App 调 `/cloud/auth/send-code`
2. App 调 `/cloud/auth/verify-code`
3. App 调 `/cloud/me/world-access/resolve`
4. `cloud-api` 按手机号查 `CloudWorld`
5. 如果不存在：
   - 创建 `CloudWorld(status=queued, desiredState=running)`
   - 自动生成 `slug`
   - 自动生成 `displayName`
   - 生成 `callbackToken`
   - 创建 `WorldLifecycleJob(jobType=provision)`
   - 创建 `WorldAccessSession(status=waiting, phase=creating)`
6. worker 消费 `provision` 任务：
   - 调 provider 创建 VM/卷
   - 下发 bootstrap 参数
   - 启动 VM
7. VM 内 agent 启动后回调 `bootstrap/heartbeat`
8. `cloud-api` 执行外部健康检查：
   - `GET {apiBaseUrl}/system/status`
9. 通过后：
   - `CloudWorld.status = ready`
   - `WorldAccessSession.status = ready`
10. App 轮询拿到 `apiBaseUrl`，进入世界

### 8.2 老用户再次进入

1. App 调 `/cloud/me/world-access/resolve`
2. `cloud-api` 查到已有 `CloudWorld`
3. 分情况处理：
   - `ready` 且健康：直接返回 `ready`
   - `sleeping`：创建去重后的 `resume` job，返回 `starting`
   - `starting/creating/bootstrapping`：直接复用当前进度
   - `failed`：返回失败原因
   - `disabled`：阻止进入
4. worker 调 provider 执行开机
5. heartbeat 恢复 + 健康检查通过
6. 状态切回 `ready`
7. App 进入原世界

### 8.3 自动/手动休眠

建议执行顺序：

- 第一阶段：先做手动休眠与手动唤起
- 第二阶段：有 `activity` 信号后再做自动休眠

自动休眠成立条件建议至少满足：

- `activeClientCount = 0`
- `lastInteractiveAt` 超过阈值
- 当前没有进行中的 access session
- 当前没有运行中的重要后台任务

### 8.4 失败修复

如果 world 处于这些异常：

- `ready` 但长时间无 heartbeat
- provider 说已关机，但 world 状态还是 `ready`
- 任务卡死超过 lease

平台应允许自动生成：

- `repair`
- `resume`
- `reconcile`

并把失败链路写入 `WorldEvent`。

## 9. 编排器与 provider 设计

### 9.1 `ComputeProvider` 抽象

首期接口建议：

- `createInstance()`
- `startInstance()`
- `stopInstance()`
- `deleteInstance()`
- `getInstance()`
- `createVolume()`
- `attachVolume()`
- `createSnapshot()`
- `restoreFromSnapshot()`

### 9.2 推荐实现顺序

先后顺序建议非常明确：

1. `MockComputeProvider`
   - 不连真实云
   - 用于把控制面、App 轮询、状态机先跑通

2. `ScriptedProvider`
   - 通过本地脚本或 HTTP bridge 调你真实环境
   - 适合先接自己的虚拟化平台

3. 真正的 `ProxmoxProvider` / 云厂商 provider
   - 等业务流已经跑通再上

原因：

- 你现在要先把“产品逻辑”落下来
- 真正的 VM provider 是外部依赖最多、最容易拖慢进度的部分
- 如果一开始就把产品流和 provider 硬绑，整个工程会很难推进

### 9.3 首期 worker 设计

首期建议：

- `@nestjs/schedule` 或应用内轮询
- 每 2-5 秒扫描一次可执行 job
- 通过 DB lease 防止重复执行
- 为 `provision/resume/suspend` 设置超时

推荐超时：

- `provision`：15 分钟
- `resume`：5 分钟
- `suspend`：5 分钟

## 10. VM 模板与启动契约

### 10.1 VM 内真正需要跑什么

根据当前产品口径，VM 内最小只需要：

- `api/`
- 数据库文件
- world agent

`apps/app` 不是每个世界单独部署的，它是统一业务前端，不应复制进每台 VM。

### 10.2 bootstrap 需要下发的参数

推荐：

- `WORLD_ID`
- `WORLD_CALLBACK_TOKEN`
- `CLOUD_API_BASE_URL`
- `PUBLIC_API_BASE_URL`
- `DATABASE_PATH`
- `PORT`
- `ADMIN_SECRET`
- provider 相关诊断信息

其中：

- `PUBLIC_API_BASE_URL` 正好和当前 `api/.env` 契约兼容
- `DATABASE_PATH` 指向独立数据盘

### 10.3 数据盘策略

建议首期直接定：

- 一世界一独立数据盘
- VM 停机不删盘
- 唤起时继续挂原盘
- 重建时优先挂回原盘或从快照恢复

这决定了“老用户回来的还是原来的世界”。

## 15. 建议的第一刀实现顺序

如果现在就准备开工，我建议按这个顺序来，不要一上来先碰真实云厂商：

1. 改 `packages/contracts/src/cloud.ts` 与 `client.ts`
2. 在 `apps/cloud-api` 新增实体与 `world-access` 模块

这样做的好处是：

- 先把产品主逻辑跑通
- 再把最不确定的外部设施接上
- 一旦 provider 接口定了，后面换 Proxmox/自建 KVM/云厂商都不会打散前面的代码

## 16. 一句话定稿

隐界云世界平台要从“手机号查询 + 人工录入世界地址”的登记系统，升级为“手机号驱动的单世界实例控制面”：

- 新用户自动创建专属 VM
- 老用户自动唤起原 VM
- App 只请求“进入我的世界”
- `cloud-api` 负责创建、唤起、回调、任务、失败恢复和运营控制
- 世界内部业务数据仍然留在各自单世界实例里，不回流云平台
