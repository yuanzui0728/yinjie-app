# 云世界实例编排与唤起平台深度规划

## 1. 目标

### 1.1 业务目标
- 用户在隐界 App 输入手机号后，不再只是查询“是否有人工开通过世界”。
- 新用户：平台自动为其分配并创建独立世界实例，底层表现为一台专属虚拟机。
- 老用户：平台识别其已有世界，若实例已休眠则自动唤起，恢复后继续服务。
- App 侧只关心“现在能不能进世界”，不关心背后是新建、唤起、重试还是排队。

### 1.2 技术目标
- 保持现有 `api/` 继续作为 `1 实例 = 1 世界主人` 的单世界运行时，不改成多租户。
- `apps/cloud-api/` 升级为“控制面 + 编排器”，负责手机号鉴权、世界注册、实例生命周期、任务调度、状态聚合。
- `apps/cloud-console/` 升级为运营控制台，负责查看世界、实例、任务、失败原因、手工干预。
- 为后续接入真实云厂商或自建虚拟化平台保留 provider 抽象层。

### 1.3 非目标
- 不把所有用户塞进同一个 `api/` 服务。
- 不让 App 直接操作云厂商 API。
- 不在首期就做复杂计费系统；先把实例生命周期跑通。

## 2. 当前仓库现状

### 2.1 已有能力
- `apps/cloud-api` 已有：
  - 手机号验证码登录
  - `CloudWorldEntity`
  - `CloudWorldRequestEntity`
  - 客户端查询 `/cloud/me/world`
  - 客户端申请 `/cloud/me/world-requests`
  - 控制台管理 `/admin/cloud/worlds`、`/admin/cloud/world-requests`
- `apps/app/src/routes/welcome-page.tsx` 当前逻辑：
  - 手机号验证后查询云世界
  - 若 `status=active` 且有 `apiBaseUrl`，直接进入世界
  - 若没有世界，只能提交申请，等待人工处理

### 2.2 关键缺口
- 没有自动建机逻辑。
- 没有“老世界休眠后重新唤起”的逻辑。
- `CloudWorldEntity` 只有手机号、名称、状态、地址，缺少实例维度字段。
- 没有实例健康检查、心跳、启动耗时、失败原因、操作日志。
- 当前控制台本质上是“人工录入地址后台”，不是“世界管理平台”。

## 3. 目标架构

### 3.1 总体分层

1. `apps/app`
   - 只负责手机号验证、轮询世界进入状态、拿到 `apiBaseUrl` 后进入具体世界。

2. `apps/cloud-api`
   - 控制面。
   - 负责用户入口编排、世界元数据、任务调度、实例生命周期、管理后台接口、运行时回调。

3. `世界虚拟机`
   - 每个用户一台独立 VM。
   - VM 内运行：
     - `api/`
     - 可选 `apps/admin` 静态资源或独立 admin 服务
     - 守护进程/agent，用于向 `cloud-api` 上报启动与心跳

4. `基础设施 provider`
   - 首期可抽象为 `ComputeProvider` 接口，后面接真实环境：
     - 自建 KVM / Proxmox
     - 公有云 ECS / EC2
     - 其他虚拟化 API

### 3.2 推荐结构调整

`apps/cloud-api/src/modules/`
- `auth`
- `worlds`
- `world-access`
- `orchestration`
- `providers`
- `runtime-callbacks`
- `admin`
- `scheduler`

说明：
- 现有文件还比较扁平，建议下一阶段按模块拆开。
- 这是结构变更；真正落代码时需要同步更新 `AGENTS.md`。

## 4. 核心业务流

### 4.1 新用户首次进入

1. App 输入手机号，调用 `/cloud/auth/send-code`
2. App 输入验证码，调用 `/cloud/auth/verify-code`
3. App 调用新的统一入口接口，例如 `POST /cloud/me/world-access/resolve`
4. `cloud-api` 根据手机号查询世界：
   - 不存在世界记录：创建 `CloudWorld`
   - 状态置为 `creating`
   - 创建 `WorldLifecycleJob(provision)`
5. 编排器消费任务：
   - 从 VM 模板创建新虚拟机
   - 挂载独立数据盘
   - 下发 cloud-init / bootstrap 参数
   - 启动 VM
6. VM 启动后，内部 agent 回调 `cloud-api`
   - 上报 `instanceId`、内网 IP、阶段状态、健康状态
7. `cloud-api` 等待世界 API 健康检查通过
8. `CloudWorld.status` 切为 `ready`
9. App 轮询到 `ready + apiBaseUrl` 后进入世界
10. App 内继续走世界主人命名 / onboarding

### 4.2 老用户回到世界

1. App 仍走 `POST /cloud/me/world-access/resolve`
2. `cloud-api` 查到已有 `CloudWorld`
3. 分情况处理：
   - `ready` 且实例在运行：直接返回可进入地址
   - `sleeping` / `stopped`：创建 `WorldLifecycleJob(resume)`，状态切为 `starting`
   - `starting`：直接返回“正在唤起”
   - `failed`：返回失败原因与重试建议
   - `disabled`：阻止进入
4. 编排器调用 provider 启动对应 VM
5. agent 心跳恢复，健康检查通过
6. 状态回到 `ready`
7. App 继续进入原来的世界

### 4.3 自动休眠

1. `cloud-api` 定时检查 `lastAccessedAt`、`lastHeartbeatAt`
2. 达到空闲阈值，例如 30-120 分钟无活跃访问
3. 创建 `WorldLifecycleJob(suspend)`
4. 通过 provider 停机或保存快照
5. `CloudWorld.status` 切为 `sleeping`

说明：
- 首期建议先做“停止 VM + 保留数据盘”。
- 如果恢复速度不够，再引入“快照恢复”或“预热池”。

## 5. 统一状态机设计

### 5.1 世界状态 `CloudWorld.status`

建议从当前的：
- `pending | provisioning | active | rejected | disabled`

扩展为：
- `requesting`
- `queued`
- `creating`
- `bootstrapping`
- `starting`
- `ready`
- `sleeping`
- `stopping`
- `failed`
- `disabled`
- `deleting`

状态含义：
- `requesting`：手机号已验证，但世界创建请求刚生成
- `queued`：已进入任务队列，等待实际编排
- `creating`：正在创建 VM / 磁盘 / 网络资源
- `bootstrapping`：VM 已起来，但世界服务仍在初始化
- `starting`：老世界正在从休眠中恢复
- `ready`：用户可进入
- `sleeping`：世界存在，但实例当前关闭
- `stopping`：正在休眠/停机
- `failed`：创建或恢复失败，需要自动/人工处理
- `disabled`：运营层面封禁或停用
- `deleting`：回收中

### 5.2 实例电源状态 `CloudInstance.powerState`

- `absent`
- `provisioning`
- `running`
- `stopped`
- `starting`
- `stopping`
- `error`

不要把“世界业务状态”和“虚拟机电源状态”混在一个字段里。

### 5.3 生命周期任务状态 `WorldLifecycleJob.status`

- `pending`
- `leased`
- `running`
- `succeeded`
- `failed`
- `cancelled`
- `deadletter`

说明：
- 平台一定要把“世界”与“任务”分开，否则无法追踪失败链路。

## 8. 进入世界的服务端具体逻辑

### 8.1 `resolveWorldAccess(phone)` 伪逻辑

1. 根据手机号查 `CloudWorld`
2. 若不存在：
   - 创建 `CloudWorld(status=queued, desiredState=running)`
   - 创建 `WorldLifecycleJob(jobType=provision)`
   - 返回 `queued`
3. 若存在且 `status in [queued, creating, bootstrapping, starting]`
   - 返回当前状态，告知客户端轮询
4. 若存在且 `status=ready`
   - 如果实例健康且最近有心跳，直接返回 `ready + apiBaseUrl`
   - 如果心跳丢失，转成 `starting` 并补一个 `repair/resume job`
5. 若存在且 `status=sleeping`
   - 创建去重后的 `resume job`
   - 返回 `starting`
6. 若存在且 `status=failed`
   - 若未超过自动重试阈值，创建 `repair job`
   - 否则返回失败态并展示提示
7. 若存在且 `status=disabled`
   - 直接拒绝进入

### 8.2 去重原则

- 同一世界同一时间只允许一个进行中的生命周期任务。
- `resume` 请求风暴时必须合并。
- 并发多端登录只创建一个世界、不重复开机。

## 9. 编排器设计

### 9.1 执行方式

首选：
- 仍在 `apps/cloud-api` 内先做 DB 驱动的轻量任务队列 + scheduler worker。

后续可升级：
- 拆成独立 worker 进程。

首期不建议直接上复杂 MQ，原因：
- 当前仓库整体仍偏单机/SQLite 阶段
- 先把状态机和 provider 抽象跑通更关键

### 9.2 Worker 周期任务

- `job-dispatcher`
  - 领取 `pending` job
- `world-health-checker`
  - 检查长时间无心跳实例
- `world-idle-reaper`
  - 回收空闲世界到 `sleeping`
- `job-reconciler`
  - 修复卡死的 `running` / `leased` job
- `provider-reconciler`
  - 定时对账云厂商真实实例状态

### 9.3 Provision job 步骤

1. 创建/分配磁盘
2. 按模板创建 VM
3. 写入 bootstrap 参数
4. 启动 VM
5. 等待 heartbeat
6. 健康检查 `GET {apiBaseUrl}/system/status`
7. 写回 `CloudWorld.apiBaseUrl`
8. 切到 `ready`

### 9.4 Resume job 步骤

1. 查询 provider instance
2. 如果已运行，直接走健康检查
3. 如果已停止，启动 VM
4. 等待 heartbeat 恢复
5. 健康检查通过后切 `ready`

### 9.5 Suspend job 步骤

1. 先确认最近没有活跃访问会话
2. 通知世界准备停机
3. 停止 VM
4. 更新 `lastSuspendedAt`
5. 切 `sleeping`

## 13. 安全与边界

### 13.1 身份边界

- 手机号是世界归属主键。
- `1 phone = 1 CloudWorld`
- 不允许一个手机号绑定多个世界。

### 13.2 平台与世界的边界

- `cloud-api` 只负责“找到/创建/唤起世界”。
- 真正的世界主人资料、API Key、角色数据，仍保存在各自世界实例中。
- 不要把世界内部业务数据复制到云平台控制面。

### 13.3 密钥管理

- `callbackToken` 单世界单独生成。
- provider 凭据只保存在平台服务端环境变量或密钥系统。
- App 永远拿不到 provider 凭据。

## 14. 失败恢复策略

### 14.1 自动重试

- `provision`：最多 2-3 次
- `resume`：最多 3-5 次
- `suspend`：失败后告警，但不立即删实例

### 14.2 死信与人工介入

- 超过阈值进入 `deadletter`
- 控制台可手工 `retry` / `rebuild`

### 14.3 对账修复

平台要允许出现这种情况：
- DB 认为实例不存在，但 provider 上其实还在
- DB 认为实例运行中，但 VM 实际已关机

必须有定时 reconcile 任务修复。

## 16. 对当前仓库最关键的设计结论

### 16.1 必须废掉“申请单就是主流程”的思路

如果目标是：
- 新用户自动开世界
- 老用户自动唤起旧世界

那客户端主入口必须从：
- `createWorldRequest`

改成：
- `resolveWorldAccess`

申请单可以保留，但应降级为：
- 审计记录
- 特殊人工审批流程
- 异常补录流程

### 16.2 `CloudWorld` 必须升级成“世界控制记录”

不能再只是：
- 手机号
- 名称
- 地址

它必须描述：
- 世界当前生命周期
- 绑定哪台 VM
- 最近是否有心跳
- 上次访问时间
- 是否应当运行

### 16.3 一定要把“世界状态”和“任务状态”拆开

否则后续无法回答这些关键问题：
- 这次创建为什么失败
- 当前是不是有唤起任务在跑
- 是世界坏了，还是任务卡死了

## 17. 建议下一步实现顺序

1. 先改 `contracts + entity + API DTO`
2. 再补 `resolve world access` 统一入口
3. 再上 `WorldLifecycleJob` 和本地 worker
4. 再接 VM provider
5. 最后改 App 欢迎页与 cloud-console 运维页

## 18. 一句话方案结论

隐界云世界平台应从“手机号 + 人工登记世界地址”的登记系统，升级为“手机号驱动的单世界实例编排控制面”：
- 新用户自动创建专属 VM
- 老用户自动唤起原有 VM
- App 永远只请求“进入我的世界”
- `cloud-api` 负责把创建、唤起、休眠、失败恢复全部吞掉
