# 游戏中心 Web 端规划

日期：2026-04-10
范围：`apps/app` Web 手机版 + Web 桌面版

## 目标

- 在现有壳层内补齐微信式游戏中心入口与主工作区。
- Web 桌面版承接一级 Tab ` /tabs/games `，形成稳定工作区。
- Web 手机版从“发现”进入游戏中心二级页，沿用微信式信息架构。
- 首版不引入后端新接口，不生成测试文件，先用共享静态目录 + 本地持久化把使用闭环跑通。

## 现状判断

- `apps/app/src/routes/games-page.tsx` 当前仅为桌面占位页。
- 桌面左侧导航已有“游戏中心”一级入口。
- 移动端“发现”目前只承载朋友圈、摇一摇、场景相遇、广场动态、视频号，没有游戏中心二级页。
- 当前仓库没有游戏目录、最近玩过、推荐位、排行位、福利位等前端模型。

## 对齐口径

- 不是做游戏平台底层，而是先把“像微信游戏中心一样的入口、浏览、推荐、最近玩过、活动位”做出来。
- 首版聚焦视觉布局、信息密度与交互节奏。
- 真正小游戏运行容器、好友战绩、服务端活动编排、云同步最近玩过，留到后续阶段。

## 信息架构

### Web 手机版

- 入口：`发现 -> 游戏`
- 页面层次：
  - 顶部栏：返回、标题、功能入口
  - Hero 区：主推活动 / 编辑推荐 / 当前主打游戏
  - 最近玩过：横向卡片
  - 好友在玩：社交热度与轻排行
  - 热门榜 / 新游榜：纵向列表
  - 为你推荐：较大卡片，承接“秒开/继续”
  - 福利活动：礼包、赛季、限时活动

### Web 桌面版

- 入口：左侧一级导航 `游戏中心`
- 页面布局：
  - 左栏：频道切换、最近玩过、固定常玩
  - 中栏：主推荐 Banner、推荐游戏网格、榜单切换、活动位
  - 右栏：好友热玩、赛季提醒、运营活动、状态说明
- 交互上不弹独立窗口，首版统一在工作区内承接。

## 数据与状态设计

### 共享目录

- 新增 `apps/app/src/features/games/`
- 目录内提供：
  - 游戏目录与专题数据
  - 好友热玩、榜单、活动位静态 mock
  - 本地状态读写工具

### 本地持久化

- 使用 `localStorage`
- 维护字段：
  - `recentGameIds`
  - `pinnedGameIds`
  - `lastOpenedAtById`
- 触发时机：
  - 点击“开始游戏 / 继续 / 秒开”时写入最近玩过
  - 点击“固定常玩”时更新 pinned

## 路由与结构变更

- 保留 `games-page.tsx` 作为统一游戏中心页面，根据布局切换移动/桌面视图。
- 新增移动端发现子路由：`/discover/games`
- 更新 `discover-page.tsx` 的移动端入口列表。
- 因新增路由与 `features/games/` 目录，需要同步更新 `AGENTS.md`。

## 实现分步

1. 新增共享游戏中心目录、静态目录数据与本地状态存储。
2. 把 `games-page.tsx` 改造成统一页面容器。
3. 新增桌面工作区组件，完成左中右三栏布局。
4. 新增移动端发现二级页布局，补齐推荐、最近玩过、榜单、活动位。
5. 在路由与发现入口接入 `/discover/games`。
6. 更新 `AGENTS.md` 结构与路由说明。
7. 执行 `pnpm --filter @yinjie/app build` 验证。

## 首版边界

- 不接小游戏真实运行容器。
- 不接后端接口。
- 不补自动化测试文件。
- 不改动已有聊天、视频号、收藏等业务逻辑。

## 后续演进

- 接服务端游戏目录与活动编排。
- 把最近玩过同步到账号级状态。
- 接入好友战绩、排行榜、房间邀请。
- 支持小游戏容器、独立游戏工作区与多开策略。
