# Web端手机版通讯录与聊天页微信化总规划

## 任务目标
- 目标范围覆盖 `apps/app` 的 Web 端手机版这几条主路径：
  - `tabs/contacts`
  - `tabs/chat`
  - `chat/$conversationId`
  - `group/$groupId`
  - `chat/$conversationId/details`
  - `group/$groupId/details`
- 目标不是“做个像微信的壳”，而是把通讯录、消息列表、单聊、群聊、聊天详情的功能和操作节奏都尽量收敛到微信手机端。
- 这次按“微信是唯一交互基线”来做，当前产品自定义痕迹要尽量退出主路径，只保留隐界必须保留的世界语义。
- 这份规划是移动端母计划，优先级高于 `通讯录微信化改造-2026-04-09.md` 里只看通讯录的旧方案。

### 微信基线
截至 `2026-04-10`，公开资料里能确认到的微信手机端相关能力：
- 中国区 iOS App Store 的微信版本历史显示，当前公开版本为 `8.0.70`，上架日期是 `2026-03-21`。
- App Store 官方描述确认了这些长期能力：
  - 图片、视频、文本、语音消息
  - 群聊和视频聊天
  - 表情商店
  - 实时位置共享
  - 聊天室墙纸自定义
  - 消息提醒自定义
  - 公众号服务
- `2025-04-03` 的公开报道提到，微信官方已把“引用回复”扩展到语音和表情包，说明手机端消息长按菜单和引用链路仍在持续增强。

### 需要明确标注的推断
- 微信官方公开资料不会把手机端所有细节交互逐项列完。
- 因此下面这些内容属于高可信推断，并按“当前微信手机端长期稳定交互”纳入规划：
  - 通讯录固定服务项结构
  - 会话列表左滑操作
  - 聊天消息长按菜单
  - 语音/键盘切换输入器
  - 图片大图查看
  - 群聊 `@` 候选和消息回复入口

## 差异总览

### A. 通讯录页还是“半微信，半自定义”

#### 1. 信息架构还不对
- 现在通讯录顶部分组是：
  - 新的朋友
  - 星标朋友
  - 群聊
  - 标签
  - 公众号
  - 世界角色
- 这一组里混入了太多项目自定义信息：
  - 每行都有副标题
  - `世界角色` 常驻首屏
  - `星标朋友` 被抬成了一级固定入口
- 微信手机端首屏通讯录的感觉是“固定服务项 + 干净联系人列表”，不是“功能卡片集合页”。

#### 2. 联系人主流程不对
- 当前移动端联系人项点击后直接 `start chat`。
- 微信手机端的高频路径是：
  - 点联系人先进入联系人资料页
  - 再从资料页发消息 / 音视频 / 设置备注 / 加星标 / 设权限
- 结论：
  - 现在的移动端通讯录更像“消息页的第二入口”，不像真正的通讯录。

#### 3. 操作深度不够
- 现有通讯录没有移动端微信常见的这些能力承接：
  - 联系人资料页主路径
  - 备注 / 标签 / 星标管理闭环
  - 从通讯录直接发起音视频或更多操作
  - 列表层级的更细状态表达
- 当前右侧索引条只是点击跳转，没有微信那种连续拖动索引的手感和浮层反馈。

#### 4. 视觉明显不像微信
- 现在用了较多暖色渐变、卡片感、说明性副文案。
- 微信手机端通讯录更平、更薄、更灰白，服务项和联系人行几乎是同一种列表语言。

### B. 消息列表页离微信还有一整层操作能力

#### 1. 会话项只有“点进去”，没有列表操作
- 当前移动端消息列表没有左滑操作。
- 微信手机端消息列表的核心效率来自左滑或长按之后的快捷动作，例如：
  - 标为未读
  - 置顶 / 取消置顶
  - 免打扰
  - 删除聊天
- 结论：
  - 现在列表只是“展示层”，不是“操作层”。

#### 2. 会话状态表达不够完整
- 当前列表只展示：
  - 标题
  - 最后一条预览
  - 时间
  - 未读 badge
  - 置顶文案
- 还缺微信式会话状态密度：
  - 免打扰小红点 / 灰点语义
  - 草稿态
  - `@我`
  - 撤回提示
  - 发送者前缀的更完整规则

#### 3. 右上角快捷菜单还只是占位
- 当前菜单里：
  - `发起群聊`
  - `添加朋友`
  - `扫一扫`
  - `收付款`
- 其中 `扫一扫`、`收付款` 还是 toast 占位。
- 如果要“完全微信化”，这个区域要么做成真实能力，要么先在视觉上对齐但明确做受控禁用态，不要停在半成品。

### C. 单聊页差距最大

#### 1. 消息长按菜单几乎没有
- 当前 `chat-message-list` 的上下文菜单只在“桌面端群聊”启用。
- 移动端聊天消息现在没有微信最关键的长按操作闭环：
  - 复制
  - 回复 / 引用
  - 转发
  - 收藏
  - 多选
  - 撤回
  - 删除
- 结论：
  - 这不是细节差异，而是聊天页主操作模型还没做。

#### 2. 单聊没有回复链路
- 当前单聊线程没有把 `onReplyMessage` 传给消息列表。
- 现在只有群聊线程保留了回复草稿能力，而且入口还是桌面右键，不是移动端入口。
- 微信手机端里，“回复某条消息”已经是日常操作，不是群聊专属，更不是桌面专属。

#### 3. 输入器模型不对
- 当前移动端左侧按钮是“浏览器语音转文字”，不是微信的“语音/键盘切换”。
- 现在的输入控件还是单行 `input`，不是微信式可伸缩输入框。
- 发送按钮颜色和整体结构也不是微信的灰白 + 绿色体系。
- 结论：
  - 当前输入区像“Web IM 输入条”，不像微信手机端输入器。

#### 4. 语音能力方向走偏了
- 当前 `Mic` 本质是语音识别转文本。
- 微信手机端主语音链路是“按住说话发送语音消息”。
- 这两者的产品语义完全不同。
- 如果真的要完全对齐，必须补“语音消息”而不是继续强化“语音转文字”。

#### 5. 更多功能面板太薄
- 当前更多功能只有：
  - 相册
  - 拍摄
  - 名片
  - 位置
  - 文件
- 跟微信手机端常见的一屏/两屏更多功能相比，缺少至少一整层高频入口。
- 即便暂时不做红包、转账，也至少要把布局、排序、空位策略、禁用策略做成微信式。

#### 6. 媒体消息交互不完整
- 当前图片消息只是直接渲染 `<img>`，没有大图查看层。
- 文件、名片、位置卡片也还偏静态展示，没有微信那种点开后的完整承接。
- 这会直接导致聊天页“能看不能玩”。

### D. 群聊页完成度比单聊高，但移动端仍然不够微信

#### 1. 群公告有了，但群内消息操作没有
- 当前群聊已经有公告条、详情页、群成员、群背景等基础能力。
- 但移动端群消息仍然没有微信式长按菜单。
- 这会导致群聊日常操作仍旧偏弱。

#### 2. `@` 能力只接了一半
- 群聊线程里已经准备了 `mentionCandidates`。
- 但真正传给输入器时只在桌面端生效，移动端没有接。
- 微信手机端群聊里，`@` 是标准能力，不应只存在桌面端。

#### 3. 回复能力入口不对
- 群聊线程支持 `replyDraft`，但移动端没有操作入口触发。
- 这会让代码里“有能力”，用户层“用不到”。

#### 4. 群顶部运营信息还不够
- 当前有公告条，但还缺微信式更稳定的群置顶消息承接和相关提示状态。

### E. 聊天详情页还混着“假设置”

#### 1. 单聊“消息免打扰”不是真设置
- 当前单聊详情页的 `消息免打扰` 和 `强提醒` 走的是 `localStorage` 偏好，不是真实会话设置。
- 这和微信手机端的预期差距很大，因为用户会默认它影响真实消息行为。

#### 2. 单聊详情项还不够像微信
- 当前已有：
  - 查找聊天记录
  - 置顶聊天
  - 保存到通讯录
  - 设置聊天背景
  - 清空聊天记录
  - 投诉 / 拉黑
- 但还缺一些更微信式的组织方式和真实设置闭环，例如：
  - 免打扰的真后端状态
  - 隐藏聊天
  - 更自然的资料页跳转和权限分组

#### 3. 群详情已经有不少能力，但交互组织仍可继续微信化
- 群详情当前比单聊详情完整很多。
- 但仍需要继续对齐微信的字段顺序、分组方式、危险操作位置和视觉语言。

## 优先级建议

### P0：先把“每天都在用”的路径变成微信
- 通讯录首屏结构和列表语言
- 消息列表左滑操作
- 单聊 / 群聊消息长按菜单
- 单聊 / 群聊统一回复链路
- 输入器重做成微信手机端结构
- 图片消息大图查看
- 单聊详情里的假设置改成真设置

### P1：再补高频增强项
- 移动端群聊 `@` 候选
- 群置顶消息承接
- 文件 / 名片 / 位置卡片点击后的完整流转
- 会话状态表达补齐

### P2：最后补真正需要新业务结构的微信能力
- 语音消息
- 转发
- 收藏
- 多选
- 撤回
- 仅聊天 / 联系人权限类能力
- 红包 / 转账类入口

## 实施方案

### 阶段 1：通讯录首屏收口
- 把移动端通讯录做成微信式扁平列表，不再用当前卡片化服务项视觉。
- 顶部保留：
  - 居中标题
  - 右上角加号入口
  - 搜索条
- 固定服务项改成微信式单列列表，不要副标题大段文案。
- `世界角色` 从主联系人区域移出，放到固定服务项末尾或独立二级页，不再和微信标准联系人列表混在一起。
- 联系人点击逻辑从“直接进聊天”改成“先进联系人资料页”，聊天按钮放资料页内。

### 阶段 2：消息列表微信化
- 新增移动端会话 cell 的左滑容器。
- 第一批接这些动作：
  - 置顶 / 取消置顶
  - 免打扰 / 取消免打扰
  - 删除 / 隐藏聊天
  - 标为未读
- 补齐会话状态表达：
  - 免打扰样式
  - 草稿位
  - `@我`
  - 撤回提示
- 右上角快捷菜单先对齐视觉和顺序，再决定哪些能力做真实落地，哪些继续占位。

### 阶段 3：统一消息操作模型
- 给 `ChatMessageList` 做移动端长按手势和底部 action sheet。
- 单聊和群聊统一走一套消息操作模型，不再“群聊有，单聊没有”。
- 第一批动作建议至少落：
  - 复制
  - 回复
  - 删除
  - 撤回自己的消息
- `replyDraft` 结构保留，但入口要移动端可达。

### 阶段 4：重做输入器
- 单行 `input` 改成微信式自增长 `textarea`。
- 左侧切成：
  - 语音 / 键盘切换
  - 真正的按住说话态
- 中间保留：
  - 文本输入
  - 表情入口
- 右侧保留：
  - `+`
  - 文本非空后切绿色发送按钮
- 更多功能面板改成微信式网格和分页规则。

### 阶段 5：补媒体和卡片交互
- 图片消息增加全屏查看器：
  - 点击放大
  - 左右切换
  - 定位回原消息
- 文件、位置、名片卡片补点击行为。
- 头像补点击跳资料页。

### 阶段 6：详情页真设置化
- 单聊详情里的 `消息免打扰` 接真实会话设置，不再走本地偏好。
- 把 `强提醒` 这类项目自定义偏好收缩到次级能力，不占主设置位。
- 群详情继续调整顺序和分组，让字段布局更贴近微信。

### 阶段 7：补群聊移动端增强项
- 移动端接入 `@` 候选选择器。
- 群置顶消息 / 顶部提醒承接到主线程。
- 群成员快捷入口、顶部信息密度继续对齐微信。

## 需要动后端/契约的点
- 如果只做 P0 前半段，基本可以主要在前端完成。
- 一旦进入这些能力，就不再是纯前端活：
  - 语音消息
  - 转发
  - 收藏
  - 多选
  - 撤回
  - 标为未读
  - 联系人权限类能力
- 这些很可能需要新增：
  - contracts 契约
  - API 路由
  - 实体字段或消息元数据
- 真要进入这一步，按仓库规则需要同步更新 `AGENTS.md` 里涉及的路由 / 实体 / 契约说明。

## 验收标准
- 通讯录第一页第一眼看上去就是微信手机端通讯录，而不是一个“联系人功能面板”。
- 消息列表可以在列表层完成高频管理动作，不用每次点进会话。
- 单聊和群聊都能在移动端长按消息并完成基础消息操作。
- 输入器切回微信式结构，不再让“语音转文字”冒充“语音消息”。
- 图片、文件、名片、位置这些消息从“静态块”变成“可操作对象”。
- 单聊详情里的设置不再是假开关。

## 风险与注意事项
- “完全抄微信”里最难的不是排版，是消息操作模型和输入器模型。
- 语音消息如果真要做，会连带牵出附件类型、上传链路和消息渲染收口。
- 移动端长按菜单和左滑手势如果实现粗糙，会直接降低体验，不能只做“能用”。
- 当前聊天相关逻辑已经散在多个大文件里，实施时应顺手拆出移动端专用组件，否则代码会继续失控。
