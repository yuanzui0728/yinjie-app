# Web端手机版群聊功能对齐微信深度规划

## 任务目标
- 只规划 `apps/app` Web 手机版群聊链路，不覆盖桌面端。
- 目标不是“做一套像微信的聊天 UI”，而是把手机版群聊的页面层级、操作路径、状态语义、消息动作尽量贴到当前微信手机端的习惯用法。
- 这里的“对齐微信”按下面的口径执行：
  - 功能和操作尽量抄路径、抄顺序、抄状态反馈
  - 视觉节奏尽量贴近，但不直接拷微信品牌资源、图标源文件、专有插画和商标文案
  - 对 2025 年公开灰度能力，按“可选增强”处理，不把所有灰度项都当作首版强依赖

## 结论先说
- 当前项目的手机版群聊已经不是空壳，主线程、群信息、群公告、成员增删、聊天记录搜索、聊天背景、`@成员`、引用回复、图片 / 文件 / 名片 / 位置发送都已经有了。
- 真正的差异不在“有没有页面”，而在三个地方：
  - 主线程和群信息页的操作细节还不够微信，很多地方还是项目自己的组件语言
  - 消息动作里有几项是“本地假动作”，不是微信群聊那种真实同步语义
  - 输入区和消息类型离微信还差一层，尤其是语音消息、消息状态、历史加载和群管理深度
- 如果要“看起来就像微信手机版群聊”，优先级不是继续堆新页面，而是先改主线程、消息动作、群信息页和输入区的状态机。

### 4. 当前已经确认的硬缺口
- 群消息是 `3s` 轮询，不是稳定实时推送。
- 消息“撤回 / 删除”当前是前端 `localStorage` 假动作，只对当前设备生效，不是服务端真实语义。
- “收藏消息”当前也是本地收藏，不是微信那种跨端一致的收藏能力。
- 群消息类型没有：
  - 语音消息
  - 视频消息
  - 链接卡片
  - 小程序卡片
  - 共享实时位置
  - 红包 / 转账 / 接龙类卡片
- 输入区的“语音”是语音转文字，不是微信默认的“按住说话发语音消息”。
- 主线程没有历史分页、没有“下拉查看更早消息”、没有“新消息跳转条”。
- 消息没有发送中 / 失败 / 重发状态。
- 群信息页里的重命名和“我在本群的昵称”编辑，仍然是 `window.prompt`，这和微信手机端差得很明显。

## 二、对照微信手机端后的判断

### 1. 截至 2026-04-10 可以确认的微信稳定能力
- 微信 App Store 中国区页面显示，微信当前版本为 `8.0.70`，发布时间是 `2026-03-21`。
- 官方描述里仍明确把这些能力作为核心聊天能力：
  - 群聊
  - 视频、图片、文本、语音消息
  - 自定义聊天背景
  - 消息提醒自定义
- 这说明对我们来说，以下四类不该再视为“锦上添花”：
  - 群聊消息基础动作
  - 语音消息
  - 背景与通知设置
  - 稳定的移动端聊天主链路

### 2. 2025 年公开可见的新变化
- 2025-03-17 的公开更新盘点里，可以确认微信在手机端已经有：
  - 长按多选后的“选择到这里”
  - 群聊里引用消息再 `@人` 时，选择提醒对象界面会带出引用来源信息
- 2025-06-17 的公开更新盘点里，可以确认：
  - 微信“发语音”是以语音消息发送为主，UI 上区分 `取消` 和 `滑到这里转文字`
  - 群信息页成员区域已经继续收口，`更多群成员` 直接展开，搜索入口挪到了右上角
- 2025-10-31 的公开报道里，可以确认：
  - 群聊免打扰后，微信允许继续接收 `@我 / @所有人 / 群公告`
  - 还支持额外添加最多 4 位重要群成员，收到他们的消息也提醒

### 3. 对齐微信时要分清两类能力
- 稳定必抄：
  - 主线程布局
  - 长按菜单顺序
  - 输入区状态机
  - 群信息页结构
  - 群成员宫格和加减人路径
  - 查找聊天记录入口
  - 群公告 / 群二维码 / 保存到通讯录 / 免打扰
- 灰度增强，可作为第二阶段：
  - 重要群成员提醒
  - 更细的群成员展开逻辑
  - 更细的引用来源展示
  - 多条消息批量撤回

## 三、差异清单

### P0：一眼不像微信的地方，必须先改

#### 1. 群聊主线程 header 还不对
- 当前：
  - `apps/app/src/features/chat/mobile-chat-thread-header.tsx` 里显示了双行 header
  - 标题下方长期显示“X 人”
- 微信：
  - 群聊主线程顶部更克制，核心是返回、群名、右上角更多
  - 人数信息不应该长期占据主线程副标题位
- 处理：
  - 移除群聊主线程常驻副标题
  - 群人数放回群信息页承载
  - 仅保留公告条承担“额外上下文”

#### 2. 群信息页还是“项目卡片感”，不是微信列表感
- 当前：
  - `ChatDetailsSection` 用的是圆角卡片 + 大写 section 标题
  - `ChatSettingRow` 和微信式分组列表相比更像设置中心
- 微信：
  - 更接近浅灰背景上的白底 list group
  - 标题弱化、分组更薄、行高更稳
- 处理：
  - 单独给群信息链路做一套更微信的 mobile details tokens
  - 不要再复用过于通用的卡片 section 视觉

#### 3. 群名称 / 群昵称编辑路径不对
- 当前：
  - `group-chat-details-page.tsx` 里通过 `window.prompt` 直接改群名和本群昵称
- 微信：
  - 都是独立编辑页，不会弹浏览器原生 prompt
- 处理：
  - 新增通用的“单字段编辑页”路由
  - 群名称、我在本群的昵称都走二级页

#### 4. 长按消息菜单虽然有，但语义有假动作
- 当前：
  - 长按菜单里有 `回复 / 转发 / 多选 / 提醒 / 收藏 / 复制 / 撤回 / 删除`
  - 但 `提醒` 是占位
  - `收藏` 是本地收藏
  - `撤回 / 删除` 是本地假动作
- 微信：
  - 长按菜单里的高频动作必须是真动作，不然用户会直接感知到不对
- 处理：
  - 先把菜单顺序和显隐规则做准
  - 再把 `撤回 / 删除 / 收藏 / 提醒` 的真语义补齐

#### 5. 输入区核心语义不对
- 当前：
  - 有麦克风按钮，但走的是语音转文字
  - `+` 面板顺序也不是严格微信逻辑
- 微信：
  - 默认“按住说话”发语音消息是主路径
  - 转文字是语音发送过程里的分支，不是替代品
- 处理：
  - 把“语音消息”补成独立消息类型
  - 保留当前语音转文字，但降级为微信式 secondary path

### P1：功能上能用，但还没抄到微信感觉的地方

#### 6. 群成员区不够微信
- 当前：
  - `group-chat-details-page.tsx` 只截前 10 个成员，再固定 `+ / -`
  - 没有“更多群成员”展开，也没有群成员搜索入口
- 微信：
  - 群成员区域本身可继续展开
  - 搜索成员入口就在成员区右上角附近
- 处理：
  - 第一版先做“更多群成员”
  - 第二版再做成员搜索

#### 7. 查找聊天记录路径还不够微信
- 当前：
  - `group-message-search-page.tsx` 已经能搜，但更像“项目自定义搜索页”
  - 分类是 `全部 / 图片与视频 / 文件 / 链接`
- 微信：
  - 路径是在群信息页进入“查找聊天记录”
  - 结果更强调筛选入口和结果类型，而不是先露一个大卡片仪表盘
- 处理：
  - 收敛入口页视觉
  - 保留已有分类能力，但不要把它做得太像“分析面板”

#### 8. 群公告页功能有，质感不够微信
- 当前：
  - 独立页 + textarea + 保存按钮
- 微信：
  - 更像标准表单页
  - 公告在群聊主线程顶部也有更弱提示
- 处理：
  - 保留当前独立页
  - 增加公告作者 / 更新时间展示
  - 增加主线程里的已读弱状态

#### 9. 消息历史加载能力不够
- 当前：
  - `getGroupMessages` 默认只拉固定数量
  - 没有上翻加载更早消息
- 微信：
  - 上滑就是连续历史
- 处理：
  - 群消息接口改 cursor / beforeId 分页
  - 前端消息列表做 prepend 历史

#### 10. 群头像、群列表预览还不够微信
- 当前：
  - 还没有明确的移动端群头像九宫格心智
  - 未命名群标题生成规则也不够微信
- 微信：
  - 未命名群展示成员名拼接
  - 群头像用组合头像
- 处理：
  - 列表页和详情页同步改
  - 群聊主线程顶部也使用相同命名规则

### P2：微信已有或接近微信，但不该卡住首版的地方

#### 11. 免打扰后的“重要群成员”提醒
- 当前：
  - 我们已支持 `@我 / @所有人 / 群公告`
  - 还不支持“额外 4 个重要成员”
- 判断：
  - 这是 2025 年公开出现的增强能力，不是最基础的微信群聊心智
- 处理：
  - 放第二阶段
  - 需要 `Group` 新增 `importantNotifierIds` 一类字段

#### 12. 语音消息 UI 的“滑到这里转文字”
- 当前：
  - 只有语音转文字流程
- 判断：
  - 如果真要抄到位，后面应该把语音消息发送和转文字分支一起做
- 处理：
  - 放第二阶段，与语音消息类型一起落

#### 13. 更深的群管理
- 当前：
  - 已有成员增删，但没有独立“群管理”页
- 微信：
  - 群主管理、管理员、邀请确认、群内权限等是更深层能力
- 判断：
  - 当前单世界 + AI 群成员模型下，不是手机版群聊第一优先级
- 处理：
  - 结构先预留
  - 真权限模型单独规划

## 四、最关键的结构性问题

### 1. 当前最危险的问题不是“少页面”，而是假语义
- 现在最影响微信感知的不是有没有群二维码页，而是：
  - 用户点“撤回”其实只是本地隐藏
  - 用户点“删除”其实也是本地隐藏
  - 用户点“收藏”不是微信收藏体系
- 这类能力如果继续往外露，只会让产品越做越像 demo。

### 2. 当前群聊消息模型不够承载微信化
- `GroupMessage` 还缺：
  - `status`
  - `recalledAt / recalledBy`
  - `deletedForOwner` 之类的本地删除语义
  - `audio` / `video` / `link_card` / `mini_program_card`
- 如果要真正做成微信式群聊，后续肯定要扩消息模型。

### 3. 轮询会把移动端聊天体验拖死
- 当前 `group-chat-thread-panel.tsx` 对群消息是 `refetchInterval: 3000`
- 这会带来：
  - 新消息延迟
  - 滚动状态不稳
  - 发送后状态跳变粗糙
  - 移动端耗电和网络浪费
- 群聊一旦微信化，必须切 socket / 推送主链路。

## 七、最终判断
- 如果目标是“用户一进来就觉得这就是微信群聊”，现阶段最该做的不是继续加零散功能，而是先把下面 4 个点做准：
  - 主线程 header 和输入区状态机
  - 群信息页结构和编辑路径
  - 长按消息动作的真假语义
