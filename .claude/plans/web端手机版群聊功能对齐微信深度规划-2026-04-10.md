# Web端手机版群聊功能对齐微信深度规划

## 任务目标
- 只规划 `apps/app` 的 Web 手机版群聊能力，不覆盖桌面端。
- 目标不是“做一个更好看的群聊”，而是尽量 1:1 对齐微信手机端群聊的布局、信息架构、操作路径和状态逻辑。
- 这里的“对齐微信”指：
  - 页面层级对齐
  - 视觉节奏对齐
  - 功能闭环对齐
  - 不直接复用微信品牌素材、图标资源、文案资产

## 范围边界
- 覆盖从消息列表进入群聊开始，到以下场景闭环：
  - 群聊主页面
  - 群聊信息页
  - 查找聊天记录
  - 发起群聊
  - 加人 / 删人
  - 群公告
  - 群通知与保存到通讯录
  - 我在本群的昵称
  - 群管理能力
- 不纳入本轮首版强制范围：
  - 红包、转账、群接龙支付
  - 群直播、语音通话、视频通话
  - 原生级录音长按体验
  - 微信表情商城级别的表情生态

  - 已有群聊信息页雏形

- `apps/app/src/components/chat-composer.tsx`
  - 已有文字、表情、图片、文件、名片、位置的输入能力

### 2. 已有后端接口
- `GET /api/groups/:id`
- `PATCH /api/groups/:id`
- `GET /api/groups/:id/members`
- `POST /api/groups/:id/members`

### 3. 已有数据结构
- `GroupEntity`
  - 已有：`name`、`avatar`、`announcement`、`isPinned`、`lastReadAt`、`lastClearedAt`、`isHidden`、`lastActivityAt`
- `GroupMemberEntity`
  - 已有：`memberName`、`memberAvatar`、`role`
- `GroupMessageEntity`
  - 已有：文本、系统消息、图片、文件、名片、位置等附件落盘能力

### 4. 当前缺口结论
- 视觉风格不对：
  - 当前群聊页仍是项目自定义的渐变卡片语言，不是微信手机端的纯平灰白体系。
- 信息架构不完整：
  - 群聊页、群详情页、成员管理页、公告页、搜索页还没有形成完整微信式路径。
- 状态持久化不对：
  - `消息免打扰`、`显示群成员昵称` 仍在 `localStorage`，没有服务端真状态。
- 实时链路不对：
  - 群聊线程现在主要依赖 `3s` 轮询，不是稳定实时推送。
- 权限模型不够：
  - 当前 `GroupMember.role` 只有 `owner/member`，没有 `admin`，无法承接微信群管理能力。
- 功能闭环不够：
  - 已有加人接口，但没有删人接口、群管理页、保存到通讯录真实能力、通知细粒度能力、消息操作能力。

## 二、微信式目标口径

### 1. 要对齐的不是单页，而是一整条群聊链路
- 消息列表里的群项
- 群聊线程页
- 输入区与附件面板
- 长按消息操作
- 群信息页
- 群成员管理页
- 群公告页
- 查找聊天记录页
- 保存到通讯录后的通讯录群聊入口

### 2. 截至 2025-10 公开可见信息，可确认的微信侧关键点
- 群信息页包含：
  - 成员宫格
  - 查找聊天记录
  - 置顶聊天
  - 保存到通讯录
  - 我在本群的昵称
  - 群公告
- 群通知在近期公开报道中已经出现更细粒度控制：
  - `以下消息仍通知`
  - `@我`
  - `@所有人`
  - `群公告`
- 群聊记录检索已经不是纯关键词列表，而是进入群信息页后再按条件查找的路径。

### 3. 本项目中的对齐标准
- P0 标准：
  - 用户打开后第一眼就像微信群聊，不像当前项目的自定义聊天页。
- P1 标准：
  - 常用群功能都能在微信同路径找到，且状态真实持久化。
- P2 标准：
  - 长按消息、成员管理、公告、搜索、通知规则都形成闭环，不再是占位入口。

## 三、目标信息架构

### A. 消息列表中的群聊项
- 维持 `/tabs/chat` 作为消息主页。
- 群聊项需要对齐微信的 4 个点：
  - 群头像使用九宫格合成头像，不再只用单个默认头像
  - 标题优先显示群名；未命名群显示成员昵称拼接
  - 副标题显示最后一条消息的微信式预览文本
  - 右侧显示时间和未读角标
- 建议补齐左滑操作：
  - `标为未读`
  - `置顶`
  - `删除`
- 当前项目已有置顶语义，`标为未读` 需要新增本地或服务端未读回滚能力。

### B. 群聊主页面

#### 1. 顶部导航
- 结构必须收敛成微信式：
  - 左：返回
  - 中：群名
  - 右：`...`
- 不显示当前项目的副标题双行头部样式。
- 标题规则：
  - 优先显示群名
  - 无群名时显示成员拼接名
  - 次级人数信息放入群信息页，不在头部持续展示

#### 2. 群公告条
- 群公告存在时，在消息区顶部显示一条可点击公告横条。
- 点击进入独立公告页，不直接弹 `prompt`。
- 未读公告需要有一次性弱提示状态。

#### 3. 消息区
- 背景、气泡、间距全部对齐微信：
  - 浅灰聊天背景
  - 白色他人气泡
  - 绿色自己气泡
  - 时间分隔居中
  - 群聊中他人消息上方显示昵称
- 首版不显示群消息已读人数，直接对齐微信群聊的“无已读回执”心智。
- 保留当前 AI 角色群回复能力，但表现上必须像正常群成员发言，不额外暴露系统感。

#### 4. 输入区
- 目标状态机对齐微信：
  - 默认文字输入
  - 语音输入模式
  - 表情面板
  - `+` 附件面板
  - 文本非空时右侧从 `+` 切换成 `发送`
- 现有输入能力可直接复用：
  - 文字
  - 表情包
  - 图片
  - 文件
  - 名片
  - 位置
- 需要新增或重构的点：
  - 更微信式的底栏布局
  - 输入区聚焦、收键盘、弹面板的状态切换
  - 发送中、失败、重发的消息状态
  - `@成员`
  - 回复某条消息

#### 5. 长按消息菜单
- 群聊必须补齐长按菜单，否则不像微信：
  - 复制
  - 转发
  - 收藏
  - 回复
  - 多选
  - 撤回
  - 删除
- 群聊特有补充：
  - `提醒` 或 `@TA`
  - 对消息内名片 / 位置 / 文件做二级动作
- 首版建议：
  - 先做 `复制/回复/多选/删除/撤回`
  - 第二阶段补 `转发/收藏`

### C. 群信息页

#### 1. 页面结构
- 群信息页视觉必须切回微信式列表页：
  - 页面背景浅灰
  - 白底 section
  - 细分隔线
  - 去掉大圆角卡片和大面积渐变
- 建议 section 顺序：
  - 成员宫格
  - 群名称
  - 群二维码
  - 群公告
  - 查找聊天记录
  - 消息免打扰
  - 置顶聊天
  - 保存到通讯录
  - 我在本群的昵称
  - 显示群成员昵称
  - 群管理
  - 清空聊天记录
  - 删除并退出

#### 2. 成员宫格
- 对齐微信 5 列宫格排布。
- 前若干个格子显示成员头像和昵称。
- 末尾固定两个功能格：
  - `+`
  - `-`
- 点击成员头像进入成员资料页或联系人资料页。

#### 3. 群二维码
- 微信群信息页里群二维码是强存在感能力，建议纳入。
- 本项目首版可不做真实扫码入群校验，但要保留真实页面和二维码展示能力。
- 由于当前实例是单世界单主人模型，二维码本质上是“群分享入口”的映射，不承担公网开放加群。

#### 4. 消息免打扰
- 不能再走本地偏好。
- 目标设计应直接按微信现网思路建模：
  - 主开关：消息免打扰
  - 子开关：以下消息仍通知
  - 子项：
    - `@我`
    - `@所有人`
    - `群公告`
- 如果首版不做细粒度，也必须先把字段设计好，避免二次拆迁。

#### 5. 群管理
- 只有群主 / 管理员可见。
- 推荐纳入以下项：
  - 群主管理权转移
  - 设置管理员
  - 加群方式
  - 邀请确认
  - 群成员邀请权限
  - 禁言能力
- 结合当前项目的 AI 群成员模型，首版不需要全部做真，但结构要先预留。

### D. 查找聊天记录页
- 当前搜索页只是“关键词过滤列表”，不够微信。
- 目标应拆成两层：
  - 搜索入口页
  - 结果页
- 搜索入口页建议提供：
  - 搜索框
  - 按日期查找
  - 按群成员查找
  - 按内容类型查找：
    - 图片及视频
    - 文件
    - 链接
    - 交易
- 考虑到当前已有消息类型，首版先支持：
  - 关键词
  - 日期
  - 群成员
  - 图片
  - 文件

### E. 发起群聊 / 加人 / 删人
- `create-group-page.tsx` 当前像业务表单，不像微信联系人选择器。
- 必须改成微信式选人页：
  - 顶部搜索
  - 联系人索引列表
  - 多选勾选
  - 底部确定按钮
- 加人页与发起群聊页共用同一套联系人选择器。
- 删人页单独进入“选择联系人”模式，只显示可删除成员。

## 五、数据模型规划

### 1. `GroupEntity` 建议新增字段
- `savedToContacts: boolean`
- `savedToContactsAt: Date | null`
- `isMuted: boolean`
- `mutedAt: Date | null`
- `notifyOnAtMe: boolean`
- `notifyOnAtAll: boolean`
- `notifyOnAnnouncement: boolean`
- `showMemberNicknames: boolean`
- `announcementUpdatedAt: Date | null`
- `announcementPublisherId: string | null`
- `announcementPublisherType: 'user' | 'character' | null`

### 2. `GroupMemberEntity` 建议调整
- `role` 从 `owner | member` 扩成：
  - `owner`
  - `admin`
  - `member`
- 新增：
  - `isRemoved: boolean`
  - `removedAt: Date | null`
- `memberName` 继续承接“我在本群的昵称 / 群名片”语义

### 3. `GroupMessageEntity` 建议新增字段
- `replyToMessageId: string | null`
- `replyPreviewPayload: string | null`
- `mentionsPayload: string | null`
- `recalledAt: Date | null`
- `recalledById: string | null`
- `clientMessageId: string | null`

### 4. 为什么不新建 `GroupPreference` 表
- 当前实例是单世界单主人模型。
- 群聊偏好本质上只有一个真实用户在操作。
- 直接扩 `GroupEntity` 成本更低，也符合当前仓库“SQLite + TypeORM synchronize”的现实。
- 如果以后真的要恢复多用户实例，再把 owner 维度偏好拆表。

## 六、接口规划

### 1. 建议保留现有接口，补齐以下接口
- `DELETE /api/groups/:id/members/:memberId`
  - 移除群成员
- `PATCH /api/groups/:id/preferences`
  - 更新：
    - `isMuted`
    - `notifyOnAtMe`
    - `notifyOnAtAll`
    - `notifyOnAnnouncement`
    - `savedToContacts`
    - `showMemberNicknames`
- `GET /api/groups/saved`
  - 获取已保存到通讯录的群聊列表
- `GET /api/groups/:id/search`
  - 支持：
    - `keyword`
    - `senderId`
    - `date`
    - `messageTypes`
- `POST /api/groups/:id/messages/:messageId/recall`
- `POST /api/groups/:id/messages/:messageId/reply`
  - 或直接在发送消息 payload 里支持 `replyToMessageId`
- `GET /api/groups/:id/qrcode`
  - 返回群二维码资源或二维码 payload
- `PATCH /api/groups/:id/announcement`
  - 如果不想复用 `PATCH /groups/:id`，可拆单独公告接口
- `PATCH /api/groups/:id/management`
  - 管理配置，二阶段使用

### 2. WebSocket 规划
- 当前群聊不应继续依赖 `3s` 轮询作为主链路。
- 建议在现有 `/chat` namespace 里补齐群房间能力：
  - `join_group`
  - `leave_group`
  - `group_new_message`
  - `group_message_recalled`
  - `group_updated`
  - `group_members_updated`
- 注意：
  - 微信群聊没有“谁正在输入”的显性提示，群聊 socket 不需要做 typing UI。

## 十、关键取舍

### 1. 必须做真状态，不能继续只做占位
- `消息免打扰`
- `保存到通讯录`
- `显示群成员昵称`
- 这三项如果继续停留在本地偏好，群聊体验永远不像微信。

### 2. 群聊不需要已读回执
- 微信群聊核心体验就是“群里没有已读压力”。
- 本项目不要为了技术炫技加入群消息已读人数展示。

### 3. Web 版语音输入要现实处理
- 微信原生长按说话体验在 Web 上很难完全等价。
- 规划上应把“语音模式”视为二阶段增强，不阻塞第一阶段整体微信化。

### 4. AI 角色群聊要服从微信心智
- 回复节奏要克制。
- 不能因为 AI 能力丰富就把群聊做成“多人 bot 乱回”。
- 群成员消息频率、回复延迟、@响应都要贴近日常群聊。

## 十一、验收标准
- 不看代码，只看页面截图和录屏，用户会把它判断为“微信式群聊”而不是“项目自定义聊天页”。
- 群信息页里的主要开关和操作都有真实状态，不是提示条占位。
- 加人、删人、公告、昵称、保存到通讯录、搜索都能闭环。
- 群消息进入、发送、刷新不再明显依赖轮询。
- 所有新增路径都挂在现有移动端壳层里，不破坏当前 Tab 结构。

## 十二、最终结论
- 当前仓库已经具备“做出微信式群聊”的 50% 基础：
  - 路由有了
  - 消息协议有了
  - 附件能力有了
  - 群 CRUD 基础有了
- 真正缺的不是零碎功能，而是：
  - 微信式信息架构
  - 微信式视觉语言
  - 服务端真状态
  - 成员管理与消息操作闭环
  - 群聊实时链路
- 推荐执行顺序不是“继续缝补当前页面”，而是直接按微信手机端重画群聊链路。
