## 当前结论

### 1. 现在不是“单聊 + 群聊”两条逻辑，而是“三条线程逻辑”

- 真单聊：
  - `Conversation.type = direct`
  - `id = direct_{characterId}`
- 假群聊：
  - 仍然存放在 `Conversation` / `Message`
  - 运行中可从 `direct` 升级成 `group`
  - 仍然走 `/chat/*` 路由和 socket `send_message`
- 真群聊：
  - `Group` + `GroupMember` + `GroupMessage`
  - 走 `/group/*` 路由和 `/groups/*` REST API

这就是当前混乱的根因。产品表面像在做“微信单聊/群聊”，但底层其实是两套半模型叠在一起。

### 2. 假群聊是后端在单聊里临时长出来的

- `ChatService.sendMessage()` 会在 direct 会话里根据意图识别把 `Conversation.type` 从 `direct` 改成 `group`
- 同时把更多角色塞进 `Conversation.participants`
- 之后该线程仍然保留在 `conversations` 表，不会迁移成正式 `Group`

结果：
- 同一个 `Conversation` 既可能是单聊，也可能被升级成群聊
- 这和正式 `Group` 模型并存，前端必须猜它到底算哪一种

### 3. 前端已经被迫靠“类型 + id 前缀”双重判断线程类别

- `Conversation.type` 已经允许 `group`
- 但前端 `isPersistedGroupConversation()` 又要求：
  - `type === "group"`
  - 并且 `id` 不能以 `direct_` 开头

结果：
- 同样是 `type = group`
- `direct_xxx` 会被当成“聊天页里的特殊会话”
- UUID `groupId` 会被当成“正式群聊”

这不是正常建模，而是前端在为后端的双模型打补丁。

### 4. 直接受影响的页面和能力已经出现语义错位

- `ChatDetailsPage` 直接拿 `participants[0]` 当“对方角色”
- 如果这个 `/chat/*` 线程已经升级成假群聊，详情页仍会按单聊展示、加好友、拉黑、举报第一位角色
- `ConversationThreadPanel` 在 `conversationType === "group"` 时显示“X 人群聊”，但这里的 `participants` 只有角色，不包含世界主人
- 全局搜索、消息列表、桌面工作区、转发都依赖 `isPersistedGroupConversation()` 决定是走单聊 API 还是群聊 API
- 后台回复逻辑观测已经不得不把分支拆成：
  - `direct`
  - `stored_group`
  - `formal_group`

这说明模型分裂已经扩散到产品层、交互层、运维层，而不是局部实现细节。

### 5. 现在很多“像微信”的功能，其实没有建立在稳定线程模型上

当前已经堆上去的很多微信式能力：
- 群公告
- 群昵称
- @我 / @所有人 通知
- 群成员增删
- 群通话
- 转发到单聊/群聊
- 搜索统一索引

这些能力只有在“线程类型唯一、消息源唯一、详情页唯一”时才可靠。现在底层模型不稳定，继续抄微信只会继续扩大歧义面。

## 建议的目标模型

### 目标一：单聊只保留一种语义

- 单聊 = 世界主人 与 1 个角色的长期对话线程
- 数据只落在 `Conversation` + `Message`
- `Conversation.type` 只保留 `direct`
- `Conversation.id` 继续允许 `direct_{characterId}`
- 单聊永远不会在运行时升级成群聊

### 目标二：群聊只保留一种语义

- 群聊 = 世界主人 + 多个角色的正式群工作区
- 数据只落在 `Group` + `GroupMember` + `GroupMessage`
- 所有群聊都统一走 `/group/*`
- 所有群聊消息都统一走 `/groups/*`
- 群聊不会伪装成 `Conversation`

### 目标三：聊天列表可以统一展示，但线程类型必须显式

- 列表可以继续混排展示单聊和群聊
- 但列表项类型不能再复用 `ConversationListItem` 硬包群聊
- 应新增统一线程摘要契约，例如：
  - `threadKind: "direct" | "group"`
  - `source: "conversation" | "group"`
  - `threadId`
  - `title`
  - `lastMessage`
  - `unreadCount`

前端不再根据 `id` 前缀猜语义。

## 具体整改方案

### Phase 1：先冻结唯一线程口径

目标：
- 先停止继续扩散“假群聊”

执行项：
- 明确产品定义：
  - 单聊不自动升级成群聊
  - 从单聊拉人时，必须创建正式 `Group`
- 把现有“stored_group / 临时群聊”标记为待迁移遗留模型
- 在新规划落地前，不再新增基于该模型的页面和交互

完成标准：
- 后续实现只围绕两种线程：
  - direct conversation
  - formal group

### Phase 2：重构后端线程模型

目标：
- 从源头删掉 `Conversation` 里的群聊语义

执行项：
- `packages/contracts/src/chat.ts`
  - `ConversationType` 收口为 `direct`
  - 群聊列表摘要从 `ConversationListItem` 拆出去
- `api/src/modules/chat/conversation.entity.ts`
  - 代码语义不再允许 `type = group`
- `api/src/modules/chat/chat.service.ts`
  - 删除 direct 升级成 group 的逻辑
  - 删除临时邀请角色进入 `Conversation.participants` 的逻辑
  - 如果需要多人协作回复：
    - 方案 A：直接创建正式 `Group`
    - 方案 B：保持单聊，但改成“主角色代为协调”的单线程回复，不改变线程类型
- `api/src/modules/chat/chat.gateway.ts`
  - `send_message` 只服务 direct conversation
- `api/src/modules/admin/reply-logic-admin.service.ts`
  - 删除 `stored_group` 分支
  - 后台只保留 `direct` 和 `formal_group`

完成标准：
- 后端不再存在“Conversation 里的 group”
- 管理后台不再需要解释三套线程分支

### Phase 3：重构列表与共享契约

目标：
- 不再让前端把群聊伪装成 conversation

执行项：
- 新增统一线程摘要契约，例如：
  - `ThreadListItem`
  - `threadKind`
  - `threadId`
  - `directConversationId?`
  - `groupId?`
- `getConversations()` 改为：
  - 方案 A：重命名为 `getThreads()`
  - 方案 B：保留旧接口但新增 `/api/threads`，前端先迁移
- `packages/contracts/src/client.ts`
  - direct 与 group 的读取接口显式分离
  - 列表接口返回统一线程摘要，而不是复用 `Conversation`
- 删除 `isPersistedGroupConversation()`

完成标准：
- 前端不再依赖 `id.startsWith("direct_")`
- 路由、搜索、转发、右键菜单都只看显式 `threadKind`

### Phase 4：重构前端线程页与详情页

目标：
- 单聊页和群聊页各自只承担一种语义

执行项：
- `useConversationThread()`
  - 只处理 direct conversation
  - `characterId` 推断只服务单聊
- `ConversationThreadPanel`
  - 不再接受 `conversationType = group`
- `ChatDetailsPage`
  - 只服务单聊
  - 只展示单角色资料、强提醒、加好友、拉黑、举报
- `GroupChatThreadPanel`
  - 只服务正式群聊
- `GroupChatDetailsPage`
  - 只服务正式群聊
- `chat-list-page.tsx`
  - 统一读取线程摘要
  - 根据 `threadKind` 决定跳转 `/chat/*` 还是 `/group/*`
- `use-search-index.ts`
  - direct 走 `getConversationMessages`
  - group 走 `getGroupMessages`
  - 逻辑不再通过 helper 猜
- `chat-message-list.tsx`
  - `threadContext.type` 继续保留 direct/group 即可
  - 但来源必须是稳定线程类型，不再从 conversation 猜

完成标准：
- `/chat/*` 页面永远只处理单聊
- `/group/*` 页面永远只处理群聊

### Phase 5：收口“抄微信但不适配世界模型”的功能面

目标：
- 只保留适合当前单世界产品的聊天能力

执行项：
- 重新审视以下群聊功能是否真需要：
  - 管理员角色
  - @所有人 通知
  - 群公告强提醒
  - 群二维码 / 邀请卡
  - 复杂群资料编辑链路
- 重新审视以下单聊功能是否真需要：
  - 强提醒是否只保留 direct
  - 语音/视频通话是否只保留 direct
  - “发起群聊”是否从单聊详情页保留，还是统一收口到独立入口
- 把“微信长得像什么”改成“隐界世界里线程应该怎么工作”

完成标准：
- 页面不再为了像微信而保留一堆不稳定 feature
- 功能边界服从产品语义，不服从微信目录结构

## 推荐执行顺序

1. 先做 Phase 1，冻结口径
2. 再做 Phase 2，删掉后端假群聊分支
3. 接着做 Phase 3，重做 contracts 和线程摘要
4. 然后做 Phase 4，收口前端线程页和详情页
5. 最后做 Phase 5，删除多余微信化功能

## 需要优先处理的具体问题

### P0

- 删除 `Conversation -> group` 运行时升级链路
- 删除后台 `stored_group` 分支
- 禁止前端继续按 `id` 前缀判断线程类型

### P1

- 修正 `ChatDetailsPage` 对假群聊按单聊处理的问题
- 修正 `ConversationThreadPanel` 对群人数统计不正确的问题
- 修正搜索、转发、聊天列表对线程类型的补丁式判断

### P2

## 最终完成定义

- 仓库里只存在两种线程模型：
  - 单聊
  - 正式群聊
- 不再存在 `stored_group`、`temporary group chat`、`type = group 且仍走 /chat/*` 这类中间态
- 前端、后端、共享契约、管理后台都使用同一套线程定义
- 后续如果继续优化聊天体验，应以“隐界世界线程模型”为基准，而不是继续按微信页面逐个抄写
