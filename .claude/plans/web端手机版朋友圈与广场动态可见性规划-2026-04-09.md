# Web 端手机版朋友圈与广场动态可见性规划

## Summary
目标是在 `apps/app` Web 端手机版里，把“朋友圈”和“广场动态”从现在的“两套页面”进一步收口成“两种明确的内容域”：

- `朋友圈 /discover/moments`：熟人域，用户在这里发的内容只面向好友关系可见。
- `广场动态 /discover/feed`：公共域，用户在这里发的内容面向整个世界可见。

这次规划的重点不是再拆一个新模块，而是把现有的：
- `moment_posts`
- `feed_posts`
- `friendships`

三者的语义真正打通，让“发在哪，就给谁看”成为稳定规则。

## 当前现状

### 已有基础
- 前端路由已经分开：
  - `apps/app/src/routes/moments-page.tsx`
  - `apps/app/src/routes/discover-feed-page.tsx`
- 后端模块已经分开：
  - `api/src/modules/moments/*`
  - `api/src/modules/feed/*`
- 好友关系已经存在：
  - `api/src/modules/social/friendship.entity.ts`
  - `api/src/modules/social/social.service.ts`

### 当前缺口
- `api/src/modules/moments/moments.service.ts` 的 `getFeed()` 现在直接返回全部朋友圈帖子，没有按好友关系过滤。
- `api/src/modules/moments/moments.service.ts` 的 `createUserMoment()` 没有把“仅好友可见”落实到互动和读取链路里。
- `api/src/modules/scheduler/scheduler.service.ts` 的朋友圈自动发帖当前面向所有角色触发，不区分是否已是好友。
- `api/src/modules/feed/feed.service.ts` 当前天然是公共流，但前端只在文案上弱区分，没有把“全世界可见”明确表达出来。

结论：
- 现在的“朋友圈”和“广场动态”主要是入口分开。
- 真正的“可见范围差异”还没有被后端规则和移动端交互完整承接。

## 核心产品决策

### 1. 不新增第三种动态模型
首版不把 `moments` 和 `feed` 合成一张总表，也不新增“发帖时切换公开范围”的混合入口。

原因：
- 现有仓库已经天然有两张表，两种内容域边界已经存在。
- 用户这次要的是“区分清楚”，不是“混成一个入口再加复杂权限”。
- 单世界模型下，先把语义做硬，比先做抽象字段更稳。

### 2. 朋友圈天然等于“好友域”
`moment_posts` 直接定义为“好友可见内容”，不额外加 `visibilityScope` 字段。

落地规则：
- 用户自己发到朋友圈的内容：
  - 自己一定可见
  - 仅 `friendships.status !== blocked` 的好友角色可见
- 角色发到朋友圈的内容：
  - 只有与当前世界主人已经建立好友关系的角色，才允许出现在朋友圈流里
- 非好友角色：
  - 不出现在朋友圈流
  - 不参与朋友圈评论/点赞
  - 不接收用户朋友圈的 AI 互动调度

### 3. 广场动态天然等于“世界公共域”
`feed_posts` 直接定义为“世界公开内容”，不与朋友圈混用。

落地规则：
- 用户发到广场的内容，整个世界里的角色都可以看到并参与互动。
- 广场流默认面向全世界，不依赖好友关系。
- 已被用户屏蔽的角色，前端继续本地过滤展示；后端的公共投放可保留，但不再回流到用户视图。

### 4. 移动端必须把范围说清楚
不能只靠用户“自己理解”。

移动端每个入口、发布区、成功提示都要明确告诉用户：
- 朋友圈：`仅好友可见`
- 广场动态：`全世界可见`

## 目标形态

### 朋友圈页
目标心智：像微信朋友圈，偏熟人、偏生活、偏关系内流通。

建议页面表达：
- 顶部标题仍叫 `朋友圈`
- 发布区副文案改成：
  - `发到朋友圈，只有好友能看到这条内容。`
- 发布按钮下增加弱提示：
  - `仅好友可见`
- 自己发布成功后的提示改成：
  - `朋友圈已发布，仅好友可见。`
- 列表里自己的帖子可加一个小标签：
  - `好友可见`

### 广场动态页
目标心智：像世界公共时间线，偏开放、偏被更多角色看到。

建议页面表达：
- 顶部标题仍叫 `广场动态`
- 发布区副文案改成：
  - `发到广场，世界里的角色都可能看到并回应。`
- 发布按钮下增加弱提示：
  - `全世界可见`
- 发布成功提示改成：
  - `广场动态已发布，正在向世界扩散。`
- 自己发布的帖子可加一个小标签：
  - `世界公开`

## 技术方案

### Phase 1：先用现有两张表落地可见范围，不做结构变更
这一阶段优先做规则落地，不改表结构。

#### 后端：朋友圈读写规则收紧
涉及文件：
- `api/src/modules/moments/moments.service.ts`
- `api/src/modules/social/social.service.ts`
- `api/src/modules/scheduler/scheduler.service.ts`

需要调整的点：
- `MomentsService.getFeed()`
  - 读取当前世界主人的好友列表
  - 只返回：
    - 用户自己发的朋友圈
    - 已建立好友关系的角色发的朋友圈
- `MomentsService.createUserMoment()`
  - 逻辑不变，但发布后触发的 AI 互动只在好友角色范围内调度
- `MomentsService.toggleOwnerLike()` / `addOwnerComment()`
  - 当目标帖子作者是角色时，需要校验该角色当前仍是可见好友
- `scheduleCharacterInteractions()`
  - 只让好友角色参与用户朋友圈互动
- `scheduleAiCommentReplies()`
  - 只允许朋友圈作者仍属于好友域时继续回复
- `SchedulerService.checkMomentSchedule()`
  - 自动发朋友圈的候选角色，限制为当前好友角色

#### 后端：补一个好友 ID 读取能力
建议在 `SocialService` 内增加一个轻量能力，例如：
- `getFriendCharacterIds()`

用途：
- 给朋友圈读取过滤
- 给朋友圈 AI 互动范围
- 给定时朋友圈投放范围

不要在 `MomentsService` 里重复写同一套关系筛选逻辑。

#### 后端：广场维持公共流
涉及文件：
- `api/src/modules/feed/feed.service.ts`

这一层原则上不需要大改：
- `getFeed()` 继续返回公共流
- `createOwnerPost()` 继续直接进入公共流
- `triggerAiReactionForPost()` 保持“世界角色都可参与”的方向

只建议补一层显式约束：
- 被用户屏蔽的角色，不再被抽中作为广场互动者

这样用户对“我明明屏蔽了，为什么还总在广场互动里看到 TA”不会困惑。

### Phase 2：移动端交互把范围标签显性化
涉及文件：
- `apps/app/src/routes/discover-page.tsx`
- `apps/app/src/routes/moments-page.tsx`
- `apps/app/src/routes/discover-feed-page.tsx`
- 如有需要，再抽公共展示组件

具体改造：
- `discover-page.tsx`
  - 入口文案强化对比
  - 朋友圈入口文案改成“好友可见”
  - 广场入口文案改成“世界公开”
- `moments-page.tsx`
  - 发布区、副标题、成功提示、空状态全部收口到熟人域语义
  - 帖子卡片为用户自己的内容增加 `好友可见` 标签
- `discover-feed-page.tsx`
  - 发布区、副标题、成功提示、空状态收口到公共域语义
  - 帖子卡片为用户自己的内容增加 `世界公开` 标签

注意：
- 这里优先做“范围认知清晰”，不做复杂筛选器。
- 先不要增加“同步发到广场”之类联动开关，否则用户又会混淆。

### Phase 3：如果后续需要，再补统一内容域字段
只有在下面需求出现时，才值得做结构变更：
- 同一条内容允许选择“朋友圈 / 广场 / 双发”
- 朋友圈再细分“全部好友 / 亲密好友 / 仅自己”
- 后续需要在统一资料页里混排两种内容

那时再考虑：
- 为 `moment_posts` / `feed_posts` 增加统一的 `surface` 或 `visibilityScope`
- 或抽一层统一发布契约

但当前这一步不建议做。

## 推荐实施顺序
1. 先改后端规则，让 `moments` 真正只面向好友域。
2. 再改移动端文案和标签，让用户明确知道“发在哪，谁能看”。
3. 最后根据体验再决定是否需要在个人主页或详情页增加“我的朋友圈 / 我的广场动态”分组视图。

## 验收标准
- 在 `/discover/moments` 发出的内容，不会再被非好友角色看到或参与互动。
- 在 `/discover/moments` 的流里，用户不会再看到非好友角色的朋友圈。
- 在 `/discover/feed` 发出的内容，仍然可以被全世界角色响应。
- 移动端入口文案、发布文案、成功提示都能让用户明确区分两种范围。
- 整个改造不新增表，不改现有路由，不打乱当前发现页信息架构。

## 风险与决策
- 当前 `friendships` 既承担“好友”也承担“屏蔽”语义，朋友圈过滤必须统一以 `status !== blocked` 为基础，再明确哪些状态算“可见好友”。
- `SchedulerService` 现在直接写 `moment_posts`，如果不把规则一起收进去，最终会出现“前端看不到、库里还在生成”的脏数据。
- 这次不建议把朋友圈和广场合成一个发布器；那会把一个清晰问题重新做复杂。

## AGENTS.md 影响
本方案首阶段按现有两张表和两套路由落地，不涉及模块/实体/路由/表结构变更，因此本次规划本身不需要更新 `AGENTS.md`。

如果后续进入 Phase 3，新增了统一内容域字段或新入口路由，再按项目规则立即更新 `AGENTS.md`。
