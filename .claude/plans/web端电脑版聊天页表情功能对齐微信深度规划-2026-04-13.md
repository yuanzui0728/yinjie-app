# Web 端电脑版聊天页表情功能对齐微信深度规划

## 任务目标

- 把 `apps/app` Web 端电脑版聊天页的表情能力，从当前“极简静态表情包”升级为更接近微信电脑端的完整心智。
- 本次规划覆盖：
  - 表情搜索
  - 更丰富的内置表情包
  - 自定义表情上传
  - 将聊天中的图片 / GIF / 已收到表情加入自定义表情
  - 桌面端表情面板、消息右键、发送闭环、最近使用、管理删除
- 本次先做规划，不直接进入实现。

## 结论先行

- 当前仓库已经具备“`sticker` 独立消息类型 + 最近使用 + 桌面面板 + 图片/GIF 附件上传”的基础，但离微信电脑端还差三层：
  - 表情面板结构不对：现在只有简单包组切换，没有搜索、自定义表情区、管理入口。
  - 数据模型不够：现在 `sticker` 只能发“内置静态包里的固定素材”，不支持用户自定义资产。
  - 操作链路缺口大：没有“添加到表情”、没有表情管理、没有上传导入、没有真正的微信式搜索。
- 推荐方案不是继续在现有 `STICKER_PACKS` 上硬堆，而是把能力拆成两条轨：
  - `builtin stickers`：内置官方风格包，仍走静态 catalog。
  - `custom stickers`：用户自定义表情，走后端实体 + 资产存储 + 前端查询。
- `sticker` 消息类型应继续保留，不建议把自定义表情退化成 `image` 消息。
  - 原因：微信心智里“表情”和“图片”是两类消息，最近使用、表情搜索、会话预览、AI 上下文、右键动作都不同。

## 目标产品形态

## 1. 总体原则

- 对齐的是微信的使用逻辑，不直接拷贝微信官方素材。
- 不能直接拿微信黄脸或官方表情包素材入库：
  - 有版权风险；
  - 也不利于后续自有品牌体系。
- 这轮应对齐：
  - 面板结构
  - 动线
  - 发送逻辑
  - 管理逻辑
  - 搜索行为
  - 自定义表情闭环
- 内置表情素材采用：
  - 自研
  - 可商用授权素材
  - 或现有品牌素材延展

## 2. 桌面端表情面板目标结构

- 点击聊天输入区左下角表情图标，打开桌面浮层。
- 面板从现在的“简单包组宫格”升级为四层结构：
  - 顶部：搜索框
  - 中部：结果区 / 当前 tab 内容区
  - 底部：tab/包组切换条
  - 右下或尾部：管理/添加入口

### 2.1 顶部搜索

- 输入即搜，默认 150ms 到 200ms 防抖。
- 搜索范围：
  - 内置表情包 label
  - 内置表情关键词
  - 自定义表情文件名衍生关键词
  - 自定义表情来源备注关键词
- 搜索结果优先级建议：
  - 最近使用且命中
  - label 前缀命中
  - 关键词命中
  - 包组标题命中

### 2.2 Tab 结构

- 建议桌面端固定顺序：
  - `最近`
  - `精选`
  - `自定义`
  - `包组 1`
  - `包组 2`
  - `包组 3...`
- 说明：
  - `最近`：本地最近使用，不需要后端同步。
  - `精选`：内置高频表情聚合，解决用户不想切包的问题。
  - `自定义`：用户上传和从聊天加入的所有个人表情。
  - 后续官方包组：补充更丰富的静态/动态内容。

### 2.3 自定义表情区

- 行为对齐微信电脑端心智：
  - 一个统一的“自定义表情”集合，而不是让用户自己建很多包。
- 面板内应有：
  - 自定义表情网格
  - “添加”按钮
  - 右键删除
  - 库存满时的提示
- 表情库默认上限建议先按 `300` 做。
  - 这与历史 Windows 微信公开资料一致；
  - 同时做成常量，便于后续微调。

### 2.4 发送行为

- 点击任意表情：
  - 直接发送
  - 记录最近使用
  - 保持微信式面板打开状态或关闭状态需要明确
- 推荐：
  - 桌面端单击发送后继续保持面板打开。
  - 原因：
    - 微信桌面重度聊天中，连续发多张表情很常见；
    - 关闭会降低效率。
- 如果要更保守，也可首版维持“发送后关闭”，但不够像微信。
- 建议最终按“发送后不关闭”执行。

## 3. 图片 / GIF / 已收到表情转成自定义表情

### 3.1 桌面消息右键菜单

- 对以下消息增加 `添加到表情`：
  - `image` 消息
  - `sticker` 消息
  - 后续若视频号/文章富媒体内有静图引用，不纳入首版
- 对 `file`、`voice`、`location_card` 等不显示该动作。

### 3.2 添加来源

- 自定义表情导入支持三种来源：
  - 本地上传
  - 聊天中的图片/GIF
  - 聊天中收到的表情

### 3.3 处理原则

- 不能直接把一张原始聊天大图原样当表情发。
- 需要一个“表情规范化”过程：
  - 静态图：压缩到表情尺寸上限
  - GIF：保留动画时长和帧，但限制体积与边长
  - 透明背景尽量保留
- 推荐策略：
  - 静态图：
    - 最大边长压到 `240px ~ 320px`
    - 默认输出 `webp/png`
  - GIF：
    - 只接受 GIF 原格式或后续动画 WebP
    - 超限时先给失败提示，不在首版做复杂的动图重编码编辑器

### 3.4 满库与重复导入

- 若自定义表情已满：
  - 阻止继续导入
  - 给出“去管理”入口
- 若导入的是已存在同一资源：
  - 不重复创建
  - 直接提示“已在自定义表情中”
  - 并将该表情顶到最近使用或更新时间靠前

## 4. 内置表情包丰富化策略

### 4.1 不要直接复制微信素材

- 这点要明确：
  - 只能对齐微信的数量级、分类密度、操作逻辑；
  - 不能直接照搬官方黄脸和官方表情包图片。

### 4.2 推荐首批内置包规模

- 当前只有 12 张，明显太少。
- 推荐第一批扩到：
  - 6 到 8 个包组
  - 每组 16 到 24 个表情
  - 总量 120 到 160 个
- 覆盖的高频语义：
  - 收到 / OK / 赞
  - 打招呼 / 早安 / 晚安
  - 开心 / 大笑 / 无语 / 崩溃
  - 疑问 / 思考 / 震惊
  - 撒娇 / 安慰 / 抱抱
  - 加油 / 鼓掌 / 庆祝
  - 工作 / 催更 / 忙 / 下班
  - 节日 / 红包 / 祝福

### 4.3 搜索词设计

- 每个内置表情不要只有 `label`，还应增加 `keywords`：
  - `收到`：`收到 ok 好 行 明白`
  - `晚安`：`晚安 睡觉 困 休息`
  - `震惊`：`震惊 吃惊 啊 什么`
- 这样搜索才像微信，而不是只支持精确匹配。

## 推荐技术方案

## 1. 数据模型

### 1.1 StickerAttachment 扩展

- 当前 `StickerAttachment` 需要扩成可区分来源：

```ts
type StickerSourceType = "builtin" | "custom";

interface StickerAttachment {
  kind: "sticker";
  sourceType: StickerSourceType;
  stickerId: string;
  packId?: string;
  url: string;
  mimeType: string;
  width: number;
  height: number;
  label?: string;
}
```

- 解释：
  - `builtin`：内置 catalog 中的表情。
  - `custom`：用户上传或从聊天导入的自定义表情。

### 1.2 发送协议扩展

- 现有 `SendMessagePayload` 里的 `sticker` 负载要改成：

```ts
{
  type: "sticker";
  text?: string;
  sticker: {
    sourceType: "builtin" | "custom";
    stickerId: string;
    packId?: string;
  };
}
```

- 后端根据 `sourceType` 决定解析方式：
  - `builtin`：查静态 catalog
  - `custom`：查 owner 自定义表情表

### 1.3 自定义表情实体

- 推荐新增实体，而不是塞进 `system_config` 或 `users` 的 JSON 字段。
- 原因：
  - 有独立增删查改
  - 有数量上限和去重
  - 后续可能要排序、搜索、最近导入、来源追踪

### 1.4 推荐实体结构

```ts
ChatCustomStickerEntity
- id
- ownerId
- label
- keywordsPayload
- fileName
- mimeType
- url
- width
- height
- sizeBytes
- assetHash
- source
- sourceMessageId?
- sourceConversationId?
- createdAt
- updatedAt
- lastUsedAt?
```

- 重点字段说明：
  - `assetHash`：做重复导入去重
  - `source`：`upload | message_image | message_sticker`
  - `keywordsPayload`：先存 JSON 数组即可
  - `lastUsedAt`：后续如果想做“自定义页按常用排序”会有用

## 2. 接口设计

### 2.1 推荐新增路由

- `GET /api/chat/stickers/catalog`
  - 返回：
    - 内置包组
    - 精选聚合
    - 自定义表情列表
- `POST /api/chat/stickers/custom`
  - 上传一个自定义表情
- `POST /api/chat/stickers/custom/from-message`
  - 从已有消息导入到自定义表情
- `DELETE /api/chat/stickers/custom/:id`
  - 删除一个自定义表情
- `POST /api/chat/stickers/custom/:id/use`
  - 可选，记录服务端最近使用或 lastUsedAt

### 2.2 是否要做后端搜索接口

- 首版不建议做单独搜索接口。
- 原因：
  - 单世界 + 单人实例
  - 表情量级可控
  - 前端本地搜索足够快
- 做法：
  - 首次打开面板拉一次 catalog
  - 之后在前端内存中过滤和排序

## 3. 资产存储

### 3.1 不复用“消息附件即表情”

- 推荐不要直接拿消息里的原始图片 URL 当表情资产。
- 因为：
  - 原图可能过大
  - 同一资源在聊天和表情使用场景下的尺寸目标不同
  - 后续清理、缓存和预览策略也不同

### 3.2 推荐单独的表情资产存储

- 新增：
  - `POST /api/chat/stickers/custom`
  - 内部生成规范化后的表情文件
- 资源访问建议：
  - `GET /api/chat/stickers/assets/:fileName`

### 3.3 与现有上传链路的关系

- 现有 `POST /api/chat/attachments` 继续服务聊天附件。
- 自定义表情上传单独走表情上传逻辑：
  - 限制不同
  - 压缩策略不同
  - 资产生命周期不同

- 重复导入与满库处理

### Phase 5：管理与细节对齐

- 自定义表情删除
- 发送后面板保持开启
- 最近使用排序优化
- 搜索排序优化

## 风险与注意事项

### 1. 素材版权

- 不能直接使用微信官方表情素材。
- 只能对齐逻辑、布局、密度和行为。

### 2. GIF 处理复杂度

- 静态图规范化相对简单；
- GIF 压缩和保真成本明显更高。
- 首版应接受“超限 GIF 先提示失败，不做复杂编辑器”的策略。

### 3. 当前组件过大

- `chat-composer.tsx` 已经承担过多职责。
- 这次若不拆，后续很难继续维护桌面输入区和表情面板。

### 4. 桌面端与移动端范围

- 本轮以桌面端为主。
- 但协议、实体和自定义表情接口最好一次设计好，让移动端后续能复用。

## 推荐执行口径

- 先不要直接“补几张表情图”。
- 正确顺序应该是：
  1. 先扩 sticker 数据模型，支持 `builtin/custom`
  2. 再做 custom sticker 实体和接口
  3. 再重做桌面端面板与搜索
  4. 最后补“添加到表情”和管理动作
- 这样做的好处是：
  - 不会返工协议
  - 不会把自定义表情临时塞成图片消息
  - 后续移动端也能共享后端能力

## 本轮规划后的明确建议

- 建议把这项需求定义为“桌面聊天页微信式表情体系升级”，而不是“补几个表情包”。
- 这次实施应至少覆盖以下验收项：
  - 桌面端可搜索表情
  - 内置表情量明显提升
  - 有自定义表情 tab
  - 可从本地上传静图/GIF 为表情
  - 可从聊天图片/GIF/表情加入自定义表情
  - 右键可删除自定义表情
  - 自定义表情可发送为 `sticker` 消息而不是 `image` 消息
  - 最近使用正常工作
