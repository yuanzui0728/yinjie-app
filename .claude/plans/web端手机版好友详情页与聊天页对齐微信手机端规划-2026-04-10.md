# Web端手机版好友详情页与聊天页对齐微信手机端规划

## 任务目标
- 面向 `apps/app` 的 Web 手机版，梳理“好友详情页 + 单聊页 + 聊天信息页”与微信手机端的差异，并按“尽量 1:1 对齐微信交互”给出改造计划。
- 本次规划不再把“通讯录点联系人直接开聊”当成既定前提，而是按微信主链路重排：
  - `通讯录列表 -> 好友详情页 -> 发消息 -> 单聊页 -> 聊天信息页 -> 再回好友详情页`
- 目标是把主路径做成用户一眼就能认出是微信式操作节奏，而不是继续保留当前“角色档案页 + 聊天页 + 设置页”三套混合模型。

## 参考基线

### 当前代码基线
- 移动端通讯录里，好友项点击后仍直接开聊，不先进好友详情页。
- 当前好友详情页实质是“角色档案页”，不是微信式好友资料页。
- 当前单聊页已经具备一部分微信式基础能力：
  - 顶部扁平导航
  - 5 分钟分隔时间条
  - 移动端长按消息
  - 回复草稿条
  - `语音/键盘` 切换外观
  - `表情 + 更多` 输入区结构
- 当前聊天信息页已有基础骨架，但和好友详情页之间没有打通。

- `2025-04-04` 的公开报道转述微信新能力，确认 `引用回复` 已可承接 `表情包 / 语音`。

### 需要明确标注的推断
- 微信手机端好友资料页、聊天信息页、长按消息菜单的分组顺序和视觉细节，官方不会在 App Store 文案里逐项列出。
- 下文中这类“分组顺序、按钮位置、操作节奏”的设计判断，属于基于当前微信手机端长期稳定交互的高可信推断，不等于来自单一官方文档的逐字说明。

## 当前差异

### A. 最大差异不在聊天页，而在“联系人主链路”

#### 1. 当前移动端通讯录主路径错了
- `apps/app/src/routes/contacts-page.tsx`
  - 移动端好友列表点击后直接 `handleStartChat`。
  - 世界角色列表点击才进入 `/character/$characterId`。
- 微信手机端不是这个模型。
- 微信主路径里，通讯录点击好友，默认进入的是“好友资料页”，不是直接跳会话。

#### 2. 当前 `character-detail-page.tsx` 不是好友详情页，而是角色介绍页
- 页面核心信息是：
  - `bio`
  - `专长`
  - `当前状态`
  - `语气`
- 主按钮只有 `开始聊天`。
- 朋友关系下只多出一个 `设为星标朋友` 开关。
- 这更像“AI 角色卡片”，不像微信好友资料页。

#### 3. 当前缺少微信式好友详情页最关键的信息和动作
- 缺少 `备注和标签` 主入口。
- 缺少 `地区 / 来源 / 标签 / 隐界号` 的微信式资料编排。
- 缺少 `发消息` 与 `音视频通话` 的双主操作结构。
- 缺少 `加入黑名单 / 删除联系人 / 更多信息` 这类关系管理入口。
- 缺少“从聊天信息页进入好友资料页”的回流链路。

### B. 当前聊天信息页已有骨架，但还是“设置页思路”，不是微信主链路中的中转页

#### 1. 结构上已经接近微信，但还差好友资料闭环
- `apps/app/src/routes/chat-details-page.tsx`
  - 已有：
    - 聊天成员宫格
    - 查找聊天记录
    - 消息免打扰
    - 置顶聊天
    - 强提醒
    - 保存到通讯录
    - 设置当前聊天背景
    - 清空聊天记录
    - 投诉
    - 加入黑名单
- 这说明“聊天信息页骨架”已经不是主要问题。

#### 2. 但它没有承担微信里“跳好友详情”的角色
- 成员宫格里的对方头像没有直达好友资料页。
- 页面里没有明确的“查看好友资料 / 更多信息”入口。
- 现在的好友关系管理被拆散到了：
  - 一部分在聊天信息页
  - 一部分在角色档案页
  - 一部分只在桌面端联系人详情里
- 这和微信的单一主路径不一致。

#### 3. 有些设置还只是“像”，不是真闭环
- `强提醒` 目前写在本地偏好里，不是服务端真实状态。
- 聊天背景虽然已有真实入口，但好友详情页没有同步暴露关系型设置。
- “保存到通讯录”和“好友资料页”的关系仍旧割裂。

### C. 单聊页已经接近微信表层，但还没到“完全抄”

#### 1. 页面骨架已经差不多了，不需要推翻重做
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 移动端是灰底、扁平头部、底部输入条。
- `apps/app/src/components/chat-message-list.tsx`
  - 已经是“按时间段插入时间条”，不是每条消息都显示时间。
- `apps/app/src/components/chat-composer.tsx`
  - 已经有 `语音/键盘切换`、`表情`、`更多`、多行 `textarea`、绿色发送按钮。
- 结论：
  - 聊天页已经不是“完全不像微信”，而是“只做到了 60 分，离完整微信化还差操作层”。

#### 2. 长按消息只做了半套
- 当前移动端长按只弹 `回复 / 复制`。
- 微信高频动作里至少还差：
  - `转发`
  - `收藏`
  - `多选`
  - `删除`
  - `撤回`
  - `提醒`
- 现在的 `mobile-message-action-sheet.tsx` 更像简化版占位菜单。

#### 3. 图片和附件还没有进入微信式承接
- 图片消息目前只是直接渲染 `<img>`，没有：
  - 点开全屏查看
  - 左右滑查看
  - 保存
  - 转发
- 文件 / 名片 / 位置卡片虽然能发，但仍旧是“能展示”，不是“微信式完整流转”。

#### 4. 语音入口外观像了，产品语义没对齐
- 当前语音模式文案已经是 `按住 说话`。
- 但实际行为仍是浏览器录音后转文字，不是真正的语音消息。
- 这会直接造成“看起来像微信，发出去却不是微信那种消息”的落差。

#### 5. 更多功能面板还是偏薄
- `apps/app/src/components/mobile-chat-plus-panel.tsx`
  - 当前只有：
    - 相册
    - 拍摄
    - 名片
    - 位置
    - 文件
- 相比微信手机端，至少还少：
  - 视频/短视频类承接
  - 收藏内容转发
  - 实时位置共享
  - 红包/转账/收款类占位
  - 更完整的两屏布局策略

### D. 数据与协议层还不够支撑“完全抄”
- 当前消息契约仍缺少这些微信高频类型或状态：
  - `voice`
  - `video`
  - `merged_forward`
  - `favorite`
  - `recall`
  - `live_location`
  - `message_reminder`
- 所以“完全像微信”不能只改前端样式，必须区分：
  - 交互先做像
  - 消息能力再补齐真功能

## 目标口径

### 1. 好友详情页要从“角色档案”改成“关系页”
- 页面第一身份必须是“我和这个人的关系资料”，不是“这个角色的人设介绍”。
- 角色世界观信息可以保留，但必须降级到次要区块，不再占首页主视图。

### 2. 聊天信息页只做“聊天相关设置 + 跳好友详情”
- 聊天信息页负责：
  - 聊天记录
  - 免打扰
  - 置顶
  - 背景
  - 发起群聊
  - 风险操作
- 好友关系资料负责：
  - 备注
  - 标签
  - 地区
  - 来源
  - 星标
  - 删除 / 拉黑

### 3. 单聊页优先补操作层，而不是继续改表皮
- 单聊页现在缺的不是大框架，而是微信高频操作：
  - 长按菜单
  - 图片查看
  - 真语音消息
  - 更多面板扩展

## 推荐改造方案

### Phase 1：先把联系人主链路改回微信模型
目标：修正“通讯录点好友直接开聊”的路径错误。

改造点：
- `apps/app/src/routes/contacts-page.tsx`
  - 移动端好友列表点击后改成进入 `/character/$characterId`。
  - “发消息”从好友详情页承接，不再直接发生在通讯录列表。
- `apps/app/src/routes/character-detail-page.tsx`
  - 重做为微信式好友资料页，而不是角色档案页。
  - 首屏结构建议改成：
    - 头像 / 备注名 / 昵称 / 隐界号
    - 个性签名
    - 发消息主按钮
    - 音视频通话次按钮
  - 资料分组建议改成：
    - `设置备注和标签`
    - `地区`
    - `来源`
    - `标签`
    - `朋友圈`
    - `更多信息`
    - `设为星标朋友`
    - `加入黑名单`
    - `删除联系人`

验收口径：
- 移动端通讯录的主路径应与微信一致。
- 好友详情页第一眼不再像“AI 角色介绍卡”。

### Phase 2：把聊天信息页变成聊天设置中转页
目标：明确好友详情页与聊天信息页的职责边界。

改造点：
- `apps/app/src/routes/chat-details-page.tsx`
  - 保留当前微信式骨架。
  - 顶部成员宫格里的对方头像增加跳转到好友详情页。
  - 新增一行 `查看好友资料` 或直接将头像区改成可点击。
  - 把更偏“关系管理”的项目移出主路径，只保留必要聊天操作。
- `apps/app/src/features/chat-details/chat-member-grid.tsx`
  - 支持成员项点击，不只保留加号入口。

验收口径：
- 用户从单聊页右上角进聊天信息页后，能自然进入好友详情页。
- 好友资料不再散落在多个页面和端形态里。

### Phase 3：补齐单聊页操作层
目标：让聊天页真正达到微信手机端的日常使用强度。

改造点：
- `apps/app/src/components/chat-message-list.tsx`
  - 移动端长按菜单扩展为：
    - `回复`
    - `复制`
    - `转发`
    - `收藏`
    - `多选`
    - `删除`
    - `撤回`
- `apps/app/src/features/chat/mobile-message-action-sheet.tsx`
  - 从简化弹层升级为微信式操作菜单。
- 图片消息补全屏查看器。
- 名片 / 位置 / 文件卡片补点击承接。

验收口径：
- 聊天页的高频操作不再只有“回复/复制”两个半成品动作。

### Phase 4：把输入器从“像微信”补到“真微信语义”
目标：消除当前语音入口的假象感。

改造点：
- `apps/app/src/components/chat-composer.tsx`
  - 保留现有布局框架。
  - 真正补 `语音消息` 发送链路，不再只走语音转文字。
  - 支持微信式手势：
    - 长按录音
    - 上滑取消
    - 右滑转文字（可后置）
- `apps/app/src/components/mobile-chat-plus-panel.tsx`
  - 扩成两屏或留占位的一屏半结构。
  - 把当前 5 个能力按微信顺序重排，并给后续能力预留固定位置。

验收口径：
- 用户看到“按住说话”时，发出的就是语音消息，而不是隐藏的转文字流程。

### Phase 5：补齐协议层，解决“完全抄”的硬边界
目标：把微信式 UI 承接成真功能。

需要补的能力：
- `voice` 消息
- `video` 消息
- `merged_forward`
- `favorite`
- `recall`
- `live_location`
- `message_reminder`

涉及范围：
- `packages/contracts/src/chat.ts`
- `packages/contracts/src/ws.ts`
- `api/src/modules/chat/*`

## 优先级

### P0：不改就仍然不像微信
- 通讯录点好友先进入好友详情页
- 好友详情页微信化重做
- 聊天信息页打通好友详情跳转
- 单聊长按菜单扩充
- 图片全屏查看

### P1：微信高频效率层
- 真语音消息
- 多选转发 / 收藏 / 删除 / 撤回
- 更多面板扩容

### P2：真正补足“完全抄”的后端能力
- 实时位置共享
- 提醒类消息
- 视频消息
- 收藏消息系统化

## 实施建议
- 不建议把好友详情页和聊天页混着改。
- 正确顺序应是：
  1. 先改联系人主链路和好友详情页
  2. 再收聊天信息页职责
  3. 最后补聊天页操作层
- 原因：
  - 如果主链路不先纠正，聊天页改得再像微信，整个产品入口仍然不像微信。

## 本次结论
- 当前 Web 手机版里，真正最不像微信的不是单聊页本体，而是“通讯录 -> 好友详情 -> 聊天信息”这条关系链。
- 单聊页本身已经有不少微信式半成品，不应该推翻重来，而应该集中补操作层和消息类型。
- 如果你要“完全抄”，首要工作不是继续修聊天皮肤，而是先把 `好友详情页` 做成真正的微信好友资料页。
