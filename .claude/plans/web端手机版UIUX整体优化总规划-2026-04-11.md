# Web端手机版 UI/UX 整体优化总规划

## 任务目标
- 面向 `apps/app` 的 Web 端手机版做一轮完整的 UI/UX 收敛。
- UX 基线明确按微信手机端主路径执行：
  - 信息架构抄微信
  - 页面骨架抄微信
  - 操作顺序抄微信
  - 反馈节奏抄微信
- UI 不做微信视觉复刻，改成“积极向上、清爽明亮”的隐界移动端风格：
  - 主色从当前暖橙玻璃感，收口到更干净的绿白体系
  - 保留少量晨光感、世界感，只作为气氛层，不进入主操作层
- 这次先出母规划，不直接进入大面积实现；后续实现按本规划分批提交。

## 当前代码基线结论

### 1. 当前不是“移动端没做”，而是“做了很多，但语言不统一”
- 聊天链路已经比旧规划阶段更接近微信：
  - `chat-composer.tsx` 已有移动端语音/键盘切换、绿色发送按钮、按住说话、表情和更多面板
  - `chat-message-list.tsx` 已有移动端长按菜单、回复、转发、收藏、多选、撤回等主动作
  - `character-detail-page.tsx` 已经在往微信好友资料页靠
- 但全局移动端仍然同时存在两套风格：
  - 一套是偏微信的灰白扁平操作面
  - 一套是偏项目自定义的暖橙玻璃卡片面
- 这会导致用户在一级页和二级页之间频繁“换 App”。

### 2. 最大的整体问题在壳层
- `mobile-shell.tsx` 现在仍然是：
  - 外层大面积暖色光斑
  - 中间 `max-w-[460px]`
  - 内层大圆角 `40px`
  - 强玻璃和悬浮壳
- 这在桌面缩窄预览时还说得过去，但在真实手机浏览器里很不对。
- Web 手机版不该再像“网页里嵌一个手机壳”，而应该直接是原生感全屏界面。

### 3. 视觉 token 方向和目标不一致
- `packages/ui/src/tokens.css` 目前主基调还是：
  - 暖奶油底
  - 橙色品牌色
  - 玻璃感 surface
  - 琥珀系阴影
- 这和“微信式移动主路径 + 积极向上”之间差了半步：
  - “积极向上”不是更厚的装饰
  - 而是更清楚、更轻、更有呼吸感的界面节奏

### 4. 一级页里，消息和通讯录比发现、我更接近目标
- `chat-list-page.tsx` 已经是列表主导，交互模型基本正确，主要问题是壳层和局部样式不统一。
- `contacts-page.tsx` 已经有微信式列表骨架，但固定服务项、搜索条、世界角色入口、区块黏性标题还可以继续收平。
- `discover-page.tsx` 和 `profile-page.tsx` 目前最不像微信手机端：
  - 发现页是长说明卡片集合
  - 我页只有一个大卡片加一个设置入口
- 这两个一级页需要重构信息密度和层级节奏。

### 5. 世界入口链路仍然偏“品牌叙事页”，不够“移动工具页”
- `splash-page.tsx`、`welcome-page.tsx` 视觉叙事偏重、说明文字偏长。
- 微信式移动入口更像“快速完成当前任务”，不是“沉浸式落地页”。
- 隐界可以保留世界感，但不能把移动端首屏做成品牌海报。

## 目标设计方向

### UX 方向：完全按微信手机端节奏
- 首页层级：
  - 底部四 Tab 固定
  - 一级页只做一级页该做的事
  - 详情页、设置页、资料页、搜索页按微信路径回退
- 列表语言：
  - 灰底
  - 白色或浅灰列表面
  - 分组头
  - cell 为主，不用大面积解释卡片
- 操作语言：
  - 顶部导航收平
  - 搜索入口就地出现
  - 底部 sheet 统一
  - 触摸反馈明确
  - 重要动作顺序贴微信

### UI 方向：晴朗绿白，不再暖橙玻璃
- 基础底色：
  - 页面底 `#ededed` / `#f7f7f7`
  - 列表面 `#ffffff`
- 主强调色：
  - 绿色优先，继续沿用聊天里已经出现的 `#07c160` 体系
  - 可允许少量更亮的品牌绿，但不要再回到橙色主导
- 气氛层：
  - 只在 `splash / welcome / 空状态 / 品牌提示` 使用淡晨光色
  - 不进入消息列表、通讯录、聊天主线程等操作层
- 形态：
  - 半径从当前 `20~40px` 收到 `8~16px`
  - 去掉大块 blur、浮空感、发光感
  - 阴影更薄，更多靠 hairline 和灰阶分层

## 核心策略

### 策略 A：先统一通用移动原语，再改页面
- 不能继续一页一页手搓，否则很快又会回到风格碎片化。
- 先建立一套移动端专用的：
  - 顶栏
  - 搜索条
  - 列表 section
  - 列表 cell
  - bottom sheet
  - 空状态
  - 通知条
  - 主要/次要/危险按钮
- 然后让一级页和二级页都吃同一套语言。

### 策略 B：移动端视觉覆盖必须与桌面端解耦
- `packages/ui/src/tokens.css` 是共享层，直接全局改会波及桌面端。
- 建议实施时采用“移动端作用域覆盖”，而不是简单全局替换：
  - 可挂在 `MobileShell` 根节点
  - 或挂到 `body` / `#root` 的 mobile 标记类
- 目标是：
  - Web 手机版整体换肤
  - 桌面端不跟着被误伤

### 策略 C：发现页和我页按微信重排，不做“功能广告墙”
- 发现页首页改成微信式入口列表，不保留大段解释文案。
- 我页改成微信式个人头卡 + 分组列表。
- 真实没做的能力不强行摆出来，避免“像微信但点进去是假”的违和感。

### 策略 D：聊天页不重写，只做一致性和完成度收口
- 聊天线程相关代码已经有不错基础。
- 本轮不建议推翻消息/输入器结构，而是做：
  - header 收平
  - 气泡、系统消息、时间条视觉统一
  - 输入区边距、按钮、面板、键盘行为微调
  - 与整体 mobile theme 对齐

## 分阶段实施计划

## Phase 1：移动壳层与设计系统收口

### 目标
- 先把“看起来像两个 App”的问题解决。

### 主要改动
- 重做 `mobile-shell.tsx`
  - 去掉中间圆角设备壳
  - 改成真正的全屏 mobile viewport
  - 保留 safe area，但不再保留假手机边框
- 收口 `tab-page-top-bar.tsx`
  - 统一一级页与二级页头部样式
  - 一级页更接近微信 Tab 头部
  - 二级页更接近微信返回导航栏
- 增补移动端样式作用域
  - 将 mobile 颜色、背景、阴影、分割线、sheet 样式独立成 mobile theme
- 收口通用组件：
  - `AppPage`
  - `AppSection`
  - `EmptyState`
  - `InlineNotice`
  - 常用搜索条

### 验收标准
- 所有 mobile 路由不再出现“网页中悬浮手机壳”观感。
- 一级页和二级页顶栏语言统一。
- 页面背景、分割线、圆角、按钮颜色统一到同一套移动规范。

## Phase 2：四个一级 Tab 重做成一套体系

### 2.1 消息页
- 基于现有 `chat-list-page.tsx` 做精修，不推翻功能。
- 重点改：
  - 顶栏与搜索条收平
  - 快捷菜单样式改微信式下拉面板
  - 订阅号/服务号/普通会话的列表语法统一
  - 置顶、免打扰、提醒态视觉更克制
  - 提醒区块从“模块卡片”改成“列表组”

### 2.2 通讯录
- 基于现有 `contacts-page.tsx` 继续收敛。
- 重点改：
  - 固定服务项顺序和视觉更贴微信
  - 搜索条与区块标题统一
  - 联系人行高度、分割线、头像、说明文案统一
  - `世界角色` 从强曝光入口降为更次级区块，不抢通讯录主路径
  - 索引条视觉和拖动反馈进一步微信化

### 2.3 发现
- `discover-page.tsx` 需要真正重排。
- 从“说明卡片列表”改成“微信式入口列表 + 必要的小标签/摘要”。
- 每个入口只保留：
  - 名称
  - 可选摘要
  - 右箭头
  - 必要时的角标/最近动态提示
- 去掉长段解释文案。

### 2.4 我
- `profile-page.tsx` 从“单卡片页”改成“微信式个人主页结构”。
- 建议结构：
  - 顶部身份卡
  - 设置/资料相关分组
  - 与当前产品真实能力匹配的条目
  - 协议与规范入口继续收口，但不抢第一屏
- 不引入没有真实能力的“支付/卡包”假入口。

### 验收标准
- 四个一级页首屏都能在 3 秒内被识别为微信式移动路径。
- 四页视觉密度统一，不再出现“有的像微信，有的像品牌落地页”。

## Phase 3：联系人与服务内容链路统一

### 目标
- 把“通讯录 -> 朋友资料 -> 发消息 / 通话 -> 聊天信息”做成同一套移动体验。

### 主要改动
- `character-detail-page.tsx`
  - 继续压缩自定义“角色介绍感”
  - 强化“朋友资料页”感
  - 将备注、标签、地区、来源等字段排版再微信化
- `official-accounts-page.tsx`
  - 从卡片列表继续收平到公众号列表语言
- `official-account-detail-page.tsx`
  - 公众号主页结构进一步微信化
- `official-account-service-page.tsx`
  - 服务号消息页贴齐消息线程语言
- `subscription-inbox-page.tsx`
  - 与消息页里的订阅号入口视觉统一

### 验收标准
- 联系人资料页、公众号列表页、服务号消息页不再跳出主体验。
- 从通讯录进入资料、从资料进入聊天、从聊天详情回资料，路径清晰且视觉连续。

## Phase 4：聊天与聊天详情精修

### 目标
- 聊天区不大改逻辑，重点做完成度、统一性和触摸体验。

### 主要改动
- `mobile-chat-thread-header.tsx`
  - 顶栏更扁平、更接近微信
  - 副标题只在必要时出现，不长期占位
- `chat-message-list.tsx`
  - 时间条、系统消息、引用块、附件卡片风格统一
  - 移动端 action sheet 与全局 bottom sheet 语言统一
  - 图片/文件/名片/位置卡片边距和层级统一
- `chat-composer.tsx`
  - 输入器与全局 theme 对齐
  - 更多面板、语音态、回复态、键盘态的切换更平顺
  - 面板底部 safe area 和键盘抬升继续打磨
- `chat-details-page.tsx` / `group-chat-details-page.tsx`
  - section 结构统一成微信式设置列表
  - 危险操作和开关行的视觉规范统一

### 验收标准
- 聊天主线程看起来不再是“局部微信、局部项目 UI”。
- 输入区、长按菜单、详情页、底部面板都是同一套移动语言。

## Phase 5：发现二级页与内容页统一

### 目标
- 让 `发现` 下面的二级页不再各写各的。

### 主要改动
- 统一这些二级页的返回顶栏、内容间距、空态和列表语言：
  - `discover/moments`
  - `discover/encounter`
  - `discover/scene`
  - `discover/feed`
  - `discover/channels`
  - `discover/games`
  - `discover/mini-programs`
- 朋友圈、广场动态、视频号、游戏、小程序都要做到：
  - 首页入口像微信
  - 二级页阅读节奏也像微信
  - 但内容语义仍然是隐界世界

### 验收标准
- 从“发现”进入任一二级页，用户不会感觉像跳到另一套产品。

## Phase 6：世界入口与设置链路瘦身

### 目标
- 把 `splash / welcome / setup / onboarding / profile settings` 收成更像移动工具链路。

### 主要改动
- `splash-page.tsx`
  - 减少叙事装饰，强化“正在进入世界”的实用反馈
- `welcome-page.tsx`
  - 云世界 / 本地世界选择做成更明确的移动步骤页
  - 表单、说明、状态提示收短
- `setup-page.tsx` / `onboarding-page.tsx`
  - 继续保持跳转，但沿用统一的移动入口视觉
- `profile-settings-page.tsx`
  - tab、表单、分组、协议页入口统一成微信式设置页语言

### 验收标准
- 首次进入世界的流程清晰、轻、快，不再像品牌故事落地页。

### 建议优先抽出的移动通用组件
- `MobilePageScaffold`
- `MobileNavBar`
- `MobileSearchBar`
- `MobileListSection`
- `MobileListCell`
- `MobileGroupedSheet`
- `MobileStatusBanner`
- `MobilePageEmpty`

### 目的
- 让一级页、二级页、设置页、详情页都能复用，而不是每个路由自己拼 className。

## 风格准则

### 需要统一保留的
- 微信式信息架构
- 微信式操作顺序
- 微信式 cell / list / header / sheet 节奏
- 绿色发送和关键正向操作

### 需要明确去掉的
- 大面积暖橙主色
- 假设备壳
- 过厚玻璃模糊
- 大段营销型解释文案
- 到处都是独立大卡片的首页式布局

### 可以保留但要降级的
- “世界”“居民”“隐界”这些语义
- 某些品牌晨光色氛围
- 个别欢迎态和空状态里的气氛插图/文案

## 风险与注意事项

### 1. 不要全局误伤桌面端
- 共享 token 改动必须有 mobile 作用域。

### 2. 不要一次性大爆改
- 正确做法是：
  - 先 foundation
  - 再一级页
  - 再主链路
  - 再二级页

### 3. 不要为了像微信而堆假入口
- 没有真实能力支撑的入口：
  - 直接不放
  - 或明确禁用
- 不能做成点击后假跳转。

### 4. 不要再让“发现”和“我”单独跑风格
- 这两个页必须服从整体移动规范。

## 最终验收口径
- 手机浏览器打开后，不再出现“网页中套手机壳”的 UI。
- 一级页和二级页一眼看上去就是同一套微信式移动产品。
- UX 路径对微信用户几乎零学习成本。
- UI 情绪从当前暖橙玻璃，切换为更积极、更干净、更有精神气的绿白体系。
- 隐界品牌感仍在，但只出现在该出现的位置，不再干扰主操作路径。
