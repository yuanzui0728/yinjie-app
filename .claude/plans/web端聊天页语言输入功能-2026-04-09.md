# Web 端聊天页语言输入功能规划

## Summary
目标是在 `apps/app` 的聊天页，为 Web 端的手机版和电脑版补齐“语音转文字输入”能力。

这次规划只处理“把说话内容转成文本，回填到现有聊天输入框，再由用户确认发送”的能力，不做语音消息气泡、不做语音通话、不改消息数据结构。

实现策略建议采用“双通路”：
- 浏览器支持原生语音识别时，优先走前端直出识别，获得更快的首字反馈。
- 浏览器不支持或识别中断时，退回“录音采集 + 服务端转写”链路，保证 Web 手机版和电脑版都可用。

## 当前现状
- `apps/app/src/components/chat-composer.tsx` 已经有移动端 `Mic` 按钮，但只是静态 UI，没有任何录音、权限、识别、状态提示。
- `apps/app/src/features/chat/conversation-thread-panel.tsx` 和 `apps/app/src/features/chat/use-conversation-thread.ts` 当前只维护纯文本输入 `text`，发送链路也只支持 `type: "text"`。
- `apps/app/src/features/shell/use-desktop-layout.ts` 已经把 Web 端拆成“桌面布局”和“移动布局”，适合作为两套语音交互入口的分流点。
- 当前仓库没有 `SpeechRecognition`、`MediaRecorder`、音频上传、转写 API 的现成实现。

## 产品定义
### 这次要做的
- 在聊天输入区支持“说话转文字”。
- 识别结果先进入输入框，用户可继续编辑，再决定是否发送。
- 支持中间态：开始、识别中、暂停、失败、取消、确认。
- Web 手机版和电脑版采用不同交互壳，但共用一套状态机和转写能力。

### 这次不做的
- 不发送语音消息文件。
- 不新增聊天消息类型。
- 不做语音通话和实时双向音频。
- 不做长音频文件管理、草稿箱、历史录音列表。

## 交互方案
### 1. Web 手机版
目标是对齐手机浏览器习惯，避免键盘、权限弹窗、录音控件三者冲突。

建议交互：
- 默认仍显示左侧 `Mic` 按钮。
- 点击 `Mic` 后，先收起键盘，再弹出一个轻量底部语音面板，不直接在原输入框里硬切状态。
- 面板核心区显示：
  - 当前状态文案：`点击开始说话` / `正在聆听...` / `识别完成`
  - 一个主录音按钮：开始 / 停止
  - 一个取消按钮
  - 一个“插入输入框”按钮
- 识别过程中，底部面板展示实时文本或分段文本，不直接自动发送。
- 用户点击“插入输入框”后，把转写文本合并回 `text`，焦点回到输入框，后续发送仍沿用现有发送按钮。

不建议首版做“按住说话”：
- 当前能力目标是“语音输入”，不是微信式语音消息。
- H5 里长按录音容易和浏览器手势、权限提示、页面滚动冲突。
- 点按开始 / 点按结束更稳定，也更容易复用到桌面 Web。

### 2. Web 电脑版
目标是保持键盘输入主路径不变，把语音输入作为增强能力。

建议交互：
- 在桌面输入框右侧新增一个 `Mic` 按钮，不替换现有发送逻辑。
- 点击后进入“正在识别”状态：
  - 输入框外侧显示一个小型状态条或胶囊：`正在聆听`
  - `Mic` 按钮变成停止态
  - 可持续把 interim transcript 写入临时文本
- 点击停止后，把最终文本合并到输入框，光标停在文本末尾。
- 识别失败时，在 composer 底部复用现有 `InlineNotice` 显示错误。

桌面端不建议弹大浮层：
- 当前桌面聊天工作区已经比较稳定。
- 语音输入是增强工具，不应打断现有阅读和输入节奏。

## 技术方案
### 1. 前端分层
建议新增 4 层职责，避免把录音逻辑直接塞进 `ChatComposer`：

- 能力检测层
  - 判断当前是否为 Web 手机版、Web 电脑版。
  - 判断浏览器是否支持 `SpeechRecognition` / `webkitSpeechRecognition`。
  - 判断是否支持 `navigator.mediaDevices.getUserMedia` 与 `MediaRecorder`。

- 语音输入控制层
  - 新增 `useSpeechInput` hook，统一管理状态机。
  - 暴露：
    - `status`
    - `supported`
    - `transcript`
    - `interimTranscript`
    - `error`
    - `start()`
    - `stop()`
    - `cancel()`
    - `commitToInput()`

- 采集 / 识别适配层
  - `browser-speech-recognition-adapter`
  - `media-recorder-adapter`
  - `server-transcription-adapter`
  - 由控制层按能力自动选择路径，不让 UI 感知底层差异。

- UI 呈现层
  - `ChatComposer` 只消费 hook 暴露的状态，不直接操控浏览器 API。
  - 移动端和桌面端分别接不同的视图组件。

### 2. 推荐状态机
统一状态建议至少包含：
- `idle`
- `requesting-permission`
- `listening`
- `processing`
- `ready`
- `error`

关键规则：
- `idle -> requesting-permission -> listening`
- 停止录音后：
  - 原生识别快路可直接进入 `ready`
  - 服务端转写链路进入 `processing`，完成后再进 `ready`
- 用户取消时回到 `idle` 并清空临时文本
- 识别文本只有在 `commitToInput()` 后才并入聊天输入框

### 3. 识别链路
建议采用以下优先级：

#### Path A：浏览器原生语音识别
适用：
- Chromium 系浏览器且 `SpeechRecognition` 可用

优点：
- 首字返回快
- 不需要上传音频文件
- 前端即可拿到 interim transcript

局限：
- 浏览器支持面不稳定，不能当唯一方案
- 部分浏览器依赖浏览器厂商的远端识别服务

适合作为：
- Web 电脑版首选快路
- Web 手机版在支持环境下的首选快路

#### Path B：录音上传到服务端转写
适用：
- 原生识别不可用
- 原生识别中途失败
- 需要统一稳定性和可控性

实现建议：
- 前端用 `getUserMedia` + `MediaRecorder` 采集短音频片段
- 停止后生成 `Blob`
- 上传到后端转写接口
- 后端统一调用可配置的转写 Provider，返回纯文本

这是首版真正的兜底主链路，建议必须做。

## 后端规划
### 1. 新增转写接口
建议新增独立语音转写接口，而不是塞进现有消息发送接口：
- `POST /api/ai/transcriptions`

请求：
- `multipart/form-data`
- 字段至少包含：
  - `file`
  - `conversationId`（用于日志和限流，不参与会话隔离）
  - `mode`：`dictation`

响应：
- `text`
- `durationMs`
- `provider`

### 2. 模块归属
建议先放在 `api/src/modules/ai/`，原因：
- 转写本质上是 AI Provider 能力编排，不是 chat domain 规则。
- 后续如果 App 里别的页面也要复用语音转文字，这个接口不需要再迁移。

### 3. Provider 设计
建议抽一层 `TranscriptionProvider`，不要在 controller/service 里直接写死某一家。

首版接口可以只保留一个可用 Provider，但代码结构要支持后续扩展：
- 读取现有服务端 AI 配置
- 能力不可用时返回明确错误
- 记录失败原因和耗时，便于后续诊断

## 前端文件落点建议

- `apps/app/src/features/chat/use-speech-input.ts`
- `apps/app/src/features/chat/speech-input-types.ts`
- `apps/app/src/features/chat/adapters/browser-speech-recognition.ts`

## 分阶段实施
### Phase 1：前端壳与状态机
目标：先把聊天页语音输入的 UI 结构和状态机接起来。

包含：
- `ChatComposer` 接入语音输入入口
- Web 手机版底部语音面板
- Web 电脑版状态胶囊
- `useSpeechInput` 状态机
- 能力检测和禁用态提示

这一阶段可以先用 mock transcript 验证交互，不接真实转写。

### Phase 2：浏览器原生识别快路
目标：在支持的浏览器中实现“点击即识别”。

包含：
- `SpeechRecognition` 适配器
- interim transcript 更新
- 停止 / 取消 / 错误处理
- 中文识别参数默认值

这一阶段完成后，支持环境下已经可用，但仍不算完整交付。

### Phase 3：服务端转写兜底
目标：补齐跨浏览器稳定可用性。

包含：
- `MediaRecorder` 采集
- 上传接口
- 后端转写 service
- 前端自动 fallback
- 超时、文件大小、录音时长限制

这一阶段完成后，Web 手机版和电脑版才算都具备可交付能力。

### Phase 4：体验打磨
包含：
- 文案优化
- 权限拒绝引导
- 自动插入标点策略
- 录音时长上限提示
- 网络慢时的 `processing` 反馈

## 验收标准
- Web 手机版聊天页可点击语音按钮开始说话，识别结果能回填输入框。
- Web 电脑版聊天页可在不离开当前工作区的前提下完成语音输入。
- 原生识别不可用时，页面仍能通过录音上传转写完成输入。
- 识别结果不会自动发送，用户始终可以编辑后再发。
- 权限拒绝、识别失败、网络失败三类异常都有明确提示。
- 不改现有聊天消息结构，不影响纯文本输入与发送。

## 风险与决策
- 最大风险是浏览器原生语音识别兼容性不稳定，所以不能只做前端 `SpeechRecognition`。
- 第二个风险是手机浏览器键盘和录音权限弹窗互相打断，因此移动端需要独立底部面板，而不是直接把输入框切成录音态。
- 第三个风险是转写 Provider 能力和成本控制。首版要限制单次录音时长，避免被长音频拖垮响应。

当前建议的核心决策：
- 产品定义限定为“语音转文字输入”，不做语音消息。
- Web 手机版和电脑版分开做交互壳，共用同一状态机和转写能力。
- 技术上采用“原生识别快路 + 服务端转写兜底”的双通路方案。

## 建议执行顺序
1. 先做 `useSpeechInput` 和前端两套 UI 壳，跑通状态机。
2. 接入浏览器原生识别，验证桌面 Web 与手机 Web 的即时反馈。
3. 再补服务端转写接口和自动 fallback。
4. 最后处理权限失败、超时、文案、时长限制这些体验问题。
