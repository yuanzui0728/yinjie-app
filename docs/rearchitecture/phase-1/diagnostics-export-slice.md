# Phase 1 Slice: Diagnostics Export and Runtime Logs

## Scope

This slice turns the system operations surface into a real diagnostics workflow:

- real runtime log path resolution
- local Core API operation log file
- real `/system/logs` indexing
- real `/system/diag/export` bundle generation

## What Landed

### 1. Runtime path module

`crates/core-api/src/runtime_paths.rs` now centralizes runtime filesystem paths for:

- runtime root
- logs directory
- core API log file
- diagnostics export directory

It also provides a local append-only operation log helper for `core-api.log`.

### 2. Real system log indexing

`GET /system/logs` no longer returns a hard-coded placeholder list.

It now reads the actual runtime logs directory and returns concrete log file paths. If the directory is still empty, it falls back to the expected Core API log path.

### 3. Real diagnostics export

`POST /system/diag/export` now creates a real diagnostics directory under the runtime root.

Each export currently contains:

- `manifest.json`
- `system-status.json`
- `realtime-status.json`
- `scheduler-status.json`
- `runtime-snapshot.json` when available
- copied runtime logs
- up to three recent backup snapshots

### 4. Local operation log fallback

Because production diagnostics should not rely only on stdout, key runtime actions now append to `runtime-data/logs/core-api.log`, including:

- Core API startup
- snapshot persistence
- scheduler executions
- backup creation
- backup restore
- diagnostics export

## Validation Notes

This slice was verified locally with a dedicated runtime-data directory:

- `cargo check`
- start Core API with custom `YINJIE_DATABASE_PATH`
- `POST /system/scheduler/run/update-world-context`
- `GET /system/logs`
- `POST /system/diag/export`

Observed results during verification:

- `/system/logs` returned the real `core-api.log` path
- `core-api.log` existed and had non-zero size
- diagnostics export created a real timestamped diagnostics directory
- exported files included manifest, system status, realtime status, scheduler status, runtime snapshot, and copied logs

## Current Behavior Notes

- diagnostics export currently produces a directory rather than a compressed archive
- operation logging is focused on high-signal infrastructure events, not every HTTP request line
- inference-gateway and desktop-shell logs are not yet generated by the new runtime
