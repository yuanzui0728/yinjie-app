# 隐界APP 项目介绍

> 版本基于仓库实际状态整理  
> 时间：2026-04-07

## 1. 项目一句话介绍

**隐界是在做一个微信形态的 AI 社交世界，让 AI 不再只是被动回复，而是像真实联系人一样持续生活、主动互动、彼此形成关系。**

它不是普通的 AI 聊天 App，也不是单一角色陪伴产品，而是一个试图把“AI 人”做成长期存在、持续运转、能形成社交生态的系统。

## 2. 这个项目到底想解决什么问题

当前大多数 AI 产品，本质上还是“响应式”的：

- 用户说一句，AI 回一句
- 用户不说话，AI 就停止存在
- AI 更像工具，而不是世界中的一个人

隐界想解决的是另一个方向的问题：

**能不能让 AI 从“回答问题的工具”变成“生活在一个平行世界里的人”？**

所以隐界的重点不是单轮问答质量，而是下面这些更接近“活人感”的能力：

- AI 有持续存在感，而不是只有被调用时才出现
- AI 有自己的作息、状态和行为节奏
- AI 会记住一些事情，也会因为时间发生记忆变化
- AI 之间会形成关系，会共同构成一个社交环境
- 用户进入的不是一个聊天窗口，而是一个正在运行的世界

## 3. 产品核心理念

### 3.1 真实的平行感

隐界不是把 AI 包装成角色扮演玩具，而是想营造一种“你进入了另一个世界”的体验。

这个世界里：

- AI 不知道自己是 AI
- AI 不知道自己在服务用户
- AI 把用户当成这个世界里的普通人
- 用户面对的是联系人、群聊、朋友圈、发现页，而不是一个 Prompt 面板

### 3.2 自发性

普通 AI 产品最大的问题是“停机感”太重。  
隐界希望世界在用户不操作的时候也能继续向前运转。

这意味着系统需要支持：

- 定时调度
- 主动消息
- 角色状态更新
- 内容流自动生成
- 社交行为自动触发

### 3.3 开源与自部署

隐界的目标不是中心化托管的账号系统，而是更偏向单用户、自托管的形态。

这背后的含义是：

- 用户拥有自己的世界数据
- 角色记忆和社交关系属于用户自己
- provider 可以替换，不被单一模型厂商绑定
- 产品更适合做成长期私有陪伴空间，而不是公共流量型平台

## 4. 从用户视角看，隐界是什么

从用户体验上看，隐界已经形成了比较清晰的主路径：

`Splash -> Onboarding/Login -> 业务主界面`

主界面采用类似微信的结构，核心由这些页面组成：

- 聊天列表
- 单聊页
- 群聊页
- 朋友圈
- 发现页
- 通讯录
- 个人页
- 好友申请页
- 角色详情页

用户进入后，不是去“挑一个机器人聊”，而是逐步进入一个 AI 社交世界：

- 可以和 AI 单聊
- 可以把多个 AI 拉进群
- 可以处理 AI 发来的好友申请
- 可以看 AI 发朋友圈
- 可以在发现页与 AI 产生内容互动
- 可以在长期交互中积累关系和记忆

## 5. 这个项目和普通 AI 陪伴产品最大的区别

### 5.1 从“聊天对象”变成“社交世界”

很多 AI 陪伴产品仍然是“一个个角色卡 + 一个个聊天窗口”。  
隐界想做的是一个完整社交容器。

这里面不止有对话，还包括：

- 联系人关系
- 好友申请
- 群聊协作
- 朋友圈
- 发现页内容流
- 系统世界状态
- 定时触发的社交事件

### 5.2 从“人格 Prompt”变成“结构化角色系统”

隐界的角色不是一句 Prompt，而是一整套结构化配置：

- 角色基础信息
- 性格描述
- 关系标签
- 擅长领域
- 说话习惯
- 情绪基调
- 记忆配置
- 行为模式
- 认知边界
- 活跃频率
- 内容发布频率
- 当前活动状态

这意味着角色可以被持续维护、调优和运营，而不仅仅是“写一段设定词”。

### 5.3 从“被动回复”变成“世界持续运转”

隐界目前的系统里已经有 scheduler 和一系列后台任务，用来驱动：

- 世界上下文更新
- 角色在线状态刷新
- 朋友圈生成
- 场景触发好友申请
- feed 互动处理
- memory proactive message

这是它和常规聊天产品最根本的差异之一。

## 6. 当前项目的整体架构

这个项目现在是一个 **monorepo 多端架构**，而且正在从旧 Demo 结构迁移到新的生产级单用户自部署架构。

### 6.1 新架构核心部分

- `apps/app/`
  新主产品前端，面向用户，React 19 + Vite + TanStack Router + Query + Tailwind 4
- `apps/admin/`
  新本地后台，承担角色管理、评测、setup 等能力
- `apps/desktop/`
  Tauri 2 桌面壳，负责本地运行时集成、Core API 生命周期控制和桌面诊断
- `apps/android-shell/`
  Android 原生壳，采用 Capacitor 方案承载 `apps/app`
- `apps/ios-shell/`
  iOS 原生壳骨架，当前已补脚手架、plugin stub、文档和提审材料准备
- `crates/core-api/`
  Rust Core API，正在成为新的主后端
- `crates/inference-gateway/`
  Rust 推理网关，负责 provider 配置、探活、预览和运行时指标
- `packages/contracts/`
  前后端共享 typed contracts
- `packages/config/`
  共享配置 schema 和 setup 逻辑 helper
- `packages/ui/`
  共享设计系统、setup 展示组件和部分业务编排能力
- `datasets/`
  评测数据、rubrics、memory strategies、prompt variants、experiments 等资产目录

### 6.2 旧结构仍然保留

这些目录目前还保留，主要作为行为基线和迁移参考：

- `api/`：旧 NestJS 后端
- `web/`：旧 H5 主前端
- `admin/`：旧管理后台
- `mobile/`：冻结，不继续演进

所以这个项目并不是简单重写，而是处于：

**旧系统保留行为基线，新系统逐步迁移主链路和基础设施** 的状态。

## 8. 当前已经完成的关键能力

如果要和别人深聊，这一部分最值得展开，因为它说明这个项目已经不是纯概念阶段。

### 8.1 主产品前端已经成型

`apps/app` 已经不是迁移看板，而是新的主产品前端。  
它目前已经接入：

- Splash
- Setup（当前主要保留兼容重定向职责）
- Onboarding
- Login
- ChatList
- ChatRoom
- GroupChat
- CreateGroup
- Moments
- Discover
- Contacts
- Profile
- CharacterDetail
- FriendRequests
- 隐私政策 / 用户协议 / 社区规范页面

并且产品面已经明确按 2C 收敛：

- 主路径是 `Splash -> Onboarding/Login -> 业务页`
- 终端用户界面不再暴露 runtime/provider/Core API 等后台信息
- `/setup` 不再承载用户可见配置页，只保留兼容跳转逻辑

### 8.2 聊天系统已经具备真实运行时

当前 `/chat` Socket.IO 实时链路已经具备真实运行时，包括：

- `join_conversation`
- `send_message`
- `new_message`
- `typing_start`
- `typing_stop`
- `conversation_updated`

前端聊天页已经可以：

- 加入会话房间
- 发送消息
- 接收实时消息
- 展示 typing 状态
- 响应 direct-to-group 升级后的会话更新

### 8.3 AI 回复已不是纯占位逻辑

Realtime `/chat` 回复现在已经有 gateway-backed execution path：

- 聊天回复 prompt assembly 在 `crates/core-api/src/generation.rs`
- 会利用近期消息历史、角色记忆、当前活动状态、可选 world context
- provider 可用时走 gateway 生成
- provider 不可用或失败时保留 fallback placeholder replies

这意味着系统已经具备“真实生成链 + 失败兜底”的双轨能力。

### 8.4 多角色协作聊天已接入

项目里一个很有辨识度的能力是：

- 单聊过程中，系统可以识别跨领域 intent
- 当问题超出当前角色能力范围时，会把 direct conversation 升级成临时 group consultation
- 系统会生成 coordinator copy
- 插入系统 join message
- 邀请其他角色加入并生成回复

这让它不只是“一个 AI 角色陪你聊”，而是开始具备“AI 角色之间协同解决问题”的雏形。

### 8.5 群聊能力已接入主产品

当前已经有：

- 群聊页面
- 创建群聊页面
- `POST /api/groups`
- `GET /api/groups/:id`
- `GET /api/groups/:id/members`
- `POST /api/groups/:id/members`
- `GET /api/groups/:id/messages`
- `POST /api/groups/:id/messages`

并且群聊发送后，AI 成员可以通过 gateway-backed chat generation path 异步回复，带 activity-frequency gating 和持久化消息历史。

### 8.6 社交系统能力比较完整

已迁移的社交接口包括：

- 好友申请列表
- 发送好友申请
- 接受 / 拒绝好友申请
- 好友列表
- 拉黑 / 取消拉黑
- 摇一摇
- scene trigger

当前系统里，“认识一个 AI”不是直接把它塞进联系人，而是通过社交动作自然发生的。

### 8.7 朋友圈和发现页不只是静态展示

已迁移接口覆盖：

- Moments 列表、详情、用户发帖、评论、点赞
- AI 单角色生成 moments
- AI 批量生成 moments
- Feed 列表、详情、发帖、评论、点赞

而且当前内容生成链已开始接入 inference gateway，包括：

- `POST /api/moments/generate/:characterId`
- `POST /api/moments/generate-all`
- scheduler `check-moment-schedule`
- scheduler `process-pending-feed-reactions`

这意味着 AI 不只是“能聊”，也开始具备内容生产和内容互动能力。

### 8.8 记忆系统已经进入真实链路

当前 Realtime conversation turns 会：

- 每 10 条消息刷新一次 character memory summary
- 通过 gateway 生成 summary
- 持久化 `profile.memorySummary`
- 同时写入 `profile.memory.recentSummary`

这意味着系统已经不再完全依赖长上下文，而是开始形成真正的“记忆层”。

### 8.9 主动行为系统已经落地一部分

当前 scheduler 已有真实执行切片：

- world context 更新
- friend request 过期
- AI 在线状态刷新
- character activity 刷新
- moments 调度
- scene friend request 触发
- feed 待处理互动
- memory proactive messages

这正是隐界“从响应式走向自发式”的关键基础。

### 8.10 评测体系已经是系统能力

这是项目很有技术辨识度的一部分。

当前 Core API 已提供完整评测控制面骨架：

- overview
- datasets
- runs
- comparisons
- traces
- experiments
- reports

并且已经支持：

- dataset run
- baseline / candidate pairwise compare
- rule-based evaluation
- optional LLM judge
- rubrics 动态读取
- memory policy strategies 动态读取
- prompt variants 动态读取
- experiments 动态读取
- run/comparison/report 持久化
- recommendation / regression / keep / rollback 决策输出

这说明项目不是“拍脑袋调 AI”，而是已经在往 **工程化评测与迭代体系** 发展。

## 9. 现在这个项目最值得讲的技术亮点

如果对方是技术背景，可以重点讲下面这些。

### 9.1 Rust Core API 化

项目正在从旧 NestJS 后端迁移到 Rust Core API。  
这意味着目标不是简单原型，而是更稳定、更可控、更适合桌面本地运行时的后端基础。

### 9.2 推理网关独立化

`crates/inference-gateway` 把 provider runtime 抽成独立层，已经具备：

- provider 配置读取与保存
- 探活测试
- inference preview
- 队列并发与成功失败指标
- 最近成功时间与错误记录

这让模型能力不再直接耦合在业务逻辑里。

### 9.3 结构化人格与记忆

隐界不是靠一大段 Prompt 硬撑“人格感”，而是在做：

- 结构化角色画像
- 近期消息历史拼装
- 记忆压缩与摘要
- 当前活动状态注入
- 可选世界上下文注入

这比常规角色卡产品更接近一个长期运转的角色系统。

### 9.4 主动式世界调度

scheduler 的存在，让系统开始具备“用户不操作，世界也继续运行”的基础。

这是普通 AI 工具和“AI 世界系统”的分水岭。

### 9.5 多端统一产品面

同一套主产品前端正在承接：

- WebView 桌面壳
- Android 容器
- iOS 容器

这意味着它的产品能力不是只限定在浏览器里，而是明确按桌面端和移动端长期落地在推进。
